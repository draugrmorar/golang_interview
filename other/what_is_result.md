## 🧠 Интересный кейс: `defer` в цикле и замыкания

### Задача
Есть слайс чисел. В цикле мы вызываем `defer` двумя способами:
1. Прямой вызов `fmt.Println`
2. Анонимная функция (замыкание)

Вопрос: что выведет программа?

### 📦 Исходный код

```go
package main

import "fmt"

func main() {
    s := []int{1, 2, 3, 4, 5}
    
    fmt.Println("=== Способ 1: прямой defer ===")
    for i, v := range s {
        defer fmt.Println(i, v)
    }
    
    fmt.Println("=== Способ 2: defer с замыканием ===")
    for i, v := range s {
        defer func() {
            fmt.Println(i, v)
        }()
    }
}
```

---

## 📊 Результаты в зависимости от версии Go

### ✅ **Go 1.22 и новее**

В современных версиях Go **на каждой итерации создаются новые переменные** `i` и `v`. Поэтому оба способа работают одинаково:

```
=== Способ 1: прямой defer ===
4 5
3 4
2 3
1 2
0 1

=== Способ 2: defer с замыканием ===
4 5
3 4
2 3
1 2
0 1
```

**Почему?**  
На каждой итерации `i` и `v` — уникальные переменные. Замыкание захватывает именно ту переменную, которая была создана на этой итерации.

---

### ⚠️ **версии младше Go 1.22**

В версиях **до 1.22** переменные `i` и `v` **переиспользовались** на всех итерациях. Результаты отличаются:

```
=== Способ 1: прямой defer ===
4 5
3 4
2 3
1 2
0 1

=== Способ 2: defer с замыканием ===
4 5
4 5
4 5
4 5
4 5
```

**Почему?**
- В способе 1 аргументы вычисляются **сразу** → запоминаются конкретные значения
- В способе 2 замыкание захватывает **сами переменные**, а они к концу цикла принимают последние значения (`i=4, v=5`)

---

## 🔍 Как проверить свою версию?

```go
package main

import (
    "fmt"
    "runtime"
)

func main() {
    fmt.Println("Версия Go:", runtime.Version())
}
```

---

## 💡 Выводы

| Версия Go | Поведение замыканий в цикле |
|-----------|-----------------------------|
| **< 1.22** | Замыкания захватывают одну переменную → все defer видят последние значения |
| **>= 1.22** | На каждой итерации создаётся новая переменная → замыкания работают интуитивно |

**Важно:** начиная с Go 1.22, можно не бояться этой ловушки — язык стал предсказуемее!


## 🧠 Итерация по мапе с удалением и добавлением

### 📦 Код

```go
package main

import "fmt"

func main() {
    m := map[string]int{
        "a": 1,
        "b": 2,
    }
    
    for k, v := range m {
        if k == "a" {
            delete(m, "b")
            m["c"] = 3
        }
        fmt.Println(k, v)
    }
    
    fmt.Println("итог:", m)
}
```

---

## 🎲 Возможные варианты вывода

### Вариант 1: Итерация начинается с `"a"`
```
a 1
c 3
итог: map[a:1 c:3]
```

**Почему:**
1. Начинаем с ключа `"a"` → печатаем `a 1`
2. Удаляем `"b"`, добавляем `"c"`
3. Go продолжает итерацию и находит новый ключ `"c"` → печатаем `c 3`
4. Ключ `"b"` уже удалён → не печатается

---

### Вариант 2: Итерация начинается с `"a"`, но `"c"` попадает позже
```
a 1
итог: map[a:1 c:3]
```
(если добавленный элемент не попал в текущую итерацию)

---

### Вариант 3: Итерация начинается с `"b"`
```
b 2
a 1
итог: map[a:1 c:3]
```

**Почему:**
1. Начинаем с ключа `"b"` → печатаем `b 2`
2. Условие `k == "a"` не сработало → ничего не удаляем и не добавляем
3. Следующая итерация: ключ `"a"` → печатаем `a 1`
4. Ключ `"b"` уже был выведен, `"c"` добавится **после** итерации

---

## 🔬 Что важно понять

| Операция | Поведение |
|----------|-----------|
| **Удаление** во время итерации | Безопасно. Удалённый ключ **не выведется**, если его итерация ещё не прошла |
| **Добавление** во время итерации | **Непредсказуемо**. Может попасть в текущую итерацию, а может нет |

---

## 📚 Цитата из спецификации Go

> If a map entry is created during iteration, it may be produced during the iteration or may be skipped.

**Перевод:** Если элемент создаётся во время итерации, он **может** появиться в текущей итерации, а может быть пропущен.

---

## 💡 Главный вывод

**Никогда не полагайся на поведение добавленных элементов во время итерации по мапе!** Результат зависит от:
- Случайного порядка начала итерации
- Внутренней реализации мапы в конкретной версии Go
- Того, успеет ли новый элемент попасть в текущий обход


## 🧠 Заметка: Замыкания в цикле и горутины

### 📦 Код

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var wg sync.WaitGroup
    
    for i := 0; i < 3; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            fmt.Print(i)
        }()
    }
    
    wg.Wait()
}
```

---

## 🎲 Результат в зависимости от версии Go

### 🟡 **Go 1.21 и старше**
```
3 3 3
```
(порядок может быть любым, но все числа одинаковые)

### 🟢 **Go 1.22 и новее**
```
0 1 2
```
(порядок может быть разным, но все числа разные)

---

## 🔬 Почему так происходит?

### 📌 Старое поведение (до 1.22)

```go
for i := 0; i < 3; i++ {
    // здесь переменная i — одна и та же на всех итерациях
    go func() {
        fmt.Print(i)  // захватывается САМА переменная i
    }()
}
```

**Что происходит:**
1. Цикл быстро пробегает значения `0 → 1 → 2 → 3`
2. Горутины планируются, но выполняются **после** завершения цикла
3. К моменту выполнения `fmt.Print(i)` переменная `i` уже равна `3`
4. ✅ Все горутины печатают `3`

👉 **Замыкание захватывает ссылку на переменную, а не её значение**

---

### 📌 Новое поведение (Go 1.22+)

В Go 1.22 изменили поведение цикла: **на каждой итерации создаётся новая переменная**

```go
for i := 0; i < 3; i++ {
    // i — новая переменная на каждой итерации!
    go func() {
        fmt.Print(i)  // захватывается СВОЯ копия
    }()
}
```

✅ Каждая горутина получает своё уникальное значение

---

## 🛠 Как сделать код предсказуемым (работает во всех версиях)

### ✅ Способ 1: Передавать параметр

```go
for i := 0; i < 3; i++ {
    wg.Add(1)
    go func(j int) {
        defer wg.Done()
        fmt.Print(j)
    }(i)  // копия значения
}
```

### ✅ Способ 2: Создавать локальную копию

```go
for i := 0; i < 3; i++ {
    iCopy := i  // копия на каждой итерации
    wg.Add(1)
    go func() {
        defer wg.Done()
        fmt.Print(iCopy)
    }()
}
```

---

## 📊 Сравнение поведения

| Версия Go | Без параметра | С параметром |
|-----------|----------------|---------------|
| **1.21** | 3 3 3 | 0 1 2 (в разном порядке) |
| **1.22+** | 0 1 2 (в разном порядке) | 0 1 2 (в разном порядке) |

---

## 💡 Главный вывод

**При работе с горутинами в цикле всегда:**
- Передавай переменную цикла как параметр
- Или создавай явную копию
- Или используй Go 1.22+ (но лучше не полагаться на это при написании библиотек)

Это гарантирует, что каждая горутина получит **своё собственное значение**, а не разделяемую переменную, которая может измениться!
