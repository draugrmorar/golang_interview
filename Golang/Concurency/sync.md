# Основные принципы пакета `sync`

- **Никогда не копируйте sync-объекты** — они содержат внутренние состояния, которые нельзя копировать (используйте указатели)
- **Zero value** — большинство типов готовы к использованию сразу (без инициализации)
- **Не используйте без необходимости** — часто каналы выразительнее
- **После использования нельзя копировать** — проверяется `go vet`

# sync.WaitGroup

**WaitGroup** — самый простой способ дождаться выполнения всех горутин. Это структура, которая содержит счетчик количества горутин, завершения которых нам нужно дождаться. Он используется для ожидания завершения всех горутин.

## Методы WaitGroup
- **Add(n int)**: сообщает, сколько горутин нужно дождаться (увеличивает счетчик на n)
- **Done()**: вызывается при завершении горутины, чтобы указать, что ее выполнение завершено (уменьшает счетчик на 1)
- **Wait()**: используется основной горутиной, пока счетчик не станет равным нулю. Когда все горутины завершаются и счетчик становится равным нулю, функция `Wait()` возвращается и продолжает дальнейшее выполнение.

## Пример использования

```go
func main() {
    var wg sync.WaitGroup
    for i := 0; i < 5; i++ {
        wg.Add(1) // 1. ДО горутины!
        go func(id int) {
            defer wg.Done() // 2. Гарантированно уменьшаем счетчик
            fmt.Printf("Горутина %d работает\n", id)
        }(i)
    }
    wg.Wait() // 3. Ждем ВСЕ горутины
}
```

## Внутреннее устройство

```go
type WaitGroup struct {
    noCopy noCopy      // форсирует передачу по указателю
    state1 [12]byte    // атомарный счетчик + семафор
    state2 uint32
}
```

**Как работает:**
- `Add` атомарно увеличивает счетчик
- `Wait` проверяет счетчик:
    - Если 0 — продолжает выполнение
    - Если >0 — паркует горутину
- `Done` уменьшает счетчик, и если стал 0 — будит все ждущие горутины

## Важные вопросы

| Вопрос | Ответ |
|--------|-------|
| **Что произойдет, если вызвать `Wait()` до `Add()`?** | Deadlock! Wait увидит счетчик 0 и продолжит выполнение, не дождавшись горутины. |
| **Что при отрицательном счетчике?** | Паника (panic) |
| **Можно ли переиспользовать WaitGroup?** | **ДА!** После `Wait()` можно снова делать `Add()` |
| **Что эффективнее — WaitGroup или каналы?** | WaitGroup — для ожидания завершения группы задач. Каналы — для передачи данных и сигнализации. Идеально комбинировать! |

# sync.Mutex

**Mutex** (мьютекс) позволяет разграничить доступ к некоторым общим ресурсам, гарантируя, что только одна горутина имеет к ним доступ в определенный момент времени. И пока одна горутина не освободит общий ресурс, другая горутина не может с ним работать.

На уровне кода мьютекс представляет тип `sync.Mutex`, который определен в пакете `sync`. Для блокирования доступа к общему разделяемому ресурсу у мьютекса вызывается метод `Lock()`, а для разблокировки доступа — метод `Unlock()`.

```go
var mutex sync.Mutex
mutex.Lock()    // блокируем доступ
counter = 0
for k := 1; k <= 5; k++{
    counter++
    fmt.Println("Goroutine", number, "-", counter)
}
mutex.Unlock()  // деблокируем доступ
```

# sync.RWMutex

**RWMutex** (мьютекс для чтения/записи) учитывает, что горутина может только считывать данные либо только изменять их. При выполнении операции записи совместное использование данных несколькими горутинами не допускается, тогда как при операции чтения несколько горутин могут считывать одни и те же данные без каких-либо проблем.

## Методы RWMutex
- **RLock()**: используется для получения блокировки на чтение. Несколько читателей могут получить эту блокировку в одной программе. Блокировка на чтение не привязана к определенному потоку.
- **RUnlock()**: освобождает полученную одну блокировку на чтение.
- **Lock()**: блокирует мьютекс для записи. Доступен только для одного потока за раз. Если другой поток получает блокировку для целей чтения/записи, `Lock` блокируется до тех пор, пока блокировка не станет доступной для получения.
- **Unlock()**: снимает блокировку на запись. При выполнении `Unlock` без получения блокировки мы получим ошибку во время выполнения.

## Пример использования

```go
// Запись
map_data.Lock()
map_data.values[keyval] = keyval * 10  // изменяем общий ресурс
map_data.Unlock()

// Чтение
map_data.RLock()
val := map_data.values // считываем данные
map_data.RUnlock()
```

## Внутреннее устройство

RWMutex имеет:
1. Счетчик читателей
2. Мьютекс для писателей
3. Условную переменную

**RLock():**
- увеличивает счетчик читателей
- если есть писатель — ждет

**Lock():**
- блокирует новые RLock
- ждет завершения всех читателей
- захватывает эксклюзивный доступ

## Тезисно о RWMutex

| Ситуация | Результат |
|----------|-----------|
| **RLock внутри Lock** | Deadlock! (Не вызывать методы с блокировками внутри блокировок, либо создать приватный метод без блокировки) |
| **Забыли RUnlock** | Всегда используй `defer` сразу после Lock/RLock |
| **Запись под RLock** | **ПАНИКА!** Нельзя писать при RLock |
| **Копирование RWMutex** | Передавай по указателю: `*Cache` |
| **Можно ли вызвать RUnlock без RLock?** | Да, но будет panic |
| **Что будет, если вызвать `Lock()` дважды в одной горутине?** | Deadlock |
| **Можно ли апгрейдить RLock до Lock?** | **НЕТ!** Это приведет к deadlock |

> **Если не уверен — бери Mutex.** RWMutex дает выигрыш **ТОЛЬКО** при реально частом чтении и редкой записи (хотя бы 90/10)!

## Starvation (Голодание)

**Проблема:** Писатель может никогда не получить доступ, если постоянно приходят читатели.

**Решение в Go 1.8+:** RWMutex не дает голодать писателям:
- Если есть ожидающие писатели, новые читатели не могут забрать блокировку
- Читатели уже внутри — продолжают, новым читателям — ждать

## Проблема рекурсивной блокировки

**Рекурсивная блокировка** возникает, когда одна и та же горутина пытается захватить уже захваченный ею же мьютекс повторно. В Go это приводит к **deadlock'у**, потому что стандартный `sync.Mutex` **не является рекурсивным (reentrant)**.

### Это проблема?
В некоторых языках (Java, C++ с рекурсивными мьютексами) повторный захват того же мьютекса одной нитью разрешен. В Go — нет. Mutex в Go устроен так, что он не помнит, какая горутина его захватила. Для него повторная попытка `Lock()` — это просто попытка захватить уже занятый ресурс.

### Когда возникает
- **Прямая рекурсия:** Функция вызывает саму себя, и обе пытаются захватить один мьютекс.
- **Косвенная рекурсия:** Функция A захватывает мьютекс и вызывает функцию B, которая тоже пытается захватить тот же мьютекс.
- **Взаимные вызовы:** Функции вызывают друг друга по кругу, используя один мьютекс.

### Почему Go не поддерживает рекурсивные мьютексы
Это осознанное архитектурное решение создателей Go:
- Рекурсивные блокировки часто маскируют плохой дизайн
- Они усложняют понимание кода и отладку
- Реализация сложнее и медленнее
- В Go "явное лучше неявного"

### Как исправить
- **Рефакторинг:** Разделить функции на публичные (с блокировкой) и приватные (без блокировки).
- **Повторное проектирование:** Пересмотреть архитектуру, чтобы избежать цепочек вызовов с одним мьютексом.
- **`sync.RWMutex` не поможет:** RWMutex тоже не рекурсивный — повторный `RLock()` после `Lock()` тоже deadlock.

# sync/atomic

Пакет `atomic` предоставляет низкоуровневые примитивы атомарной памяти, полезные для реализации алгоритмов синхронизации. Для правильного использования этих функций требуется большая осторожность. За исключением специальных низкоуровневых приложений, синхронизацию лучше выполнять с помощью каналов или средств пакета `sync`.

**Атомарные операции** без блокировок (lock-free) работают на уровне процессора. Это операции, которые выполняются полностью (как единое целое) без возможности прерывания их другими потоками.

## Основные типы
- `int32`, `int64`, `uint32`, `uint64`, `uintptr`
- Указатели (через `unsafe.Pointer`)

## Основные функции
- **Add** — атомарное сложение/вычитание
- **Load/Store** — атомарное чтение/запись
- **Swap** — атомарная замена
- **CompareAndSwap (CAS)** — сердце lock-free структур

## `atomic.Value` — для любых типов
Хранит значение любого типа атомарно.
- `Store` — сохранить значение
- `Load` — загрузить значение

**Важные ограничения:**
- `nil` нельзя хранить
- Все `Store` должны быть одного типа
- `Value` нельзя копировать после первого Store

## Когда использовать atomic

| ✅ **МОЖНО** | ❌ **НЕЛЬЗЯ** |
|--------------|---------------|
| Счетчики (метрики, статистика) | Сложные структуры данных |
| Флаги состояния | Операции с несколькими полями |
| Кэширование конфигурации | Когда нужна сложная логика синхронизации |
| Простые семафоры | Если не уверены в правильности (используйте Mutex) |
| Lock-free структур с CAS | |

> ‼️ **Важно:** Atomic операции **не детектятся райс-детектором**, но это не значит, что их использование корректно!

# sync.Map

Добавлен в **Go 1.9**. Структура `sync.Map` с самого начала проектировалась как оптимизация для узких, но критически важных сценариев: когда большая часть операций — это `Load`, и ключи редко изменяются.

Обычный `map + RWMutex` использовать предпочтительнее, `sync.Map` имеет преимущество только в случаях, если много чтения и мало записи.

## Строение

Внутри `sync.Map` — нестандартная организация памяти. Структура устроена вокруг двух отдельных хеш-таблиц: одна только для чтения, вторая для всех новых ключей и записей. Доступ к ним регулируется через один `sync.Mutex`, но используется он только в случае записи или промаха — поэтому на чистых чтениях мьютекс не блокирует конкуренцию.

```go
type Map struct {
    mu    sync.Mutex      // защита dirty и miss-счётчика
    read  atomic.Value    // быстрая карта только для чтений
    dirty map[any]entry   // временное хранилище новых ключей
    miss  int             // количество промахов по read
}
```

## Как работает

1. **Чтение** всегда первым делом обращается к `read`. Эта карта обёрнута в `atomic.Value`, что делает доступ безмьютексным и безопасным для конкурентных горутин.
2. Если ключ найден в `read` — отлично, возврат значения без единого системного вызова.
3. Если ключ **не найден** — выполняется **промах**:
    - Тогда под мьютексом запускается поиск в `dirty`.
    - Если и там нет — возвращается `nil`.
    - А если есть — счётчик `miss` увеличивается.
4. Когда `miss > len(dirty)`, считается, что `read` уже сильно устарела, и запускается процедура **«промоушена»**: вся `dirty` копируется в `read`, а старая `read` отбрасывается.

> ‼️ Вот здесь и возникает лаг — поток, обнаруживший промах, блокирует других, пока не завершит копирование. **Это и есть узкое место, создающее пики латентности при большом количестве уникальных ключей.**

## Узкое место

Фича в том, что `read` не обновляется на каждую запись, а только когда промахов становится слишком много. Это позволяет снизить overhead при частом чтении и редких записях. Но в случаях с постоянно меняющимися ключами (`id_1`, `id_2`, `id_3`, ...) `read` всё время неактуальна, и промоушены происходят постоянно. Именно здесь `sync.Map` превращается из оптимизации в проблему: постоянное копирование карты под мьютексом создаёт контеншн.

## Когда `sync.Map` действительно хорош

Есть очень узкий, но важный класс задач, под который `sync.Map` и создавался — так называемые **read-mostly сценарии**:
- Чтения происходят постоянно, в сотнях и тысячах потоков одновременно
- Записи — крайне редки
- Набор ключей стабилен
- Пропуски случаются редко или вообще не случаются

**Классический пример:** система метрик или логирования, где по ходу работы приложения нужно отслеживать счётчики событий, тайминги, фичи — и всё это без тормозов, аллокаций и мьютексов.

## Методы sync.Map

| Метод | Описание |
|-------|----------|
| `Store(key, value interface{})` | Добавляет или обновляет значение по ключу |
| `Load(key interface{}) (value interface{}, ok bool)` | Возвращает значение по ключу. `ok` равно `false`, если ключ не найден |
| `LoadOrStore(key, value interface{}) (actual interface{}, loaded bool)` | Возвращает существующее значение, если ключ есть (`loaded = true`). Если нет, сохраняет переданное значение и возвращает его (`loaded = false`) |
| `LoadAndDelete(key interface{}) (value interface{}, loaded bool)` | Удаляет ключ и возвращает его старое значение (если оно было) |
| `Delete(key interface{})` | Удаляет значение по ключу |
| `Range(f func(key, value interface{}) bool)` | Последовательно вызывает функцию `f` для каждого ключа и значения. Если `f` возвращает `false`, обход прекращается |

## Особенности использования
> ‼️ **Типизация:** Принимает и возвращает `interface{}`, что требует приведения типов при чтении.

# sync.Once

**Синглтон (Singleton)** — это порождающий паттерн проектирования, ограничивающий создание экземпляра типа одним объектом.

`sync.Once` — это примитив синхронизации, гарантирующий, что переданная функция выполнится **ровно один раз**, независимо от того, сколько горутин вызывают `Do()` одновременно.

## Пример

```go
var once sync.Once
onceBody := func() {
    fmt.Println("Only once")
}
for i := 0; i < 10; i++ {     // не смотря на многократный вызов
    go func() {
        once.Do(onceBody)     // будет вызван лишь единожды
        done <- true
    }()
}
```

## Метод `Do`

```go
func (o *Once) Do(f func())
```

Вызывает функцию `f` только если `Do` был вызван впервые для этого инстанса `Once`. Если `once.Do(f)` вызывается несколько раз, только первый вызов выполнит `f`, даже если `f` имеет различное значение в каждом вызове.

Если требуется выполнять `f` для каждого вызова, тогда необходим новый экземпляр `Once` на каждый вызов.

## Как работает

1. Имеет внутренний флаг (`uint32`), который принимает значения 0 или 1
2. При первом вызове `Do()` блокирует остальные горутины с помощью мьютекса
3. После выполнения функции флаг устанавливается в 1
4. Все последующие вызовы `Do()` ничего не делают и сразу возвращают управление

## Где используется

- **Ленивая инициализация** — создание ресурса только когда он реально нужен
- **Синглтоны** — гарантия единственного экземпляра объекта
- **Однократная настройка** — инициализация логгера, подключение к БД
- **Отложенная загрузка конфигурации**
- **Graceful shutdown** — закрытие ресурсов только один раз

## Важные особенности

- Переданная функция **не должна паниковать** — если это случится, Once все равно посчитает, что выполнение состоялось
- **Нельзя проверить**, выполнялась ли функция без ее вызова
- Once **нельзя копировать** после использования
- Гарантирует, что все горутины увидят результаты инициализации (синхронизация памяти)

## Подводные камни

- Функция не должна блокироваться надолго — все остальные горутины будут ждать
- **Deadlock**, если функция внутри Once вызовет тот же Once
- После паники в функции повторный вызов не запустит ее снова

## Практические советы

**Используйте Once когда:**
- Нужна потокобезопасная одноразовая инициализация
- Хотите избежать двойной проверки с мьютексами
- Ресурс может не понадобиться вообще

**Не используйте** для простых флагов — `atomic` достаточно.
