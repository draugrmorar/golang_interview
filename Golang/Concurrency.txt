Горутины
    это функции или методы, выполняемые конкурентно с другими горутинами в одном и
    том же адресном пространстве. Они легковеснее традиционных потоков,
    занимают меньше памяти и позволяют эффективно использовать ядра процессора.
    Запуск горутины происходит с помощью ключевого слова go, за которым следует вызов функции
    Ожидание завершения горутин — критически важная часть управления
    конкурентными операциями в Go. Без явного ожидания горутин главный поток может
    завершиться раньше, чем горутины успеют выполнить свою работу,
    что приведет к тому, что горутины просто не выполнятся.

    Очень лёгкие — тысячи горутин могут работать одновременно.
    Не требуют явного завершения. Они завершаются, когда заканчивается их функция.
    Работают параллельно, но не обязательно одновременно (зависит от планировщика Go).

    Строение горутины: (type g struct {})
        Первое поле структуры g имеет тип stack - Динамический стек (начинается с ~2-4KB, может расти до 1GB)
            Сам стек представляет собой не что иное, как два значения, обозначающих его начало и конец.
            Горутина стартует с минимального размера стека в 2 килобайта,
            который увеличивается и уменьшается по мере необходимости без риска
            когда-либо закончиться.
            Перед выполнением любой функции Go проверяет, доступен ли объем стека, необходимый для функции,
            которую он собирается выполнить; если нет, то выполняется вызов runtime.morestack,
            который выделяет новую страницу, и только после этого выполняется функция.
            Наконец, когда эта функция завершается, ее возвращаемые аргументы копируются
            обратно в исходный фрейм стека, а все невостребованное пространство стека высвобождается.

            Хотя минимальный размер стека определен как 2048 байтов, рантайм Go также не позволяет горутинам
            превышать максимальный размер стека; этот максимум зависит от архитектуры и
            составляет 1 ГБ для 64-разрядных систем и 250 МБ для 32-разрядных систем.

            Если этот предел достигнут, будет выполнен вызов runtime.abort.
            Если превысить этот размер стека с помощью рекурсивной функции мы увидим, что приложение паникует,
            стек больше не сможет расти и выполнит вышеупомянутый runtime.abort.

    Итак, сколько горутин вы можете запустить?
        Сколько угодно, ограничены памятью,
        по мере роста числа горутин возникают две основные проблемы:
        использование памяти (и начинается свопинг) и более медленная сборка мусора.
    Когда использовать?
        - Если нужна асинхронность. Например, когда мы работаем с сетью, диском, базой данных,
          защищенным мьютексом ресурсом и т.п.
        - Если время выполнения функции достаточно велико и можно получить выигрыш, нагрузив другие ядра.
    Порядок выполнения горутин НЕДЕТЕРМИРОВАННЫЙ

Data Race
    это ситуация, когда два или более потока/горутины одновременно обращаются к одной памяти,
    и хотя бы одно обращение — запись, без синхронизации.
    go run -race main.go
    Go race detector покажет конфликтующие доступы с точностью до строки!

Race Condition
    это ситуация, когда поведение программы зависит от непредсказуемого порядка
    выполнения операций в разных горутинах/потоках.
    МЕХАНИЗМ ВОЗНИКНОВЕНИЯ
    Состояние гонки развивается по схеме:
        Критическая секция — участок кода с обращением к общим ресурсам
        Отсутствие синхронизации — несколько процессов входят в критическую секцию одновременно
        Непредсказуемый результат — конечное состояние ресурса становится неопределенным
    ТИПЫ RACE CONDITION
        Limit-overrun / TOCTOU (Time of Check — Time of Use)
            Наиболее простой тип — уязвимости, ограничивающие количество выполнений действия:
                - Многократное использование одного купона скидки
                - Погашение подарочной карты несколько раз
                - Многократное выставление оценок
                - Снятие/перевод средств сверх остатка
                - Повторное использование CAPTCHA
                - Обход anti-brute-force механизмов
        Hidden substates (Скрытые состояния)
            Более сложный тип — использование подсистем в состоянии машины, где злоумышленник
            получает доступ к состояниям в коротком временном окне.
            Алгоритм поиска скрытых состояний:
                - Определение конечных точек — выявить все точки, читающие или пишущие в подсистему
                - Три ключевых вопроса:
                    Как хранится состояние? (на сервере — интересно, на клиенте/JWT — пропускаем)
                    Мы редактируем или добавляем? (редактирование опаснее добавления)
                    На чем основана операция? (нужны операции с одним ключом)
                - Поиск подсказок — тестирование с RC-атаками, поиск отклонений в ответах
                - Доказательство концепции — минимизация до 2 запросов, автоматизация
        TIME-SENSITIVE ATTACKS (Атаки, чувствительные ко времени)
            Даже без условий гонки, точная синхронизация запросов может выявить уязвимости:
            Пример: маркер сброса пароля, генерируемый только по временной метке —
            два одновременных запроса могут получить одинаковый токен.
    ОБНАРУЖЕНИЕ RACE CONDITION
        Процесс обнаружения:
            Определить одноразовую или rate-limited конечную точку с влиянием на безопасность
            Отправить несколько запросов в быстрой последовательности
            Проверить возможность превышения лимита
        Проблемы:
            Сетевой джиттер (jitter) — непредсказуемые задержки
            Внутренние задержки сервера
            Техники отправки запросов:
                HTTP/1: классическая синхронизация по последнему байту
                HTTP/2: однопакетная атака (20-30 запросов в одном TCP-пакете)
            Техника расщепления HTTP-запроса:
                Отправка 99% запроса, удержание последнего байта
                Одновременная отправка финальных байтов для множества соединений
                Серверы ждут завершения запроса, создавая окно для гонки
    ИНСТРУМЕНТЫ ДЛЯ ЭКСПЛУАТАЦИИ
        Burp Turbo Intruder
            Высокоскоростной стек HTTP
            Автоматическая фильтрация результатов
            Плоское потребление памяти (возможны многодневные атаки)
            Конфигурация через Python
            Поддержка 22 встроенных типов атак
        RacePWN
            Утилита на Go с конфигурацией через JSON. Два режима:
                Консольная утилита: чтение конфигурационного файла
                Веб-сервис: POST-запросы на /race с конфигурацией
            Параметры конфигурации:
                Режим гонки
                Временная задержка между частями запроса
                Размер последнего чанка
    МЕТОДЫ ЗАЩИТЫ
        Семафоры — присвоение ресурсу значения доступности, ожидание при недоступности
        Мьютексы — разрешение доступа только одному процессу
        Атомарные операции — выполнение операций за один неделимый шаг

    Data Race vs Race Condition (гонка данных / состояние гонки)
        Data Race
        Два или более потока одновременно обращаются к одной памяти, и хотя бы одно обращение — запись,
        без синхронизации
        Ключевое: Прямое состязание за доступ к памяти.
        Результат — битые данные, неопределенное поведение, паника.
        Решение: всегда требует синхронизации

        Race Condition
        Поведение программы зависит от неконтролируемой последовательности или тайминга событий
        Ключевое: Логическая ошибка в проектировании.
        Результат зависит от того, в каком порядке выполняются операции.
        Решение: перепроектирование логики


Примитивы синхронизации
    это инструменты, которые позволяют управлять доступом горутин/потоков к общим ресурсам и
    координировать их выполнение.
        Примитив	Пакет	        Назначение
        Mutex	    sync	        Взаимное исключение (только одна горутина) Защита данных от одновременной записи.
        RWMutex	    sync	        Чтение многими, запись одной               Данные, которые часто читаются, редко пишутся
        WaitGroup	sync	        Ожидание группы горутин                    Нужно дождаться завершения всех горутин
        Once	    sync	        Выполнить один раз                         Ленивая инициализация, синглтоны.
        Cond	    sync	        Условные переменные                        Сложные сценарии ожидания условий.
        Atomic	    sync/atomic	    Атомарные операции                         Простые счетчики, флаги, CAS операции.
        Channel	    встроенный	    Каналы для коммуникации                    Коммуникация между горутинами
        Semaphore	нет встроенного	Ограничение доступа N горутин              Ограничение параллелизма, пулы соединений.

    Простые правила:
    - Счетчик → atomic
    - Защита данных → mutex
    - Много читаем, редко пишем → RWMutex
    - Ждем группу горутин → WaitGroup
    - Передача данных → channel
    - Инициализация один раз → Once
    - Сложное ожидание условий → Cond
    - Ограничение параллелизма → semaphore (канал)

sync.WaitGroup
    самый простой способ дождаться выполнения всех горутин
    Это структура, которая содержит счетчик количества горутин,
    завершения которых нам нужно дождаться. Он используется для ожидания завершения всех горутин.
    Для отслеживания всех горутин, которые вызываются в программе,
    WaitGroup предоставляет ряд методов:
        Add: сообщает, сколько горутин нужно дождаться.
        Done: вызывается при завершении горутины, чтобы указать, что ее выполнение завершено.
        Wait: используется основной горутиной, пока счетчик не станет равным нулю.
              Когда все горутины завершаются и счетчик становится равным нулю,
              функция Wait() возвращается и продолжает дальнейшее выполнение.
    func main() {
        var wg sync.WaitGroup
        for i := 0; i < 5; i++ {
            wg.Add(1) // 1. ДО горутины!
            go func(id int) {
                defer wg.Done() // 2. Гарантированно уменьшаем счетчик
                fmt.Printf("Горутина %d работает\n", id)
            }(i)
        }
        wg.Wait() // 3. Ждем ВСЕ горутины
    }

    - wg.Add(1) добавляет единицу к счетчику
    - wg.Done() вычитает единицу из счетчика
    - wg.Wait() ожидает когда счетчик будет равен нулю
    Таким образом wg.Wait() дождется пока каждая из горутин выполнит wg.Done()

    Внутреннее устройство
        type WaitGroup struct {
            noCopy noCopy // форсирует передачу по указателю
            state1 [12]byte // атомарный счетчик + семафор
            state2 uint32
        }
        Как работает:
            Add атомарно увеличивает счетчик
            Wait проверяет счетчик:
                Если 0 — продолжает выполнение
                Если >0 — паркует горутину
            Done уменьшает счетчик, и если стал 0 — будит все ждущие горутины
    Что произойдет, если вызвать Wait() до Add()?
        Deadlock! Wait увидит счетчик 0 и продолжит выполнение, не дождавшись горутины.
        Знаю про отрицательный счетчик → panic
    Можно ли переиспользовать WaitGroup?
        ДА! После Wait() можно снова делать Add()
    Что эффективнее — WaitGroup или каналы?
        WaitGroup — для ожидания завершения группы задач
        Каналы — для передачи данных и сигнализации
        Идеально комбинировать!

Семафор
    это примитив синхронизации, который ограничивает доступ к ресурсу
    определенным количеством горутин/потоков одновременно.
    В Go нет встроенного типа Semaphore, но его легко сделать через каналы (в паттернах проектирования есть код)

    Где используется:
    - Ограничение соединений с БД
    - Rate Limiter (ограничитель скорости)
    - "Worker Pool" с семафором

    Виды семафоров:
    - Счетный семафор - Разрешает N одновременных доступов
    - Бинарный семафор - Только 0 или 1 (как мьютекс)

    Чем семафор отличается от мьютекса?
        - Мьютекс — это бинарный семафор (0 или 1).
          У него есть "владелец" — только захватившая горутина может освободить.
        - Семафор может иметь значение >1, и любая горутина может вызвать
          Release (не обязательно та, что Acquire)
    Зачем нужен семафор, если есть буферизированные каналы?
        Семафор — это паттерн поверх каналов. Каналы — инструмент, семафор — концепция.
        В Go мы часто используем каналы как семафоры.
    Что такое "вес" семафора?
        Некоторые реализации позволяют захватывать несколько "единиц" сразу
    Может ли семафор привести к deadlock?
        Да!

Mutex
    Позволяют разграничить доступ к некоторым общим ресурсам, гарантируя,
    что только одна горутина имеет к ним доступ в определенный момент времени.
    И пока одна горутина не освободит общий ресурс, другая горутина не может с ним работать.
    На уровне кода мьютекс представляет тип sync.Mutex, который определен в пакете sync.
    Для блокирования доступа к общему разделяемому ресурсу у мьютекса вызывается метод Lock(),
    а для разблокировки доступа - метод Unlock().

    var mutex sync.Mutex
    mutex.Lock()    // блокируем доступ к переменной counter
    counter = 0
    for k := 1; k <= 5; k++{
        counter++
        fmt.Println("Goroutine", number, "-", counter)
    }
    mutex.Unlock()  // деблокируем доступ

RWMutex
    Горутина может только считывать данные, либо только изменять их.
    При выполнении операции записи совместное использование данных несколькими горутинами
    не допускается, тогда как при операции чтения несколько горутин могут считывать
    одни и те же данные без каких-либо проблем.
    Go предоставляет специальный мьютекс для чтения/записи - структуру sync.RWMutex,
    которая содержит различные поля для отслеживания доступа к памяти несколькими потоками.
    sync.RWMutex реализует несколько методов для блокировки/разблокировки:
        RLock(): используется для получения блокировки на чтение.
                 Несколько читателей могут получить эту блокировку в одной программе.
                 Блокировка на чтение не привязан к определенному потоку.
        RUnlock(): освобождает полученную одну блокировку на чтение.
        Lock(): блокирует мьютекс для записи. Доступен только для одного потока за раз.
                Если другой поток получает блокировку для целей чтения/записи,
                Lock блокируется до тех пор, пока блокировка не станет доступной для получения.
        Unlock(): снимает блокировку на запись. При выполнении Unlock
                без получения блокировки мы получим ошибку во время выполнения.

        map_data.Lock()     // блокировка на запись
        map_data.values[keyval] = keyval * 10  // изменяем общий ресурс
        map_data.Unlock()   // сбрасываем блокировку на запись

        map_data.RLock()        // устанавливаем блокировку на чтение
        val := map_data.values // считываем данные
        map_data.RUnlock()    // сбрасываем блокировку на чтение

    RWMutex имеет:
    1. Счетчик читателей
    2. Мьютекс для писателей
    3. Условную переменную
    RLock():
       - увеличивает счетчик читателей
       - если есть писатель - ждет
    Lock():
       - блокирует новые RLock
       - ждет завершения всех читателей
       - захватывает эксклюзивный доступ

    ТЕЗИСНО:
        RLock внутри Lock (Deadlock!)
            Не вызывать методы с блокировками внутри блокировок, либо создать приватный метод без блокировки.
        Забыли RUnlock
            Всегда используй defer сразу после Lock/RLock
        Запись под RLock
            ПАНИКА! Нельзя писать при RLock
        Копирование RWMutex
            Передавай по указателю: *Cache
        Можно ли вызвать RUnlock без RLock?
            Да, но будет panic:
        Что будет, если вызвать Lock() дважды в одной горутине?
            Deadlock
        Можно ли апгрейдить RLock до Lock?
            НЕТ! Это приведет к deadlock
        Если не уверен — бери Mutex. RWMutex дает выигрыш ТОЛЬКО при реально частом чтении
        и редкой записи (хотя бы 90/10)!

    STARVATION (ГОЛОДАНИЕ)
        Проблема: Писатель может никогда не получить доступ, если постоянно приходят читатели.
        Решение в Go 1.8+: RWMutex не дает голодать писателям
            - Если есть ожидающие писатели, новые читатели не могут забрать блокировку
            - Читатели уже внутри — продолжают, новым читателям — ждать
SELECT
    Оператор select позволяет go-процедуре находиться в ожидании нескольких операций передачи данных.
    select блокируется до тех пор, пока один из его блоков case не будет готов к запуску,
    а затем выполняет этот блок.
    Если сразу несколько блоков могут быть запущены, то выбирается произвольный.
    Ключевая особенность: select работает ТОЛЬКО с каналами (chan).
    Нельзя использовать select с другими типами синхронизации напрямую.
    default в select
        Блок default в select запускается, если никакой другой блок не готов.
        Используйте default, чтобы посылать и получать данные без блокировок.
    select {} без кейсов
        это оператор, который создает вечную блокировку горутины.
        Последствия:
            Горутина приостанавливается навечно
            Не потребляет CPU
            Остается жить в памяти до завершения процесса
            При массовом использовании (сотни тысяч горутин) → утечка памяти и системных ресурсов
    ПРОБЛЕМА — 4 УРОВНЯ РИСКА
        Уровень 1: Давление на планировщик
                   Заблокированные горутины остаются в очередях планировщика (runq, локальные очереди P)
                   При каждом кванте планирования происходит лишняя проверка состояния
                   Чем больше заблокированных горутин → медленнее работает планировщик
        Уровень 2: Потребление памяти
                   Каждая горутина = структура g + стек (несколько килобайт) + метаданные
                   Тысячи заблокированных горутин → неконтролируемый рост памяти
                   Рост происходит без видимой нагрузки на CPU
        Уровень 3: Захламление диагностики
                   runtime.Stack() и pprof возвращают массивы стеков
                   Реальные активные горутины тонут в массе "мертвых" ожиданий
                   Диагностика становится бесполезной из-за шума
        Уровень 4: Лимиты ОС на потоки
                   Блокировки на внешних вызовах (сеть, файлы, БД) могут приводить к росту M-воркеров
                   В Linux soft limit на треды = 1024 на процесс
                   Достижение лимита → критические сбои
    ЧТЕНИЕ ИЗ ЗАКРЫТОГО КАНАЛА
        Проблема: При закрытии канала:
            Чтение не блокируется
            Мгновенно возвращается нулевое значение
            Цикл крутится с бешеной скоростью
            CPU растет до потолка
            Система перестает отвечать
        Решение: Всегда проверять открыт ли канал
            select {
            case v, ok := <-ch:
                if !ok {
                    return // Канал закрыт - завершаем горутину
                }
                fmt.Println(v)
            }
    КОГДА SELECT {} ОПРАВДАН (ЛЕГИТИМНЫЕ СЛУЧАИ)
        1. Блокировка до внешнего сигнала. Ожидание системных сигналов (SIGTERM/SIGINT)
        2. В тестах с таймаутом
    Разные каналы
        Чтение из канала (<-ch):
            Открытый с данными: кейс готов немедленно
            Открытый пустой: блокировка до появления данных
            Закрытый: кейс готов немедленно, возвращается нулевое значение и false
            nil: кейс игнорируется (никогда не будет готов)
        Запись в канал (ch <-):
            Открытый с буфером: если есть место — готов немедленно
            Открытый без буфера: готов только когда есть получатель
            Закрытый: паника!
            nil: кейс игнорируется
        nil-каналы:
            Кейс с nil-каналом никогда не выбирается
            Используется для динамического отключения кейсов
            Полезно в циклах для временного игнорирования каналов
    СПЕЦИАЛЬНЫЕ СЛУЧАИ
        Пустой select {}:
            Нет кейсов, нет default
            Блокирует горутину навсегда
            Никогда не просыпается
            Причина утечек ресурсов
        Select с default:
            Неблокирующая проверка каналов
            Если ни один кейс не готов — выполняется default
            Позволяет реализовать неблокирующие операции
        Select в цикле for:
            Типичный паттерн для долгоживущих горутин
            Требует осторожности с закрытием каналов
            Необходима проверка ok при чтении
    ОСОБЕННОСТИ РАБОТЫ С ПЛАНИРОВЩИКОМ
        Состояние горутины:
            При блокировке в select горутина переходит в состояние waiting
            Освобождает поток (M) для других горутин
            Не потребляет CPU, но остается в памяти
        Пробуждение:
            При готовности канала горутина перемещается в run queue
            Планировщик назначает ее на доступный поток
            Возможно переключение на другой поток (не обязательно тот же)
        Множественные готовые кейсы:
            Рантайм выбирает случайный
            Остальные готовые кейсы не теряются — они остаются готовыми
            Гарантируется, что хотя бы один выполнится
    ТОНКОСТИ И НЮАНСЫ
        Порядок выполнения:
            Нет гарантии, что кейс, который первым стал готов, выполнится первым
            Выбор происходит только когда select достигается в коде
            Операции на каналах не буферизируются в select
        Завершение горутины:
            Выход из горутины через return в кейсе
            break в select выходит только из select, не из цикла
            Для выхода из цикла нужны метки (labels) или return
        Обработка паники:
            Паника в одном кейсе не влияет на другие
            Паника всплывает как обычно, если не обработана
            Закрытие канала во время select — особая ситуация
        Производительность:
            Небольшой оверхед на проверку состояния каналов
            Линейно зависит от количества кейсов (O(n))
            На практике оптимизирован для небольшого числа кейсов
    ЧАСТЫЕ ОШИБКИ
        Отсутствие проверки закрытия канала:
            При чтении из закрытого канала кейс срабатывает бесконечно
            В цикле приводит к 100% CPU
            Решение: всегда проверять ok
        Забытый default:
            Может привести к нежелательной блокировке
            Или наоборот — к активному ожиданию (spin loop)
        Модификация каналов:
            Нельзя менять каналы во время выполнения select
            Каналы фиксируются на момент входа в select
        Забытый выход:
            Горутина с select без условия выхода остается навсегда
            Причины утечек в долгоживущих программах
    ДИАГНОСТИКА ПРОБЛЕМ
        Как увидеть заблокированные select:
            pprof показывает горутины в состоянии chan receive/send
                            go tool pprof http://localhost:6060/debug/pprof/goroutine
            runtime.Stack показывает точное место блокировки
                            buf := make([]byte, 1<<20)
                            runtime.Stack(buf, true)
                            fmt.Printf("%s\n", buf)
            В дампе видно, на каких каналах ждет горутина
        Признаки проблем:
            Рост числа горутин без нагрузки
            Утечки памяти
            Деградация производительности планировщика
            Зашумленные профили
    ОПТИМИЗАЦИЯ И ЛУЧШИЕ ПРАКТИКИ
        Проектирование:
            Каждая горутина с select должна иметь сценарий завершения
            Использовать сигнальные каналы (chan struct{}) для остановки
            Документировать вечные блокировки явно
        Обработка ошибок:
            Всегда проверять ok при чтении
            Добавлять таймауты через time.After
            Использовать default для неблокирующих проверок
        Производительность:
            Минимизировать количество кейсов
            Избегать select в горячих циклах без необходимости
            Для простых случаев достаточно single channel receive

Основные принципы пакета sync
    Никогда не копируйте sync-объекты — они содержат внутренние состояния, которые нельзя копировать (используйте указатели)
    Zero value — большинство типов готовы к использованию сразу (без инициализации)
    Не используйте без необходимости — часто каналы выразительнее
    После использования нельзя копировать — проверяется go vet
