СТРОКИ
        Строка физически - это массив байтов. По умолчанию строки кодируются в UTF-8 в языке Go.
        Строка логически - это массив рун.
        Строка состоит из двух составляющих: это указатель на массив байт и длина массива.
        При передаче строки не копируется весь текст, передается только указатель и длина (16 байт на 64-битной системе).
        Строка иммутабельна (Immutable)
    Руны
        Rune - это псевдоним для int32
        Литерал типа данных rune - это одинарные кавычки.
        runes := []rune("Хай") - сделать строку массивом рун
    strings.Builder{}
        Для построения строк этот подход менее затратный чем конкатенация
        Для построения новых строк/ перестроения старых/ склеивания и тд
        builder := strings.Builder{}
        builder.WriteString(strconv.Itoa(i))
        str := builder.String()
    strconv.Itoa(i) / strconv.Atoi() "Integer to ASCII" / "ASCII to Integer"
        Использовать эти функции для превращения чисел в строки и обратно, так как в случае явного приведения типа,
        приведение происходит байт, а не рун
    Подстрока из строки
        Делается так же как создаются слайсы. Но тут опять не следует забывать что строка это массив байт.
        Подстрока начинает указывать на элемент из старой строки. - это помешает памяти очистить старую строку:
        как решение этой проблемы лучше создать копию
        firstWord := string([]byte(message[:5]))
        firstWord := strings.Clone(message[:5]) // Go1.18
    Обратные кавычки ``
        сохраняют расставленные переносы строк, не схлопывают пробелы и
        позволяют удобно расставлять внутри себя одинарные и двойные кавычки без экранирования.
    Сравнение строк
        оператор ==
            производительный - быстрее чем strings.Compare()
        strings.Compare()
            самый медленный способ сравнения строк
            возвращает -1/0/1 - первая строка меньше второй/ строки равны / первая строка больше второй
        strings.EqualFold()
            самый производительный, но он не учитывает верхний или нижний регистр -
            поэтому можно использовать если регистр не важен
МАССИВЫ
    массив - коллекция элементов одного типа. Длина массива не может изменяться.
        Если мы создадим два массива в Go с разными длинами, то два массива будут иметь разные типы,
        так как длина массива в Go, входит в его тип.
        Копируется при передаче в другую переменную или при отправке в другую область видимости.
        arr := [4]int{3,2,5,4} // создание массива
    Копирование
        Копирование массива — это просто присвоение одного массива другому.
        Поскольку в Go массив ведет себя как значение, а не как ссылка,
        то изменение любого значения одного массива не оказывает никакого влияния на другой.
    Сравнение
        Два массива называются равными, если они оба имеют одинаковый тип и размер, имеют одинаковые элементы,
        и все элементы находятся в одинаковом порядке в обоих массивах.
СЛАЙСЫ
    Срезы (slice) представляют последовательность элементов одного типа переменной длины.
    В отличие от массивов длина в срезах не фиксирована и динамически может меняться,
    то есть можно добавлять новые элементы или удалять уже существующие.
    bar := []byte{} // создание слайса
    Способ с make является более интересным, так как дает нам возможность задать тип, длину и емкость.

    Строение слайса под капотом:
        struct {
    	    array *[]T // ссылка на исходный массив
    	    length int
    	    capacity int
        }
    len()
        Длина слайса. Общее количество элементов, которые содержит срез.
    cap()
        Емкость среза — это количество элементов в базовом массиве, начиная с первого элемента среза.
        Это означает, что емкость среза относится к размеру, до которого может быть расширен срез.
    append
        Добавляет элементы в конец среза (slice), автоматически управляя базовым массивом.
        Если емкости (capacity) достаточно, она просто увеличивает длину (length) и записывает данные.
        Если места нет, она создает новый массив большего размера, копирует туда старые данные и добавляет новые.
        Динамические расширяясь, капасити нового слайса растет в два раза для маленьких слайсов (до 256),
        а потом капасити растет в 1.25–1.5.
    копирование copy()
        Эта функция принимает два аргумента: исходный срез, который мы хотим скопировать,
        и целевой срез, в который мы хотим скопировать элементы.
        Она всегда возвращает целочисленное число, которое представляет количество скопированных элементов.
        Возвращаемое значение всегда является минимальной длиной исходного среза и целевого среза.
    сортировка
        Несколько методов сортировки для среза по типам sort.Ints(), sort.Float64s() и sort.Strings.
        Эти методы сортировки всегда сортируют срез в порядке возрастания
    поиск
        sort.SearchInts(), sort.SearchFloat64s() и sort.SearchStrings
        Эти функции выполняют двоичный поиск в отсортированном срезе в порядке возрастания.
        Если элемент найден в отсортированном срезе, функция поиска возвращает его индекс;
        если нет, она возвращает индекс, по которому элемент будет помещен в срез
    сравнение
        Два среза считаются равными, если они хранят данные одного типа и содержат одни и те же элементы.
        Go предоставляет функцию reflect.DeepEqual(slice1, slice2)

МАПЫ
    Что это
        Хэш-таблица — это структура данных, ассоциативный массив.
        Она позволяет хранить пары «ключ-значение».
        Хэш-таблица нужна для хранения и организации информации с помощью хэш-функции.
        - В качестве ключа можно использовать любой тип данных для которого определена операция сравнения.
          Например, можно использовать структуру с тем же условием для всех ее полей.
        - При отсутствии элемента возвращается нулевое значение для типа.
          Вторым параметром можно получить флаг наличия элемента по ключу
        - Нельзя получить адрес элемента. Потому что при росте мапы оно переедет в другой бакет и
          адрес у него, соответственно, поменяется
        - Мапа не безопасна для конкурентного использования.
          Для этого можно использовать обертку из sync.Map или мьютекс
        - Порядок итерации не сохраняется. При каждой новой итерации мапы
          последовательность возвращаемых элементов может отличаться.
          Под капотом каждый раз выбирается рандомный бакет, с которого начинается итерация.
          Для сохранения нужного порядка придется сохранять ключи в отдельном массиве и итерироваться по нему
        - При каждом создании мапы генерируется seed для рандомизации хэш-функции.
          Это сделано для безопасности, так как зная хэш-функцию можно подобрать такие ключи,
          что все значения попадут в один бакет и мы получим линейную скорость поиска.
        - При коллизиях используется стратегия сlosed addressing.
          Мы перебираем все ячейки бакета (их 8) и ищем первое свободное место.
        - OverloadFactor равен 6.5 (~81% заполненности бакетов).
          Когда бакеты в среднем заполнены больше чем на 6.5 элементов, триггерится рост мапы,
          и все элементы перемещаются в новые бакеты, которых создается в два раза больше.
        - При росте элементы переносятся в новые бакеты постепенно, а не все сразу.
        - В Go 1.24+ - мапа поменялась и внутренняя логика тоже
    Мапа строение структуры
        type hmap struct {
        	count     int // размер мапы. используется функцией len()
        	---- flags     uint8
        	B         uint8  // log_2 количества бакетов. Для 8 бакетов B=3, для 16 B=4 и так далее.
        	noverflow uint16 // примерное число переполненных бакетов
        	hash0     uint32 // seed для хэш-функции, генерируется при создании мапы. Нужен для рандомизации хэш-функции
        	buckets    unsafe.Pointer // ссылка на массив из 2^B бакетов; nil, если count==0
        	oldbuckets unsafe.Pointer // ссылка на массив предыдущих бакетов.
        	                          // в процессе роста мапы здесь будет массив старых бакетов,
        	                          // откуда потихоньку будут переноситься значения в новые бакеты.
        	nevacuate  uintptr        // количество "эвакуированных" бакетов.
        	extra *mapextra // опциональные поля
        }
        // Бакет
        type bmap struct {
          tophash [bucketCnt]uint8     // массив, содержит 8 верхних байт хэшей каждого ключа в бакете.
                                       // Для оптимизации поиска храним массив первых восьми бит от хэша ключа
                                       // После массива tophash идет массив размера bucketCnt ключей и
                                       // массив размера bucketCnt элементов.
        }

    Мапа строение логическое
        Состоит из бакетов: в бакете 8 элементов
    Коллизии (способы решения /реиндексация)
         это ситуация, когда два различных ключа имеют одинаковое хеш-значение и, как следствие,
         попадают в одну и ту же «корзину» (ячейку) массива данных.
         Способы решения коллизий: метод открытой адресации и метод цепочек
         Открытая адресация
            Мы помещаем ключ в свободную ячейку, следующую за текущей или с определённым шагом.
            Не забываем, что при работе с хэш-таблицей надо не только записывать в неё,
            но и выполнять по ней поиск.
            Он будет происходить в том же порядке, что и при вставке,
            и до тех пор, пока не будет найден элемент с искомым ключом или свободная ячейка.
            Последнее означает отсутствие элемента в хэш-таблице.
         Метод цепочек
            При использовании метода цепочек мы добавляем элементу в текущей ячейке ссылку на новый.
            Так появляются связь между предыдущим и новым элементами,
            образуется цепочка элементов с одинаковыми хэшами (двусвязный список).
         Сложность O(N)
    Сложность
        O(1)
    Overflow-бакет
        это дополнительный бакет, который создается, когда основной бакет (и его стандартные 8 ячеек) переполняется.
        Фактически это "запасной" бакет, связанный с основным через указатель.
        - Разрешение коллизий
        - Предотвращение немедленного роста
        - Баланс производительности и памяти
    load factor
        Коэффициент заполняемости таблицы — это отношение количества хранимых пар «ключ-значение» к размеру таблицы.
        Если они достигают пограничного значения, то хэш-мапу расширяют, а индексы пар пересчитывают и перераспределяют.
    Расширение Эвакуация
        Увеличение количества бакетов в Go и распределение существующих пар называется эвакуацией.
        Рост мапы начинается в двух случаях:
            - бакеты в среднем заполнены на больше чем ~80% (loadFactor равно 6.5).
            - частный случай (не закодирован) - количество overflow бакетов примерно равно количеству самих бакетов.
        При эвакуации количество бакетов увеличивается в два раза.
        Текущие пары с топхэшами распределяются по этим бакетам.
        В Go используются два подхода, которые сводятся к увеличению количества бакетов:
            - Увеличение общего количества бакетов и перераспределение пар по этим бакетам.
            - Создание бакета, на который даём ссылку в заполненный бакет.
              Если понадобится записать в заполненный, то мы запишем в тот, на который есть ссылка.
        три стратегии:
        - мапа не растет, и тогда программист выделяет мапу достаточного размера заранее
        - мапа растет за время операции вставки, и некоторые вставки очень долгие
        - данные переносятся в новую мапу в фоне — это процесс эвакуации
    Процесс эвакуации
        Ключевая особенность реализации Go в том, что эвакуация происходит постепенно, во время штатных операций с map.
        Это сделано для того, чтобы избежать длительных stop-the-world) при копировании большого объема данных
        1 Инициализация роста (hashGrow)
          При срабатывании одного из триггеров вызывается функция hashGrow . Она:
          - Выделяет память под новый массив бакетов (buckets) в два раза большего размера (или того же, в случае "равного роста").
          - Поле oldbuckets начинает указывать на старый массив бакетов.
          - Поле nevacuate сбрасывается в 0.
          - На этом этапе данные не копируются, только создается новый массив.
        2 Постепенная эвакуация (growWork и evacuate)
          Каждый раз при последующих операциях записи (mapassign) или удаления (mapdelete) из map,
          вызывается функция growWork, которая запускает процесс эвакуации для 1-2 бакетов :
          - Первый бакет для эвакуации — это тот, к которому происходит обращение (например, бакет, куда должна произойти вставка).
          - Второй бакет — это бакет с индексом h.nevacuate (текущий счетчик прогресса).
          - Логика эвакуации бакета (evacuate)
        3 Функция evacuate выполняет перенос данных из одного конкретного старого бакета
        (и всей его цепочки overflow-бакетов) в новый массив. Алгоритм для случая удвоения:
          - Берется старый бакет.
          - Для каждой пары ключ-значение в бакете и его overflow-бакетах вычисляется хеш ключа.
          - Анализируется один новый бит хеша (который появился из-за увеличения B).
            Если этот бит равен 0, элемент перемещается в бакет с тем же номером в первой половине нового массива.
            Если бит равен 1 — в бакет с номером старый_номер + старое_количество_бакетов во второй половине.
          - Элементы в новом бакете размещаются компактно.
          - После успешного переноса всех данных старый бакет помечается как "эвакуированный".
            Для этого используется специальное значение в поле tophash[0] (например, evacuatedX, evacuatedY).
            Это позволяет операциям чтения понимать, где искать данные.
        4. Поведение операций во время эвакуации
           Запись (mapassign) и Удаление (mapdelete):
           - Как уже говорилось, они являются движущей силой процесса.
             Перед выполнением основной операции гарантированно эвакуируется целевой бакет (и один по счетчику).
           Чтение (mapaccess):
           - Сначала проверяется, не находится ли map в состоянии роста (h.oldbuckets != nil).
           - Вычисляется хеш ключа и индекс предполагаемого бакета в новом массиве.
           - Проверяется, эвакуирован ли уже соответствующий старый бакет (по флагу evacuated).
           - Если старый бакет еще не эвакуирован, поиск ключа происходит в нем (и его overflow-бакетах).
             Если ключ не найден в старом бакете, это означает, что его там нет
             (либо его никогда не было, либо он был перенесен при эвакуации этого бакета ранее).
           - Если старый бакет уже эвакуирован, поиск идет сразу в новом массиве.
        5. Окончание эвакуации
           Процесс продолжается до тех пор, пока все старые бакеты не будут перенесены
           (h.nevacuate не достигнет количества старых бакетов).
           Когда эвакуация завершена, h.oldbuckets устанавливается в nil, и
           память, занятая старым массивом бакетов и связанных с ним overflow-бакетов,
           становится доступной для сборщика мусора (GC).
           Map снова входит в нормальный режим работы.
    Алгоритм добавления элемента
        В цикле ищем совпадения по tophash и ключу или свободное место.
        Сохраняем данные и возвращаем соответствующий флаг для подсчета элементов в мапе.
        Если не нашли нужного места, то повторяем упражнения на overflow бакете.
    Удаляем элемент
        Вместо фактического удаления просто помечаем ячейку пустой.
    Итерация по мапе
        Происходит магия "рандомной" последовательности элементов при каждой новой итерации:
        выбирается случайный бакет, с которого начнется итерация, и отступ,
        с которого будут возвращаться элементы внутри каждого бакета.
        Алгоритм итерации можно разделить на две части:
        - простой случай - когда мапа не в процессе роста;
        - сложный - в процессе роста. Для этого потребуются дополнительные проверки.
МАПА Go 1.24+
        Различия:
            - Новая структура данных
            - Отсутствие оverflow-бакетов
            - Тип решения коллизий: раньше был: Закрытый тип индексации (цепочки переполнения),
              стал Открытый (линейное/квадратичное пробирование)
            - Разный тип расширения
            - load factor был 80%, стал 87%
        Новая структура данных:
            Slot (Слот): Самая маленькая единица — это пара "ключ-значение".
            Group (Группа): Группа объединяет 8 слотов.
                Это аналог старого бакета, но теперь все группы в рамках одной таблицы имеют фиксированный размер.
            Table (Таблица): Это и есть основная хеш-таблица, которая состоит из множества групп.
                Таблица имеет ограничение на максимальный размер (например, не более 1024 слотов или 128 групп).
            Directory (Директория): Это массив указателей на таблицы.
                Когда одна таблица переполняется, создается новая, и указатель на нее добавляется в директорию.
                Выбор нужной таблицы происходит по первым битам хеша ключа (Extendible Hashing).
        Расширение:
            Рост инициируется, когда коэффициент загрузки (load factor) достигает 7/8 (87%)
            Рост для небольших map (до ~1024 элементов) Если map небольшая и состоит из одной таблицы, то при переполнении:
                - Создается новая таблица большего размера (обычно в 2 раза).
                - Все элементы из старой таблицы немедленно перехешируются и копируются в новую таблицу.
                - Указатель в директории начинает указывать на новую таблицу,
                  старая таблица становится доступной для сборщика мусора (GC).
            Рост для больших map (более 1024 элементов) — расщепление таблиц
            Для больших map используется механизм, позволяющий избежать копирования всей огромной таблицы целиком.
            Этот подход называется Extendible Hashing (расширяемое хеширование).
            Вместо увеличения одной таблицы, Go создает новую таблицу, а старая таблица может быть расщеплена.
                - Создается новая, пустая таблица.
                - Часть элементов из переполненной таблицы перемещается в новую таблицу.
                  Критерий для перемещения — значение старших битов хеша ключа
                  (биты, которые отвечают за выбор таблицы в директории).
                - В директории появляется новая запись, указывающая на созданную таблицу.
                  Глубина директории (globalDepth) увеличивается, и некоторые старые указатели
                  могут быть продублированы или перенаправлены.
                - В результате этого процесса большая часть данных остается на месте,
                  и копируется только часть элементов из одной переполненной таблицы, а не вся map целиком.
                  Это значительно сокращает задержки при росте.
