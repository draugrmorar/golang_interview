## Обработка ошибок (Error handling) в Go

---

### Философия ошибок в Go

**Ошибки — это значения (errors are values)** . В Go нет исключений (exceptions) как в Java/Python. Ошибки возвращаются как обычные значения и обрабатываются явно.

```go
result, err := doSomething()
if err != nil {
    // обрабатываем ошибку
}
```

---

### Тип error

`error` — это **встроенный интерфейс**:

```go
type error interface {
    Error() string
}
```

Любой тип, реализующий метод `Error() string`, автоматически удовлетворяет интерфейсу `error`.

---

### Создание ошибок

#### 1. `errors.New()` — простая ошибка

```go
import "errors"

err := errors.New("что-то пошло не так")
```

#### 2. `fmt.Errorf()` — с форматированием

```go
err := fmt.Errorf("пользователь %s не найден", username)
```

#### 3. Свои типы ошибок (для доп. данных)

```go
type ValidationError struct {
    Field string
    Value interface{}
    Msg   string
}

func (e ValidationError) Error() string {
    return fmt.Sprintf("%s: %v — %s", e.Field, e.Value, e.Msg)
}

// Использование
if age < 0 {
    return ValidationError{Field: "age", Value: age, Msg: "must be positive"}
}
```

---

### Проверка типа ошибки

#### 1. `errors.As()` — извлечение конкретного типа

```go
var valErr ValidationError
if errors.As(err, &valErr) {
    fmt.Printf("ошибка валидации в поле %s", valErr.Field)
}
```

#### 2. `errors.Is()` — сравнение с конкретным значением

```go
var ErrNotFound = errors.New("not found")

if errors.Is(err, ErrNotFound) {
    // обрабатываем "не найдено"
}
```

#### 3. Type assertion (старый способ)

```go
if valErr, ok := err.(ValidationError); ok {
    fmt.Println(valErr.Field)
}
```

---

### Паника (panic) и восстановление (recover)

**Паника** — это исключительная ситуация, которую нельзя обработать через `error`.

```go
panic("что-то ужасное случилось")
```

**recover** — ловит панику (только внутри defer):

```go
defer func() {
    if r := recover(); r != nil {
        fmt.Println("паника перехвачена:", r)
    }
}()
```

**Когда использовать панику:**
- Инициализация программы (нельзя продолжить без конфига)
- Фатальные ошибки, которые не должны происходить
- В тестах (для проверки, что функция паникует)

**В обычном коде — только ошибки, не паника!**

---

### Оборачивание ошибок (error wrapping)

Go 1.13+ позволяет добавлять контекст к ошибкам.

```go
if err := readFile(); err != nil {
    return fmt.Errorf("чтение файла: %w", err)  // %w — оборачивает
}
```

**Распаковка:**
```go
err := fmt.Errorf("ошибка: %w", originalErr)

if errors.Is(err, originalErr) {       // true
    // ...
}

var pathErr *os.PathError
if errors.As(err, &pathErr) {          // можно извлечь
    fmt.Println(pathErr.Path)
}
```

**Unwrap:** `errors.Unwrap(err)` возвращает оригинальную ошибку.

---

### Полезные пакеты

| Пакет | Назначение |
|-------|------------|
| `errors` | Базовые операции (New, Is, As, Unwrap) |
| `fmt` | Errorf с `%w` для оборачивания |
| `github.com/pkg/errors` | (исторически) Добавлял стек-трейсы, но сейчас есть встроенные средства |
| `errors.Join` (Go 1.20+) | Объединение нескольких ошибок |

---

### Объединение ошибок (Go 1.20+)

```go
var errs error
for _, op := range operations {
    if err := op(); err != nil {
        errs = errors.Join(errs, err)
    }
}
if errs != nil {
    return errs  // вернутся все ошибки через '\n'
}

// Проверка
if errors.Is(errs, ErrSpecific) { /* ... */ }
```

---

### Частые ошибки (no pun intended)

#### 1. Игнорирование ошибок

```go
db.Query("...")  // ❌ ошибка потеряна

// ✅ всегда проверяйте или явно игнорируйте (с _
_, _ = db.Query("...")
```

#### 2. Сравнение через `==` с кастомными ошибками

```go
if err == MyError{}  // ❌ не работает, сравниваются значения

var myErr MyError
if errors.As(err, &myErr)  // ✅
```

#### 3. Перетирание ошибки в defer

```go
func bad() (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panic: %v", r)
        }
    }()
    err = doSomething()
    return  // может перетереть ошибку из panic!
}
```

#### 4. Оборачивание nil

```go
var err error
return fmt.Errorf("ошибка: %w", err)  // ❌ запаникует, если err == nil
if err != nil {
    return fmt.Errorf("ошибка: %w", err)  // ✅
}
```

---

### Паттерны обработки ошибок

#### 1. Продолжаем даже с ошибкой (например, чтение из кеша)

```go
data, err := cache.Get(key)
if err != nil {
    // логируем, но не падаем
    log.Printf("cache error: %v", err)
    data = fetchFromDB(key)  // fallback
}
```

#### 2. Аккумулируем ошибки

```go
var errs []error
for _, item := range items {
    if err := process(item); err != nil {
        errs = append(errs, err)
    }
}
if len(errs) > 0 {
    return errors.Join(errs...)
}
```

#### 3. Пакетные операции с остановкой

```go
for _, item := range items {
    if err := process(item); err != nil {
        return fmt.Errorf("stop on %s: %w", item.Name, err)
    }
}
```

#### 4. Retry с экспоненциальной задержкой

```go
var err error
for attempt := 0; attempt < 3; attempt++ {
    if err = doHTTPRequest(); err == nil {
        return nil
    }
    time.Sleep(time.Second << attempt)
}
return fmt.Errorf("failed after retries: %w", err)
```

---

### Резюме для собеса

> **Обработка ошибок в Go** строится на возврате значений типа `error`, который является интерфейсом с методом `Error() string`. Ошибки создаются через `errors.New()` или `fmt.Errorf()`, с Go 1.13 появилось оборачивание через `%w` и функции `errors.Is`/`errors.As` для проверки. **Паники** используются только для фатальных ситуаций, их ловят через `recover` в `defer`. Ключевой принцип — явная обработка: ошибки не игнорируются, а проверяются сразу после вызова функции.

## Рефлексия (Reflection) в Go 

---

### Что такое рефлексия?

**Рефлексия** — это способность программы **исследовать и модифицировать свою собственную структуру** во время выполнения . В Go рефлексия реализована в пакете **`reflect`** .

Простыми словами: это код, который смотрит на другой код и спрашивает: *"Что ты за тип? Какие у тебя поля? А вызвать твой метод можно?"*

---

### Зачем нужна рефлексия?

| Сценарий | Пример |
|----------|--------|
| **Сериализация** | JSON, XML, YAML — пакеты смотрят на структуры |
| **ORM** | GORM смотрит на теги структур (`gorm:"column:name"`) |
| **Тестирование** | Сравнение сложных структур (`reflect.DeepEqual`) |
| **Фреймворки** | Инъекция зависимостей, валидация |
| **Плагины** | Вызов методов по именам |

Без рефлексии пришлось бы писать код для каждого типа вручную.

---

### Два главных типа в reflect

```go
// Тип — информация о типе
var t reflect.Type = reflect.TypeOf(x)

// Значение — содержимое переменной
var v reflect.Value = reflect.ValueOf(x)
```

---

### Получение информации о типе

```go
var x float64 = 3.4
t := reflect.TypeOf(x)

fmt.Println(t.Name())        // "float64"
fmt.Println(t.Kind())        // reflect.Float64
fmt.Println(t.Size())        // 8 (байт)
fmt.Println(t.String())      // "float64"
fmt.Println(t.Align())       // 8 (выравнивание)
```

**Kind vs Name:**
- `Kind()` — базовая категория (struct, slice, int, float64...)
- `Name()` — имя типа (например, "MyStruct", "float64" — если не кастомный, то совпадает)

---

### Работа со значениями

```go
v := reflect.ValueOf(x)
fmt.Println(v.Kind())        // reflect.Float64
fmt.Println(v.Float())       // 3.4 (получить значение)
fmt.Println(v.Interface())   // interface{} с исходным значением

// Изменение значения (только если передано как указатель!)
v = reflect.ValueOf(&x).Elem()  // Elem() разыменовывает указатель
v.SetFloat(7.1)                  // x теперь 7.1
```

---

### Рефлексия структур — самое частое применение

```go
type User struct {
    Name string `json:"name" validate:"required"`
    Age  int    `json:"age" validate:"min=0"`
}

u := User{Name: "Анна", Age: 25}
t := reflect.TypeOf(u)
v := reflect.ValueOf(u)

// Перебор полей
for i := 0; i < t.NumField(); i++ {
    field := t.Field(i)
    value := v.Field(i)
    
    fmt.Printf("Поле: %s\n", field.Name)
    fmt.Printf("  Тип: %s\n", field.Type)
    fmt.Printf("  Значение: %v\n", value.Interface())
    fmt.Printf("  Тег json: %s\n", field.Tag.Get("json"))
    fmt.Printf("  Тег validate: %s\n", field.Tag.Get("validate"))
}
```

---

### Рефлексия методов

```go
type Calculator struct{}

func (c Calculator) Add(a, b int) int { return a + b }

// Вызов метода по имени
c := Calculator{}
v := reflect.ValueOf(c)
method := v.MethodByName("Add")
args := []reflect.Value{reflect.ValueOf(2), reflect.ValueOf(3)}
result := method.Call(args)
fmt.Println(result[0].Int()) // 5
```

---

### reflect.DeepEqual — глубокое сравнение

```go
m1 := map[string]int{"a": 1, "b": 2}
m2 := map[string]int{"a": 1, "b": 2}

// == с мапами не работает
if reflect.DeepEqual(m1, m2) {
    fmt.Println("мапы равны")
}
```

**Что умеет сравнивать:**
- Срезы (даже nil и пустой считает равными!)
- Мапы
- Структуры (рекурсивно)
- Циклические ссылки (осторожно!)

---

### Производительность — главный минус

```go
BenchmarkDirectCall         1 ns/op
BenchmarkReflectionCall     50-100 ns/op   // в 50-100 раз медленнее
BenchmarkReflectionFieldAccess // тоже медленно
```

**Почему:**
- Динамические проверки вместо статических
- Escape-анализ не всегда работает
- Дополнительные аллокации

---

### Когда использовать рефлексию

| ✅ **Можно и нужно** | ❌ **Лучше избегать** |
|---------------------|----------------------|
| Библиотеки общего назначения | Внутренний код приложения |
| JSON/XML сериализация | Горячие пути (тысячи вызовов в секунду) |
| ORM и валидаторы | Где можно обойтись интерфейсами |
| Тесты и отладка | Критичный к производительности код |
| Фреймворки | Там, где тип известен на этапе компиляции |

---

### Частые ошибки

#### 1. Передача значения вместо указателя для изменения

```go
v := reflect.ValueOf(x)  // x — значение
v.SetFloat(7.1)           // PANIC! нелья изменить

v := reflect.ValueOf(&x).Elem()  // ✅ правильно
```

#### 2. Путаница между Kind и Type

```go
type MyInt int
var x MyInt = 5
t := reflect.TypeOf(x)
fmt.Println(t.Name()) // "MyInt"
fmt.Println(t.Kind()) // "int"  — базовая категория
```

#### 3. Паника при вызове метода с неправильными аргументами

```go
method.Call([]reflect.Value{}) // если метод принимает параметры — паника!
```

#### 4. Забыть проверить CanSet

```go
if v.CanSet() {
    v.Set(newValue)
}
```

---

### Полезные функции

```go
// Создание нового экземпляра типа
t := reflect.TypeOf(MyStruct{})
newStruct := reflect.New(t).Elem()  // создаёт новый, как new()

// Проверка, реализует ли тип интерфейс
var reader io.Reader
t := reflect.TypeOf(os.Stdin)
if t.Implements(reflect.TypeOf(&reader).Elem()) {
    fmt.Println("реализует io.Reader")
}
```

---

### Рефлексия vs Дженерики

| | Рефлексия | Дженерики |
|---|-----------|-----------|
| **Когда работает** | В рантайме | На этапе компиляции |
| **Типобезопасность** | Нет (паника!) | ✅ Да |
| **Производительность** | Медленно | ✅ Быстро (как ручной код) |
| **Гибкость** | Максимальная | Ограничена |
| **Что можно** | Всё что угодно | Только то, что описано в ограничениях |

**Золотое правило:**  
> Сначала думай о дженериках. Если не получается — используй рефлексию. Но будь готов платить за это скоростью.

---

### Резюме для собеса

> **Рефлексия** (`reflect`) позволяет программе анализировать и изменять свою структуру во время выполнения. Используется в сериализации (JSON), ORM, тестировании (`DeepEqual`), фреймворках. Два основных типа: `reflect.Type` (информация о типе) и `reflect.Value` (значение). Главные минусы — медленная работа и потеря типобезопасности (ошибки в рантайме). Начинать всегда стоит с дженериков или интерфейсов, а рефлексию оставлять для библиотек общего назначения.

## Сериализация / Десериализация в Go

---

### Что это простыми словами

**Сериализация** — превращение объекта/структуры данных в **последовательность байт** для:
- Хранения (в файл)
- Передачи (по сети)
- Обмена между программами

**Десериализация** — обратный процесс, восстановление объекта из байт.

---

### Зачем нужна

- Сохранение состояния программы между запусками
- Передача данных по сети (клиент-сервер, микросервисы)
- Кэширование объектов на диске или в Redis
- Межпроцессное взаимодействие
- Логирование и отладка (превратить объект в текст)

---

### Форматы сериализации в Go

| Формат | Пакет | Когда использовать |
|--------|-------|-------------------|
| **JSON** | `encoding/json` | **Самый популярный**, веб-API, человек-читаемый |
| **Gob** | `encoding/gob` | Специфичный для Go, эффективный, только для Go-сервисов |
| **XML** | `encoding/xml` | Легаси, интеграция со старыми системами |
| **Protocol Buffers (protobuf)** | `google.golang.org/protobuf` | Высокая производительность, строгая схема, микросервисы |
| **MessagePack** | `github.com/vmihailenco/msgpack` | Компактнее JSON, быстрее |
| **YAML** | `gopkg.in/yaml.v3` | Конфиги, человек-читаемый |
| **Binary** | `encoding/binary` | Низкоуровневая работа, бинарные протоколы |

---

### JSON — самый частый на собесах

#### Сериализация (Marshal)

```go
type User struct {
    Name string `json:"name"`
    Age  int    `json:"age"`
    Password string `json:"-"`              // игнорировать
    Email    string `json:"email,omitempty"` // пропустить, если пустое
}

user := User{Name: "Анна", Age: 25}
data, err := json.Marshal(user)
// data: {"name":"Анна","age":25}

// С отступами (pretty print)
data, _ := json.MarshalIndent(user, "", "  ")
```

#### Десериализация (Unmarshal)

```go
jsonData := []byte(`{"name":"Анна","age":25}`)
var user User
err := json.Unmarshal(jsonData, &user)
```

---

### Важные правила для JSON

1. **Поля должны быть экспортируемыми** (с большой буквы)
2. **Теги структур** — управление именами в JSON
3. **omitempty** — не включать пустые поля
4. **`-`** — игнорировать поле
5. **`string`** — число как строка (например, для больших чисел в JS)

```go
type Product struct {
    ID    int64   `json:"id,string"` // "id": "123456"
    Price float64 `json:"price"`      // "price": 99.99
}
```

---

### Что можно сериализовать

✅ Встроенные типы (int, string, slice, map)  
✅ Структуры с экспортируемыми полями  
✅ Массивы и слайсы  
✅ Мапы (с ключами, поддерживающими сравнение)  

❌ Каналы  
❌ Функции  
❌ Циклические ссылки (зациклится)  
❌ Некоторые сложные структуры (например, `time.Time` — ок, но осторожно)

---

### Gob — родной формат Go

```go
var buf bytes.Buffer
enc := gob.NewEncoder(&buf)
err := enc.Encode(user)

dec := gob.NewDecoder(&buf)
var decoded User
err = dec.Decode(&decoded)
```

**Особенности:**
- Только для Go (не для межъязыкового обмена)
- Эффективнее JSON по размеру и скорости
- Поддерживает интерфейсы (нужно регистрировать типы)

---

### Protocol Buffers (protobuf)

**Требует определения схемы в .proto файле:**

```proto
syntax = "proto3";
message User {
  string name = 1;
  int32 age = 2;
}
```

**В Go:**
```go
user := &pb.User{Name: "Анна", Age: 25}
data, _ := proto.Marshal(user)

var newUser pb.User
proto.Unmarshal(data, &newUser)
```

**Плюсы:**
- Компактный бинарный формат
- Очень быстрый
- Строгая схема, обратная совместимость
- Поддержка многих языков

---

### Custom Marshal/Unmarshal (своя логика)

```go
type Duration time.Duration

func (d Duration) MarshalJSON() ([]byte, error) {
    return json.Marshal(time.Duration(d).String())
}

func (d *Duration) UnmarshalJSON(b []byte) error {
    var s string
    if err := json.Unmarshal(b, &s); err != nil {
        return err
    }
    dur, err := time.ParseDuration(s)
    *d = Duration(dur)
    return err
}
```

---

### Проблемы и частые ошибки

#### 1. Неэкспортируемые поля

```go
type User struct {
    name string  // маленькая буква — не сериализуется!
}
```

#### 2. Циклические ссылки

```go
type Node struct {
    Next *Node
}
node.Next = &node  // цикл — json зациклится!
```

#### 3. Большие числа (int64) в JSON для JS

```go
ID int64 `json:"id,string"`  // решение
```

#### 4. Потеря типа у чисел в JSON

JSON не различает int и float — в interface{} всё float64.

---

### Производительность

| Формат | Скорость | Размер | Человек-читаемость |
|--------|----------|--------|-------------------|
| **JSON** | Средне | Средний | ✅ Отлично |
| **Gob** | Быстро | Маленький | ❌ Нет |
| **Protobuf** | Очень быстро | Очень маленький | ❌ Нет |
| **MessagePack** | Быстро | Маленький | ⚠️ Частично |
| **XML** | Медленно | Большой | ✅ Отлично |

**Бенчмарки:** Protobuf и MessagePack обычно быстрее JSON в 2-5 раз и занимают меньше места.

---

### Резюме

> **Сериализация** — преобразование данных в байты для хранения/передачи. В Go чаще всего используется **JSON** (`encoding/json`) с тегами структур для управления именами и `omitempty`. Для высоконагруженных систем или межъязыкового обмена выбирают **Protocol Buffers**. Внутри Go-экосистемы можно использовать **Gob**. Важно помнить про экспортируемые поля, теги и ограничения (нельзя сериализовать каналы, функции, циклы).

## Дженерики (Generics) в Go

---

### Что такое дженерики?

**Дженерики (обобщённое программирование)** — это возможность писать функции и типы, которые работают **с любым типом данных**, сохраняя при этом типобезопасность .

Появились в **Go 1.18 (февраль 2022)** — это было самое большое изменение в языке за всю его историю.

---

### Зачем нужны дженерики?

**До дженериков** приходилось либо:
1. Копипастить код для разных типов (писать `IntSlice`, `StringSlice`)
2. Использовать `interface{}` и терять типобезопасность (проверки в рантайме)

**Дженерики решают эти проблемы:** код пишется один раз, но работает с разными типами, а компилятор проверяет корректность на этапе компиляции.

---

### Синтаксис

```go
// Функция с параметром типа
func PrintSlice[T any](s []T) {
    for _, v := range s {
        fmt.Println(v)
    }
}

// Использование
PrintSlice[int]([]int{1, 2, 3})
PrintSlice[string]([]string{"a", "b"})
// Часто компилятор сам выводит тип:
PrintSlice([]int{1, 2, 3})
```

**Квадратные скобки** `[]` — место для параметров типа.

---

### Ограничения (Constraints)

Ограничения говорят, какие типы можно подставлять. Это **интерфейсы**!

```go
// any — любой тип (псевдоним для interface{})
func Print[T any](v T)

// comparable — типы, которые можно сравнивать (==, !=)
func Find[T comparable](slice []T, value T) int

// Своё ограничение (union типов)
type Number interface {
    int | int64 | float64 | float32
}

func Sum[T Number](nums []T) T {
    var sum T
    for _, v := range nums {
        sum += v
    }
    return sum
}
```

**Тильда `~`** — разрешает и производные типы:
```go
type MyInt int

type Number interface {
    ~int | ~float64  // работает и с int, и с MyInt
}
```

---

### Дженерики в структурах

```go
type Stack[T any] struct {
    items []T
}

func (s *Stack[T]) Push(item T) {
    s.items = append(s.items, item)
}

func (s *Stack[T]) Pop() (T, bool) {
    if len(s.items) == 0 {
        var zero T  // нулевое значение типа
        return zero, false
    }
    item := s.items[len(s.items)-1]
    s.items = s.items[:len(s.items)-1]
    return item, true
}
```

---

### Дженерики и интерфейсы

В Go 1.20+ можно комбинировать дженерики с интерфейсами:

```go
type Stringer interface {
    String() string
}

func PrintAll[T Stringer](items []T) {
    for _, item := range items {
        fmt.Println(item.String())
    }
}
```

---

### Дженерики в стандартной библиотеке

| Пакет | Что даёт |
|-------|----------|
| `slices` | Обобщённые функции для работы со слайсами |
| `maps` | Для работы с мапами |
| `sync` | `Pool[T any]`, `Map[K comparable, V any]` |
| `golang.org/x/exp/constraints` | Полезные ограничения (`Signed`, `Unsigned`, `Ordered`) |

---

### Важные ограничения (что НЕЛЬЗЯ)

| Ограничение | Почему |
|-------------|--------|
| **Нельзя использовать операции типа `+`, `-`** | Для них нужны ограничения (только через интерфейсы) |
| **Методы не могут иметь дополнительные параметры типа** | `func (s Stack[T]) Merge[U any](other Stack[U])` — нельзя! |
| **Типы-дженерики нельзя использовать в switch type** | `switch v.(type)` не работает с параметризованными типами |
| **Дженерики и производительность** | Компилятор генерирует код для каждого типа (как шаблоны C++), может увеличить размер бинарника |

---

### Типовые примеры использования

#### 1. Функция Map (как в функциональных языках)

```go
func Map[T, U any](slice []T, f func(T) U) []U {
    result := make([]U, len(slice))
    for i, v := range slice {
        result[i] = f(v)
    }
    return result
}
```

#### 2. Функция Filter

```go
func Filter[T any](slice []T, pred func(T) bool) []T {
    result := []T{}
    for _, v := range slice {
        if pred(v) {
            result = append(result, v)
        }
    }
    return result
}
```

#### 3. Min для упорядоченных типов

```go
type Ordered interface {
    ~int | ~int8 | ~int16 | ~int32 | ~int64 |
    ~uint | ~uint8 | ~uint16 | ~uint32 | ~uint64 |
    ~float32 | ~float64 | ~string
}

func Min[T Ordered](a, b T) T {
    if a < b {
        return a
    }
    return b
}
```

---

### Производительность

- **Без оверхеда в рантайме** — компилятор генерирует отдельные реализации для каждого типа (как если бы вы написали вручную) 
- **Размер бинарника** — может расти, если много разных типов
- **Скорость компиляции** — немного медленнее из-за мономорфизации

---

### Ошибки новичков

1. **Переусложнение** — не везде нужны дженерики, иногда интерфейсы проще
2. **Забывают про `any`** — многие пишут `[T interface{}]` вместо `[T any]`
3. **Пытаются использовать операторы** — `+`, `-` не работают с `T any`
4. **Забывают про `~`** — своё ограничение не работает с кастомными типами
5. **Дженерики в рантайме** — информация о типах стирается (как в Java, но на уровне компиляции)

---

### Когда использовать, а когда нет

| Стоит использовать | Лучше не использовать |
|-------------------|----------------------|
| Структуры данных (стек, очередь, список) | Простые функции с одним типом |
| Алгоритмы (сортировка, поиск) | Когда нужны разные методы для разных типов |
| Вспомогательные функции (Map, Filter) | Если тип известен и не изменится |
| Библиотеки общего назначения | В коде, который и так понятен без дженериков |

---

### Резюме

> **Дженерики в Go (с 1.18)** позволяют писать типобезопасный код, работающий с разными типами. Синтаксис — квадратные скобки `[T any]`. Ограничения задаются интерфейсами (можно объединять типы через `|` и `~`). Дженерики компилируются в отдельные реализации для каждого типа (мономорфизация), поэтому оверхеда в рантайме нет, но бинарник может стать больше. Используются для структур данных, алгоритмов и библиотек. Главная ошибка — пытаться применять их там, где хватило бы интерфейса.

# ЗАМЫКАНИЕ (Closure)

**Замыкание** — это функция, которая "запоминает" переменные из области видимости, где она была создана, даже после того, как эта область перестала существовать.

```go
func main() {
    message := "Привет"
    
    // Анонимная функция - замыкание
    sayHello := func() {
        fmt.Println(message) // "захватила" переменную message
    }
    
    sayHello() // Выведет: Привет
}
```

---

## ГДЕ ИСПОЛЬЗУЮТ ЗАМЫКАНИЯ

- **Фабрики функций**
- **Инкапсуляция состояния** (приватные переменные)
- **Callback функции**
- **Middleware** в веб-фреймворках

---

## КОГДА НЕ ИСПОЛЬЗОВАТЬ ЗАМЫКАНИЯ

- В **горячих циклах** — могут быть накладные расходы
- Если переменные **большие** — они не освобождаются, пока живо замыкание
- Для **простых операций** — иногда проще передать параметры явно

---

## ЗОЛОТЫЕ ПРАВИЛА

### 1. Замыкание захватывает **ССЫЛКУ**, а не значение

```go
i := 0
defer func() {
    fmt.Println(i) // выведет 5, а не 0
}()
i = 5
```

### 2. В цикле `for` — создавай копию или передавай параметр

```go
go func(val int) { ... }(i) // ✅ правильно
```

### 3. Замыкания полезны для фабрик и инкапсуляции

```go
multiplyBy(2) // фабрика, возвращающая функцию
```

### 4. Не злоупотребляй, если можно проще

```go
func add(a, b int) int // лучше, чем замыкание
```

## Defer

---

### Что такое defer?
**defer** — это ключевое слово в Go, которое откладывает выполнение функции до выхода из окружающей функции (обычно используется для cleanup-операций: закрыть файл, освободить ресурс, разблокировать мьютекс).

```go
file, _ := os.Open("file.txt")
defer file.Close() // закроется при выходе из функции
```

---

### Три золотых правила defer

#### 1. Аргументы вычисляются **сразу** (при вызове defer), а не когда функция выполняется

```go
x := 10
defer fmt.Println(x) // выведет 10
x = 20
// результат: 10, а не 20
```

#### 2. Выполняются в порядке **LIFO** (последний отложенный — выполнится первым)

```go
defer fmt.Println("первый")
defer fmt.Println("второй")
defer fmt.Println("третий")
// Вывод:
// третий
// второй
// первый
```

#### 3. Defer-функции могут обращаться к именованным возвращаемым значениям и изменять их

```go
func example() (result int) {
    defer func() {
        result = 42 // меняем возвращаемое значение
    }()
    return 0 // вернется 42!
}
```

---

### Типичные ошибки на собеседованиях

**Ошибка 1:** Думать, что defer выполняется сразу
```go
x := 1
defer fmt.Println(x)
x = 2 // всё равно выведет 1
```

**Ошибка 2:** Забывать про замыкание
```go
x := 1
defer func() {
    fmt.Println(x) // выведет 2, потому что это замыкание!
}()
x = 2
```

**Ошибка 3:** Переменная цикла в defer
```go
for i := 0; i < 3; i++ {
    defer fmt.Println(i) // выведет 2 1 0 (но значения захвачены правильно, т.к. аргументы копируются)
}
// Но если через замыкание:
for i := 0; i < 3; i++ {
    defer func() { fmt.Println(i) }() // опасно! выведет 3 3 3 (в Go 1.21-)
}
```

---

### Где используют defer

| Ситуация | Пример |
|----------|--------|
| Закрытие файла | `defer file.Close()` |
| Разблокировка мьютекса | `defer mu.Unlock()` |
| Закрытие соединения | `defer conn.Close()` |
| Обработка паники | `defer func() { recover() }()` |
| Логирование времени работы | `defer logTime(time.Now())` |

---

### Defer и производительность

- В Go 1.14+ defer стал очень быстрым (раньше был overhead)
- В горячих циклах лучше не использовать, но для обычного кода — норм

---

### Резюме

> Defer — это отложенный вызов функции. Аргументы вычисляются сразу, выполняются в порядке LIFO, могут изменять именованные результаты. Используется для cleanup-операций. Главная ловушка — переменные в замыканиях внутри defer.

## Приведение типов

---

### Что такое приведение типов?

В Go это преобразование значения из одного типа в другой. Но в Go есть **три разных механизма**, и их важно различать.

---

### 1. Преобразование типов (Type Conversion)

**Для совместимых типов** (например, числовые типы, строки и слайсы байт/рун).

```go
var i int = 42
var f float64 = float64(i)  // int → float64
var u uint = uint(f)         // float64 → uint

var b []byte = []byte("hello")  // string → []byte
var s string = string(b)        // []byte → string
var r []rune = []rune("hello")  // string → []rune
```

**Важно:**
- Требует **явного** преобразования (автоматического приведения нет!)
- Типы должны быть совместимы по структуре
- Может обрезать данные (например, int64 → int8)

---

### 2. Утверждение типа (Type Assertion)

**Для интерфейсов.** Проверяет/извлекает конкретный тип из интерфейсного значения.

```go
var any interface{} = "hello"

// Один возвращаемый параметр (будет паника, если тип не тот)
str := any.(string)

// Два параметра — безопасно
str, ok := any.(string)  // ok = true, str = "hello"
num, ok := any.(int)     // ok = false, num = 0 (zero value)
```

**Синтаксис:** `x.(T)` где `x` — интерфейс, `T` — конкретный тип или другой интерфейс.

**Паника:** если тип не совпадает и используется форма с одним возвращаемым значением.

---

### 3. Type Switch (переключатель по типу)

**Для интерфейсов.** Позволяет обработать несколько возможных типов.

```go
func process(v interface{}) {
    switch val := v.(type) {
    case string:
        fmt.Println("Строка:", val)
    case int:
        fmt.Println("Число:", val * 2)
    case []int:
        fmt.Println("Сумма слайса:", sum(val))
    default:
        fmt.Printf("Неизвестный тип: %T\n", val)
    }
}
```

---

### Сравнение

| Механизм | Когда использовать | Пример |
|----------|-------------------|--------|
| **Преобразование (Conversion)** | Числа, строки ↔ байты, руны | `float64(42)` |
| **Утверждение (Assertion)** | Интерфейс → конкретный тип | `x.(string)` |
| **Type Switch** | Несколько вариантов из интерфейса | `switch v := x.(type)` |

---

### Частые ошибки на собесах

**Ошибка 1:** Путают преобразование и утверждение
```go
var x interface{} = 42
y := int(x) // ОШИБКА! int(x) работает только с конкретными типами, а x — интерфейс
y := x.(int) // ✅ правильно
```

**Ошибка 2:** Забывают про `ok` при утверждении
```go
str := x.(string) // если x не string — паника!
str, ok := x.(string) // безопасно
```

**Ошибка 3:** Думают, что в Go есть автоматическое приведение
```go
var i int = 42
var f float64 = i // ОШИБКА! Нужно явно: float64(i)
```

---

### Особенности преобразования строк

```go
// string ↔ []byte (копирует данные!)
b := []byte("hello")
s := string(b)

// string ↔ []rune (для Unicode)
r := []rune("привет")
s2 := string(r)

// Число ↔ строка (не путать с rune!)
s3 := string(65)     // "A" (не "65"!)
s4 := strconv.Itoa(65) // "65" ✅ правильно
```

---

### Производительность

- **Преобразование чисел** — практически бесплатно
- **string ↔ []byte** — копирование данных (O(n))
- **Утверждение типа** — очень быстро (пара наносекунд)
- **Type switch** — тоже быстро, оптимизировано компилятором

---

### Резюме

> В Go есть три механизма работы с типами:  
> **Преобразование** (conversion) — для совместимых конкретных типов,  
> **Утверждение** (assertion) — для извлечения типа из интерфейса,  
> **Type switch** — для ветвления по типу интерфейса.  
> Важно помнить: автоматического приведения типов нет, всё делается явно.

## Написание тестов в Go

---

### Базовые правила

- Тесты живут в файлах с суффиксом `_test.go`
- Тестовая функция называется `TestXxx(t *testing.T)`
- Запускаются командой `go test`
- Для деталей: `go test -v`

```go
func TestAdd(t *testing.T) {
    result := Add(2, 3)
    expected := 5
    if result != expected {
        t.Errorf("Add(2,3) = %d; want %d", result, expected)
    }
}
```

---

### Виды тестов

| Тип | Сигнатура | Назначение |
|-----|-----------|------------|
| **Test** | `TestXxx(t *testing.T)` | Обычные тесты |
| **Benchmark** | `BenchmarkXxx(b *testing.B)` | Тесты производительности |
| **Example** | `ExampleXxx()` | Примеры в документации |
| **Fuzz** | `FuzzXxx(f *testing.F)` | Фаззинг-тесты (случайные данные) |

---

### Основные методы *testing.T

```go
t.Error()      // ошибка + продолжение теста
t.Fatal()      // ошибка + остановка теста
t.Errorf()     // форматированная ошибка
t.Fatalf()     // форматированная + остановка
t.Skip()       // пропустить тест
t.Parallel()   // запускать параллельно
t.Log()        // логирование (видно с -v)
```

---

### Табличные тесты (Table Driven Tests)

**Самый популярный паттерн в Go:**

```go
func TestDivide(t *testing.T) {
    tests := []struct {
        name     string
        a, b     int
        expected int
        err      bool
    }{
        {"positive", 10, 2, 5, false},
        {"by zero", 10, 0, 0, true},
        {"negative", -6, 3, -2, false},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            result, err := Divide(tt.a, tt.b)
            if (err != nil) != tt.err {
                t.Errorf("error = %v, wantErr %v", err, tt.err)
            }
            if result != tt.expected {
                t.Errorf("got %d, want %d", result, tt.expected)
            }
        })
    }
}
```

**Преимущества:**
- Легко добавлять новые кейсы
- Каждый кейс можно запустить отдельно: `go test -run TestDivide/positive`
- Читаемо и поддерживаемо

### Моки (тестирование с зависимостями)

#### 1. Через интерфейсы (идиоматичный Go)

```go
// Объявляем интерфейс
type UserRepo interface {
    GetUser(id int) (User, error)
}

// В тесте — мок
type mockRepo struct {
    getUserFunc func(id int) (User, error)
}

func (m *mockRepo) GetUser(id int) (User, error) {
    return m.getUserFunc(id)
}

func TestService(t *testing.T) {
    mock := &mockRepo{
        getUserFunc: func(id int) (User, error) {
            return User{Name: "Test"}, nil
        },
    }
    service := NewService(mock)
    // тестируем
}
```

- **testify/mock** — встроенное мокирование

### Фаззинг (Fuzz Testing) с Go 1.18+

```go
func FuzzDivide(f *testing.F) {
    f.Add(10, 2)  // seed-кортежи
    f.Add(5, 0)
    
    f.Fuzz(func(t *testing.T, a, b int) {
        result, err := Divide(a, b)
        if b == 0 && err == nil {
            t.Errorf("expected error for b=0")
        }
        if b != 0 && result != a/b {
            t.Errorf("wrong result")
        }
    })
}
```

Запуск: `go test -fuzz=FuzzDivide`

### Частые ошибки

1. **Не проверять ошибки** в тестах
2. **Глобальное состояние** — тесты влияют друг на друга
3. **Нет изоляции** — тесты работают с реальной БД/API
4. **Слишком сложные тесты** — трудно понять, что пошло не так
5. **Забыть `t.Parallel()`** — тесты выполняются последовательно

✅ Тесты должны быть **изолированы** и детерминированны  
✅ Одна тестовая функция — одна логическая проверка  
✅ Используйте **табличные тесты**  
✅ Мокируйте внешние зависимости (интерфейсы спасают)  
✅ Тестируйте **публичное API**, не лезьте в приватное  
✅ Запускайте `-race` для поиска гонок  
✅ Поддерживайте покрытие, но не гонитесь за 100%

## Линтеры в Go

---

### Что такое линтеры?

**Линтеры (linters)** — это инструменты **статического анализа кода**, которые проверяют код без его выполнения и находят потенциальные проблемы: ошибки, стилистические нарушения, неэффективные конструкции и уязвимости . Они помогают поддерживать качество кода и выявлять проблемы на ранних этапах разработки .

---

### Ключевые линтеры в экосистеме Go

| Инструмент | Назначение | Когда использовать |
|------------|------------|-------------------|
| **golangci-lint** | Металинтер, объединяет десятки линтеров | **Стандарт индустрии** для большинства проектов  |
| **go vet** | Встроенный в Go анализатор | Базовые проверки, идёт "из коробки"  |
| **staticcheck** | Продвинутый статический анализ (200+ проверок) | Глубокий анализ, поиск багов  |
| **errcheck** | Проверяет, что ошибки не игнорируются | Критически важен для надёжности  |
| **gosec** | Анализ безопасности | Поиск уязвимостей  |
| **revive** | Современная замена устаревшему `golint` | Гибкая настройка стиля  |
| **gofmt/goimports** | Форматирование и управление импортами | **Обязательно** для всех Go-проектов  |

---

### golangci-lint — де-факто стандарт

**Почему все его используют?**

1. **Скорость** — запускает десятки линтеров **параллельно**, кеширует результаты, превращает 3 минуты в 10 секунд 
2. **Умная архитектура** — делает общую работу (парсинг AST) **один раз** для всех линтеров, а не дублирует 
3. **Единая конфигурация** — всё в одном `.golangci.yml` файле 
4. **Поддержка "только изменённых строк"** — можно запускать только на новом коде, игнорируя старые ошибки в большой кодовой базе 

**Как родился:** Денис Исаев создал его в 2018 году, разобравшись с проблемами gometalinter. Рискнул, сделал форки существующих линтеров с новыми интерфейсами — и выиграл .

---

### Пример конфигурации .golangci.yml

```yaml
linters:
  enable:
    - staticcheck
    - gosimple
    - govet
    - errcheck
    - gosec
    - revive
    - gocyclo
    - misspell

linters-settings:
  errcheck:
    check-type-assertions: true
    check-blank: true
  govet:
    enable-all: true
  gocyclo:
    min-complexity: 15

run:
  timeout: 5m
  tests: true
```

### Как работают линтеры под капотом

1. **Парсинг** — код превращается в AST (абстрактное синтаксическое дерево) через пакет `go/ast` 
2. **Типизация** — через `go/types` определяется семантика (типы переменных, что за идентификаторы) 
3. **Анализ** — проход по дереву, поиск проблемных паттернов
4. **Фреймворк `go/analysis`** — стандартный способ создания линтеров, абстрагирует рутину 

**Как написать свой линтер:** используешь `go/analysis.Analyzer`, реализуешь функцию `Run(pass *analysis.Pass)` — и можно встраивать в `golangci-lint` .

# Type Switch в Go

**Type switch** — это конструкция для проверки типа интерфейсной переменной. Позволяет выполнять разный код в зависимости от конкретного типа значения.

```go
switch v := someInterface.(type) {
case string:
    // v — string
    fmt.Println("Строка:", v)
case int:
    // v — int
    fmt.Println("Число:", v)
case MyStruct:
    // v — MyStruct
    fmt.Println("Моя структура:", v)
case nil:
    fmt.Println("nil значение")
default:
    fmt.Printf("Неизвестный тип: %T\n", v)
}
```

## Ключевые моменты

- **Синтаксис**: `.(type)` можно использовать **только в switch**
- **Переменная**: `v` получает значение с правильным типом в каждой ветке
- **Порядок**: ветки проверяются сверху вниз
- **default**: срабатывает, если ни один тип не подошел

## Где используется

- Обработка разных типов в функции, принимающей `interface{}`
- Работа с данными из JSON (где типы могут быть разными)
- Реализация полиморфизма без заранее известных типов

## Важные ограничения

- **Type switch работает только с интерфейсами**
- Если переменная уже конкретного типа — компилятор ругнется
- Можно комбинировать несколько типов в одной ветке: `case int, float64:`

## Стандартный профилировщик Go - pprof

**pprof** — это встроенный в Go инструмент для профилирования и оптимизации программ. Он помогает ответить на главные вопросы: *"Почему моя программа тормозит?"*, *"Куда уходит память?"* и *"Где узкие места?"* .

Это семплирующий профайлер: он периодически (обычно каждые 10 мс) заглядывает в программу, снимает стек-вызовы и записывает, чем в этот момент занята программа. Из этих "семплов" потом собирается статистика .

## Основные типы профилей

pprof умеет собирать разные данные :

| Тип профиля | Что показывает | Как получить |
| :--- | :--- | :--- |
| **CPU** | Где программа проводит больше всего процессорного времени | `/debug/pprof/profile?seconds=30` |
| **Heap (память)** | Какие объекты сейчас живут в куче и сколько места занимают | `/debug/pprof/heap` |
| **Allocs (аллокации)** | Что и как часто выделяет память (даже если потом сразу освобождается) | `/debug/pprof/allocs` |
| **Goroutine** | Сколько горутин запущено, и где они находятся | `/debug/pprof/goroutine` |
| **Mutex** | Кто и как долго ждет блокировок (мьютексов) | `/debug/pprof/mutex` |
| **Block** | Где горутины блокируются (на каналах, сетевых вводах-выводах) | `/debug/pprof/block` |
| **Trace** | Не профиль, а трассировка исполнения (для `go tool trace`) | `/debug/pprof/trace?seconds=5` |

## Как включить pprof?

Есть три основных способа :

### 1️⃣ Через HTTP (самый удобный для веб-сервисов)

Просто добавьте импорт:
```go
import _ "net/http/pprof"
```
Если у вас уже запущен HTTP-сервер, профили станут доступны по адресу `http://ваш-сервер:порт/debug/pprof/` .

**Важно для продакшена**: лучше запускать профайлер на отдельном порту, доступном только локально, чтобы никто со стороны не мог смотреть внутренности вашего сервиса .

```go
go func() {
    // Админский сервер только на localhost
    log.Println(http.ListenAndServe("localhost:6060", nil))
}()
```

### 2️⃣ Через бенчмарки

При запуске тестов производительности можно сохранить профиль в файл :
```bash
go test -bench=. -cpuprofile cpu.out -memprofile mem.out
```

### 3️⃣ Напрямую в коде

Можно вручную включать и выключать запись профиля функциями из пакета `runtime/pprof` .

## Как смотреть результаты?

### Интерактивный режим в консоли
```bash
# Для запущенного сервиса
go tool pprof http://localhost:6060/debug/pprof/heap

# Для файла с профилем
go tool pprof cpu.out
```
Внутри можно вводить команды :
- `top` — показать самые "горячие" функции (где `flat` — время в самой функции, `cum` — вместе с вызванными)
- `list ИмяФункции` — показать исходный код с разбивкой по строкам
- `peek Имя` — кто вызывает функцию и кого вызывает она

### Визуализация (огонь 🔥)
```bash
# Откроет браузер с красивыми графами
go tool pprof -http=:8080 cpu.out
```
Можно смотреть:
- **Граф** (call graph) — стрелками показано, кто кого вызывает
- **Flame Graph** (очень наглядно) — где шире полоска, там больше времени

## Heap vs Allocs: в чем разница?

Очень важный момент :
- **Heap (inuse)** — показывает, что *сейчас* лежит в памяти и не отпускается (полезно для поиска утечек)
- **Allocs (alloc_space)** — показывает, что *вообще* создавалось за время профилирования (даже если сразу удалилось). Помогает найти места, где программа "мусорит" и нагружает сборщик мусора.

## Что искать в профиле?

- Функции с большим `flat` — они сами много работают (сложные вычисления, циклы)
- Функции с большим `cum` и маленьким `flat` — они просто вызывают другие функции, которые и делают всю работу 
- В профиле памяти — функции, которые создают много объектов (часто это работа с JSON, создание временных структур в циклах) 

## Коротко и ясно

| Что хотите узнать | Какой профиль брать |
| :--- | :--- |
| "Почему процессор грузится?" | CPU профиль |
| "Почему память растет и не падает?" | Heap (inuse) |
| "Почему GC (сборщик мусора) часто работает?" | Allocs (alloc_space) |
| "Почему все тормозит и много горутин?" | Goroutine + Mutex/Block |
| "Что вообще происходит?" | Trace |

### Что такое «семплирующий профайлер»?
Это как **фотосъёмка программы**: ты не снимаешь видео каждую наносекунду, а просто щёлкаешь кадры через равные промежутки времени (например, каждые 10 миллисекунд). Потом смотришь: на какой строке кода чаще всего был сделан снимок — значит, там программа и проводит больше всего времени.

**pprof в Go делает именно так:** он периодически останавливает программу, смотрит, какая функция сейчас выполняется, и записывает это в статистику. Чем больше семплов попало на функцию — тем она "горячее".

---

### Overhead (накладные расходы) от стандартного профайлера
Очень маленький. **Обычно 1–5%** для CPU-профилирования.

**Почему так мало:**
- Программа **не останавливается** на время профилирования (кроме короткого момента снятия семпла).
- Семплинг происходит не постоянно, а с частотой **100 Гц** (по умолчанию, каждые 10 миллисекунд). Это настраивается, но увеличивать частоту не рекомендуется.
- Сам сбор данных очень легковесный — просто читаем program counter и сохраняем стек вызовов.

**Для памяти (heap) overhead ещё меньше** — просто периодически снимается "слепок" кучи, программа при этом вообще не тормозит.

### Можно ли включать в проде?
**Да, можно и нужно!** Именно для этого он и сделан. Но с умом:
- Не держите включённым 24/7, только когда ищете проблему.
- Лучше настраивать через отдельный `localhost` порт, чтобы никто со стороны не дёргал.
- На production-серверах можно включать на короткое время (30–60 секунд) и смотреть, что происходит под реальной нагрузкой. Это безопасно.
