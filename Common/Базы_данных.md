### Базы данных (БД)

это некий объём перенесённой в электронный вид информации, которую хранят в файле или наборе файлов.

----
## Типы
#### Реляционные
Хранят данные в таблицах (строки и столбцы) со строгой структурой.
Идеальны для транзакционных систем (банки, CRM), где важна целостность (ACID).
#### Не реляционные (NoSQL)
Обладают гибкой схемой данных, подходят для больших объемов, высокой скорости и неструктурированной информации.
- Документоориентированные:
Данные хранятся как JSON-документы (MongoDB, CouchDB).
- Ключ-значение (Key-Value):
Быстрые хранилища, где каждая запись — пара ключ-значение (Redis).
- Графовые:
Базируются на узлах и связях, эффективны для анализа социальных сетей или рекомендаций (Neo4j).
Маршрутизация, рекомендации
- Колоночные (Столбцовые):
Оптимизированы для аналитики, хранят данные по столбцам, а не строкам (ClickHouse, Cassandra)
- Time_series
Хранение по времени (Prometheus)

## Сравнение реляционных и нереляционных БД
#### Основные отличия

| Характеристика | Реляционные (SQL) | Нереляционные (NoSQL) |
|----------------|-------------------|----------------------|
| **Модель данных** | Таблицы, строки, столбцы | Документы, ключ-значение, графы, колонки |
| **Схема** | Фиксированная (строгая) | Гибкая (динамическая) |
| **Язык запросов** | SQL (стандартизированный) | Специфичный для каждой БД |
| **ACID** | Полная поддержка | Обычно BASE (кроме некоторых) |
| **Масштабирование** | Вертикальное (горизонтальное сложно) | Горизонтальное (sharding) |
| **Связи** | Foreign keys, JOIN-ы | Вложенные документы, денормализация |
| **Транзакции** | Поддерживаются | Ограниченно или отсутствуют |
| **Примеры** | PostgreSQL, MySQL, Oracle | MongoDB, Redis, Cassandra, Neo4j |

#### Таблица 2: Когда что использовать

| Сценарий | Реляционные (SQL) | Нереляционные (NoSQL) |
|----------|-------------------|----------------------|
| **Данные с четкой структурой** | ✅ Идеально | ⚠️ Возможно, но необязательно |
| **Частые изменения схемы** | ❌ Тяжело (миграции) | ✅ Легко |
| **Сложные запросы и JOIN-ы** | ✅ Отлично | ❌ Плохо (или обходные пути) |
| **Большие объемы, высокая нагрузка** | ⚠️ Сложно горизонтально | ✅ Хорошо масштабируется |
| **Транзакции, финансы** | ✅ Обязательно | ❌ Рискованно |
| **Иерархические данные** | ⚠️ Через JOIN-ы | ✅ Документы/графы |
| **Кэширование, сессии** | ❌ Избыточно | ✅ Идеально (Redis) |
| **Аналитика, Big Data** | ⚠️ Тяжело | ✅ Колоночные БД |
| **Связи многие-ко-многим** | ✅ Естественно | ⚠️ Сложнее |
| **Прототип, стартап** | ⚠️ Может сковывать | ✅ Гибкость на старте |

#### Вывод:

> **SQL** — когда важны целостность, связи и транзакции  
> **NoSQL** — когда нужны масштабирование, гибкость и скорость

## SQL (Structured Query Language — «язык структурированных запросов»)
это специализированный язык программирования, используемый для создания, изменения, удаления и управления данными в реляционных базах данных. Он позволяет быстро извлекать информацию, фильтровать её, сортировать и объединять данные из различных таблиц.

---
## **Нормализация** — это процесс организации данных для устранения избыточности и аномалий.
Три основные нормальные формы (1НФ, 2НФ, 3НФ)

#### 1НФ (Первая нормальная форма)
**Правило:** Все значения **атомарны** (неделимы). Нет повторяющихся групп.

*   В каждой ячейке — **одно значение**
*   Нет списков, массивов, вложенных таблиц


#### 2НФ (Вторая нормальная форма)
**Правило:** Находится в 1НФ + **нет частичной зависимости** от составного ключа.

*   Все неключевые поля зависят **от всего ключа целиком**, а не от его части
*   Актуально только для таблиц с **составным первичным ключом**


#### 3НФ (Третья нормальная форма)
**Правило:** Находится в 2НФ + **нет транзитивной зависимости**.

*   Неключевые поля не должны зависеть **от других неключевых полей**
*   Все неключевые поля зависят только от первичного ключа


### Простая аналогия

| НФ | Что проверяем | Нарушение |
|-----|---------------|-----------|
| **1НФ** | Ячейки | В столбце "Телефоны" записано "+7-123, +7-456" |
| **2НФ** | Часть ключа | В таблице заказов (id_товара + id_заказа) хранится название товара — зависит только от id_товара |
| **3НФ** | Цепочки | В таблице сотрудников есть город и страна — страна зависит от города, а не от id сотрудника |

| Тип | Что делает | Примеры                            |
|-----|------------|------------------------------------|
| **DDL** (Data Definition Language) | Определяет структуру БД | CREATE, ALTER, DROP                |

---

### **DML** (Data Manipulation Language)

* SELECT — выборка данных
```sql
SELECT name, age FROM users WHERE age > 18;
```

* INSERT — добавление данных
```sql
INSERT INTO users (name, age) VALUES ('Анна', 25);
```

* UPDATE — обновление данных
```sql
UPDATE users SET age = 26 WHERE name = 'Анна';
```

* DELETE — удаление данных
```sql
DELETE FROM users WHERE age < 18;
```

### **DDL** (Data Definition Language)

* CREATE — создание таблицы
```sql
CREATE TABLE users (id INT, name VARCHAR(100));
```

* ALTER — изменение структуры
```sql
ALTER TABLE users ADD COLUMN email VARCHAR(255);
```

* DROP — удаление таблицы/БД
```sql
DROP TABLE users;
```

### **DCL** (Data Control Language)

* GRANT — дать права
```sql
GRANT SELECT ON users TO 'user'@'localhost';
```

* REVOKE — забрать права
```sql
REVOKE INSERT ON users FROM 'user'@'localhost';
```

### **TCL** (Transaction Control Language)
* COMMIT — зафиксировать транзакцию
* ROLLBACK — откатить транзакцию
```sql
  BEGIN; INSERT....; COMMIT;
  BEGIN; INSERT....; ROLLBACK; COMMIT;
```

## UNION
**UNION** — это оператор для **объединения результатов двух (или более) SELECT-запросов** в один результирующий набор.

### Как работает

1. Выполняются оба SELECT-запроса
2. Результаты **склеиваются вертикально** (строки из второго запроса добавляются под строки первого)
3. **Удаляются дубликаты** (если не указано UNION ALL)
4. Возвращается один общий результат

### Важные правила

- Количество и порядок колонок должны быть **одинаковыми** во всех запросах
- Типы данных должны **совместимы** (можно преобразовать к общему типу)
- Имена колонок берутся из **первого** запроса

### Пример

```sql
SELECT name, age FROM teachers
UNION
SELECT name, age FROM students;
```
Вернет всех учителей и учеников в одном списке.

### UNION vs UNION ALL

| Оператор | Дубликаты | Скорость |
|----------|-----------|----------|
| **UNION** | Удаляет | Медленнее (нужна сортировка) |
| **UNION ALL** | Оставляет | Быстрее |

**Совет:** Если знаешь, что дубликатов нет или они допустимы — используй **UNION ALL**.

---

### Агрегатные функции

Вычисляют одно значение на основе набора строк.

| Функция | Что делает | Пример |
|---------|------------|--------|
| **COUNT()** | Количество строк | `COUNT(*)` |
| **SUM()** | Сумма значений | `SUM(price)` |
| **AVG()** | Среднее | `AVG(age)` |
| **MIN()** | Минимум | `MIN(salary)` |
| **MAX()** | Максимум | `MAX(salary)` |

### GROUP BY

**Группирует** строки с одинаковыми значениями в указанных колонках, чтобы применить агрегатные функции к каждой группе отдельно.

```sql
SELECT department, AVG(salary) 
FROM employees 
GROUP BY department;
```
Средняя зарплата по каждому отделу.

### HAVING

**Фильтрует группы** (как WHERE, но для групп, а не для строк).

```sql
SELECT department, AVG(salary) as avg_salary
FROM employees 
GROUP BY department
HAVING AVG(salary) > 50000;
```
Только отделы со средней зарплатой выше 50000.

### Порядок выполнения

1. **WHERE** — фильтруем строки
2. **GROUP BY** — группируем
3. **HAVING** — фильтруем группы
4. **SELECT** — выбираем поля и агрегаты

```sql
SELECT department, COUNT(*) 
FROM employees
WHERE hire_date > '2020-01-01'  -- сначала строки
GROUP BY department               -- потом группы
HAVING COUNT(*) > 5;              -- фильтр групп
```

## EXISTS и NOT EXISTS

### EXISTS

**Проверяет, есть ли хотя бы одна строка в подзапросе.** Возвращает `true`, если подзапрос вернул хоть что-то.

```sql
SELECT name FROM employees e
WHERE EXISTS (
    SELECT 1 FROM departments d 
    WHERE d.id = e.dept_id
);
```
Сотрудники, которые приписаны к существующему отделу.


### NOT EXISTS

**Проверяет, что подзапрос не вернул ни одной строки.** Возвращает `true`, если подзапрос пустой.

```sql
SELECT name FROM employees e
WHERE NOT EXISTS (
    SELECT 1 FROM departments d 
    WHERE d.id = e.dept_id
);
```
Сотрудники без отдела (или с несуществующим отделом).


### Самое важное

| Особенность | Пояснение |
|-------------|-----------|
| **Работает с коррелированными подзапросами** | Подзапрос может ссылаться на поля внешнего запроса |
| **Останавливается на первой же строке** | Не читает все данные, если нашел совпадение |
| **Возвращает только true/false** | Содержимое SELECT в подзапросе неважно (пишут обычно `SELECT 1`) |
| **EXISTS часто быстрее IN** | Особенно если много данных и есть индекс |


### EXISTS vs IN

```sql
-- EXISTS (часто быстрее)
SELECT * FROM orders o
WHERE EXISTS (SELECT 1 FROM customers c WHERE c.id = o.cust_id);

-- IN (может быть медленнее с большими списками)
SELECT * FROM orders
WHERE cust_id IN (SELECT id FROM customers);
```

## Views, Хранимые процедуры, Триггеры (Знать и не использовать (не прозрачные и легаси, не поддерживается))

### Представления (Views)

**Виртуальная таблица**, которая хранит только запрос, а не данные.

```sql
CREATE VIEW active_users AS 
SELECT * FROM users WHERE status = 'active';
```

**Важно:**
- Упрощают сложные запросы
- Ограничивают доступ к данным (показать только нужные колонки)
- Некоторые обновляемы, некоторые нет
- Не занимают место (хранят только запрос)

### Хранимые процедуры (Stored Procedures)

**Набор SQL-команд**, сохранённый на сервере БД.

```sql
CREATE PROCEDURE GetUser(IN user_id INT)
BEGIN
    SELECT * FROM users WHERE id = user_id;
END;
```

**Важно:**
- Уменьшают сетевой трафик (всё выполняется на сервере)
- Переиспользование кода
- Безопасность (можно дать права на процедуру, но не на таблицы)
- Сложнее отлаживать и версионировать

### Триггеры (Triggers)

**Автоматически выполняются** при событии (INSERT, UPDATE, DELETE).

```sql
CREATE TRIGGER update_timestamp
BEFORE UPDATE ON users
FOR EACH ROW
SET NEW.updated_at = NOW();
```

**Важно:**
- Гарантируют выполнение (нельзя пропустить)
- Для аудита, валидации, каскадных изменений
- Могут замедлять операции
- Неочевидны (сложно отлаживать)

## Constraints (Ограничения)
### Виды ограничений

| Constraint | Что делает | Пример |
|------------|------------|--------|
| **PRIMARY KEY** | Уникальный идентификатор строки (UNIQUE + NOT NULL) | `id INT PRIMARY KEY` |
| **UNIQUE** | Все значения в колонке уникальны (можно NULL) | `email VARCHAR UNIQUE` |
| **NOT NULL** | Запрещает NULL | `name VARCHAR NOT NULL` |
| **FOREIGN KEY** | Ссылка на другую таблицу | `FOREIGN KEY (dept_id) REFERENCES depts(id)` |
| **CHECK** | Проверяет условие перед вставкой | `CHECK (age >= 18)` |
| **DEFAULT** | Значение по умолчанию | `status VARCHAR DEFAULT 'active'` |

### SERIAL (автоинкремент)
```sql
id SERIAL PRIMARY KEY  -- автоматически увеличивается
```
- Не совсем constraint, а псевдо-тип
- Создаёт последовательность (sequence) и DEFAULT


### PRIMARY KEY vs UNIQUE

| | PRIMARY KEY | UNIQUE |
|---|-------------|--------|
| **NULL** | Запрещён | Разрешён (один NULL) |
| **Количество** | Один на таблицу | Много |
| **Цель** | Идентифицирует строку | Запрещает дубликаты |
| **Индекс** | Кластерный (часто) | Некластерный |


### CASCADE DELETE

Что происходит при удалении строки из родительской таблицы:

| Опция | Результат |
|-------|-----------|
| **CASCADE** | Удалить связанные строки в дочерней таблице |
| **SET NULL** | Поставить NULL в внешнем ключе |
| **RESTRICT** | Запретить удаление, если есть ссылки |
| **NO ACTION** | То же, что RESTRICT (проверка в конце транзакции) |
| **SET DEFAULT** | Поставить значение по умолчанию |

```sql
FOREIGN KEY (dept_id) REFERENCES depts(id) ON DELETE CASCADE
```

## Связи, Композиция, Агрегация, JOIN

### Связи между таблицами

| Тип связи | Описание | Реализация |
|-----------|----------|------------|
| **1:1** (один к одному) | Одной строке в А соответствует одна строка в Б | Уникальный FK в одну из таблиц |
| **1:M** (один ко многим) | Одной строке в А соответствует много строк в Б | FK в "многой" таблице |
| **M:N** (многие ко многим) | Много строк в А связано с много строк в Б | Таблица-связка (junction table) |
| **Самосвязь** | Связь внутри одной таблицы | FK на тот же PK (например, менеджер-сотрудник) |

---

### Композиция vs Агрегация (в контексте БД)

| | Композиция | Агрегация |
|---|-----------|-----------|
| **Суть** | Часть не может существовать без целого | Часть может существовать отдельно |
| **Жизненный цикл** | Зависимый | Независимый |
| **ON DELETE CASCADE** | Да (удалили заказ — удалились позиции) | Нет (удалили отдел — сотрудники остаются) |
| **Пример** | Заказ и его позиции | Отдел и сотрудники |

---

### Виды JOIN-ов

| JOIN | Что возвращает |
|------|----------------|
| **INNER JOIN** | Только строки, где есть совпадение в обеих таблицах |
| **LEFT JOIN** | Все строки из левой + совпадения из правой (NULL справа, если нет) |
| **RIGHT JOIN** | Все строки из правой + совпадения из левой |
| **FULL OUTER JOIN** | Все строки из обеих (NULL где нет совпадения) |
| **CROSS JOIN** | Декартово произведение (каждая с каждой) |
| **SELF JOIN** | Таблица соединяется сама с собой (через алиасы) |

```sql
-- Самый частый
SELECT * FROM orders o
LEFT JOIN customers c ON o.cust_id = c.id;
```

## Оконные функции

### **Оконные функции** 
выполняют вычисления над набором строк (окном), но при этом **не схлопывают строки** в одну (в отличие от GROUP BY).

Каждая строка сохраняется, а рядом добавляется результат вычисления.

```sql
SELECT  name, 
    salary,
    AVG(salary) OVER() AS avg_salary  -- среднее по всем
FROM employees;
```

### Три типа оконных функций

#### 1. Агрегатные функции над окнами

Работают те же, что и в GROUP BY: **SUM, AVG, COUNT, MIN, MAX**

```sql
SELECT 
    name,
    department,
    salary,
    AVG(salary) OVER(PARTITION BY department) AS dept_avg
FROM employees;
```

#### 2. Ранжирующие функции

| Функция | Что делает |
|---------|------------|
| **ROW_NUMBER()** | Уникальный номер строки (1,2,3...) |
| **RANK()** | Ранг с пропусками (1,1,3) |
| **DENSE_RANK()** | Ранг без пропусков (1,1,2) |
| **NTILE(n)** | Делит на n групп |

```sql
SELECT 
    name,
    salary,
    RANK() OVER(ORDER BY salary DESC) AS salary_rank
FROM employees;
```

#### 3. Функции смещения (LAG / LEAD)

Доступ к соседним строкам:

| Функция | Что делает |
|---------|------------|
| **LAG(column)** | Значение из предыдущей строки |
| **LEAD(column)** | Значение из следующей строки |

```sql
SELECT 
    name,
    salary,
    LAG(salary) OVER(ORDER BY hire_date) AS prev_salary
FROM employees;
```

### OVER — главное слово

Определяет **окно**, по которому считается функция:

- **OVER()** — все строки
- **OVER(PARTITION BY dept)** — по группам
- **OVER(ORDER BY salary)** — с сортировкой (для рангов, LAG/LEAD)
- **OVER(PARTITION BY dept ORDER BY salary)** — и то, и другое

### Порядок выполнения

1. **FROM + WHERE + GROUP BY + HAVING**
2. **SELECT** (здесь вычисляются оконные функции)
3. **ORDER BY**

**Важно:** оконные функции нельзя использовать в WHERE

### Итог

> Оконные функции считают агрегаты/ранги/смещения **без потери строк** и незаменимы для аналитических запросов.

## CTE (Common Table Expressions)

### **CTE**
это временный именованный результат запроса, который существует только во время выполнения основного запроса. Упрощает чтение и организацию сложных запросов.

```sql
WITH high_earners AS (
    SELECT * FROM employees WHERE salary > 70000 )
SELECT * FROM high_earners WHERE department = 'IT';
```

### Зачем нужны

- **Читаемость** — разбивают сложный запрос на части
- **Переиспользование** — можно обращаться несколько раз в одном запросе
- **Рекурсия** — единственный способ сделать рекурсивные запросы в SQL
- **Альтернатива подзапросам** — часто понятнее, чем вложенные SELECT-ы

### RECURSIVE CTE

**Рекурсивная CTE** вызывает саму себя для обработки иерархических данных (деревья, графы).

```sql
WITH RECURSIVE subordinates AS (
    -- База: выбираем начального сотрудника
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE id = 1
    
    UNION ALL
    
    -- Рекурсия: присоединяем подчиненных
    SELECT e.id, e.name, e.manager_id, s.level + 1
    FROM employees e
    JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates;
```

**Как работает:**
1. **Anchor member** (первая часть до UNION) — начальный набор
2. **Recursive member** (после UNION) — присоединяет следующие уровни
3. Выполняется, пока рекурсивная часть возвращает строки


### Где применяется RECURSIVE

- Деревья (категории,组织结构)
- Графы (друзья, маршруты)
- Иерархии (менеджер-подчиненные)
- Генерация последовательностей


### Важные ограничения

- В рекурсивной части **нельзя использовать** агрегатные функции, GROUP BY, ORDER BY
- Нужно указывать **UNION ALL** (обычно)
- Есть риск бесконечной рекурсии — можно ограничить глубину через `WHERE level < 10`

## EXPLAIN и EXPLAIN ANALYZE

### Что это

**Инструменты для анализа и оптимизации запросов.** Показывают, как PostgreSQL (и другие БД) выполняет запрос.


### EXPLAIN

Показывает **план выполнения запроса** — что и в каком порядке будет делать БД.

```sql
EXPLAIN SELECT * FROM users WHERE age > 30;
```

**Вывод:** Seq Scan, Index Scan, Hash Join и т.д. + оценки стоимости.

**Что видно:**
- Типы сканирования (Seq Scan, Index Scan)
- Используемые индексы
- Способы соединения таблиц (Nested Loop, Hash Join, Merge Join)
- Оценка количества строк (rows)
- Стоимость (cost) — стартовая и общая


### EXPLAIN ANALYZE

**Реально выполняет запрос** и показывает:
- План выполнения + **фактические данные**
- Время выполнения каждого узла (actual time)
- Реальное количество строк (actual rows)
- Количество обращений к буферам (loops)

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE age > 30;
```

**Важно:** Запрос **реально выполняется**, поэтому на UPDATE/DELETE нужно быть осторожным (можно обернуть в транзакцию с ROLLBACK).


### Дополнительные параметры

| Параметр | Что добавляет |
|----------|---------------|
| **ANALYZE** | Выполняет запрос и показывает реальную статистику |
| **VERBOSE** | Детальная информация по колонкам, схемам, функциям |
| **BUFFERS** | Информация о чтении/записи буферов (только с ANALYZE) |
| **TIMING** | Включить/выключить замер времени (по умолчанию ON с ANALYZE) |
| **SUMMARY** | Добавляет сводку по времени (по умолчанию ON с ANALYZE) |
| **FORMAT** | JSON, XML, YAML — для машинной обработки |

```sql
EXPLAIN (ANALYZE, BUFFERS, VERBOSE) SELECT * FROM users;
```

### Что искать в выводе

| Показатель | На что смотреть |
|------------|-----------------|
| **Seq Scan на большой таблице** | Нужен индекс |
| **cost** | Большая разница между оценкой и реальностью — статистика устарела |
| **actual rows vs estimated rows** | Сильное расхождение — проблема со статистикой |
| **Buffers** | Много чтения с диска — мало в кэше |
| **loops > 1** | Многократное выполнение (часто проблема) |


### Типичный сценарий

```sql
BEGIN;
EXPLAIN (ANALYZE, BUFFERS) DELETE FROM users WHERE id = 1;
ROLLBACK;
```

Так можно проанализировать изменение данных, не меняя их реально.

## Блокировки строк и SELECT FOR UPDATE

### Зачем нужны блокировки строк

Предотвращают **одновременное изменение** одних и тех же данных разными транзакциями.  
Нужны для целостности при конкурентном доступе.

### Виды блокировок (основные)

| Тип | Что делает | Когда берется |
|-----|------------|---------------|
| **Блокировка чтения (SHARE)** | Другие могут читать, но не менять | SELECT (обычно не берется), явные LOCK |
| **Блокировка записи (EXCLUSIVE)** | Никто не может читать/писать | UPDATE, DELETE, INSERT |
| **Блокировка строки (Row-level)** | Блокирует только конкретные строки | SELECT FOR UPDATE, UPDATE, DELETE |

### SELECT FOR UPDATE

**Явная блокировка выбранных строк** для последующего обновления.

```sql
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- строка id=1 заблокирована для других транзакций
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

**Что происходит:**
- Другие транзакции не могут изменить эти строки
- Другие SELECT FOR UPDATE будут ждать
- Обычные SELECT читают (в PostgreSQL — видят старую версию, если READ COMMITTED)


### Варианты FOR UPDATE

| Вариант | Что делает |
|---------|------------|
| **FOR UPDATE** | Блокирует строки для обновления |
| **FOR NO KEY UPDATE** | Мягче, не блокирует ключи |
| **FOR SHARE** | Другие могут читать, но не менять |
| **FOR KEY SHARE** | Только блокировка ключей |

```sql
SELECT * FROM orders FOR SHARE;  -- другие могут читать, но не обновлять
```

### WHERE и NOWAIT / SKIP LOCKED

```sql
-- Не ждать, если строка заблокирована — ошибка
SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;

-- Пропустить заблокированные строки
SELECT * FROM accounts FOR UPDATE SKIP LOCKED;
```

**SKIP LOCKED** полезен для очередей (воркеры берут свободные задачи).

### Важные нюансы

1. **Блокировки снимаются** при COMMIT или ROLLBACK
2. **Мертвые блокировки (deadlock)** — когда транзакции ждут друг друга (БД сама их обнаруживает и убивает одну)
3. **FOR UPDATE блокирует даже строки, которых еще нет** (при работе с внешними ключами)
4. **Уровни изоляции** влияют на поведение (READ COMMITTED, REPEATABLE READ, SERIALIZABLE)


### Типичный сценарий: инвентарь

```sql
BEGIN;
-- Блокируем товар, который хотим заказать
SELECT * FROM products WHERE id = 1 FOR UPDATE;
-- Проверяем остаток, обновляем
UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;
```

## MVCC, Dead Tuples, VACUUM

### MVCC (Multi-Version Concurrency Control)

**Как PostgreSQL обеспечивает конкурентный доступ без блокировок чтения.**

**Принцип:** При обновлении строки PostgreSQL не перезаписывает её, а создаёт **новую версию** строки.

- Читающие транзакции видят "старую" версию (на момент своего начала)
- Пишущие создают "новую" версию
- **Читатели не блокируют писателей, писатели не блокируют читателей**

### xmin и xmax — главные часы MVCC

Каждая строка (версия строки) в PostgreSQL хранит две скрытые системные колонки:

| Колонка | Что хранит |
|---------|------------|
| **xmin** | ID транзакции, которая **создала** эту версию строки |
| **xmax** | ID транзакции, которая **удалила/заменила** эту версию (0 если жива) |

```sql
-- Увидеть xmin и xmax своими глазами
SELECT xmin, xmax, * FROM your_table;
```

### Dead Tuples (Мёртвые строки)

**Что это:** Старые версии строк, которые уже никто не видит.

**Откуда берутся:**
- UPDATE = INSERT новой версии + старая помечается мёртвой
- DELETE = строка помечается мёртвой

**Проблема:**
- Занимают место на диске
- Замедляют сканирование таблиц (нужно просматривать и мёртвые)
- Раздувают таблицу (bloat)

### VACUUM

**Обычный VACUUM** — помечает место мёртвых строк как **доступное для переиспользования**.

```sql
VACUUM table_name;
```

**Что делает:**
- Удаляет ссылки на мёртвые строки из индексов
- Освобождает место внутри страниц (но не возвращает ОС)
- Обновляет статистику
- **Не блокирует** чтение и запись

**Не делает:** Не уменьшает файл на диске.

### VACUUM FULL

**Полная перепаковка таблицы** — создаёт новую копию без мёртвых строк.

```sql
VACUUM FULL table_name;
```

**Что делает:**
- Перестраивает таблицу заново
- Возвращает место операционной системе
- **Блокирует таблицу** (эксклюзивная блокировка)

**Когда использовать:** Только когда нужно реально уменьшить файл (ночью, в обслуживание).

### AUTOVACUUM

**Автоматический фоновый процесс**, который сам запускает VACUUM когда нужно.

**Как работает:**
- Следит за количеством мёртвых строк
- При превышении порога (`autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor * rows`) запускается
- Работает в фоне, не мешая работе

**Важные параметры:**
- `autovacuum_vacuum_scale_factor` = 0.2 (по умолчанию 20% мёртвых)
- `autovacuum_vacuum_threshold` = 50 (минимум мёртвых строк)
- `autovacuum_naptime` = 1min (частота проверки)


### Итог одной фразой

MVCC позволяет читать и писать одновременно, но оставляет мёртвые строки. VACUUM убирает ссылки на них, AUTOVACUUM делает это автоматически, а VACUUM FULL реально уменьшает файл (но с блокировкой).

## pg_stat_statements и аппроксимация

### pg_stat_statements

**Это расширение PostgreSQL для сбора статистики по выполненным запросам.**

**Что собирает:**
- Текст запроса (нормализованный — с $1 вместо конкретных значений)
- Количество вызовов
- Общее время выполнения
- Среднее время
- Время чтения/записи блоков
- Количество возвращаемых строк

```sql
-- Включить расширение
CREATE EXTENSION pg_stat_statements;

-- Посмотреть топ медленных запросов
SELECT query, calls, total_time, mean_time
FROM pg_stat_statements
ORDER BY mean_time DESC
LIMIT 10;
```

**Зачем:**
- Найти медленные запросы
- Отследить частоту выполнения
- Увидеть нагрузку на систему

**Важно:**
- Данные сбрасываются при перезагрузке (или вручную через `pg_stat_statements_reset()`)
- Требует настройки в postgresql.conf (`shared_preload_libraries`)

### Аппроксимация (приблизительные вычисления)

**Когда точный подсчет слишком дорогой**, PostgreSQL использует приблизительные оценки.

**Где применяется:**
- **EXPLAIN** — показывает приблизительную стоимость запроса (не точную)
- **Статистика планировщика** — оценивает количество строк (`reltuples` в pg_class)
- **Анализ больших таблиц** — VACUUM/ANALYZE обновляют приблизительные значения

**Как работает:**
- Хранит не точные числа, а **гистограммы** и **частотные списки**
- Планировщик использует их для выбора плана
- Точность настраивается (`default_statistics_target`)

```sql
-- Посмотреть приблизительное количество строк
SELECT reltuples FROM pg_class WHERE relname = 'users';
```

**Зачем:**
- Скорость вместо точности
- EXPLAIN ANALYZE показывает реальные цифры
- Для больших таблиц аппроксимация — единственный вариант

### Главное отличие

| | pg_stat_statements | Аппроксимация |
|---|-------------------|----------------|
| **Что дает** | Реальная статистика по выполненным запросам | Приблизительные оценки для планировщика |
| **Для кого** | Администратор/разработчик | Сама БД (планировщик) |
| **Точность** | Точная (факт) | Приблизительная |
| **Цель** | Поиск проблемных запросов | Выбор оптимального плана |

## CAP-теорема

**CAP-теорема** (теорема Брюера) утверждает, что в распределенной системе можно гарантировать **только два из трех** свойств одновременно:

### Три свойства

| Свойство | Что значит |
|----------|------------|
| **C** (Consistency) | **Согласованность** — все узлы видят одни и те же данные в один момент времени |
| **A** (Availability) | **Доступность** — каждый запрос получает ответ (успех или ошибку) |
| **P** (Partition Tolerance) | **Устойчивость к распределению** — система продолжает работать при потере связи между узлами |

### Три варианта

| Тип | Что выбираем | Примеры |
|-----|--------------|---------|
| **CP** | Согласованность + Устойчивость | Банковские системы, Zookeeper, etcd |
| **AP** | Доступность + Устойчивость | DNS, Cassandra, CouchDB |
| **CA** | Согласованность + Доступность | Реляционные БД (но в распределенной среде CA невозможен, т.к. P обязателен) |


### Важно понимать

1. **P (Partition Tolerance) — обязателен** для распределенных систем. Сеть может упасть всегда.
2. Реальный выбор: **CP или AP** (CA только в нераспределенных)
3. При отказе сети:
    - **CP** — жертвуем доступностью (часть узлов не отвечают)
    - **AP** — жертвуем согласованностью (узлы могут расходиться)


### Примеры

| Система | Тип | Почему |
|---------|-----|--------|
| **PostgreSQL (кластер)** | CP | Согласованность важнее, при расщеплении часть не пишет |
| **Cassandra** | AP | Всегда можно писать, но данные могут разойтись |
| **MongoDB** | CP (по умолчанию) | Приоритет согласованности |
| **DynamoDB** | AP | Amazon выбрал доступность |

## Горизонтальное масштабирование
### Горизонтальное vs Вертикальное масштабирование

| | Горизонтальное (Scale Out) | Вертикальное (Scale Up) |
|---|----------------------------|-------------------------|
| **Что делаем** | Добавляем **новые серверы/узлы** | Улучшаем **один сервер** (CPU, RAM, диск) |
| **Плюсы** | Дешевле, отказоустойчивость | Проще, нет распределенных проблем |
| **Минусы** | Сложность, сетевые задержки | Упирается в максимум железа |


### Партиционирование (Partitioning)

**Разбиение одной таблицы на несколько физических частей внутри одной БД.**

```sql
-- Пример в PostgreSQL (декларативное партиционирование)
CREATE TABLE orders (
    id INT,
    date DATE
) PARTITION BY RANGE (date);

CREATE TABLE orders_2023 PARTITION OF orders
FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');
```

**Важно:**
- Все партиции **в одном экземпляре БД**
- Прозрачно для приложения (пишем в ту же таблицу)
- Ускоряет запросы (pruning — отсечение партиций)
- Упрощает удаление старых данных (DROP PARTITION)

### Шардинг (Sharding)

**Разбиение таблицы по разным физическим серверам (нодам).**

```sql
-- Концептуально (не SQL, а логика приложения)
if user_id % 4 == 0 → шард 1
if user_id % 4 == 1 → шард 2
...
```

**Важно:**
- Данные распределены **по разным серверам**
- Приложение или прокси знает, куда идти
- Нет автоматических JOIN-ов между шардами

### Отличие шардинга от партиционирования

| | Партиционирование | Шардинг |
|---|-------------------|---------|
| **Где живут данные** | На одном сервере | На разных серверах |
| **Прозрачность** | Полная (БД сама) | Требует логики в приложении |
| **JOIN между частями** | Возможен | Сложен или невозможен |
| **Цель** | Управляемость, скорость запросов | Масштабирование нагрузки |


### Ключ шардирования (Shard Key)

**Поле, по которому решаем, на какой шард отправить данные.**

```sql
-- Пример: user_id — ключ шардирования
shard = hash(user_id) % number_of_shards
```

**Требования к хорошему ключу:**
- **Равномерное распределение** (чтобы не было "горячих" шардов)
- **Высокая селективность** (много уникальных значений)
- **Часто используется в WHERE** (чтобы запрос шел на один шард)

**Плохой ключ:** дата (если все пишут сегодня → один шард перегружен)

### Вертикальное партиционирование

**Разделение по колонкам** (одна таблица -> несколько таблиц с разными наборами колонок).

```sql
-- Было: users (id, name, login, password, bio, avatar_url, last_login)
-- Стало:
users_main (id, name, login, password)
users_profile (id, bio, avatar_url)
users_stats (id, last_login)
```

**Когда нужно:**
- Редко используемые большие поля (JSON, текст, картинки)
- Разные паттерны доступа к разным колонкам
- Безопасность (разделение доступа)

### Итог одной таблицей

| | Горизонтальное | Вертикальное |
|---|---------------|--------------|
| **По строкам** | Шардинг, партиционирование | — |
| **По колонкам** | — | Вертикальное партиционирование |

## PgBouncer и Connection Pool

### Connection Pool — что это

**Пул соединений** — это набор заранее открытых подключений к БД, которые переиспользуются.

**Зачем:**
- Открытие нового соединения — дорого (TCP handshake, аутентификация)
- Ограничение количества одновременных подключений (БД не резиновая)


### PgBouncer

**Легкий менеджер пула соединений для PostgreSQL.** Сидит между приложением и БД.

```sql
Приложение → PgBouncer (порт 6432) → PostgreSQL (порт 5432)
```

### Режимы работы PgBouncer

| Режим | Как работает | Когда использовать |
|-------|--------------|---------------------|
| **Session pooling** | Соединение закрепляется за клиентом на всю сессию | Когда нужно много сессий, но мало активности |
| **Transaction pooling** | Соединение возвращается в пул после **каждой транзакции** | Самый популярный, экономит соединения |
| **Statement pooling** | После каждого запроса (опасно, редкий режим) | Почти не используется |


### Что дает PgBouncer

1. **Экономит соединения** — 1000 приложений могут использовать 50 реальных соединений к БД
2. **Быстрое переподключение** — не создает новые соединения, берет из пула
3. **Защита от перегрузки** — можно ограничить очередь
4. **Устойчивость** — переживает падение БД (держит пул и переподключается)

### Важные параметры

| Параметр | Что делает |
|----------|------------|
| **pool_mode** | session/transaction/statement |
| **default_pool_size** | Сколько соединений к БД держать |
| **max_client_conn** | Сколько клиентов может подключиться к PgBouncer |
| **reserve_pool_size** | Дополнительные соединения для пиков |
| **server_idle_timeout** | Закрывать неактивные соединения к БД |

### Минусы

- Не понимает подготовленные запросы (в transaction mode)
- Добавляет еще один слой (точка отказа)
- Нет полноценного мониторинга запросов (только метрики)

### Типичная связка

**Приложение (1000 подов)** → **PgBouncer (100 соединений)** → **PostgreSQL (100 max_connections)**

Без PgBouncer пришлось бы держать 1000 соединений, что убило бы БД.
