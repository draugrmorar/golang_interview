## 📚 1. Основные понятия и архитектура

**Что это и зачем:**
- **Apache Kafka** — распределённая платформа для потоковой передачи данных (event streaming). Работает по модели **publish-subscribe**
- Используется для: построения пайплайнов реального времени, сбора логов, метрик, микросервисного взаимодействия, CDC (Change Data Capture)

**Ключевые компоненты:**
| Компонент | Что делает |
|-----------|------------|
| **Producer** | Отправляет сообщения в топики |  |
| **Consumer** | Читает сообщения из топиков |  |
| **Broker** | Сервер Kafka (хранит данные) |  |
| **Topic** | Категория/поток сообщений |  |
| **Partition** | Подразделение топика для параллелизма |  |
| **Offset** | Уникальный ID сообщения в партиции |  |
| **Consumer Group** | Группа консьюмеров для распределения нагрузки |  |
| **Replication** | Копии данных для отказоустойчивости |  |
| **ISR** (In-Sync Replicas) | Реплики, синхронизированные с лидером |  |

**ZooKeeper vs KRaft** :
- До версии 2.8: **ZooKeeper** — управление кластером, выбор лидера, хранение метаданных
- С версии 3.3+: **KRaft** (Kafka Raft) — режим без ZooKeeper (рекомендованный для новых установок)

## 🔄 2. Producer: как это работает

**Процесс отправки:**
1. Сериализация ключа и значения
2. Определение партиции (по ключу или раунд-робин)
3. Накопление в батчах (для производительности)
4. Отправка

**Ключевые параметры:**
- **`bootstrap.servers`** — адреса брокеров
- **`key.serializer` / `value.serializer`** — сериализаторы
- **`acks`** — уровень подтверждения :
  - `0` — не ждать ответа (быстро, но можно потерять)
  - `1` — ждать от лидера
  - `all` (или `-1`) — ждать от всех ISR
- **`retries`** — количество повторов при ошибке
- **`batch.size`** — размер батча (слишком большой = задержка)
- **`linger.ms`** — задержка перед отправкой для накопления

**Проблемы:**
- **QueueFullException** — продюсер не успевает отправлять
- **Большой batch.size** + **linger.ms** = задержки

---

## 📥 3. Consumer: как это работает

**Процесс получения:**
1. Подписка на топики
2. Получение назначения партиций (assignment)
3. Периодическое чтение батчей
4. Коммит оффсетов

**Ключевые параметры:**
- **`group.id`** — идентификатор группы
- **`enable.auto.commit`** — автосохранение оффсетов (true/false)
- **`auto.offset.reset`** — что делать, если нет оффсета: `earliest` / `latest` / `none`
- **`fetch.min.bytes`** / **`fetch.max.wait.ms`** — оптимизация чтения
- **`max.poll.records`** — сколько записей за раз
- **`session.timeout.ms`** / **`heartbeat.interval.ms`** — таймауты сессии

**Ручной коммит оффсетов** :
- **`commitSync()`** — синхронно (ждёт ответа)
- **`commitAsync()`** — асинхронно (быстрее, но не ждёт)

**Что происходит при:**
- **Партиций > консьюмеров** — консьюмеры читают несколько партиций
- **Партиций < консьюмеров** — лишние простаивают

---

## 👥 4. Consumer Group и ребалансировка

**Consumer Group** — набор консьюмеров, которые делят партиции между собой

**Лидер группы** :
- Выбирается автоматически (первый подключившийся)
- Отвечает за распределение партиций (assignment)
- Получает события ребалансировки и пересчитывает назначение

**Ребалансировка (Rebalance)** случается когда :
- Новый консьюмер подключается
- Консьюмер выходит/падает
- Меняются топики
- Проблема: **Rebalancing storm** — частые ребалансы из-за коротких таймаутов

**Стратегии назначения партиций:**
- **Range** — последовательное
- **RoundRobin** — равномерное
- **Sticky** — минимизация перемещений

---

## 🛡️ 5. Отказоустойчивость и надёжность

**Как Kafka обеспечивает отказоустойчивость** :
- **Репликация** — каждая партиция имеет копии на разных брокерах
- **Лидер и фолловеры** — все операции идут через лидера, фолловеры копируют
- **ISR** — набор синхронизированных реплик
- **Выбор нового лидера** — при падении текущего

**Роль `min.insync.replicas`** :
- Минимальное количество реплик ISR для успешной записи
- Если ISR < `min.insync.replicas` — **NotEnoughReplicasException**

**Acks = all** гарантирует, что данные записаны на все ISR

## 📨 6. Гарантии доставки (Delivery Semantics)

Это **обязательно** спросят!

| Семантика | Что значит | Проблемы |
|-----------|------------|----------|
| **At-most-once** (0 или 1 раз) | Сообщение может потеряться, но дубликатов нет | Потеря данных |
| **At-least-once** (≥1 раз) | Не теряется, но могут быть дубликаты | Дубликаты |
| **Exactly-once** (строго 1 раз) | Не теряется и не дублируется | Сложно реализовать |

**Как достичь Exactly-once:**
- Идемпотентный продюсер (`enable.idempotence=true`) + транзакции
- На стороне консьюмера: фильтрация по ID (Outbox pattern)

---

## 🔥 7. Проблемы и их решения

| Проблема | Причина | Решение |
|----------|---------|---------|
| **Message duplication** | At-least-once + падение консьюмера | Идемпотентность, фильтрация по ID |  |
| **Consumer lag** | Медленная обработка, мало партиций | Добавить партиций/консьюмеров, оптимизировать код |  |
| **Stuck partitions** | Блокирующая обработка, долгий commit | Асинхронность, отдельные потоки |  |
| **Hot partitions** (skew) | Неравномерные ключи | Равномерные ключи, custom partitioner |  |
| **Rebalancing storm** | Частые входы/выходы | Увеличить session.timeout.ms |  |
| **Не обновляется offset/lag** | fetch.bytes > размера сообщения | Увеличить fetch.max.bytes |  |

## 🐇 8. Kafka vs RabbitMQ

Это классический вопрос на сравнение :

| Характеристика | Kafka | RabbitMQ |
|----------------|-------|----------|
| **Модель** | Publish-subscribe (лог) | Очереди (Queue) |
| **Хранение** | Хранит все сообщения до retention | Удаляет после обработки |
| **Потребление** | Pull (консьюмер сам тянет) | Push (брокер отправляет) |
| **Производительность** | Миллионы/сек | Десятки тысяч/сек |
| **Маршрутизация** | Простая (по топикам) | Сложная (Exchanges, bindings) |
| **Когда использовать** | Потоки событий, логи, метрики | Задачи, RPC, сложная маршрутизация |

### 1. Расскажи архитектуру Kafka

**Kafka** — это распределённая платформа для потоковой передачи данных, работающая по модели **publish-subscribe**.  
Основные компоненты архитектуры:

- **Producer** — отправляет сообщения в топики.
- **Consumer** — читает сообщения из топиков.
- **Broker** — сервер Kafka, хранит данные и обрабатывает запросы. Кластер состоит из нескольких брокеров.
- **Topic** — категория сообщений (как таблица в БД).
- **Partition** — подразделение топика для параллелизма. Каждая партиция — это упорядоченный, неизменяемый лог.
- **Offset** — уникальный идентификатор сообщения внутри партиции.
- **Consumer Group** — группа консьюмеров, которые вместе читают топик, распределяя партиции между собой.
- **ZooKeeper / KRaft** — координация брокеров, выбор лидера, хранение метаданных.

**Принцип работы:**  
Продюсеры пишут данные в партиции топиков. Данные хранятся на диске и реплицируются для отказоустойчивости. Консьюмеры читают данные, используя оффсеты для отслеживания прочитанного.

---

### 2. Чем отличается Producer от Consumer?

| Producer | Consumer |
|----------|----------|
| Отправляет (пишет) сообщения в топики | Читает сообщения из топиков |
| Определяет, в какую партицию писать (по ключу или round-robin) | Получает назначение партиций от группы |
| Может работать идемпотентно и транзакционно | Отслеживает прогресс с помощью оффсетов |
| Отвечает за сериализацию данных | Отвечает за десериализацию данных |
| Настраивается через `acks`, `retries`, `batch.size` | Настраивается через `group.id`, `auto.offset.reset` |

---

### 3. Что такое партиция и оффсет?

**Партиция (Partition)** — это физическое подразделение топика.
- Топик может состоять из нескольких партиций, распределённых по разным брокерам.
- Каждая партиция — это **упорядоченный, неизменяемый лог** сообщений.
- Партиции позволяют **распараллеливать** чтение и запись.

**Оффсет (Offset)** — это уникальный **порядковый номер** сообщения внутри партиции.
- Начинается с 0 и монотонно увеличивается.
- Консьюмеры сохраняют оффсет, чтобы знать, какие сообщения уже прочитаны.
- Оффсет управляется либо автоматически, либо вручную коммитится консьюмером.

---

### 4. Как работает Consumer Group?

**Consumer Group** — это группа консьюмеров, которые вместе читают один или несколько топиков.

**Как это работает:**
1. Каждая партиция топика назначается **только одному** консьюмеру внутри группы.
2. Если партиций больше, чем консьюмеров — некоторые консьюмеры будут читать несколько партиций.
3. Если консьюмеров больше, чем партиций — лишные будут простаивать.
4. При изменении состава группы (подключение/отключение) происходит **ребалансировка** — перераспределение партиций.
5. Оффсеты хранятся для **каждой партиции** и для **каждой группы** отдельно (то есть разные группы могут читать один топик независимо).

**Зачем нужно:**
- Балансировка нагрузки
- Отказоустойчивость (при падении одного консьюмера его партиции переходят другим)

---

### 5. Что такое ISR?

**ISR (In-Sync Replicas)** — это набор реплик партиции, которые **синхронизированы с лидером** (то есть содержат все актуальные сообщения).

**Как работает:**
- У каждой партиции есть **лидер** (leader) и несколько **фолловеров** (followers).
- Фолловеры постоянно копируют данные от лидера.
- Если фолловер отстаёт или падает — он исключается из ISR.
- При записи с `acks=all` продюсер ждёт подтверждения от **всех** реплик из ISR.

**Параметр `min.insync.replicas`** задаёт минимальное количество ISR для успешной записи. Если ISR становится меньше — запись отклоняется.

---

### 6. Что такое ZooKeeper в Kafka?

**ZooKeeper** — это координационный сервис, который использовался в Kafka для:
- Выбора **лидера** партиции (leader election)
- Хранения метаданных о брокерах, топиках, конфигурациях
- Отслеживания состояния брокеров (liveness)
- Управления Consumer Group (в старых версиях)

**Важно:** Начиная с Kafka 2.8, появился режим **KRaft**, который позволяет запускать Kafka **без ZooKeeper**. В новых версиях (3.3+) это рекомендуемый способ.

---

### 7. Как обеспечить Exactly-once доставку?

**Exactly-once** означает, что сообщение будет обработано **ровно один раз**, без потерь и без дубликатов.

**Как достигается в Kafka:**

**На стороне продюсера:**
- **Идемпотентность** (`enable.idempotence=true`) — гарантирует, что повторная отправка не создаст дубликатов в логе (используется Producer ID + Sequence Number).
- **Транзакции** — позволяют атомарно писать в несколько партиций.

**На стороне консьюмера:**
- Хранить оффсет **в той же транзакции**, что и результат обработки.
- Использовать **источник идемпотентности** на стороне получателя (например, уникальный ID сообщения для дедупликации).
- Применять **Transactional API** и `isolation.level=read_committed`.

**Самый надёжный паттерн:** **Transactional Outbox** — запись в БД и в Kafka в одной транзакции.

### 8. Почему Rebalance — это плохо и как его избежать?

**Почему Rebalance плох:**
- Во время ребалансировки **консьюмеры не читают данные** (простой).
- Частые ребалансы снижают пропускную способность.
- Может привести к повторной обработке сообщений (при auto-commit).
- **Rebalancing storm** — лавина ребалансов из-за коротких таймаутов.

**Как избежать:**
- Увеличить `session.timeout.ms` и `heartbeat.interval.ms` (чтобы консьюмер реже считался мёртвым)
- Увеличить `max.poll.interval.ms` (дать больше времени на обработку)
- Использовать **Static Group Membership** (статическое членство) — консьюмеры получают уникальный `group.instance.id`
- Настроить `partition.assignment.strategy` (StickyAssignor минимизирует перемещения)
- Избегать слишком частых коммитов

---

### 9. Что будет, если партиций меньше, чем консьюмеров?

**Правило:** Одна партиция может читаться **только одним** консьюмером внутри группы.

- Если партиций **5**, а консьюмеров **7**, то:
    - 5 консьюмеров будут активно читать (каждый по одной партиции)
    - 2 консьюмера будут **простаивать** (idle) и не получать данные

**Вывод:** Бессмысленно запускать больше консьюмеров, чем партиций — лишние будут просто ждать.

### 10. Чем Kafka отличается от RabbitMQ и когда что выбирать?

| Характеристика | Kafka | RabbitMQ |
|----------------|-------|----------|
| **Модель** | Публикация-подписка (лог) | Очереди (Queue) |
| **Хранение** | Хранит все сообщения до истечения retention (долго) | Удаляет после подтверждения |
| **Потребление** | Pull (консьюмер сам тянет) | Push (брокер отправляет) |
| **Маршрутизация** | Простая (по топикам) | Гибкая (Exchanges, bindings, routing keys) |
| **Производительность** | Очень высокая (миллионы/сек) | Средняя (десятки тысяч/сек) |
| **Гарантии порядка** | Внутри партиции | Только в одной очереди |
| **Когда выбирать** | Потоки событий, логи, метрики, CDC, аналитика | Задачи (tasks), RPC, сложная маршрутизация |

**Простое правило:**
- Нужна высокая пропускная способность и хранение потока событий → **Kafka**
- Нужна гибкая маршрутизация и задачи → **RabbitMQ**

### 11. Как продюсер выбирает партицию?

**Стратегии выбора партиции:**

1. **Если указан ключ (`key`)** :
    - Вычисляется хеш ключа (murmur2)
    - По хешу определяется партиция: `hash(key) % numPartitions`
    - Гарантирует, что сообщения с одинаковым ключом попадают в одну партицию (и сохраняют порядок)

2. **Если ключ не указан (key = null)** :
    - Используется **round-robin** (по умолчанию) или **sticky** (для батчинга)

3. **Кастомный partitioner**:
    - Можно написать свою логику распределения (например, по содержимому)

**Важно:** Количество партиций должно быть фиксированным, иначе хеширование сломается.

### 12. Что такое min.insync.replicas и acks?

**`acks`** — параметр продюсера, определяющий, сколько подтверждений нужно ждать:

| acks | Значение | Риск |
|------|----------|------|
| `0` | Не ждать подтверждения | Потеря данных при падении |
| `1` | Ждать подтверждения от лидера | Потеря при падении лидера (до репликации) |
| `all` (или `-1`) | Ждать подтверждения от всех ISR | Максимальная надёжность |

**`min.insync.replicas`** — минимальное количество синхронизированных реплик (ISR), которые должны подтвердить запись при `acks=all`.

- Если `min.insync.replicas=2`, а текущий ISR = 1 (одна реплика упала) → запись будет отклонена (NotEnoughReplicasException)
- Защищает от записи в нестабильный кластер

**Типичная настройка:**
- `replication.factor = 3`
- `min.insync.replicas = 2`
- `acks = all`

## replication.factor — коротко

**Количество копий (реплик) каждой партиции, которые хранятся в кластере Kafka.**

- **`replication.factor = 1`** — данные хранятся только на одном брокере. Если он упадет — данные потеряны.
- **`replication.factor = 3`** — три копии на трех разных брокерах. Можно потерять два брокера — данные сохранятся.
- **Рекомендация:** в продакшене обычно ставят **3** (баланс надежности и ресурсов).

**Как работает:**
- Одна реплика — **лидер** (leader), остальные — **фолловеры** (followers).
- Все записи идут через лидера, фолловеры синхронно копируют данные.
- Если лидер падает — новый лидер выбирается из ISR.

**Влияние на другие настройки:**
- `min.insync.replicas` должен быть **меньше или равен** `replication.factor`
- `acks=all` ждет подтверждения от `min.insync.replicas` реплик
