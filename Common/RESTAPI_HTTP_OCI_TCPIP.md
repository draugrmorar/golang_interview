# OSI (Open Systems Interconnection)

**OSI** — это эталонная (теоретическая) модель взаимодействия открытых систем, разработанная ISO в 1984 году. Она описывает, как данные должны передаваться по сети, разделяя процесс на 7 уровней. В отличие от практической модели TCP/IP (4 уровня), OSI — это идеальная модель, к которой нужно стремиться. В реальности чистой OSI никто не реализует, но её используют для обучения и понимания сетевого взаимодействия.

| Ур-нь | Название | Что делает |
|------|----------|------------|
| 7 | Прикладной (Application) | Взаимодействие с приложениями пользователя |
| 6 | Представления (Presentation) | Кодировка, шифрование, сжатие |
| 5 | Сеансовый (Session) | Управление сеансом связи (открытие/закрытие) |
| 4 | Транспортный (Transport) | Надежная доставка, контроль потока, порты |
| 3 | Сетевой (Network) | Маршрутизация, логическая адресация |
| 2 | Канальный (Data Link) | Физическая адресация (MAC), доступ к среде |
| 1 | Физический (Physical) | Передача битов по проводу/воздуху |

# TCP/IP

**TCP/IP** — это набор (стек) протоколов, которые обеспечивают передачу данных между компьютерами в сети. Название происходит от двух основных протоколов: TCP (Transmission Control Protocol) и IP (Internet Protocol).

### Модель TCP/IP (4 уровня)

В отличие от теоретической модели OSI (7 уровней), TCP/IP использует 4 уровня:

| Уровень | Название | Что делает | Примеры протоколов |
|--------|----------|------------|-------------------|
| 4 | Прикладной (Application) | Взаимодействие с приложениями пользователя | HTTP, HTTPS, FTP, SMTP, DNS, SSH |
| 3 | Транспортный (Transport) | Гарантия доставки, контроль потока, порты | TCP, UDP |
| 2 | Сетевой (Internet) | Адресация, маршрутизация, фрагментация | IP, ICMP, ARP, RIP |
| 1 | Канальный/Физический (Network Access) | Передача битов по проводу/воздуху | Ethernet, Wi-Fi, DSL, Fiber |

При передаче данных происходит **ИНКАПСУЛЯЦИЯ** данных по слоям в заголовки на каждом уровне. При получении происходит обратный процесс (декапсуляция) — каждый уровень снимает "свой" заголовок.

---

## IP (Internet Protocol) — доставка

IP отвечает за адресацию и маршрутизацию пакетов от источника к получателю.

- IPv4 (самый распространенный) / IPv6 (будущее)
- **Ненадежный**: пакет может потеряться, прийти не по порядку, дублироваться
- **Без установки соединения**: каждый пакет идет сам по себе
- **Маршрутизация**: пакеты могут идти разными путями

---

## TCP (Transmission Control Protocol) — гарантия

Обеспечивает надежную передачу информации от отправителя к получателю, а также контролирует поток передаваемых данных: предотвращает перегрузку сети, разбивая информацию на пакеты и контролируя ее путь до получателя.

| Свойство | Как обеспечивает |
|----------|------------------|
| Надежность | Подтверждения (ACK), таймауты, повторная отправка потерянных сегментов |
| Порядок | Номера последовательности (SEQ) — собирает пакеты в правильном порядке |
| Контроль потока | Окно (Window) — получатель говорит, сколько данных готов принять |
| Контроль перегрузки | Алгоритмы (Slow Start, Congestion Avoidance) — не перегружать сеть |
| Дуплекс | Данные могут идти в обе стороны одновременно |
| Без потерь | Контрольная сумма (Checksum) — проверка целостности |

## Соединение БД

Каждая БД имеет свой собственный протокол прикладного уровня, который работает поверх TCP:

| СУБД | Протокол прикладного уровня | Транспорт | Порт по умолчанию |
|------|----------------------------|-----------|-------------------|
| PostgreSQL | Frontend/Backend Protocol | TCP | 5432 |
| MySQL | MySQL Client/Server Protocol | TCP | 3306 |
| MongoDB | MongoDB Wire Protocol | TCP | 27017 |
| Redis | RESP (REdis Serialization Protocol) | TCP | 6379 |

---

## UDP (User Datagram Protocol) — быстрый, но ненадежный

UDP — "облегченная" альтернатива TCP, когда скорость важнее надежности.

| Характеристика | TCP | UDP |
|----------------|-----|-----|
| Соединение | Ориентирован на соединение | Без соединения |
| Надежность | Гарантирует доставку | Не гарантирует (пакеты могут теряться) |
| Порядок | Собирает пакеты по порядку | Порядок не гарантирован |
| Скорость | Медленнее (накладные расходы) | Быстрее (минимум проверок) |
| Заголовок | 20-60 байт | 8 байт |
| Когда использовать | HTTP, Email, SSH, FTP | Видеозвонки, DNS, Gaming, VoIP |

# HTTP (HyperText Transfer Protocol)

**HTTP** — это протокол прикладного уровня (7 уровень OSI), который лежит в основе передачи данных в вебе. Протокол передачи гипертекста. **Гипертекст** — это система текстовых страниц с перекрестными ссылками.

### Запрос
```
[МЕТОД] [URI] [ВЕРСИЯ]          ← Стартовая строка
Host: example.com                ↓
User-Agent: Mozilla/5.0          ← Заголовки (Headers)
Accept: text/html                ↓
Content-Type: application/json   ↓
← Пустая строка (обязательно!)
{                                 ↓
"key": "value"               ← Тело (Body) - опционально
}
```

### Ответ
```
[ВЕРСИЯ] [КОД] [ФРАЗА]           ← Стартовая строка
Content-Type: application/json    ↓
Cache-Control: no-cache           ← Заголовки (Headers)
Content-Length: 27                ↓
← Пустая строка (обязательно!)
{                                 ↓
"id": 123,                    ← Тело (Body) - опционально
"status": "created"
}
```

### Методы

```
GET POST PUT PATCH DELETE HEAD OPTIONS CONNECT TRACE
```

### Коды ответа (статусы)

#### 1xx — Informational (информационные)
| Код | Фраза | Значение |
|-----|-------|----------|
| 100 | Continue | Продолжай, я принял заголовки |
| 101 | Switching Protocols | Переключаюсь на другой протокол (WebSocket) |

#### 2xx — Success (успех)
| Код | Фраза | Значение |
|-----|-------|----------|
| 200 | OK | Успешно (стандартный ответ для GET) |
| 201 | Created | Создано (для POST) |
| 202 | Accepted | Принято, но обработаю позже (асинхронно) |

#### 3xx — Redirection (перенаправление)
| Код | Фраза | Значение |
|-----|-------|----------|
| 301 | Moved Permanently | Ресурс навсегда переехал (сохраняй новый адрес) |
| 302 | Found (Moved Temporarily) | Временно здесь (не сохраняй) |
| 304 | Not Modified | Не изменилось (используй кэш) |
| 307 | Temporary Redirect | Временно (метод не меняется) |
| 308 | Permanent Redirect | Навсегда (метод не меняется) |

#### 4xx — Client Error (ошибка клиента)
| Код | Фраза | Значение |
|-----|-------|----------|
| 400 | Bad Request | Кривой запрос (неправильный JSON и т.д.) |
| 401 | Unauthorized | Требуется аутентификация |
| 403 | Forbidden | Доступ запрещен (даже с логином) |
| 404 | Not Found | Ресурс не найден |
| 405 | Method Not Allowed | Нельзя DELETE на этот URL |
| 409 | Conflict | Конфликт (например, дубликат) |
| 413 | Payload Too Large | Тело слишком большое |
| 418 | I'm a teapot | Я чайник (апрельская шутка RFC) |
| 422 | Unprocessable Entity | Данные валидны, но логика не позволяет |
| 429 | Too Many Requests | Слишком много запросов (rate limit) |

#### 5xx — Server Error (ошибка сервера)
| Код | Фраза | Значение |
|-----|-------|----------|
| 500 | Internal Server Error | Все сломалось (общая ошибка) |
| 502 | Bad Gateway | Прокси получил кривой ответ от бэкенда |
| 503 | Service Unavailable | Сервер перегружен или на обслуживании |
| 504 | Gateway Timeout | Прокси не дождался ответа от бэкенда |

### HTTP vs HTTPS

- **HTTP** — передача данных в открытом виде (текст)
- **HTTPS** — HTTP поверх **SSL/TLS** (шифрование)
    - S = Secure
    - Данные шифруются, нельзя перехватить
    - Используются сертификаты для подтверждения подлинности сайта

## CORS (Cross-Origin Resource Sharing)

Браузер запрещает JavaScript с сайта A обращаться к сайту B (Same-Origin Policy). CORS — механизм, позволяющий сайтам разрешать такие запросы.

## URL (Uniform Resource Locator)

**URL** — это адрес ресурса в интернете (унифицированный указатель ресурса). Состоит из 3 частей:

- Название протокола (HTTP/HTTPS/ftp...)
- Адрес сервера (Доменный или IP адрес)
- Адрес ресурса на сервере (не обязательно html, возможна ссылка на какой-нибудь файл)

## REST API

**REST (Representational State Transfer — передача состояния представления)** — это архитектурный стиль построения API для распределенных систем, чаще всего веб-сервисов. **RESTful API** — это API, следующий принципам REST.

Чтобы API считался RESTful, он должен соблюдать 6 правил (по Roy Fielding, 2000):

| Принцип | Суть | Почему важно |
|---------|------|--------------|
| 1. Client-Server | Разделение клиента и сервера | Клиент не заботится о хранении данных, сервер — об интерфейсе. Меняются независимо. |
| 2. Stateless | Сервер не хранит состояние клиента между запросами | Каждый запрос содержит всю информацию. Легче масштабировать. |
| 3. Cacheable | Ответы должны явно указывать, можно ли кэшировать | Уменьшение нагрузки, ускорение ответов (через заголовки HTTP). |
| 4. Uniform Interface | Единый интерфейс (одинаково работаем с ресурсами) | Упрощает архитектуру. Достигается через стандарты HTTP. |
| 5. Layered System | Система может иметь слои (прокси, шлюзы) | Клиент не знает, общается он с сервером или с прокси. |
| 6. Code on Demand (опц.) | Сервер может передавать клиенту исполняемый код (скрипты) | Например, JavaScript на веб-странице. |

### Принципы REST API
#### 1. Единый интерфейс (Uniform Interface)
Это главный отличительный признак REST. Взаимодействие клиента с сервером должно быть унифицированным и простым:
*   **Ресурсы идентифицируются по URL:** Каждый объект (пользователь, товар) имеет свой уникальный адрес (например, `/users/123`).
*   **Представление ресурса:** Сервер передает не сам объект в памяти, а его *представление* (JSON, XML, HTML). Клиент без разницы, как данные хранятся в базе.
*   **Самодостаточность сообщений (HATEOAS):** Идеальный REST сам подсказывает клиенту, какие действия можно совершить дальше (ссылки в ответе). *На практике часто этим пренебрегают.*

#### 2. Клиент-серверная архитектура (Client-Server)
Разделение ответственности:
*   **Клиент** отвечает за интерфейс и пользовательский опыт.
*   **Сервер** отвечает за хранение данных, бизнес-логику и безопасность.
*   Они взаимодействуют через запросы, при этом смена интерфейса не ломает серверную часть (и наоборот).

#### 3. Отсутствие состояния (Stateless)
**Самый важный принцип для масштабирования.**
*   Сервер **не помнит** предыдущие запросы клиента.
*   Каждый запрос от клиента должен содержать **всю необходимую информацию** для его понимания (аутентификацию, параметры).
*   Сессии хранятся на стороне клиента (в токенах JWT, например).
*   *Плюсы:* Легко восстановиться после сбоя, добавить новые сервера (горизонтальное масштабирование).
*   *Минусы:* Каждый запрос тяжелее, так как нужно каждый раз авторизовывать пользователя.

#### 4. Кэширование (Cacheable)
Ответы сервера должны явно указывать, можно ли их кэшировать клиенту или промежуточным узлам (прокси).
*   Заголовки `Cache-Control`, `Expires`, `ETag` позволяют клиенту сохранить данные и не дергать сервер впустую.
*   Это радикально повышает производительность и снижает нагрузку.

#### 5. Многоуровневая система (Layered System)
Клиент при запросе не знает, взаимодействует ли он напрямую с сервером приложений, с балансировщиком или с кэширующим прокси.
*   Можно добавлять промежуточные слои (аутентификация, балансировка, кэш).
*   Клиент видит только конечный результат.

#### 6. Код по требованию (Code on Demand) — *необязательный*
Единственный необязательный принцип. Сервер может временно расширить функционал клиента, отправив ему исполняемый код (например, JavaScript или Java-апплет).

---

### Как эти принципы влияют на практику?

**Как НАДО делать:**
*   Использовать HTTP-методы по назначению: `GET` (чтение), `POST` (создание), `PUT` (замена), `PATCH` (частичное изменение), `DELETE` (удаление).
*   Использовать HTTP-статусы: `200` (ОК), `201` (Создано), `400` (Ошибка клиента), `401` (Не авторизован), `404` (Не найдено).
*   Не хранить состояние сессии на сервере (не использовать `HttpSession` в Java/PHP по умолчанию).

**Чего делать НЕЛЬЗЯ:**
*   Хранить `isLoggedIn=true` в переменной сервера для конкретного пользователя (нарушение Stateless).
*   Использовать `GET` для удаления товара (нарушение Uniform Interface).
*   Возвращать `200 OK` с текстом "Товар не найден" (нужно `404`).

#### Простой пример сравнения:

| Действие | Плохо (не REST) | Хорошо (почти REST) |
| :--- | :--- | :--- |
| Получить юзера | `GET /getUser?id=123` | `GET /users/123` |
| Создать юзера | `GET /createUser?name=John` | `POST /users` |
| Удалить юзера | `GET /deleteUser?id=123` | `DELETE /users/123` |
| Авторизация | Сервер хранит сессию `JSESSIONID` | Клиент шлет JWT токен в каждом запросе |

**Главная боль:** Принцип Stateless часто нарушают (привязывают запросы к IP или пишут кэш на самом сервере вместо отдельного слоя). Чем строже вы следуете принципам, тем лучше API будет масштабироваться.

### Методы REST

| Метод | Идемпотентный? | Безопасный? | Повтор запроса |
|-------|----------------|-------------|----------------|
| GET | ✅ Да | ✅ Да | Можно повторять сколько угодно |
| HEAD | ✅ Да | ✅ Да | Можно повторять |
| OPTIONS | ✅ Да | ✅ Да | Можно повторять |
| DELETE | ✅ Да | ❌ Нет | Повторный DELETE вернет 404 (но состояние не меняется) |
| PUT | ✅ Да | ❌ Нет | Повторный PUT с теми же данными = тот же результат |
| POST | ❌ Нет | ❌ Нет | Повторный POST создаст еще один ресурс |
| PATCH | ❌ (может быть) | ❌ Нет | Зависит от реализации |

# BGP (Border Gateway Protocol)

**Border Gateway Protocol (BGP)** — это протокол маршрутизации прикладного уровня. BGP выбирает наиболее эффективные маршруты для доставки интернет-трафика.

## Принцип работы BGP

BGP можно назвать **службой доставки Интернета**. Когда вы кому-то что-то отправляете, для отправки вашей посылки выбирается самый быстрый и эффективный маршрут. BGP работает по очень похожей схеме: когда кто-то отправляет данные через Интернет, BGP проверяет все доступные пути, по которым могут перемещаться данные, и выбирает наилучший маршрут.

> **Пример:** когда пользователь в Санкт-Петербурге загружает веб-сайт с исходными серверами в Бейруте, BGP обеспечивает самое быстрое и эффективное взаимодействие.

📖 *Читайте также в нашем блоге: как работает защита от сетевых атак по BGP.*

## Соседи BGP

**Соседи BGP** — это одноранговые узлы (пиры), устанавливающиеся **вручную** между маршрутизаторами. Для поддержания соединения BGP-спикер каждые **60 секунд** посылает *keepalive* сообщения.

> **Важное отличие:** BGP использует **TCP** (порт 179) в качестве транспортного протокола, в отличие от других протоколов маршрутизации.

BGP бывает двух типов:

| Тип | Описание |
|-----|----------|
| **iBGP** (Internal) | Работает внутри одной автономной системы (AS). |
| **eBGP** (External) | Работает между различными автономными системами (AS). |

### Различия в распространении маршрутов

- Маршруты, полученные от **eBGP**, обычно перераспределяются между **всеми iBGP узлами**, а также всеми другими eBGP соседями.
- Маршруты, объявленные на **iBGP пиринге**, повторно объявляются **только всем одноранговым узлам eBGP**.

> Это означает, что все **iBGP соседи должны быть соединены в одну сеть** (полносвязная топология).

---

## Формат сообщений BGP

Сообщение BGP состоит из двух частей — **заголовка** и **части данных**.

- **Транспорт:** TCP (порт 179)
- **Длина:** от 19 до 4096 октетов
- **Заголовок:** составляет 19 октетов и состоит из трех полей (формат един для всех типов сообщений)

---

## Типы сообщений BGP

BGP использует четыре типа сообщений:

| Тип | Назначение |
|-----|------------|
| **OPEN** | Устанавливает и настраивает смежность BGP. Содержит версию BGP, ASN, время удержания и идентификатор BGP. |
| **UPDATE** | Объявляет, обновляет или отменяет маршруты. Включает информацию о доступности сетевого уровня (NLRI). |
| **NOTIFICATION** | Указывает на состояние ошибки. Отправляется при истечении таймера или ошибках сеанса. Приводит к закрытию BGP-соединения. |
| **KEEPALIVE** | Обеспечивает работоспособность соседей. Отправляются каждые треть таймера удержания. |

---

## Состояния соседства BGP

BGP использует **конечный автомат (FSM)** для ведения таблицы всех одноранговых узлов и состояния их работоспособности.

| Состояние | Описание |
|-----------|----------|
| **Idle** | Начальный этап. BGP ожидает стартового события и инициирует TCP-соединение. |
| **Connect** | BGP запускает TCP-соединение. При успешном трехстороннем подтверждении отправляется сообщение Open. |
| **Active** | BGP начинает новое трехстороннее квитирование TCP. Если попытка неудачна, возвращается в состояние Connect. |
| **OpenSent** | Отправлено сообщение OPEN, ожидается OPEN от соседа. Проверка сообщений на ошибки. |
| **OpenConfirm** | Ожидается сообщение Keepalive или Notification. При получении Keepalive переход в Established. |
| **Established** | **Сеанс BGP установлен.** Соседи обмениваются маршрутами через сообщения Update. |
Важно: Если таймер удержания истекает в состоянии Established, обнаруживается ошибка, и сосед переводится обратно в состояние **Idle**.

---

**OSPF (англ. Open Shortest Path First)** — протокол динамической маршрутизации, основанный на технологии отслеживания состояния канала (link-state technology) и использующий для нахождения кратчайшего пути алгоритм Дейкстры.
**VRRP (Virtual Router Redundancy Protocol)** — это сетевой протокол, повышающий отказоустойчивость шлюза по умолчанию путем объединения нескольких физических маршрутизаторов в один виртуальный с общим IP-адресом. Если основной маршрутизатор (Master) выходит из строя, резервный (Backup) берет на себя его функции за несколько секунд.
**DNS (Domain Name System)** — это иерархическая распределенная система и протокол, преобразующий понятные человеку доменные имена (например, google.com) в числовые IP-адреса, необходимые компьютерам для соединения. Протокол работает преимущественно через UDP (реже TCP) на 53 порту, обеспечивая быстрый поиск IP-адреса через цепочку серверов: рекурсивный резолвер, корневой сервер, TLD-сервер и авторитетный сервер.

**Основные аспекты протокола DNS:**
- Принцип работы: Когда пользователь вводит адрес сайта, браузер запрашивает IP у DNS-сервера. Если данных нет в кэше, серверы ищут информацию от корневого уровня до авторитетного сервера, который хранит точную запись.
- Иерархия: Система состоит из корневых серверов, серверов доменов верхнего уровня (TLD, например, .com, .ru) и авторитетных серверов, хранящих точные записи о доменах.
- Кэширование: Для ускорения работы DNS-серверы временно сохраняют результаты запросов, что снижает нагрузку и ускоряет открытие сайтов.
- Безопасность: Классический DNS-запрос незашифрован. Для защиты используются протоколы DNS-over-HTTPS (DoH) и DNS-over-TLS (DoT), которые предотвращают перехват запросов.
- Типы запросов: Рекурсивные (сервер ищет ответ сам) и итеративные (сервер дает ссылку на другой сервер)

# NGFW (Next-Generation Firewall)

**Next-Generation Firewall (NGFW)** — это межсетевой экран нового поколения, объединяющий возможности классического брандмауэра, системы предотвращения вторжений (IPS) и контроля приложений в едином решении.

В отличие от старых экранов, NGFW анализирует не только порты и IP-адреса, но и **содержимое трафика** (уровень приложений) для блокировки сложных атак.

---

## 🎯 Ключевые особенности и функции NGFW

| Функция | Описание |
|---------|----------|
| **Глубокая инспекция пакетов (DPI)** | Анализ контента, а не только заголовков, позволяет выявлять вредоносные программы внутри легитимного трафика. |
| **Контроль приложений (Application Control)** | Возможность видеть, какие именно приложения (Skype, Telegram, торренты) используются, и управлять ими, независимо от порта. |
| **Интеграция с пользователями** | Привязка политик безопасности к пользователям и группам (через интеграцию с Active Directory), а не просто к IP-адресам. |
| **Предотвращение вторжений (IPS)** | Активная защита от эксплойтов, уязвимостей и сетевых атак. |
| **SSL/TLS-инспекция** | Возможность расшифровать и проверить зашифрованный трафик, который составляет большинство современного сетевого обмена. |
| **Антивирусная защита и фильтрация контента** | Блокировка подозрительных сайтов и загрузок. |

---

## 🔄 Отличие от традиционного фаервола

| Характеристика | Традиционный фаервол | NGFW |
|----------------|----------------------|------|
| **Уровень работы** | L3/L4 (IP/порт) | L7 (приложение) |
| **Анализ трафика** | Заголовки пакетов | Контент и контекст |
| **Пример политики** | Закрыть порт 80 | Разрешить веб-сайты, но запретить передачу файлов через соцсети |

> **Ключевое отличие:** NGFW позволяет, например, не просто закрыть порт 80, а разрешить сотрудникам посещать веб-сайты, но запретить им передавать файлы через соцсети.

---

## ✅ Преимущества NGFW

- **Комплексная защита** — снижение количества разрозненных средств защиты (UTM, антивирус, IPS), что упрощает управление безопасностью.
- **Высокая производительность** — специализированные аппаратные блоки (ASIC) позволяют проверять трафик без потери скорости сети.
- **Актуальность** — защита от современных угроз, таких как таргетированные атаки (APT).

## 📌 Резюме

NGFW — это эволюция сетевой безопасности, объединяющая в одном устройстве:

1. Традиционную фильтрацию (L3–L4)
2. Контекстный анализ приложений (L7)
3. Активные методы защиты (IPS, антивирус)
4. Работу с зашифрованным трафиком (SSL/TLS)

Это делает NGFW ключевым элементом современной защиты периметра сети, особенно в условиях распространения зашифрованного трафика и сложных целевых атак.
