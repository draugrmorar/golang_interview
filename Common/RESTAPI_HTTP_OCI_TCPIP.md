# OSI (Open Systems Interconnection)

**OSI** — это эталонная (теоретическая) модель взаимодействия открытых систем, разработанная ISO в 1984 году. Она описывает, как данные должны передаваться по сети, разделяя процесс на 7 уровней. В отличие от практической модели TCP/IP (4 уровня), OSI — это идеальная модель, к которой нужно стремиться. В реальности чистой OSI никто не реализует, но её используют для обучения и понимания сетевого взаимодействия.

| Ур-нь | Название | Что делает |
|------|----------|------------|
| 7 | Прикладной (Application) | Взаимодействие с приложениями пользователя |
| 6 | Представления (Presentation) | Кодировка, шифрование, сжатие |
| 5 | Сеансовый (Session) | Управление сеансом связи (открытие/закрытие) |
| 4 | Транспортный (Transport) | Надежная доставка, контроль потока, порты |
| 3 | Сетевой (Network) | Маршрутизация, логическая адресация |
| 2 | Канальный (Data Link) | Физическая адресация (MAC), доступ к среде |
| 1 | Физический (Physical) | Передача битов по проводу/воздуху |

# TCP/IP

**TCP/IP** — это набор (стек) протоколов, которые обеспечивают передачу данных между компьютерами в сети. Название происходит от двух основных протоколов: TCP (Transmission Control Protocol) и IP (Internet Protocol).

### Модель TCP/IP (4 уровня)

В отличие от теоретической модели OSI (7 уровней), TCP/IP использует 4 уровня:

| Уровень | Название | Что делает | Примеры протоколов |
|--------|----------|------------|-------------------|
| 4 | Прикладной (Application) | Взаимодействие с приложениями пользователя | HTTP, HTTPS, FTP, SMTP, DNS, SSH |
| 3 | Транспортный (Transport) | Гарантия доставки, контроль потока, порты | TCP, UDP |
| 2 | Сетевой (Internet) | Адресация, маршрутизация, фрагментация | IP, ICMP, ARP, RIP |
| 1 | Канальный/Физический (Network Access) | Передача битов по проводу/воздуху | Ethernet, Wi-Fi, DSL, Fiber |

При передаче данных происходит **ИНКАПСУЛЯЦИЯ** данных по слоям в заголовки на каждом уровне. При получении происходит обратный процесс (декапсуляция) — каждый уровень снимает "свой" заголовок.

---

## IP (Internet Protocol) — доставка

IP отвечает за адресацию и маршрутизацию пакетов от источника к получателю.

- IPv4 (самый распространенный) / IPv6 (будущее)
- **Ненадежный**: пакет может потеряться, прийти не по порядку, дублироваться
- **Без установки соединения**: каждый пакет идет сам по себе
- **Маршрутизация**: пакеты могут идти разными путями

---

## TCP (Transmission Control Protocol) — гарантия

Обеспечивает надежную передачу информации от отправителя к получателю, а также контролирует поток передаваемых данных: предотвращает перегрузку сети, разбивая информацию на пакеты и контролируя ее путь до получателя.

| Свойство | Как обеспечивает |
|----------|------------------|
| Надежность | Подтверждения (ACK), таймауты, повторная отправка потерянных сегментов |
| Порядок | Номера последовательности (SEQ) — собирает пакеты в правильном порядке |
| Контроль потока | Окно (Window) — получатель говорит, сколько данных готов принять |
| Контроль перегрузки | Алгоритмы (Slow Start, Congestion Avoidance) — не перегружать сеть |
| Дуплекс | Данные могут идти в обе стороны одновременно |
| Без потерь | Контрольная сумма (Checksum) — проверка целостности |

## Соединение БД

Каждая БД имеет свой собственный протокол прикладного уровня, который работает поверх TCP:

| СУБД | Протокол прикладного уровня | Транспорт | Порт по умолчанию |
|------|----------------------------|-----------|-------------------|
| PostgreSQL | Frontend/Backend Protocol | TCP | 5432 |
| MySQL | MySQL Client/Server Protocol | TCP | 3306 |
| MongoDB | MongoDB Wire Protocol | TCP | 27017 |
| Redis | RESP (REdis Serialization Protocol) | TCP | 6379 |

---

## UDP (User Datagram Protocol) — быстрый, но ненадежный

UDP — "облегченная" альтернатива TCP, когда скорость важнее надежности.

| Характеристика | TCP | UDP |
|----------------|-----|-----|
| Соединение | Ориентирован на соединение | Без соединения |
| Надежность | Гарантирует доставку | Не гарантирует (пакеты могут теряться) |
| Порядок | Собирает пакеты по порядку | Порядок не гарантирован |
| Скорость | Медленнее (накладные расходы) | Быстрее (минимум проверок) |
| Заголовок | 20-60 байт | 8 байт |
| Когда использовать | HTTP, Email, SSH, FTP | Видеозвонки, DNS, Gaming, VoIP |

# HTTP (HyperText Transfer Protocol)

**HTTP** — это протокол прикладного уровня (7 уровень OSI), который лежит в основе передачи данных в вебе. Протокол передачи гипертекста. **Гипертекст** — это система текстовых страниц с перекрестными ссылками.

### Запрос
```
[МЕТОД] [URI] [ВЕРСИЯ]          ← Стартовая строка
Host: example.com                ↓
User-Agent: Mozilla/5.0          ← Заголовки (Headers)
Accept: text/html                ↓
Content-Type: application/json   ↓
← Пустая строка (обязательно!)
{                                 ↓
"key": "value"               ← Тело (Body) - опционально
}
```

### Ответ
```
[ВЕРСИЯ] [КОД] [ФРАЗА]           ← Стартовая строка
Content-Type: application/json    ↓
Cache-Control: no-cache           ← Заголовки (Headers)
Content-Length: 27                ↓
← Пустая строка (обязательно!)
{                                 ↓
"id": 123,                    ← Тело (Body) - опционально
"status": "created"
}
```

### Методы

```
GET POST PUT PATCH DELETE HEAD OPTIONS CONNECT TRACE
```

### Коды ответа (статусы)

#### 1xx — Informational (информационные)
| Код | Фраза | Значение |
|-----|-------|----------|
| 100 | Continue | Продолжай, я принял заголовки |
| 101 | Switching Protocols | Переключаюсь на другой протокол (WebSocket) |

#### 2xx — Success (успех)
| Код | Фраза | Значение |
|-----|-------|----------|
| 200 | OK | Успешно (стандартный ответ для GET) |
| 201 | Created | Создано (для POST) |
| 202 | Accepted | Принято, но обработаю позже (асинхронно) |

#### 3xx — Redirection (перенаправление)
| Код | Фраза | Значение |
|-----|-------|----------|
| 301 | Moved Permanently | Ресурс навсегда переехал (сохраняй новый адрес) |
| 302 | Found (Moved Temporarily) | Временно здесь (не сохраняй) |
| 304 | Not Modified | Не изменилось (используй кэш) |
| 307 | Temporary Redirect | Временно (метод не меняется) |
| 308 | Permanent Redirect | Навсегда (метод не меняется) |

#### 4xx — Client Error (ошибка клиента)
| Код | Фраза | Значение |
|-----|-------|----------|
| 400 | Bad Request | Кривой запрос (неправильный JSON и т.д.) |
| 401 | Unauthorized | Требуется аутентификация |
| 403 | Forbidden | Доступ запрещен (даже с логином) |
| 404 | Not Found | Ресурс не найден |
| 405 | Method Not Allowed | Нельзя DELETE на этот URL |
| 409 | Conflict | Конфликт (например, дубликат) |
| 413 | Payload Too Large | Тело слишком большое |
| 418 | I'm a teapot | Я чайник (апрельская шутка RFC) |
| 422 | Unprocessable Entity | Данные валидны, но логика не позволяет |
| 429 | Too Many Requests | Слишком много запросов (rate limit) |

#### 5xx — Server Error (ошибка сервера)
| Код | Фраза | Значение |
|-----|-------|----------|
| 500 | Internal Server Error | Все сломалось (общая ошибка) |
| 502 | Bad Gateway | Прокси получил кривой ответ от бэкенда |
| 503 | Service Unavailable | Сервер перегружен или на обслуживании |
| 504 | Gateway Timeout | Прокси не дождался ответа от бэкенда |

### HTTP vs HTTPS

- **HTTP** — передача данных в открытом виде (текст)
- **HTTPS** — HTTP поверх **SSL/TLS** (шифрование)
    - S = Secure
    - Данные шифруются, нельзя перехватить
    - Используются сертификаты для подтверждения подлинности сайта

## CORS (Cross-Origin Resource Sharing)

Браузер запрещает JavaScript с сайта A обращаться к сайту B (Same-Origin Policy). CORS — механизм, позволяющий сайтам разрешать такие запросы.

---

## URL (Uniform Resource Locator)

**URL** — это адрес ресурса в интернете (унифицированный указатель ресурса). Состоит из 3 частей:

- Название протокола (HTTP/HTTPS/ftp...)
- Адрес сервера (Доменный или IP адрес)
- Адрес ресурса на сервере (не обязательно html, возможна ссылка на какой-нибудь файл)

## REST API

**REST (Representational State Transfer — передача состояния представления)** — это архитектурный стиль построения API для распределенных систем, чаще всего веб-сервисов. **RESTful API** — это API, следующий принципам REST.

Чтобы API считался RESTful, он должен соблюдать 6 правил (по Roy Fielding, 2000):

| Принцип | Суть | Почему важно |
|---------|------|--------------|
| 1. Client-Server | Разделение клиента и сервера | Клиент не заботится о хранении данных, сервер — об интерфейсе. Меняются независимо. |
| 2. Stateless | Сервер не хранит состояние клиента между запросами | Каждый запрос содержит всю информацию. Легче масштабировать. |
| 3. Cacheable | Ответы должны явно указывать, можно ли кэшировать | Уменьшение нагрузки, ускорение ответов (через заголовки HTTP). |
| 4. Uniform Interface | Единый интерфейс (одинаково работаем с ресурсами) | Упрощает архитектуру. Достигается через стандарты HTTP. |
| 5. Layered System | Система может иметь слои (прокси, шлюзы) | Клиент не знает, общается он с сервером или с прокси. |
| 6. Code on Demand (опц.) | Сервер может передавать клиенту исполняемый код (скрипты) | Например, JavaScript на веб-странице. |

### Методы REST

| Метод | Идемпотентный? | Безопасный? | Повтор запроса |
|-------|----------------|-------------|----------------|
| GET | ✅ Да | ✅ Да | Можно повторять сколько угодно |
| HEAD | ✅ Да | ✅ Да | Можно повторять |
| OPTIONS | ✅ Да | ✅ Да | Можно повторять |
| DELETE | ✅ Да | ❌ Нет | Повторный DELETE вернет 404 (но состояние не меняется) |
| PUT | ✅ Да | ❌ Нет | Повторный PUT с теми же данными = тот же результат |
| POST | ❌ Нет | ❌ Нет | Повторный POST создаст еще один ресурс |
| PATCH | ❌ (может быть) | ❌ Нет | Зависит от реализации |
