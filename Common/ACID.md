# Требования ACID

**ACID** — набор требований, которые обеспечивают сохранность ваших данных.

### Atomicity — Атомарность
Атомарность гарантирует, что каждая транзакция будет выполнена полностью или не будет выполнена совсем. Не допускаются промежуточные состояния.

### Consistency — Согласованность
Благодаря тому, что транзакция не допускает промежуточных результатов, база остается консистентной.  
Есть такое определение транзакции:  
*«Упорядоченное множество операций, переводящих базу данных из одного согласованного состояния в другое»*.  
То есть до выполнения операции и после база остается консистентной (в переводе на русский — согласованной).

### Isolation — Изолированность
Во время выполнения транзакции параллельные транзакции не должны оказывать влияния на её результат.

### Durability — Надёжность
Если пользователь получил подтверждение от системы, что транзакция выполнена, он может быть уверен, что сделанные им изменения не будут отменены из-за какого-либо сбоя.

# 4 возможные проблемы

| Проблема | Описание |
|----------|----------|
| **Потерянная запись** | Первая транзакция не завершилась, началась 2я. В итоге данные не консистентны. |
| **Грязное чтение** | Чтение данных, которые другая транзакция еще не зафиксировала (может откатить). |
| **Неповторяющееся чтение** | В рамках одной транзакции один и тот же запрос дает РАЗНЫЕ результаты (кто-то изменил и закоммитил строку). |
| **Фантомное чтение** | В рамках одной транзакции один и тот же запрос выдает РАЗНОЕ КОЛИЧЕСТВО строк (кто-то добавил или удалил). |

# Блокировки СУБД

### По типу доступа

- **Разделяемая (Shared Lock / Read Lock)**: Транзакции могут только читать данные. Если кто-то читает, другие тоже могут читать, но не могут писать.
- **Эксклюзивная (Exclusive Lock / Write Lock)**: Транзакция пишет данные. Никто другой не может ни читать, ни писать в этот ресурс.

### По логике работы

- **Оптимистичная (Optimistic)**: Думаем, что конфликтов не будет. Работаем без блокировок. В конце проверяем, не изменил ли кто данные. Если изменил — перезапускаем транзакцию (или ошибка).
- **Пессимистичная (Pessimistic)**: Думаем, что конфликты неизбежны. Блокируем данные сразу при чтении, чтобы никто не влез.

# Уровни изоляции транзакций (ANSI/ISO)

Чем выше уровень, тем меньше проблем, но медленнее работа.

| Уровень | Грязное чтение | Неповтор. чтение | Фантомы | Как работает |
|---------|----------------|------------------|---------|--------------|
| **Read Uncommitted** | ✅ Есть | ✅ Есть | ✅ Есть | PSQL не поддерживает!!! «Можно читать всё, даже незакоммиченное». Никакой защиты. |
| **Read Committed** | ❌ Нет | ✅ Есть | ✅ Есть | PSQL по умолчанию!! Видим только чужие коммиты. Защита от Dirty Read. |
| **Repeatable Read** | ❌ Нет | ❌ Нет | ✅ Есть | Читаем строки на момент начала. Защита от Non-Repeatable Read. |
| **Serializable** | ❌ Нет | ❌ Нет | ❌ Нет | Полная изоляция. Транзакции идут строго друг за другом. |

# MVCC (Многоверсионность)

СУБД не блокирует данные, а хранит несколько версий одной строки.

**Как работает**: Каждая транзакция видит «снапшот» (срез) данных на момент своего старта.

**Особенность**: Чтение никогда не блокирует запись, запись не блокирует чтение.

**Какую проблему решает**: Позволяет избежать большинства блокировок на чтение.

**Кто использует**: PostgreSQL, Oracle, MySQL (InnoDB).

---

# Какую проблему что решает

| Проблема | Чем лечат |
|----------|-----------|
| **Lost Update** | Блокировка строки при чтении (SELECT FOR UPDATE) или оптимистичная блокировка с версией. |
| **Dirty Read** | Уровень Read Committed (или выше) + MVCC. |
| **Non-Repeatable Read** | Уровень Repeatable Read (или выше) + MVCC (держит старую версию строки). |
| **Phantom Read** | Уровень Serializable или блокировка диапазона (Gap Lock в MySQL). |

**Коротко**: Либо блокируем данные (пессимистично), либо храним версии (MVCC) и выбираем правильный уровень изоляции под задачу.
