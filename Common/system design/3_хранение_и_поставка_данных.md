## 1. Реляционные базы данных (Relational Databases)

### Что это
Данные организованы в **таблицы** со строго определенной схемой (schema). Таблицы связаны между собой через **foreign keys**. Для запросов используется **SQL** (Structured Query Language).

### Ключевые принципы
- **ACID** — атомарность, согласованность, изоляция, долговечность
- **Нормализация** — минимизация дублирования данных
- **Constraints** — foreign keys, unique, check, not null
- **Транзакции** — возможность группировать операции

### Популярные представители

| Система | Особенности | Типичное применение |
|---------|-------------|---------------------|
| **PostgreSQL** | Самый мощный open-source, расширяемость, JSONB, full-text search, GIS | Универсальный выбор, сложные транзакции, аналитика |
| **MySQL** | Простота, высокая скорость чтения, широкое распространение | Веб-приложения, read-heavy нагрузки |
| **Oracle** | Корпоративный, дорогой, мощный | Банки, крупный enterprise |
| **Microsoft SQL Server** | Интеграция с .NET, BI инструменты | Корпоративные Windows-среды |
| **Amazon Aurora** | Cloud-native, совместим с PostgreSQL/MySQL, отделяет storage от compute | Облачные приложения, высокая доступность |

### Когда выбирать

✅ **Подходит для:**
- **Сложные транзакции:** Платежи, заказы, бронирования (нужны ACID, foreign keys)
- **Связанные данные:** Данные имеют четкие связи (users → orders → items)
- **Сложные запросы:** JOIN-ы, агрегации, подзапросы
- **Целостность данных:** Нельзя потерять или повредить данные (финансы, медицина)
- **Схема стабильна:** Структура данных известна заранее и редко меняется

❌ **Не подходит для:**
- **Горизонтального масштабирования:** Реляционные БД сложно шардировать (хотя есть Citus, Vitess, Aurora)
- **Гибкой схемы:** Если структура данных постоянно меняется
- **Очень высокой нагрузки (> 100k RPS):** Требуют сложного шардирования
- **Хранения больших файлов:** Неэффективно (используйте object storage)

### Пример на интервью

> *«Для обработки заказов я выбираю PostgreSQL. Нам нужны строгие ACID-гарантии, foreign keys между заказами, пользователями и товарами, а также сложные запросы для отчетности. Postgres также поддерживает JSONB, что даст нам гибкость для хранения метаданных заказа без изменения схемы»*.

## 2. Документоориентированные базы данных (Document Stores)

### Что это
Данные хранятся как **документы** (обычно JSON, BSON). Каждый документ содержит все данные об объекте, включая вложенные структуры. Схема гибкая — разные документы в одной коллекции могут иметь разную структуру.

### Ключевые принципы
- **Документ как единица хранения** — всё, что нужно об объекте, в одном месте
- **Гибкая схема** — можно добавлять поля без миграций
- **Денормализация** — дублирование данных допустимо ради производительности чтения
- **Масштабируемость** — горизонтальное масштабирование (шардирование) из коробки

### Популярные представители

| Система | Особенности | Типичное применение |
|---------|-------------|---------------------|
| **MongoDB** | Самый популярный, шардирование из коробки, богатый язык запросов, транзакции (с 4.0) | Каталоги товаров, профили пользователей, логи, контент |
| **Couchbase** | In-memory first, высокопроизводительный, SQL++ (SQL-like) | Real-time приложения, кэш + БД в одном |
| **CouchDB** | REST API, репликация P2P, eventual consistency | Мобильные приложения, офлайн-синхронизация |

### Когда выбирать

✅ **Подходит для:**
- **Гибкая схема:** Структура данных эволюционирует, разные объекты имеют разные поля
- **Агрегаты (Domain-Driven Design):** Данные естественно представляются как единый документ (корзина покупок, профиль)
- **Высокая скорость чтения:** Все данные объекта в одном месте — нет JOIN-ов
- **Горизонтальное масштабирование:** Легко шардируется по ключу
- **JSON-нативная работа:** Если приложение уже использует JSON

❌ **Не подходит для:**
- **Сложные JOIN-ы:** Хотя MongoDB имеет $lookup, это не его сильная сторона
- **Сложные транзакции между документами:** Транзакции есть, но тяжелые
- **Строгая целостность:** Нет foreign keys на уровне БД
- **Частые обновления одного поля в большом документе:** Документ переписывается целиком

### Пример на интервью

> *«Для каталога товаров я выбираю MongoDB. У товаров разная структура в зависимости от категории (одежда, электроника, книги) — гибкая схема здесь идеальна. Мы можем хранить весь товар в одном документе: характеристики, отзывы, наличие. Это дает быстрые чтения без JOIN-ов. MongoDB также легко шардируется по category_id или product_id при росте»*.

## 3. Key-Value базы данных (Key-Value Stores)

### Что это
Самый простой тип БД: **хранилище пар «ключ — значение»**. Аналог хеш-таблицы (dict, map), но распределенный и персистентный.

### Ключевые принципы
- **Простейший интерфейс:** GET(key), SET(key, value), DELETE(key)
- **Максимальная производительность:** O(1) доступ
- **Масштабируемость:** Легко шардируется по ключу
- **Отсутствие структуры:** Значение — это просто blob (байты)

### Популярные представители

| Система | Особенности | Типичное применение |
|---------|-------------|---------------------|
| **Redis** | In-memory, персистентность опционально, богатые структуры данных (strings, hashes, lists, sets, sorted sets, streams) | Кэш, сессии, лидерборды, очереди, rate limiting |
| **Memcached** | In-memory, простейший, нет персистентности, только key-value | Кэш для статики, сессии |
| **Amazon DynamoDB** | Managed, key-value + document, consistent hashing, автоскейлинг | Высоконагруженные приложения, serverless |
| **RocksDB** | Embedded LSM-дерево, высокопроизводительная запись | База для других систем (Kafka, Flink, MySQL) |

### Когда выбирать

✅ **Подходит для:**
- **Кэширование:** Redis, Memcached как L2 кэш перед БД
- **Хранение сессий:** Быстрый доступ по session_id
- **Лидерборды:** Redis sorted sets — идеальный инструмент
- **Очереди и стримы:** Redis Streams, Pub/Sub
- **Rate limiting:** Счетчики с TTL
- **Простые lookup-таблицы:** User_id → профиль (кэш)

❌ **Не подходит для:**
- **Сложные запросы:** Нет фильтрации по значению (кроме Redis с индексами через структуры)
- **Связи между объектами:** Нужно управлять связями на прикладном уровне
- **Транзакции:** Redis поддерживает транзакции, но ограниченно
- **Хранение больших данных:** In-memory БД ограничены RAM

### Redis против Memcached

| Характеристика | Redis | Memcached |
|----------------|-------|-----------|
| **Структуры данных** | Богатые (5+ типов) | Только строки |
| **Персистентность** | Да (RDB, AOF) | Нет |
| **Репликация** | Да (master-replica) | Нет |
| **Шардирование** | Cluster mode | Клиентское |
| **Сложность** | Выше | Проще |

### Пример на интервью

> *«Для хранения пользовательских сессий я выбираю Redis. Нам нужен быстрый доступ по session_id (O(1)), автоматическое удаление сессий по TTL, и репликация для отказоустойчивости. Redis также даст нам возможность хранить дополнительные данные (корзина покупок как hash, лидерборды как sorted set) без введения новых хранилищ»*.

## 4. Time Series базы данных (Time Series Databases — TSDB)

### Что это
Специализированные БД для хранения и анализа **данных, индексированных по времени**. Оптимизированы для высокой скорости записи и эффективных агрегаций по временным интервалам.

### Ключевые принципы
- **Высокая скорость записи:** Миллионы точек данных в секунду
- **Эффективное сжатие:** Колоночное хранение, дельта-кодирование
- **Агрегации:** Предварительно вычисленные агрегаты (rollups)
- **Retention policies:** Автоматическое удаление старых данных

### Популярные представители

| Система | Особенности | Типичное применение |
|---------|-------------|---------------------|
| **InfluxDB** | Самый популярный open-source, SQL-like (InfluxQL, Flux), Telegraf для сбора | Метрики инфраструктуры, IoT |
| **TimescaleDB** | Расширение PostgreSQL, SQL, гибрид реляционной + TSDB | Когда нужны сложные SQL-запросы к временным данным |
| **Prometheus** | Pull-модель, встроенный алертинг, интеграция с Kubernetes | Мониторинг инфраструктуры, метрики |
| **VictoriaMetrics** | Совместим с Prometheus, лучше масштабируется, ниже потребление ресурсов | Крупномасштабный мониторинг |
| **QuestDB** | Высокопроизводительная, SIMD-оптимизации, PostgreSQL wire protocol | Финансовые данные, рыночные котировки |

### Когда выбирать

✅ **Подходит для:**
- **Метрики инфраструктуры:** CPU, память, сеть, диск
- **IoT данные:** Сенсоры, устройства, телеметрия
- **Финансовые данные:** Котировки акций, курс валют
- **Логи и события:** Если нужны агрегации по времени
- **Мониторинг бизнес-метрик:** RPS, ошибки, конверсии

❌ **Не подходит для:**
- **Транзакционные данные:** Заказы, пользователи
- **Частые обновления:** TSDB оптимизированы на вставку, не на update
- **Сложные JOIN-ы:** Не сильная сторона (кроме TimescaleDB)

### Выбор между TSDB

| Сценарий | Выбор |
|----------|-------|
| Метрики инфраструктуры в Kubernetes | Prometheus + VictoriaMetrics |
| Сложные SQL-запросы к временным данным | TimescaleDB (поверх PostgreSQL) |
| IoT с миллиардами точек | InfluxDB или QuestDB |
| Финансовые котировки (низкая задержка) | QuestDB |

### Пример на интервью

> *«Для сбора метрик инфраструктуры мы используем Prometheus. Он отлично интегрируется с Kubernetes, собирает метрики по pull-модели, имеет мощный язык запросов PromQL и встроенный алертинг. Для долгосрочного хранения (больше 30 дней) мы используем VictoriaMetrics как remote storage — он эффективнее сжимает данные и лучше масштабируется»*.

## 5. Колоночные базы данных (Columnar Databases)

### Что это
Данные хранятся **по колонкам, а не по строкам**. Это кардинально меняет производительность для аналитических запросов.

### Row-based vs Columnar

| Характеристика | Row-based (PostgreSQL) | Columnar (ClickHouse) |
|----------------|------------------------|------------------------|
| **Хранение** | Вся строка последовательно | Каждая колонка отдельно |
| **Чтение** | Хорошо для «всей строки» (OLTP) | Отлично для агрегаций по колонкам (OLAP) |
| **Запись** | Быстрая (вставка строк) | Медленная (вставка колонок), лучше батчами |
| **Сжатие** | Низкое (разные типы в строке) | Высокое (одинаковые типы в колонке) |

### Ключевые принципы
- **Колоночное хранение:** Каждая колонка хранится отдельно
- **Высокое сжатие:** Однородные данные сжимаются в 5–10 раз
- **Векторизованные вычисления:** SIMD-оптимизации для агрегаций
- **OLAP-оптимизация:** Сложные аналитические запросы, сканирование больших объемов

### Популярные представители

| Система | Особенности | Типичное применение |
|---------|-------------|---------------------|
| **ClickHouse** | Самый производительный open-source, колоночное хранение, сжатие, векторизованные вычисления | Аналитика, логи, observability, real-time отчеты |
| **Apache Druid** | OLAP, sub-second запросы, потоковый и батч-ингeст | Event-аналитика, дашборды в реальном времени |
| **Google BigQuery** | Serverless, колоночное хранение, отдельно storage и compute | Аналитика на больших данных (PB) |
| **Amazon Redshift** | Managed, колоночное, интеграция с AWS | Data warehouse |

### Когда выбирать

✅ **Подходит для:**
- **OLAP нагрузки:** Аналитические запросы, агрегации, дашборды
- **Большие объемы данных:** Терабайты — петабайты
- **Сжатие критично:** Хранение логов, событий, исторических данных
- **Real-time аналитика:** ClickHouse дает миллисекундные ответы на миллиардах строк
- **Data Lakehouse:** Parquet формат (колоночный) в S3 + Trino/Presto

❌ **Не подходит для:**
- **OLTP нагрузки:** Частые точечные обновления, вставки по одной строке
- **Транзакции:** Нет ACID в классическом понимании (у ClickHouse нет транзакций)
- **Сложные JOIN-ы:** Хотя ClickHouse поддерживает JOIN, это не его сильная сторона

### Пример на интервью

> *«Для аналитики по событиям пользователей мы выбираем ClickHouse. У нас 100 млрд событий в месяц, мы строим агрегации по пользователям, сессиям, воронкам. Колоночное хранение и векторизованные вычисления дают нам ответы на сложные запросы за миллисекунды-секунды, а сжатие снижает объем хранилища в 5–10 раз»*.

## 6. Wide-Column базы данных (Wide-Column Stores)

### Что это
Гибрид между key-value и колоночными БД. Данные организованы в **таблицы, но колонки могут динамически добавляться**. Каждая строка может иметь свой набор колонок.

### Ключевые принципы
- **Модель:** Key → набор колонок (column families)
- **Горизонтальное масштабирование:** Из коробки, без единой точки отказа
- **Высокая доступность:** Eventually consistent (AP в CAP)
- **Высокая скорость записи:** LSM-деревья

### Популярные представители

| Система | Особенности | Типичное применение |
|---------|-------------|---------------------|
| **Apache Cassandra** | Masterless, линейная масштабируемость, высокая доступность, CQL (SQL-like) | High-write нагрузки, IoT, мессенджеры, временные ряды |
| **ScyllaDB** | Cassandra-совместимая, на C++ (вместо Java), в 10x быстрее | High-performance Cassandra |
| **Apache HBase** | Поверх HDFS, сильная согласованность (CP), интеграция с Hadoop | Big data, batch analytics |

### Cassandra против традиционных БД

| Характеристика | Cassandra | PostgreSQL | MongoDB |
|----------------|-----------|------------|---------|
| **Масштабирование** | Линейное (masterless) | Вертикальное/шардирование (сложно) | Горизонтальное (но есть мастер) |
| **Запись** | Молнии (LSM) | Быстрая | Быстрая |
| **Чтение** | Очень быстрое (по ключу) | Быстрое | Быстрое |
| **Сложные запросы** | Ограничены (только по ключам) | Да | Да (MongoDB Query API) |
| **Консистенность** | Настраиваемая (tunable consistency) | Strong (ACID) | Strong (с 4.0) |
| **Availability** | AP (по CAP) | CP | CP |

### Когда выбирать

✅ **Подходит для:**
- **Высокая скорость записи:** Миллионы операций записи в секунду
- **Распределенная архитектура:** Десятки/сотни нод, географически распределенные
- **Глобальная доступность:** Multi-DC репликация из коробки
- **Простые запросы:** Доступ по ключу (user_id, device_id)
- **No downtime:** Можно добавлять/удалять ноды без остановки

❌ **Не подходит для:**
- **Сложные запросы:** JOIN-ы, агрегации, secondary indexes ограничены
- **Строгая согласованность:** Настраиваемая, но цена — latency
- **ACID транзакции:** Нет (только lightweight transactions на уровне одной строки)

### Cassandra Data Model (ключевая особенность)

В Cassandra данные **проектируются под запросы**. Нельзя просто взять реляционную схему и перенести.

**Правила:**
1. Определите запросы заранее
2. Денормализуйте данные (храните их в нескольких таблицах)
3. Primary key = (partition key, clustering columns)

**Пример для мессенджера:**
```sql
-- Сообщения в чате (доступ по chat_id)
CREATE TABLE messages_by_chat (
    chat_id uuid,
    created_at timestamp,
    message_id uuid,
    text text,
    user_id uuid,
    PRIMARY KEY (chat_id, created_at, message_id)
) WITH CLUSTERING ORDER BY (created_at DESC);

-- Сообщения пользователя (доступ по user_id)
CREATE TABLE messages_by_user (
    user_id uuid,
    created_at timestamp,
    message_id uuid,
    chat_id uuid,
    text text,
    PRIMARY KEY (user_id, created_at, message_id)
);
```

### Пример на интервью

> *«Для хранения сообщений в мессенджере я выбираю Cassandra. Нам нужна высокая скорость записи (миллионы сообщений в секунду в пике), глобальная доступность (пользователи по всему миру) и линейное масштабирование без даунтайма. Cassandra из коробки поддерживает multi-DC репликацию и настраиваемую консистентность — мы можем выбрать баланс между latency и consistency в зависимости от региона»*.

## 7. Object Storage (Объектное хранилище)

### Что это
Хранилище для **неструктурированных данных** — файлов, изображений, видео, бэкапов, логов. Данные хранятся как объекты с метаданными.

### Ключевые принципы
- **Объект:** Данные + метаданные + уникальный идентификатор (key)
- **Плоское пространство:** Нет иерархии папок (но эмулируется префиксами)
- **Высокая durability:** 11 девяток (99.999999999%) — данные не теряются
- **Стоимость:** Дешево (особенно для холодных данных)
- **Масштабируемость:** Практически бесконечная

### Популярные представители

| Система | Особенности | Типичное применение |
|---------|-------------|---------------------|
| **Amazon S3** | Стандарт де-факто, 11 девяток, tiered storage, интегрирован со всем AWS | Файлы, статика, бэкапы, Data Lake |
| **Google Cloud Storage** | Аналог S3, интеграция с GCP | Файлы, Data Lake |
| **Azure Blob Storage** | Аналог S3, интеграция с Azure | Файлы, Data Lake |
| **MinIO** | Open-source, S3-совместимый, self-hosted | On-premise S3, Kubernetes |
| **OpenStack Swift** | Open-source объектное хранилище | On-premise |

### Tiered Storage в S3

| Уровень | Задержка | Стоимость (относительная) | Применение |
|---------|----------|--------------------------|------------|
| **S3 Standard** | ms | 1x | Часто используемые данные |
| **S3 Intelligent-Tiering** | ms | ~0.8x | Автоматическое перемещение |
| **S3 Standard-IA** | ms | ~0.6x | Редко используемые |
| **S3 One Zone-IA** | ms | ~0.5x | Некритичные, один AZ |
| **S3 Glacier Instant Retrieval** | ms | ~0.3x | Долгосрочное, быстрый доступ |
| **S3 Glacier Flexible Retrieval** | минуты–часы | ~0.1x | Архив, редкий доступ |
| **S3 Glacier Deep Archive** | часы | ~0.05x | Долгосрочный архив (годы) |

### Когда выбирать

✅ **Подходит для:**
- **Статические файлы:** Изображения, видео, аудио, документы
- **Бэкапы:** Дампы БД, логи, снапшоты
- **Data Lake:** Хранение больших данных в форматах Parquet, ORC, Avro
- **Логи:** Долгосрочное хранение логов
- **CDN origin:** Исходник для CDN (CloudFront, Cloudflare)
- **Архив:** Данные, которые нужно хранить годами

❌ **Не подходит для:**
- **Частые обновления:** Объектное хранилище не предназначено для транзакционных обновлений
- **Поиск внутри объектов:** Нет индексов (нужен отдельный слой)
- **Мелкие объекты:** Эффективность падает при миллиардах мелких объектов (но работает)

### Пример на интервью

> *«Для хранения пользовательских аватаров, изображений и видео мы используем S3. Это дает нам практически бесконечное масштабирование, высокую durability и дешевое хранение. Для часто запрашиваемого контента мы добавляем CDN (CloudFront) на уровне выше — это снижает нагрузку на S3 и уменьшает задержку для пользователей. Для старых видео (старше 30 дней) автоматически перемещаем в S3 Glacier, что снижает стоимость хранения в 10 раз»*.

# ХАРАКТЕРИСТИКИ БАЗ ДАННЫХ

## 1. ACID / BASE

Это две философии обеспечения согласованности данных. Выбор между ними — один из самых фундаментальных в системном дизайне.

### ACID (Atomicity, Consistency, Isolation, Durability)

**Что это:**
Набор свойств, гарантирующих надежную обработку транзакций. Классический подход реляционных БД.

| Свойство | Описание | Пример |
|----------|----------|--------|
| **Atomicity (Атомарность)** | Транзакция выполняется полностью или не выполняется вообще. Нет «половинчатых» состояний | Перевод денег: списание и зачисление — одна операция. Если одно не удалось, откатывается всё |
| **Consistency (Согласованность)** | Транзакция переводит БД из одного корректного состояния в другое. Все constraints (foreign keys, unique) соблюдены | После перевода сумма на счетах не изменилась (100 + 200 = 300 до и после) |
| **Isolation (Изоляция)** | Параллельные транзакции не влияют друг на друга. Результат такой же, как если бы они выполнялись последовательно | Два перевода с одного счета одновременно не приведут к отрицательному балансу |
| **Durability (Долговечность)** | После подтверждения транзакции данные сохранены и не будут потеряны (даже при сбое) | Деньги списались — даже если сервер упал через секунду, запись осталась |

**Где применяется:**
- **PostgreSQL, MySQL, Oracle, SQL Server**
- Все реляционные БД
- Некоторые NoSQL добавили ACID транзакции (MongoDB 4.0+, CockroachDB)

**Когда выбирать ACID:**
- Финансовые транзакции (платежи, переводы)
- Системы бронирования (билеты, отели)
- Заказы в e-commerce (нельзя создать заказ без резерва товара)
- Любые сценарии, где потеря или дублирование данных недопустимы

---

### BASE (Basically Available, Soft state, Eventually consistent)

**Что это:**
Альтернативный подход, жертвующий строгой согласованностью ради доступности и масштабируемости. Основа большинства NoSQL систем.

| Свойство | Описание | Пример |
|----------|----------|--------|
| **Basically Available (Базовая доступность)** | Система отвечает на запросы, даже если некоторые узлы упали. Может вернуть устаревшие данные, но не откажет | Cassandra: при падении узла чтение идет с реплик, даже если они не самые свежие |
| **Soft State (Мягкое состояние)** | Состояние системы может меняться со временем даже без новых запросов (из-за репликации, сходимости) | В DynamoDB после записи данные распространяются по узлам асинхронно |
| **Eventually Consistent (Согласованность в итоге)** | Если новых записей нет, через некоторое время все реплики придут к согласованному состоянию. В момент после записи данные могут быть несогласованы | В Instagram: после поста лента у разных пользователей может показывать разное в течение секунд |

**Где применяется:**
- **Cassandra, DynamoDB, Riak, CouchDB**
- **MongoDB** (по умолчанию, но есть опции strong consistency)
- **Redis Cluster** (асинхронная репликация)
- **Kafka** (в некоторых сценариях)

**Когда выбирать BASE:**
- Социальные сети (лайки, посты могут немного расходиться)
- Системы аналитики (eventual consistency приемлема)
- IoT (потеря нескольких показаний не критична)
- Глобально распределенные системы (нужна высокая доступность)

---

### ACID vs BASE: сравнение

| Характеристика | ACID | BASE |
|----------------|------|------|
| **Философия** | Consistency first | Availability first |
| **Согласованность** | Строгая (strong) | В итоге (eventual) |
| **Доступность** | Может снижаться при сетевых разделениях (CP в CAP) | Высокая (AP в CAP) |
| **Задержка** | Выше (синхронные операции) | Ниже (асинхронные) |
| **Масштабируемость** | Вертикальное или сложное шардирование | Горизонтальное (из коробки) |
| **Сложность** | Ниже для разработки (гарантии на уровне БД) | Выше (нужна идемпотентность, разрешение конфликтов) |
| **Примеры** | PostgreSQL, MySQL | Cassandra, DynamoDB |

### На интервью

> *«Для платежной системы мы выбираем ACID-совместимую БД (PostgreSQL). Нам критична строгая согласованность — нельзя списать деньги и не зачислить получателю. Для ленты новостей, где допустима небольшая задержка видимости, мы можем использовать BASE-подход с Cassandra или DynamoDB, что даст нам высокую доступность и линейное масштабирование»*.

## 2. OLAP / OLTP / HTAP

Это классификация по **типу нагрузки**, для которой оптимизирована БД.

### OLTP (Online Transaction Processing)

**Что это:**
Системы для обработки транзакций в реальном времени. Много мелких операций (INSERT, UPDATE, DELETE). Основной тип нагрузки для приложений.

**Характеристики:**
- **Много мелких операций:** Тысячи-миллионы RPS
- **Точечные запросы:** `SELECT * FROM users WHERE id = 123`
- **Низкая задержка:** < 100 мс
- **Высокая скорость записи:** Часто write-heavy
- **Нормализованные данные:** Чтобы избежать аномалий при обновлениях

**Примеры БД:**
- PostgreSQL, MySQL
- MongoDB
- Cassandra (high-write OLTP)

**Пример сценария:**
- Обработка заказа в интернет-магазине
- Авторизация платежа
- Создание поста в соцсети

---

### OLAP (Online Analytical Processing)

**Что это:**
Системы для аналитики и отчетности. Сложные запросы, агрегации, сканирование больших объемов данных. Скорость записи не критична.

**Характеристики:**
- **Сложные запросы:** `SELECT category, SUM(revenue) FROM orders GROUP BY category`
- **Большие объемы:** Терабайты — петабайты
- **Сканирование данных:** Читаются миллиарды строк
- **Высокая латентность допустима:** Секунды — минуты
- **Денормализованные данные:** Схемы типа «звезда», «снежинка»

**Примеры БД:**
- ClickHouse
- Google BigQuery
- Amazon Redshift
- Snowflake
- Apache Druid

**Пример сценария:**
- Дашборд продаж за месяц
- Анализ воронки конверсии
- BI отчетность

---

### HTAP (Hybrid Transactional/Analytical Processing)

**Что это:**
Современный подход, объединяющий OLTP и OLAP в одной системе. Позволяет выполнять аналитические запросы по свежим транзакционным данным без ETL.

**Характеристики:**
- **Одна система** для транзакций и аналитики
- **Реальная свежесть:** Аналитика по данным, которые только что записали
- **Изоляция нагрузок:** Часто через columnar engine + row engine

**Примеры БД:**
- **TiDB** — распределенная, совместима с MySQL, columnar storage (TiFlash) для OLAP
- **SingleStore** (бывший MemSQL) — гибрид, in-memory
- **CockroachDB** — с columnar capabilities
- **Google AlloyDB** — PostgreSQL-совместимая с columnar engine

**Когда нужен HTAP:**
- Real-time дашборды по операционным данным
- Fraud detection (анализ транзакций в реальном времени)
- Personalization (мгновенные рекомендации на основе действий)

---

### Сравнение OLTP vs OLAP vs HTAP

| Характеристика | OLTP | OLAP | HTAP |
|----------------|------|------|------|
| **Нагрузка** | Много мелких операций | Сложные агрегации | Оба типа |
| **Задержка** | Миллисекунды | Секунды-минуты | Миллисекунды (OLTP) + секунды (OLAP) |
| **Свежесть аналитики** | N/A | Часы/дни (ETL) | Real-time |
| **Модель данных** | Нормализованная | Денормализованная (звезда) | Гибрид |
| **Масштабирование** | Вертикальное/шардирование | Горизонтальное (MPP) | Горизонтальное |
| **Примеры** | PostgreSQL, MySQL | ClickHouse, BigQuery | TiDB, SingleStore |

### На интервью

> *«Для операционной части (заказы, пользователи) мы используем PostgreSQL как OLTP-базу. Для аналитики мы выгружаем данные в ClickHouse (OLAP) через CDC. Если бы нам требовалась real-time аналитика по свежим данным без ETL, мы бы рассмотрели HTAP-решение вроде TiDB»*.

## 3. Persistent / In-memory базы данных

Классификация по тому, **где и как долго хранятся данные**.

### Persistent (Дисковые) базы данных

**Что это:**
Данные хранятся на диске (SSD/HDD) и сохраняются между перезапусками. Это стандартный тип БД.

**Характеристики:**
- **Durability:** Данные не теряются при перезапуске
- **Объем:** Ограничен диском (терабайты, петабайты)
- **Скорость:** Дисковый I/O (миллисекунды), но с кэшированием
- **Стоимость:** Дешевле RAM

**Примеры:**
- PostgreSQL, MySQL (данные на диске + кэш в RAM)
- Cassandra, MongoDB (data on disk)
- ClickHouse (на диске, но с горячим кэшем)

---

### In-memory базы данных

**Что это:**
Данные хранятся **в оперативной памяти (RAM)**. Персистентность опциональна (обычно есть snapshot или WAL на диск).

**Характеристики:**
- **Скорость:** Микросекунды (100–1000x быстрее диска)
- **Объем:** Ограничен RAM (обычно до сотен ГБ на ноду)
- **Durability:** Опциональна (можно потерять при падении, если не включена персистентность)
- **Стоимость:** RAM дороже диска

**Примеры:**
- **Redis** — in-memory с опциональной персистентностью (RDB, AOF)
- **Memcached** — только in-memory, без персистентности
- **Aerospike** — in-memory + flash (гибрид)
- **SingleStore** — in-memory OLTP

---

### Гибридные подходы

| Подход | Описание | Пример |
|--------|----------|--------|
| **In-memory + snapshot** | Периодически сохраняет весь дамп на диск | Redis RDB |
| **In-memory + WAL** | Пишет каждую операцию в журнал (AOF), данные в памяти | Redis AOF |
| **Tiered storage** | Hot data в RAM, cold data на диске | Aerospike, SingleStore |
| **Caching layer** | Дисковая БД + внешний кэш в Redis | PostgreSQL + Redis |

### Когда выбирать in-memory

| Сценарий | Почему |
|----------|--------|
| **Кэш** | Redis для быстрого доступа к горячим данным |
| **Сессии** | Быстрый доступ по session_id, TTL |
| **Лидерборды** | Redis sorted sets |
| **Rate limiting** | Счетчики с TTL |
| **Real-time аналитика** | In-memory для sub-millisecond ответов |

### Когда in-memory НЕ подходит

| Сценарий | Почему |
|----------|--------|
| **Долгосрочное хранение** | RAM дорогая, объем ограничен |
| **Данные > RAM** | Не помещаются, нужна эвактуация |
| **Строгая durability** | При падении можно потерять неперсистированные данные |

### На интервью

> *«Для основного хранилища заказов мы используем PostgreSQL (persistent). Нам нужна долговременная сохранность данных. Для кэша часто запрашиваемых заказов добавляем Redis (in-memory) — это снижает нагрузку на PostgreSQL и дает микросекундную задержку. Для Redis настраиваем AOF, чтобы минимизировать потери данных при перезапуске»*.

## 4. Embedded / Single file базы данных

Классификация по **способу развертывания и использования**.

### Embedded (встраиваемые) базы данных

**Что это:**
Библиотека, которая работает внутри приложения. Нет отдельного серверного процесса. Приложение запускает и останавливает БД как часть своего кода.

**Характеристики:**
- **Нет сети:** Все вызовы внутри процесса (быстрее)
- **Простота развертывания:** Один бинарник
- **Zero configuration:** Не нужен администратор БД
- **Ограниченное масштабирование:** Только для одного процесса

**Примеры:**
- **SQLite** — самый популярный embedded SQL
- **RocksDB** — embedded key-value (LSM-дерево)
- **LMDB** — embedded key-value, memory-mapped
- **LevelDB** — embedded key-value от Google
- **DuckDB** — embedded OLAP, in-memory

**Когда применять:**
- Мобильные приложения (iOS, Android)
- Десктопные приложения
- Edge/IoT устройства
- Тестирование (in-memory база для unit-тестов)
- Serverless (нельзя держать постоянное соединение)

---

### Single file базы данных

**Что это:**
Частный случай embedded БД, где вся база данных хранится в **одном файле** на диске.

**Характеристики:**
- **Портативность:** Можно копировать, перемещать, бэкапить одним файлом
- **Простота:** Нет сложной структуры директорий
- **Транзакции:** SQLite поддерживает ACID на уровне файла

**Примеры:**
- **SQLite** — вся БД в одном `.db` файле
- **Microsoft Access** — `.accdb`
- **Parquet / DuckDB** — аналитические данные в файлах

---

### Embedded vs Client-Server

| Характеристика | Embedded (SQLite) | Client-Server (PostgreSQL) |
|----------------|------------------|---------------------------|
| **Архитектура** | Библиотека в приложении | Отдельный процесс/кластер |
| **Сеть** | Нет | Есть |
| **Конкурентная запись** | Ограничена (одна запись) | Много одновременных записей |
| **Масштабирование** | Только вертикальное | Горизонтальное (шардирование) |
| **Сложность** | Минимальная | Высокая |
| **Использование** | Мобильные, IoT, desktop | Web-приложения, микросервисы |

### Когда выбирать embedded

✅ **Подходит для:**
- **Мобильные приложения:** SQLite — стандарт iOS/Android
- **Edge/IoT:** Устройства с ограниченными ресурсами
- **Десктопные приложения:** Локальное хранилище
- **Тестирование:** In-memory SQLite для unit-тестов
- **Serverless functions:** Не держит соединения, быстро стартует

❌ **Не подходит для:**
- **Высокая конкурентность:** SQLite не подходит для тысяч одновременных записей
- **Распределенные системы:** Нет встроенной репликации
- **Большие объемы (ТБ):** Embedded БД не масштабируются

### На интервью

> *«Для мобильного клиента мы используем SQLite как embedded базу данных. Это позволяет приложению работать офлайн, хранить данные локально и синхронизироваться с сервером при появлении сети. SQLite идеально подходит — это один файл, не требует отдельного процесса и поддерживает ACID транзакции»*.

---

## Сводная таблица: характеристики БД

| Характеристика | Варианты | Ключевое отличие |
|----------------|----------|------------------|
| **Согласованность** | ACID / BASE | Строгая vs итоговая |
| **Нагрузка** | OLTP / OLAP / HTAP | Транзакции vs аналитика vs гибрид |
| **Хранение** | Persistent / In-memory | Диск vs RAM |
| **Развертывание** | Embedded / Client-Server | Внутри приложения vs отдельный процесс |

---

## Частые ошибки на интервью

| Ошибка | Правильный подход |
|--------|-------------------|
| **«ACID всегда лучше»** | ACID дает гарантии, но снижает доступность и масштабируемость. Для глобальных систем BASE часто предпочтительнее |
| **«OLAP можно на PostgreSQL»** | Можно, но на больших объемах (десятки ТБ) производительность будет низкой. Специализированные OLAP дают 100x ускорение |
| **«Redis как основное хранилище»** | In-memory БД не предназначены для долгосрочного хранения без персистентности. Используйте их как кэш или specialized store |
| **«SQLite для веб-приложения»** | SQLite не выдерживает high-concurrency записи. Для веб-приложений нужна client-server БД |
| **«HTAP заменит всё»** | HTAP — компромисс. Для чистого OLTP или OLAP специализированные БД будут эффективнее |

# ИНДЕКСЫ

## Что это и зачем

**Индекс** — это вспомогательная структура данных, которая ускоряет операции чтения (SELECT, UPDATE, DELETE с WHERE) за счет дополнительных затрат на запись (INSERT, UPDATE, DELETE) и хранение.

**Аналогия:** Индекс в книге — вы ищете тему не перелистывая все страницы, а смотрите в алфавитный указатель.

**Компромисс:**
- **Чтение:** Ускоряется (иногда в 1000+ раз)
- **Запись:** Замедляется (нужно обновлять индекс)
- **Хранение:** Дополнительное место (иногда больше самих данных)

---

## 1. Типы индексов по структуре данных

### B-Tree (Balanced Tree)

**Что это:**
Сбалансированное дерево, где все листья находятся на одном уровне. Самый распространенный тип индекса.

**Структура:**
```
        [50]
       /    \
   [20,30]   [70,80]
   /  |  \    /  |  \
 [10][25][40][60][75][90]
```

**Характеристики:**
- **Сложность:** O(log n) для поиска, вставки, удаления
- **Поддерживает:** `=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `LIKE 'prefix%'`
- **Не поддерживает:** `LIKE '%suffix'`, полнотекстовый поиск

**Где используется:**
- **Все реляционные БД** (PostgreSQL, MySQL) по умолчанию
- **MongoDB** (по умолчанию на _id)
- **Cassandra** (для первичного ключа)

**Когда выбирать B-Tree:**
- Универсальный выбор для большинства запросов
- Нужны диапазонные запросы (`BETWEEN`, `>`)
- Нужна сортировка (`ORDER BY`)

---

### Hash Index

**Что это:**
Хеш-таблица, где ключ — хеш от значения, значение — указатель на строку.

**Структура:**
```
Hash Function: "user123" → 0x7F3A
Table:
0x7F3A → [row_pointer_1, row_pointer_2]
```

**Характеристики:**
- **Сложность:** O(1) для точного поиска
- **Поддерживает:** Только точное равенство (`=`, `IN`)
- **Не поддерживает:** Диапазоны, сортировку, `LIKE`

**Где используется:**
- **PostgreSQL:** `CREATE INDEX USING HASH` (но ограничен)
- **MySQL:** Только для MEMORY таблиц
- **Redis:** По сути вся БД — большая хеш-таблица
- **Cassandra:** Для некоторых secondary indexes

**Когда выбирать Hash:**
- Только точные поиски по ключу
- Нет потребности в диапазонах и сортировке
- Максимальная скорость

**Пример:**
```sql
-- Хорош для hash
SELECT * FROM sessions WHERE session_id = 'abc123';

-- Плох для hash (не поддерживается)
SELECT * FROM users WHERE created_at > '2024-01-01';
```

---

### Bitmap Index

**Что это:**
Индекс, представляющий данные в виде битовых карт. Каждое уникальное значение — это битовая строка, где каждый бит соответствует строке.

**Структура для колонки `status` со значениями 'active', 'inactive', 'deleted':**
```
Строка 1: active
Строка 2: inactive
Строка 3: active
Строка 4: deleted
Строка 5: active

bitmap 'active':    [1, 0, 1, 0, 1]
bitmap 'inactive':  [0, 1, 0, 0, 0]
bitmap 'deleted':   [0, 0, 0, 1, 0]
```

**Характеристики:**
- **Эффективность:** Отлично для колонок с **низкой селективностью** (мало уникальных значений)
- **Битовые операции:** `AND`, `OR`, `NOT` — очень быстрые
- **Компактность:** Хорошо сжимаются (run-length encoding)

**Где используется:**
- **Oracle** (широко)
- **PostgreSQL** (через расширения)
- **ClickHouse** (внутренне)
- **Data warehouses**

**Когда выбирать Bitmap:**
- Колонки с малым количеством уникальных значений (пол, статус, категория)
- Сложные комбинации условий (`WHERE status = 'active' AND region = 'EU' AND plan = 'premium'`)
- OLAP нагрузки (больше чтений, редкие записи)

**Пример:**
```sql
-- Отлично для bitmap (низкая селективность)
SELECT * FROM users WHERE status = 'active' AND country = 'US';
```

---

### Spatial Index (Пространственный индекс)

**Что это:**
Индекс для геопространственных данных. Позволяет эффективно искать точки, линии, полигоны.

**Структуры:**
- **R-Tree:** Разбивает пространство на вложенные прямоугольники
- **QuadTree:** Рекурсивно делит пространство на 4 квадранта
- **GeoHash:** Преобразует координаты в строку (геохеш), затем B-Tree

**Характеристики:**
- **Поддерживает:** `ST_Within`, `ST_Distance`, `ST_Intersects`, поиск ближайших соседей
- **Сложность:** Логарифмическая для пространственных запросов

**Где используется:**
- **PostgreSQL** (PostGIS) — самая мощная гео-БД
- **MySQL** (Spatial extensions)
- **MongoDB** (2dsphere, GeoJSON)
- **Elasticsearch** (geo_point, geo_shape)

**Когда выбирать Spatial:**
- Поиск точек рядом (`NEARBY`)
- Определение принадлежности области (доставка в зоне)
- Геоаналитика (тепловые карты)

**Пример:**
```sql
-- Найти все рестораны в радиусе 1 км
SELECT * FROM restaurants
WHERE ST_DWithin(
    location, 
    ST_MakePoint(37.617, 55.755), 
    1000
);
```

---

### Reversed Index (Обратный индекс)

**Что это:**
Индекс, где ключ инвертирован. Используется для поиска по суффиксам или в полнотекстовом поиске.

**Структура:**
```
Обычный индекс: "apple" → [doc1, doc2]
Reversed index: "elppa" → [doc1, doc2]
```

**Характеристики:**
- **Поддерживает:** Поиск по суффиксам (`LIKE '%pattern'`), полнотекстовый поиск
- **Инверсия:** Позволяет использовать B-Tree для суффиксов

**Где используется:**
- **Elasticsearch** (inverted index для полнотекстового поиска — основа)
- **PostgreSQL** (reverse index через `pg_trgm` для LIKE)
- **Lucene** (базовый механизм)

**Когда выбирать Reversed:**
- Полнотекстовый поиск
- Поиск по окончаниям (`LIKE '%something'`)
- Автокомплит (поиск по префиксам)

**Пример:**
```sql
-- PostgreSQL с триграммами
CREATE INDEX idx_users_email_trgm ON users USING gin (email gin_trgm_ops);
SELECT * FROM users WHERE email LIKE '%@gmail.com';
```

## 2. Кластерные / Некластерные индексы

### Кластерный индекс (Clustered Index)

**Что это:**
Индекс, который определяет **физический порядок данных на диске**. Таблица может иметь только **один** кластерный индекс.

**Как работает:**
- Данные хранятся в отсортированном порядке по ключу индекса
- Листья индекса содержат **сами данные**, а не указатели

**Характеристики:**
- **Скорость чтения:** Очень быстро для диапазонных запросов (данные физически рядом)
- **Запись:** Медленнее (нужно переупорядочивать данные при вставке)
- **Ограничение:** Только один на таблицу

**Где используется:**
- **MySQL InnoDB:** Первичный ключ — всегда кластерный индекс
- **SQL Server:** Можно выбрать
- **PostgreSQL:** Нет кластерных индексов (есть `CLUSTER` команда, но не поддерживается автоматически)

**Пример (MySQL):**
```sql
CREATE TABLE users (
    id INT PRIMARY KEY,  -- Это кластерный индекс
    name VARCHAR(100)
);
-- Данные физически хранятся в порядке id
```

### Некластерный индекс (Non-Clustered Index)

**Что это:**
Индекс, который хранится отдельно от данных. Таблица может иметь **много** некластерных индексов.

**Как работает:**
- Листья индекса содержат **указатели на строки** (ROWID, primary key)
- Данные хранятся в другом месте (куча или кластерный индекс)

**Характеристики:**
- **Скорость:** Может требовать дополнительного чтения (lookup в таблицу)
- **Гибкость:** Можно создавать много индексов под разные запросы
- **Размер:** Меньше кластерного (хранят не все данные)

**Сравнение:**

| Характеристика | Кластерный | Некластерный |
|----------------|------------|--------------|
| **Количество** | 1 на таблицу | Много |
| **Физический порядок** | Данные отсортированы | Индекс отсортирован, данные нет |
| **Скорость чтения по ключу** | Очень быстро | Быстро |
| **Скорость диапазонного чтения** | Очень быстро | Медленнее (нужен lookup) |
| **Скорость записи** | Медленнее | Быстрее |
| **Хранение** | Данные в листьях | Указатели в листьях |

## 3. Селективность индекса (Selectivity)

**Что это:**
Мера того, насколько хорошо индекс отфильтровывает строки. Высокая селективность = мало строк соответствует значению.

**Формула:**
```
Selectivity = уникальные_значения / общее_количество_строк
```

| Селективность | Пример | Эффективность индекса |
|---------------|--------|----------------------|
| **Высокая** | `user_id` (1/N) | Отлично — индекс очень полезен |
| **Средняя** | `category_id` (1/100) | Хорошо — все еще полезен |
| **Низкая** | `status` ('active', 'inactive') | Плохо — БД может проигнорировать индекс |

### Почему низкая селективность плоха

Если индекс имеет низкую селективность (например, `status = 'active'` для 50% строк), то:
1. **Индексное чтение:** Нужно прочитать много указателей
2. **Random I/O:** Каждый указатель → отдельное чтение (медленно)
3. **Оптимизатор:** Может выбрать full table scan, который быстрее

**Пример:**
```sql
-- Высокая селективность — индекс эффективен
SELECT * FROM users WHERE user_id = 12345;  -- 1 строка

-- Низкая селективность — индекс может не использоваться
SELECT * FROM users WHERE status = 'active';  -- 50% строк
```

### Когда индекс с низкой селективностью полезен

1. **Покрывающий индекс:** Если все нужные поля в индексе
2. **Bitmap индекс:** Специально для низкой селективности
3. **В комбинации:** `WHERE status = 'active' AND category_id = 5` (второе условие высокоселективное)

## 4. Функциональные индексы (Functional / Expression Indexes)

**Что это:**
Индекс, построенный не на колонке, а на **результате функции или выражения**.

**Зачем:**
Запросы с функциями не могут использовать обычные индексы.

**Пример проблемы:**
```sql
-- Без функционального индекса — full table scan
SELECT * FROM users WHERE LOWER(email) = 'john@example.com';

-- С функциональным индексом — быстрый поиск
CREATE INDEX idx_users_email_lower ON users (LOWER(email));
```

**Примеры использования:**

| Сценарий | Индекс |
|----------|--------|
| **Case-insensitive поиск** | `CREATE INDEX ON users (LOWER(email))` |
| **Поиск по дате без времени** | `CREATE INDEX ON orders (DATE(created_at))` |
| **JSON поле** | `CREATE INDEX ON users ((data->>'phone'))` |
| **Вычисляемые поля** | `CREATE INDEX ON products ((price * (1 - discount)))` |
| **Математические операции** | `CREATE INDEX ON points (SQRT(x^2 + y^2))` |

**Где поддерживается:**
- **PostgreSQL:** Полноценно
- **MySQL:** С MySQL 8.0 (функциональные индексы)
- **SQL Server:** Computed columns + index
- **Oracle:** Да

## 5. Покрывающие индексы (Covering Index)

**Что это:**
Индекс, который содержит **все колонки**, необходимые для запроса. БД может ответить на запрос, **не обращаясь к таблице** (без lookup).

**Структура:**
```
Обычный индекс:  ключ → указатель на строку → данные
Покрывающий:     ключ + нужные данные → ответ готов
```

**Пример:**
```sql
-- Запрос: нужно имя и email пользователя по id
SELECT name, email FROM users WHERE id = 123;

-- Обычный индекс на id (не покрывающий):
-- 1. Найти в индексе id=123
-- 2. Взять указатель
-- 3. Прочитать строку из таблицы
-- 4. Вернуть name, email

-- Покрывающий индекс на (id, name, email):
CREATE INDEX idx_users_covering ON users (id) INCLUDE (name, email);
-- 1. Найти в индексе id=123
-- 2. Вернуть name, email (уже в индексе)
-- 3. Нет обращения к таблице!
```

### PostgreSQL: INCLUDE (Covering Index)

```sql
-- INCLUDE добавляет колонки в листья индекса, но не в дерево поиска
CREATE INDEX idx_orders_user_date ON orders (user_id) INCLUDE (total, status);

-- Запрос использует индекс без обращения к таблице
SELECT user_id, total, status FROM orders WHERE user_id = 123;
```

### MySQL: Covering Index

```sql
-- Если индекс содержит все нужные колонки
CREATE INDEX idx_users_covering ON users (id, name, email);

-- EXPLAIN покажет "Using index" (не "Using index condition")
EXPLAIN SELECT name, email FROM users WHERE id = 123;
```

### Преимущества покрывающих индексов

| Преимущество | Объяснение |
|--------------|------------|
| **Скорость** | В 2–10x быстрее (нет random I/O в таблицу) |
| **Снижение нагрузки** | Меньше чтений с диска |
| **Улучшение p99** | Предсказуемая задержка |

### Недостатки

| Недостаток | Объяснение |
|------------|------------|
| **Размер** | Индекс занимает больше места |
| **Запись медленнее** | Нужно обновлять больше данных |
| **Дублирование** | Те же данные в индексе и таблице |

## 6. Разряженные индексы (Sparse Index)

**Что это:**
Индекс, который содержит **не все строки**, а только часть. Экономит место, но может потребовать обращения к таблице.

### Виды разреженности

**1. По условию (Partial Index)**
Индекс только для строк, удовлетворяющих условию.

```sql
-- Индекс только для активных пользователей
CREATE INDEX idx_users_active ON users (last_login) WHERE status = 'active';

-- Запрос использует только маленький индекс
SELECT * FROM users WHERE status = 'active' AND last_login > '2024-01-01';
```

**2. По объему (Sample)**
Не все строки включены (редко, только для очень больших таблиц).

**3. Отложенная запись (Write-once)**
Индекс строится не сразу, а после накопления данных.

### Partial Index (условный индекс)

**Когда применять:**
- **Фильтрация редких значений:** Индекс для `deleted = false`, когда большинство строк удалены
- **Мягкое удаление:** Индекс только для активных записей
- **Архивные данные:** Индекс только для последнего месяца

**Пример:**
```sql
-- 90% пользователей inactive, нужен быстрый поиск только по active
CREATE INDEX idx_users_active_email ON users (email) WHERE status = 'active';

-- Быстрый поиск (использует маленький индекс)
SELECT * FROM users WHERE status = 'active' AND email LIKE '%@gmail.com';

-- Поиск по inactive (не использует индекс, но это редко)
SELECT * FROM users WHERE status = 'inactive' AND email LIKE '%@gmail.com';
```

### Преимущества Partial Index

| Преимущество | Объяснение |
|--------------|------------|
| **Меньше места** | Индекс содержит только нужные строки |
| **Быстрее обновление** | Меньше строк в индексе |
| **Лучшая производительность чтения** | Меньше индексных страниц для сканирования |

---

## Сводная таблица: выбор типа индекса

| Сценарий | Тип индекса | Почему |
|----------|-------------|--------|
| **Точный поиск по ключу** | Hash | O(1) скорость |
| **Диапазонные запросы** | B-Tree | Поддерживает `<`, `>`, `BETWEEN` |
| **Сортировка (`ORDER BY`)** | B-Tree | Данные уже отсортированы |
| **Низкая селективность (статус, пол)** | Bitmap | Эффективные битовые операции |
| **Геопоиск (Nearby)** | Spatial (R-Tree) | Поддержка пространственных запросов |
| **Полнотекстовый поиск** | Reversed (Inverted) | Основа поисковых движков |
| **Поиск по суффиксам** | Reversed | `LIKE '%pattern'` |
| **Запросы с функциями** | Functional | `LOWER()`, `DATE()` |
| **Частые запросы с фиксированным набором колонок** | Covering | Нет обращения к таблице |
| **Только часть строк (active, не deleted)** | Partial (Sparse) | Меньше места, быстрее |

## Практические рекомендации для интервью

### 1. Когда создавать индекс

✅ **Создавайте индекс, если:**
- Колонка часто используется в `WHERE`, `JOIN`, `ORDER BY`
- Селективность высокая (>5-10% уникальных значений)
- Соотношение чтения к записи > 10:1

❌ **Не создавайте индекс, если:**
- Колонка редко используется в запросах
- Низкая селективность (<5%) и нет покрывающего индекса
- Таблица часто обновляется (индексы замедляют запись)
- Есть очень много индексов (каждый замедляет запись)

### 2. Анализ запросов (EXPLAIN)

В интервью можно упомянуть, что вы анализируете запросы через `EXPLAIN`:

```sql
EXPLAIN (ANALYZE, BUFFERS) 
SELECT * FROM users WHERE email = 'user@example.com';
```

Смотрим:
- **Seq Scan** → full table scan (плохо для больших таблиц)
- **Index Scan** → индексный поиск (хорошо)
- **Index Only Scan** → покрывающий индекс (отлично)
- **Bitmap Index Scan** → битовая карта (хорошо для нескольких условий)

### 3. Составные индексы (Composite Indexes)

Порядок колонок важен:

```sql
-- Индекс (status, created_at)
CREATE INDEX idx_orders_status_created ON orders (status, created_at);

-- Хорошо: используется статус
SELECT * FROM orders WHERE status = 'completed' AND created_at > '2024-01-01';

-- Хорошо: используется статус
SELECT * FROM orders WHERE status = 'completed';

-- Плохо: статус не указан (индекс не используется)
SELECT * FROM orders WHERE created_at > '2024-01-01';
```

**Правило:** Помещайте сначала колонки с высокой селективностью или часто используемые в условиях.

### 4. На интервью

> *«Для таблицы заказов я создам составной индекс на (user_id, created_at). Это позволит быстро находить заказы пользователя с сортировкой по дате. Для запросов, которые возвращают только user_id, total и status, я добавлю покрывающий индекс: (user_id, created_at) INCLUDE (total, status). Это даст Index Only Scan без обращения к таблице. Для статусных отчетов (WHERE status = 'completed') с низкой селективностью рассмотрю partial index только для completed заказов»*.

# ТРАНЗАКЦИИ

## Что это

**Транзакция** — это логическая единица работы, которая выполняется как единое целое: либо полностью, либо не выполняется вовсе.

Классический пример — перевод денег:
1. Списать 100₽ со счета А
2. Зачислить 100₽ на счет Б

Если после первого шага произошел сбой, транзакция откатывается — деньги не теряются.

---

## 1. WAL (Write-Ahead Logging)

### Что это

**WAL** — это механизм, обеспечивающий **долговечность (Durability)** транзакций. Принцип: **сначала пишем в журнал, потом в данные**.

### Как работает

```
1. Приходит запрос на изменение данных
2. Запись добавляется в WAL (журнал) на диск
3. После подтверждения записи в WAL, транзакция считается зафиксированной
4. Данные в основной таблице обновляются позже (асинхронно)
```

### Почему это важно

| Сценарий | Без WAL | С WAL |
|----------|---------|-------|
| **Сбой после COMMIT** | Данные потеряны (были только в памяти) | Данные восстановлены из WAL |
| **Сбой во время транзакции** | Могут быть частичные изменения | WAL позволяет откатить незавершенные транзакции |
| **Восстановление после краша** | Непонятно, что было зафиксировано | WAL воспроизводится, данные восстановлены |

### Структура WAL

```
WAL (журнал) — это последовательный лог всех изменений:

| LSN | Transaction ID | Operation | Page | Old Value | New Value |
|-----|----------------|-----------|------|-----------|-----------|
| 100 | T1             | INSERT    | 42   | NULL      | ...       |
| 101 | T1             | UPDATE    | 15   | 100       | 150       |
| 102 | T2             | DELETE    | 87   | ...       | NULL      |
| 103 | T1             | COMMIT    | -    | -         | -         |
```

**LSN (Log Sequence Number)** — уникальный номер записи в журнале.

### Где используется

| БД | Реализация WAL |
|----|----------------|
| **PostgreSQL** | Полноценный WAL, используется также для репликации |
| **MySQL InnoDB** | redo log (аналог WAL) + undo log |
| **SQL Server** | Transaction log |
| **MongoDB** | WiredTiger storage engine использует WAL |
| **Cassandra** | Commit log (аналог WAL) |

### WAL и репликация

WAL используется не только для восстановления, но и для **репликации**:

```
Primary:
    Изменения → WAL → отправка WAL на реплики

Replica:
    Получение WAL → воспроизведение изменений
```

В PostgreSQL это называется **physical replication** (streaming replication).

### На интервью

> *«Для обеспечения durability мы используем механизм WAL (Write-Ahead Log). При каждом изменении данных сначала пишем в журнал на диск, и только после подтверждения считаем транзакцию зафиксированной. При сбое мы восстанавливаемся, проигрывая WAL с последней контрольной точки. WAL также используется для репликации — реплики читают и применяют те же записи»*.

## 2. Deferrable Constraints

### Что это

**Deferrable constraints** — это ограничения (constraints), которые проверяются **не сразу**, а в конце транзакции.

### Обычные (NOT DEFERRABLE) constraints

```sql
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INT REFERENCES users(id)  -- NOT DEFERRABLE по умолчанию
);

BEGIN;
INSERT INTO orders (user_id) VALUES (100);  -- Проверка foreign key СРАЗУ
-- Если user_id 100 не существует → ОШИБКА
COMMIT;
```

### Deferrable constraints

```sql
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INT REFERENCES users(id) DEFERRABLE INITIALLY DEFERRED
);

BEGIN;
INSERT INTO orders (user_id) VALUES (100);  -- Проверка НЕ выполняется
INSERT INTO users (id) VALUES (100);        -- Создаем пользователя
COMMIT;  -- Только здесь проверяется, что user_id 100 существует
```

### Типы deferrable constraints

| Тип | Описание |
|-----|----------|
| **INITIALLY IMMEDIATE** | Проверяется после каждой команды (как обычный) |
| **INITIALLY DEFERRED** | Проверяется только в конце транзакции |
| **SET CONSTRAINTS DEFERRED** | Можно переключать внутри транзакции |

### Когда применять

| Сценарий | Пример |
|----------|--------|
| **Циклические зависимости** | Таблица А ссылается на Б, Б ссылается на А — нельзя вставить без deferrable |
| **Пакетная вставка связанных данных** | Сначала вставляем заказы, потом пользователей (если order.user_id → users.id) |
| **Сложные миграции** | Обновление схемы с новыми foreign keys |

### Пример циклической зависимости

```sql
CREATE TABLE employee (
    id SERIAL PRIMARY KEY,
    name TEXT,
    manager_id INT REFERENCES employee(id) DEFERRABLE INITIALLY DEFERRED
);

BEGIN;
INSERT INTO employee (id, name, manager_id) VALUES (1, 'Alice', 2);
INSERT INTO employee (id, name, manager_id) VALUES (2, 'Bob', 1);
COMMIT;  -- ✅ OK, проверка в конце транзакции
```

### На интервью

> *«При импорте больших объемов связанных данных мы используем DEFERRABLE constraints. Это позволяет сначала загрузить все данные (даже если порядок нарушает foreign keys), а затем проверить целостность в конце транзакции. Также это полезно при циклических зависимостях, например, в иерархических структурах»*.

## 3. MVCC / 2PL

Это два основных подхода к **управлению конкурентным доступом** и реализации изоляции транзакций.

### MVCC (Multi-Version Concurrency Control)

**Что это:**
БД хранит **несколько версий** одной строки. Читающие транзакции видят snapshot данных на момент начала транзакции, не блокируя пишущие.

**Как работает:**

```
Время:
t1: Транзакция A начинает чтение → видит версию v1
t2: Транзакция B обновляет строку → создает версию v2
t3: Транзакция A продолжает чтение → все еще видит v1 (стабильный snapshot)
t4: Транзакция B коммитится → v2 становится видимой для новых транзакций
```

**Структура строки с версиями:**

| xmin (создано транзакцией) | xmax (удалено транзакцией) | data |
|----------------------------|----------------------------|------|
| T1 | NULL | значение v1 |
| T2 | NULL | значение v2 |

- **xmin** — ID транзакции, которая создала эту версию
- **xmax** — ID транзакции, которая удалила эту версию (NULL если активна)

**Преимущества MVCC:**
- **Читатели не блокируют писателей** (и наоборот)
- Высокая конкурентность
- Нет блокировок для чтения

**Недостатки:**
- **Управление версиями:** Нужно чистить старые версии (VACUUM в PostgreSQL)
- **Увеличение размера данных:** Старые версии хранятся до очистки

**Где используется MVCC:**
- **PostgreSQL** — полноценный MVCC
- **MySQL InnoDB** — MVCC с undo log
- **Oracle** — MVCC
- **MongoDB** — MVCC (WiredTiger)
- **Cassandra** — MVCC для tombstone management

---

### 2PL (Two-Phase Locking)

**Что это:**
Классический механизм на основе блокировок. Все операции чтения и записи требуют получения блокировок. Блокировки **удерживаются до конца транзакции**.

**Две фазы:**

| Фаза | Что происходит |
|------|----------------|
| **Фаза 1: Growing Phase** | Транзакция получает блокировки, но не отпускает |
| **Фаза 2: Shrinking Phase** | Транзакция отпускает блокировки, но не может получить новые |

```
Время:
├─ Growing Phase ────────────┼─ Shrinking Phase ──┤
   ↑                          ↑                    ↑
   получаем блокировки       первое освобождение   конец
   (можно только LOCK)       (можно только UNLOCK)
```

**Типы блокировок:**

| Тип | Назначение |
|-----|------------|
| **Shared Lock (S)** | Для чтения. Много транзакций могут читать одновременно |
| **Exclusive Lock (X)** | Для записи. Только одна транзакция может иметь X-lock |

**Совместимость:**
|        | S | X |
|--------|---|---|
| **S**  | ✅ | ❌ |
| **X**  | ❌ | ❌ |

**Проблема 2PL:**
- **Deadlocks:** Транзакции могут заблокировать друг друга

**Где используется 2PL:**
- **MySQL InnoDB** (в комбинации с MVCC)
- **SQL Server** (для некоторых операций)
- **PostgreSQL** (для операций, требующих строгой сериализуемости)

---

### MVCC vs 2PL: сравнение

| Характеристика | MVCC | 2PL |
|----------------|------|-----|
| **Конкурентность** | Высокая (читатели не блокируются) | Средняя (блокировки) |
| **Читатели vs писатели** | Не блокируют друг друга | Могут блокировать |
| **Deadlocks** | Редко (обычно при обновлениях) | Часто |
| **Накладные расходы** | Управление версиями, очистка | Управление блокировками |
| **Snapshot reads** | Естественно | Нет |
| **SERIALIZABLE** | SSI (Serializable Snapshot Isolation) | Strict 2PL |

## 4. Изоляции транзакций

Это **четыре уровня**, определяющих, какие аномалии допустимы. Стандарт SQL определяет 4 уровня.

### Аномалии (проблемы) конкурентного доступа

| Аномалия | Описание | Пример |
|----------|----------|--------|
| **Dirty Read** | Чтение незафиксированных данных другой транзакции | Транзакция A читает данные, которые B еще не закоммитила, а B потом откатывается |
| **Non-Repeatable Read** | Повторное чтение дает другой результат из-за UPDATE другой транзакции | A читает строку, B обновляет, A читает снова → другое значение |
| **Phantom Read** | Повторное чтение с WHERE дает другое количество строк из-за INSERT/DELETE другой транзакции | A читает список заказов, B добавляет новый, A читает снова → новые строки |
| **Serialization Anomaly** | Результат набора транзакций не эквивалентен никакому последовательному выполнению | Сложные аномалии при конкурентных операциях |

---

### Уровни изоляции

| Уровень | Dirty Read | Non-Repeatable Read | Phantom Read | Serialization Anomaly |
|---------|------------|---------------------|--------------|----------------------|
| **READ UNCOMMITTED** | ✅ (возможна) | ✅ | ✅ | ✅ |
| **READ COMMITTED** | ❌ | ✅ | ✅ | ✅ |
| **REPEATABLE READ** | ❌ | ❌ | ⚠️ (зависит от БД) | ✅ |
| **SERIALIZABLE** | ❌ | ❌ | ❌ | ❌ |

---

### 4.1. READ UNCOMMITTED (грязное чтение)

**Что это:**
Самый низкий уровень. Транзакция может читать незафиксированные изменения других транзакций.

**Пример:**
```
Транзакция A: UPDATE accounts SET balance = balance - 100 WHERE id = 1;
Транзакция B: SELECT balance FROM accounts WHERE id = 1;  -- видит -100
Транзакция A: ROLLBACK;  -- но B уже считал несуществующий баланс
```

**Когда использовать:**
- Практически никогда в production
- Может быть для очень больших таблиц, где допустима неточность (например, приблизительные агрегаты)

**Где поддерживается:**
- PostgreSQL: READ UNCOMMITTED работает как READ COMMITTED
- MySQL: поддерживается
- Oracle: не поддерживает

---

### 4.2. READ COMMITTED (чтение зафиксированных данных)

**Что это:**
Стандартный уровень в большинстве БД. Транзакция видит только **зафиксированные** данные. Но повторное чтение может дать другой результат.

**Как работает:**
- Каждый запрос видит snapshot на момент **начала запроса**, а не транзакции
- Dirty Read исключены

**Пример проблемы (Non-Repeatable Read):**
```
Транзакция A (READ COMMITTED):
    BEGIN;
    SELECT balance FROM accounts WHERE id = 1;  -- 100

Транзакция B:
    BEGIN;
    UPDATE accounts SET balance = 200 WHERE id = 1;
    COMMIT;

Транзакция A:
    SELECT balance FROM accounts WHERE id = 1;  -- 200 (другое значение!)
    COMMIT;
```

**Когда использовать:**
- Большинство OLTP приложений
- Когда допустимы небольшие неконсистентности при повторных чтениях
- Высокая производительность

**Где по умолчанию:**
- **PostgreSQL:** READ COMMITTED (по умолчанию)
- **Oracle:** READ COMMITTED (по умолчанию)
- **SQL Server:** READ COMMITTED (по умолчанию)
- **MySQL:** REPEATABLE READ (по умолчанию)

---

### 4.3. REPEATABLE READ (повторяемое чтение)

**Что это:**
Гарантирует, что если строка была прочитана, повторное чтение даст **то же значение**. Но новые строки (phantom) могут появиться.

**Как работает:**
- В MVCC БД транзакция видит snapshot на момент **начала транзакции**
- Non-Repeatable Read исключен

**Пример Phantom Read (в MySQL):**
```
Транзакция A (REPEATABLE READ):
    BEGIN;
    SELECT COUNT(*) FROM orders WHERE user_id = 1;  -- 5

Транзакция B:
    BEGIN;
    INSERT INTO orders (user_id) VALUES (1);
    COMMIT;

Транзакция A:
    SELECT COUNT(*) FROM orders WHERE user_id = 1;  -- 6 (новые строки!)
    COMMIT;
```

**Различия между БД:**

| БД | REPEATABLE READ | Phantom Read |
|----|-----------------|--------------|
| **PostgreSQL** | Исключает phantom read (использует SSI) | Нет |
| **MySQL InnoDB** | Phantom read возможен (без gap locks) | Да |
| **Oracle** | Не поддерживает (READ COMMITTED или SERIALIZABLE) | — |

**Когда использовать:**
- Когда нужна консистентность при повторных чтениях
- Отчеты, где нужно стабильное представление данных

**Где по умолчанию:**
- **MySQL InnoDB:** REPEATABLE READ (по умолчанию)

---

### 4.4. SERIALIZABLE (сериализуемость)

**Что это:**
Самый строгий уровень. Транзакции выполняются так, как если бы они шли **последовательно** одна за другой.

**Как работает:**
- В MVCC БД: **SSI (Serializable Snapshot Isolation)**
- В 2PL БД: Strict 2PL (блокировки до конца)

**Пример проблемы, которую решает SERIALIZABLE:**

```
Два врача смотрят свободные слоты и записывают пациента:

Транзакция A: SELECT COUNT(*) FROM slots WHERE doctor_id=1 AND booked=false;  -- 1
Транзакция B: SELECT COUNT(*) FROM slots WHERE doctor_id=1 AND booked=false;  -- 1
Транзакция A: UPDATE slots SET booked=true WHERE id=10;
Транзакция B: UPDATE slots SET booked=true WHERE id=11;
Транзакция A: COMMIT;
Транзакция B: COMMIT;
-- Результат: оба записались, хотя был 1 слот! (serialization anomaly)

SERIALIZABLE: одна из транзакций получит ошибку "could not serialize access"
```

**Когда использовать:**
- Финансовые операции (нельзя перевести деньги дважды)
- Бронирование (нельзя продать один билет двум)
- Инвентаризация (нельзя продать товар, которого нет)

**Цена:**
- **Производительность ниже** (конфликты, откаты)
- **Retry logic нужна** (при ошибке сериализации повторить транзакцию)

---

### Сравнение уровней изоляции в PostgreSQL

| Уровень | Грязное чтение | Неповторяемое чтение | Фантомы | Сериализация |
|---------|----------------|---------------------|---------|--------------|
| READ UNCOMMITTED | ❌ (как READ COMMITTED) | ✅ | ✅ | ✅ |
| READ COMMITTED | ❌ | ✅ | ✅ | ✅ |
| REPEATABLE READ | ❌ | ❌ | ❌ | ✅ |
| SERIALIZABLE | ❌ | ❌ | ❌ | ❌ |

В PostgreSQL REPEATABLE READ исключает фантомы благодаря MVCC.

---

### Выбор уровня изоляции на интервью

| Сценарий | Рекомендуемый уровень | Почему |
|----------|----------------------|--------|
| **Платежи, переводы** | SERIALIZABLE | Недопустимы аномалии, двойные списания |
| **Бронирование (билеты, отели)** | SERIALIZABLE | Нельзя продать один товар дважды |
| **Большинство веб-приложений** | READ COMMITTED | Хороший баланс производительности и консистентности |
| **Отчеты, аналитика** | REPEATABLE READ | Стабильный snapshot на время отчета |
| **Чтение статистики (приблизительно)** | READ UNCOMMITTED | Максимальная скорость, допустима неточность |

### На интервью

> *«Для нашей платежной системы мы используем уровень изоляции SERIALIZABLE. Это гарантирует, что даже при конкурентных переводах с одного счета не возникнет отрицательного баланса или двойного списания. Цена — возможные конфликты и необходимость retry логики на приложении, но для финансовых операций это оправдано»*.

## 1. Сжатие данных (Data Compression)

**Что это:**
Уменьшение объема данных, занимаемого на диске или передаваемого по сети, за счет алгоритмов сжатия.

**Зачем:**
- Снижение затрат на хранение (диски дешевле, но не бесплатны)
- Ускорение I/O (меньше данных читается/пишется с диска)
- Уменьшение сетевого трафика (при репликации, бэкапах, передаче клиентам)

### Где применяется

| Слой | Пример |
|------|--------|
| **Базы данных** | PostgreSQL TOAST (сжатие больших полей), InnoDB page compression, ClickHouse (колоночное сжатие по умолчанию) |
| **Файловые системы** | ZFS, Btrfs со сжатием на лету |
| **Объектное хранилище** | S3 интеллектуальное сжатие, Parquet/ORC форматы в Data Lake |
| **Очереди сообщений** | Kafka поддерживает сжатие (gzip, snappy, lz4, zstd) на уровне продюсера |
| **Сеть** | HTTP compression (gzip, brotli) между клиентом и сервером |

### Алгоритмы сжатия

| Алгоритм | Степень сжатия | Скорость | Использование |
|----------|----------------|----------|---------------|
| **gzip** | Высокая | Средняя | Универсальный, стандарт для HTTP |
| **brotli** | Очень высокая | Средняя | HTTP (современные браузеры), лучше чем gzip |
| **lz4** | Низкая/средняя | Очень высокая | Kafka, базы данных (нужна скорость) |
| **snappy** | Средняя | Высокая | Kafka, Google-экосистема |
| **zstd (Zstandard)** | Высокая | Высокая | Современный выбор, баланс скорости и степени |
| **Parquet/ZSTD** | Очень высокая | Средняя | Аналитические БД, Data Lake |

### Компромиссы

- **CPU vs Storage:** Сжатие потребляет CPU, но экономит диск и I/O. Для аналитических систем (ClickHouse) это выгодно, для high-frequency trading — нет.
- **Размер блока:** Мелкие объекты плохо сжимаются. Для них выгода минимальна.
- **Тип данных:** Текстовые (JSON, XML, логи) сжимаются отлично (5–10x). Бинарные, изображения, видео — плохо.

**На интервью:** *«Для логов и исторических данных мы используем сжатие zstd — хороший баланс между степенью сжатия и скоростью. Для Kafka между микросервисами включаем lz4 — он быстрее, а накладные расходы на CPU минимальны»*.

---

## 2. Охлаждение данных (Data Cooling / Tiered Storage)

**Что это:**
Разделение данных на «горячие» (часто используемые) и «холодные» (редко используемые) с хранением на разных носителях.

### Иерархия хранения

| Уровень | Носитель | Стоимость | Скорость | Срок хранения |
|---------|----------|-----------|----------|---------------|
| **Hot** | NVMe SSD, RAM | Высокая | Микросекунды | Часы/дни |
| **Warm** | SSD (SATA) | Средняя | Миллисекунды | Недели/месяцы |
| **Cold** | HDD | Низкая | Десятки мс | Месяцы |
| **Archive** | Ленточные библиотеки, S3 Glacier | Очень низкая | Минуты/часы | Годы |

### Стратегии охлаждения

| Стратегия | Как работает | Пример |
|-----------|--------------|--------|
| **TTL / Retention** | Данные автоматически удаляются после истечения срока | Логи хранятся 30 дней в Elasticsearch hot, затем удаляются |
| **Tiered Storage** | Данные автоматически перемещаются между уровнями по мере старения | Kafka: последние 24 часа в SSD, старше — в S3 |
| **Archival** | Данные выгружаются в холодное хранилище, могут быть восстановлены по запросу | Старые заказы выгружаются из PostgreSQL в S3 Glacier |
| **Сжатие при охлаждении** | Холодные данные хранятся в максимально сжатом виде | Parquet + zstd в Data Lake |

### Применение

- **Kafka Tiered Storage:** Старые сегменты топиков выгружаются в S3, снижая стоимость хранения на 80%+.
- **ClickHouse:** Автоматическое перемещение партиций между томами (SSD → HDD).
- **TimescaleDB:** Continuous aggregates + data retention.
- **S3 Lifecycle Policies:** Автоматический переход из S3 Standard в S3 Infrequent Access → Glacier.

**На интервью:** *«Для хранения логов мы используем tiered storage. Последние 7 дней — в горячем хранилище (быстрый поиск). После 7 дней данные сжимаются и перемещаются в холодное хранилище (S3 Glacier), где хранятся 90 дней. Это снижает стоимость хранения примерно в 5 раз»*.

## 3. Запись батчами (Batch Writes)

**Что это:**
Группировка множества операций записи в один запрос к хранилищу вместо отправки каждой операции по отдельности.

### Зачем

| Проблема одиночных записей | Решение через батчи |
|----------------------------|---------------------|
| Высокий сетевой overhead (RTT на каждую запись) | Один RTT на N записей |
| Много мелких I/O операций | Один крупный I/O |
| Низкая пропускная способность | Рост throughput в разы |
| Высокая нагрузка на CPU (обработка каждого запроса) | Снижение CPU |

### Примеры

| Хранилище | Механизм батчирования |
|-----------|------------------------|
| **PostgreSQL** | `INSERT INTO table VALUES (1), (2), (3)`; COPY command |
| **Cassandra / ScyllaDB** | Batch statements (но с осторожностью — не для разных партиций) |
| **Kafka** | Продюсеры батчат сообщения (batch.size, linger.ms) |
| **Redis** | Pipelining, MSET, HMSET |
| **Elasticsearch** | Bulk API (одна операция на множество документов) |
| **ClickHouse** | Вставка пачками по 10k–100k строк |

### Компромиссы

| Плюсы | Минусы |
|-------|--------|
| Высокая пропускная способность | Задержка одной записи увеличивается (ждем накопления батча) |
| Снижение нагрузки на БД | Риск атомарности (в некоторых системах батч может быть частично применен) |
| Экономия сетевых ресурсов | Сложнее обработка ошибок (какая запись в батче упала?) |

**Практика:**
- **Время накопления:** Обычно 10–100 мс или по количеству записей (например, 1000 записей или 100 мс, что наступит раньше)
- **Размер батча:** Подбирается под конкретное хранилище (для Kafka — до 1 МБ, для ClickHouse — 10k–100k строк)

**На интервью:** *«Для записи событий аналитики мы используем батчирование на стороне продюсера: накапливаем события в памяти в течение 50 мс или до 500 событий, затем отправляем в ClickHouse одной вставкой. Это повышает throughput в 20 раз по сравнению с построчной вставкой»*.

## 4. Расчет железа (Capacity Planning)

**Что это:**
Оценка необходимых ресурсов (CPU, RAM, диск, сеть) для поддержки требуемой нагрузки.

### Что нужно считать

| Параметр | Как считать |
|----------|-------------|
| **RPS (requests per second)** | Пиковая нагрузка (обычно 2–5x от средней) |
| **Данные на один запрос** | Средний размер запроса и ответа |
| **Сетевой трафик** | RPS × размер (входящий + исходящий) |
| **CPU** | Зависит от логики. Обычно 2–4 vCPU на 1k–10k RPS (в зависимости от сложности) |
| **RAM** | Рабочий набор данных (working set) + буферы + ОС |
| **Диск** | Хранение данных + индексы + WAL/logs + свободное место (обычно +30%) |
| **IOPS** | Для transactional БД критичны случайные операции (SSD обязателен) |

### Пример расчета для PostgreSQL

* «У нас 1 млн активных пользователей, каждый делает 10 запросов в день. Пик — 10x от среднего. Средний размер строки — 2 КБ, храним 3 года.»*

```
Средний RPS = 1M × 10 / 86400 ≈ 115 RPS
Пиковый RPS = 115 × 10 ≈ 1150 RPS

Хранение:
1M пользователей × 2 КБ = 2 ГБ (пользователи)
Количество операций в день = 10M записей
3 года = 10M × 365 × 3 ≈ 11 млрд записей
Объем = 11B × 2 КБ ≈ 22 ТБ
+ индексы (обычно 50–100% от данных) = 33–44 ТБ
+ 30% запас = 43–57 ТБ

Железо для PostgreSQL:
- Мастер: 16 vCPU, 64 ГБ RAM
- Реплики: 2 шт по 16 vCPU, 64 ГБ RAM
- Диск: NVMe SSD, 60 ТБ (можно шардировать по 10 ТБ на шард)
```

### Важные принципы

| Принцип | Объяснение |
|---------|------------|
| **Запас (buffer)** | Всегда добавляйте 20–30% к расчетам. Нагрузка всегда растет быстрее ожиданий |
| **Разделение compute/storage** | Где возможно, используйте облачные решения с отдельным масштабированием (AWS Aurora, Snowflake) |
| **Бюджетирование** | Если система приносит деньги, железо должно быть достаточным. Если нет — ищем баланс |

**На интервью:** *«Оценка нагрузки: пиковые 10k RPS, средний размер ответа 50 КБ → сетевой трафик 500 МБ/с. Хранилище: 1 млрд объектов по 1 КБ = 1 ТБ, с репликацией и бэкапами — 3 ТБ. База данных: рабочая нагрузка read-heavy, достаточно мастера и одной read replica на 8 vCPU, 32 ГБ RAM. Для кэша — Redis кластер из 3 нод по 16 ГБ»*.

## 5. Хранимые процедуры (Stored Procedures)

**Что это:**
Код (SQL, PL/pgSQL, etc.), который хранится и выполняется на стороне базы данных.

### Когда применять

| Сценарий | Почему |
|----------|--------|
| **Сложная бизнес-логика в БД** | Многоступенчатые обновления, каскадные операции |
| **Минимизация сетевых вызовов** | Один вызов вместо N round-trips |
| **Пакетная обработка** | Обработка больших объемов данных внутри БД без пересылки на приложение |
| **Атомарность операций** | Несколько операций в одной транзакции без риска race conditions |

### Аргументы ПРОТИВ (в микросервисной архитектуре)

| Аргумент | Объяснение |
|----------|------------|
| **Связывание с конкретной БД** | Смена БД (PostgreSQL → CockroachDB) становится невозможной |
| **Сложность версионирования** | Код в БД сложнее версионировать, тестировать, деплоить |
| **Масштабирование** | Хранимые процедуры масштабируются только вертикально (БД) |
| **Нарушение принципов микросервисов** | Бизнес-логика оказывается не в сервисе, а в БД |

### Современный подход

- **Хранимые процедуры** допустимы для:
    - Тривиальных операций с высокой производительностью
    - Пакетной обработки внутри ETL
    - Сложных валидаций целостности данных
- **Бизнес-логика** должна быть в приложении, а не в БД

**На интервью:** *«Хранимые процедуры мы используем ограниченно — только для операций, где критична производительность и атомарность, например, для каскадного обновления счетчиков (лайков, подписчиков). Основная бизнес-логика остается в приложении, чтобы сохранять гибкость архитектуры»*.

## 6. Materialized View (Материализованные представления)

**Что это:**
Результат запроса, сохраненный на диске как физическая таблица. Обновляется либо по расписанию, либо при изменении исходных данных (инкрементально).

### Обычное View vs Materialized View

| Обычное View | Materialized View |
|--------------|-------------------|
| Виртуальная таблица (хранится только запрос) | Физическая таблица (хранятся данные) |
| Всегда актуальна (читает исходные таблицы) | Может быть stale (если не обновлена) |
| Каждый запрос выполняет запрос | Запрос читает готовые данные (быстро) |

### Применение

| Сценарий | Пример |
|----------|--------|
| **Агрегации и отчеты** | Ежедневные продажи по категориям |
| **Денормализация для быстрых чтений** | Готовые данные для API |
| **Сложные JOIN-ы** | Объединение 5 таблиц в одну для чтения |
| **Кэширование сложных запросов** | Альтернатива внешнему кэшу (Redis) |

### Обновление Materialized View

| Способ | Описание | Задержка |
|--------|----------|----------|
| **Full refresh** | Полное перестроение (TRUNCATE + INSERT) | Высокая, блокирует чтение |
| **Incremental refresh** | Обновление только измененных данных | Низкая |
| **Trigger-based** | Триггеры на исходных таблицах обновляют MV | Минимальная, но сложно |
| **По расписанию** | CRON: каждые 5 минут / каждый час | Зависит от интервала |

### Инструменты

| БД | Поддержка MV |
|----|--------------|
| **PostgreSQL** | Поддерживает, но обновление только полное (CONCURRENTLY для неблокирующего) |
| **ClickHouse** | Материализованные представления с инкрементальным обновлением |
| **Snowflake** | Dynamic Tables (инкрементальное обновление) |
| **BigQuery** | Материализованные представления с автообновлением |

**На интервью:** *«Для дашборда с аналитикой по продажам мы используем материализованные представления, которые обновляются каждые 5 минут. Это позволяет показывать агрегированные данные (продажи по категориям, топ-товары) без выполнения тяжелых запросов к сырым данным в реальном времени»*.

## 7. Триггеры (Triggers)

**Что это:**
Код (процедура), который автоматически выполняется при наступлении определенного события в БД (INSERT, UPDATE, DELETE).

### Типы триггеров

| Тип | Событие | Пример |
|-----|---------|--------|
| **BEFORE** | До выполнения операции | Валидация данных, установка default значений |
| **AFTER** | После выполнения операции | Логирование изменений, обновление счетчиков |
| **INSTEAD OF** | Вместо операции | Для представлений (views) |

### Когда применять

| Сценарий | Описание |
|----------|----------|
| **Аудит (Audit Log)** | Автоматическое логирование всех изменений важных таблиц |
| **Обновление денормализованных данных** | При изменении родителя обновить счетчик в родительской таблице |
| **Валидация сложной целостности** | Проверки, которые не покрываются foreign keys |
| **Soft delete каскады** | При удалении пользователя пометить все его записи как deleted |

### Проблемы и риски

| Проблема | Объяснение |
|----------|------------|
| **Неявное поведение** | Разработчики могут не знать, что триггер делает дополнительную работу |
| **Производительность** | Триггеры выполняются в транзакции и увеличивают время операции |
| **Каскадные триггеры** | Триггер может вызвать другой триггер, сложно отлаживать |
| **Масштабирование** | Триггеры привязаны к одной БД, не работают в шардированной архитектуре |

### Современный подход

В микросервисной архитектуре триггеры часто заменяются на:
- **Application-level события** — при изменении данных сервис публикует событие в Kafka
- **CDC (Change Data Capture)** — Debezium читает WAL и публикует изменения в Kafka
- **Явная логика в сервисе** — вместо неявных триггеров

**На интервью:** *«Триггеры мы используем осторожно — в основном для аудита (автоматическое логирование изменений) и для поддержания целостности в legacy-модулях. В новой архитектуре мы выносим такую логику на уровень приложения или используем CDC для публикации событий об изменениях»*.

## 1. Сценарии использования (Use Cases)

Очереди сообщений — это backbone асинхронной архитектуры.

### Основные сценарии

| Сценарий | Описание | Пример |
|----------|----------|--------|
| **Разделение сервисов (decoupling)** | Сервисы общаются через очередь, не зная друг о друге | Сервис заказов → очередь → сервис доставки |
| **Сглаживание пиков (load buffering)** | Очередь принимает пиковую нагрузку, consumer-ы обрабатывают в своем темпе | Black Friday: заказы идут в Kafka, а не в БД напрямую |
| **Асинхронная обработка** | Операции, не требующие немедленного результата | Отправка email, генерация отчетов, транскодинг видео |
| **Распределение задач (fan-out)** | Одно сообщение обрабатывается несколькими consumer-ами | Событие «новый заказ» → уведомление складу, аналитике, CRM |
| **Гарантированная доставка** | Сообщения не теряются, даже если consumer временно недоступен | Платежные транзакции, аудит-логи |
| **Событийная архитектура (event-driven)** | Микросервисы реагируют на события | Новый пользователь → welcome email + создание корзины + аналитика |

### Пример архитектуры с очередями

```
[API Gateway]
      │
      ▼
[Order Service] ──► Kafka (orders topic)
      │                    │
      │                    ├──► [Payment Service]
      │                    ├──► [Inventory Service]
      │                    ├──► [Notification Service]
      │                    └──► [Analytics Service]
      │
      └──► PostgreSQL (order storage)
```

**На интервью:** *«Очереди сообщений мы используем для асинхронной обработки: при создании заказа мы синхронно сохраняем его в БД и отправляем событие в Kafka. Дальше сервисы платежей, инвентаризации и уведомлений обрабатывают его независимо. Это снижает задержку для клиента и делает систему устойчивой к сбоям отдельных компонентов»*.

## 2. Гарантии доставки (Delivery Guarantees)

Это ключевой выбор при проектировании: сколько раз сообщение будет доставлено и что произойдет при сбоях.

### Три уровня гарантий

| Гарантия | Описание | Риски |
|----------|----------|-------|
| **At most once** | Сообщение доставляется 0 или 1 раз. Может быть потеряно | Потеря данных |
| **At least once** | Сообщение доставляется 1 или более раз. Может быть дублировано | Дубликаты (нужна идемпотентность) |
| **Exactly once** | Сообщение доставляется строго 1 раз | Высокая сложность, низкая производительность |

### Как это реализовано в популярных системах

| Система | Гарантии | Механизм |
|---------|----------|----------|
| **Kafka** | Все три (настраиваются) | acks=all + транзакции + идемпотентный продюсер |
| **RabbitMQ** | At most once, At least once | Подтверждения (ack), persistent queues |
| **AWS SQS** | At least once (по умолчанию) | Visibility timeout, dead letter queue |
| **AWS SQS FIFO** | Exactly once (в рамках группы) | FIFO очередь, дедупликация |

### Настройки в Kafka

| Параметр | Значение | Гарантия |
|----------|----------|----------|
| `acks=0` | Продюсер не ждет подтверждения | At most once |
| `acks=1` | Ждет подтверждения от лидера | At least once (возможна потеря при падении лидера) |
| `acks=all` | Ждет подтверждения от всех реплик | At least once |
| **+ idempotent producer** | `enable.idempotence=true` | Exactly once в рамках партиции |
| **+ transactions** | Транзакции между несколькими партициями/топиками | Exactly once |

### Выбор гарантий

| Сценарий | Рекомендуемая гарантия | Почему |
|----------|------------------------|--------|
| **Аналитика, логи** | At most once | Потеря пары сообщений не критична, скорость важнее |
| **Платежи, заказы** | Exactly once или At least once + идемпотентность | Потеря данных недопустима, дубликаты тоже |
| **Уведомления (email)** | At least once + идемпотентность | Дубликат письма лучше, чем потеря |
| **Системы реального времени** | At most once | Минимальная задержка важнее гарантий |

**На интервью:** *«Для платежной системы мы используем exactly once семантику: идемпотентный продюсер в Kafka + потребители с идемпотентной обработкой (проверяем, не обработано ли сообщение по уникальному ID). Для аналитических событий допускаем at most once — потеря 0.01% событий не влияет на бизнес-решения»*.

## 3. Data Retention (Хранение данных в очереди)

**Что это:**
Срок и политика хранения сообщений в очереди/топике после того, как они были прочитаны (или не прочитаны).

### Retention в разных системах

| Система | Механизм retention | Типичные настройки |
|---------|---------------------|-------------------|
| **Kafka** | Время (retention.ms) или размер (retention.bytes) | 24 часа — 7 дней (для transient), до бесконечности для event sourcing |
| **RabbitMQ** | TTL для очереди и сообщений, Lazy queues | Сообщение удаляется после ack |
| **SQS** | Retention period: 1 минута — 14 дней | По умолчанию 4 дня |
| **Pulsar** | Tiered storage: hot (bookkeeper) + cold (S3) | Бесконечное хранение с выгрузкой в S3 |

### Kafka Tiered Storage

```
[Hot tier]     Последние 24 часа → NVMe SSD
     │
     ▼
[Warm tier]    24 часа — 7 дней → HDD
     │
     ▼
[Cold tier]    > 7 дней → S3 (дешево, бесконечно)
```

### Сценарии retention

| Сценарий | Retention | Объяснение |
|----------|-----------|------------|
| **Транзакционные события** | 7 дней | Достаточно для восстановления после сбоя |
| **Логи и метрики** | 30–90 дней | Аудит, дашборды |
| **Event sourcing** | Бесконечно | Журнал событий — источник истины |
| **Очереди задач** | Пока не обработаны | Dead letter queue для упавших задач |
| **CDC (Debezium)** | Долго (недели) | Для перечитывания изменений при перезапуске |

### Dead Letter Queue (DLQ)

**Что это:**
Специальная очередь для сообщений, которые не удалось обработать после нескольких попыток.

**Применение:**
1. Сообщение не обработалось (ошибка, валидация)
2. Повторные попытки (retries) с exponential backoff
3. После N неудач → сообщение отправляется в DLQ
4. Инженеры анализируют DLQ, исправляют ошибки, переигрывают сообщения

**На интервью:** *«Для очередей задач мы настраиваем retention в 7 дней — этого достаточно для восстановления после сбоя. Для сообщений, которые не удалось обработать после 3 попыток, используем Dead Letter Queue. Это позволяет не терять проблемные сообщения и анализировать причины ошибок отдельно»*.

## Связка с другими темами

| Тема | Связь с очередями сообщений |
|------|----------------------------|
| **Кэширование** | Очереди могут использоваться для асинхронной инвалидации кэша |
| **Микросервисы** | Очереди — основной способ асинхронной коммуникации |
| **Отказоустойчивость** | Очереди буферизируют сообщения, если consumer временно недоступен |
| **Репликация** | Kafka реплицирует партиции для отказоустойчивости |
| **CDC** | Debezium читает WAL БД и публикует изменения в Kafka |

1. **Оценить нагрузку** в цифрах (RPS, throughput, storage)
2. **Перевести нагрузку в ресурсы** (CPU, RAM, Disk, Network)
3. **Учесть отказоустойчивость и рост** (redundancy, buffer, scaling)

# РАСЧЕТ ЖЕЛЕЗА (CAPACITY PLANNING)

## Подход: от нагрузки к ресурсам

Общая схема расчета:

```
Пользователи → RPS → Throughput → CPU / RAM / Disk / Network
```

---

## Шаг 1. Оценка нагрузки

### 1.1. Определяем метрики

| Метрика | Что это | Формула |
|---------|---------|---------|
| **DAU / MAU** | Активная аудитория | Daily / Monthly Active Users |
| **Requests per User** | Запросов на пользователя | Например, 10 запросов в день |
| **Average RPS** | Среднее число запросов в секунду | DAU × запросов_на_пользователя / 86400 |
| **Peak RPS** | Пиковое число запросов | Average RPS × peak_factor (обычно 3–10) |
| **Read / Write ratio** | Соотношение чтений и записей | Например, 10:1 |

### 1.2. Данные от интервьюера

В систем дизайн интервью вы должны **задать уточняющие вопросы**:

> *«Сколько у нас активных пользователей? Какой пиковый RPS? Каков размер одного запроса/ответа? Какое соотношение чтения и записи?»*

Если интервьюер не дает цифры — **предложите свои реалистичные допущения**:

> *«Давайте примем, что у нас 10 млн DAU, каждый делает 10 запросов в день, пиковая нагрузка в 5 раз выше средней. Размер одного запроса — 10 КБ, ответа — 50 КБ. Чтение:запись = 10:1»*.

---

### 1.3. Расчет RPS (Requests Per Second)

**Формула:**
```
Average RPS = DAU × запросов_в_день_на_пользователя / 86 400 секунд
Peak RPS = Average RPS × пик-фактор
```

**Пример:**
```
DAU = 10 000 000
Запросов в день на пользователя = 10
Average RPS = 10M × 10 / 86 400 ≈ 1157 RPS
Peak factor = 5
Peak RPS = 1157 × 5 ≈ 5800 RPS
```

**Отдельно для чтения и записи:**
```
Read RPS = Peak RPS × (read_ratio / (read_ratio + write_ratio))
Write RPS = Peak RPS × (write_ratio / (read_ratio + write_ratio))

При read:write = 10:1:
Read RPS = 5800 × 10/11 ≈ 5270 RPS
Write RPS = 5800 × 1/11 ≈ 530 RPS
```

---

### 1.4. Расчет Throughput (сетевой трафик)

**Формула:**
```
Inbound throughput = Write RPS × размер_запроса
Outbound throughput = Read RPS × размер_ответа
Total throughput = inbound + outbound
```

**Пример:**
```
Write RPS = 530
Размер запроса = 10 КБ = 0.01 МБ
Inbound = 530 × 0.01 ≈ 5.3 МБ/с

Read RPS = 5270
Размер ответа = 50 КБ = 0.05 МБ
Outbound = 5270 × 0.05 ≈ 263.5 МБ/с

Total = 5.3 + 263.5 ≈ 269 МБ/с ≈ 2.15 Гбит/с
```

**На интервью:** *«Исходящий трафик составляет около 270 МБ/с — это ~2.2 Гбит/с. Один балансировщик (Nginx) может обработать такой трафик, но для надежности мы поставим два»*.

---

## Шаг 2. Расчет CPU

CPU — сложная метрика, так как она сильно зависит от:
- Языка программирования (Go vs Python)
- Сложности операций (простой прокси vs ML инференс)
- Эффективности кода

### 2.1. Эвристики для оценки CPU

| Тип нагрузки | CPU на 1k RPS |
|--------------|---------------|
| **Простой прокси / кэш** | 0.5–1 vCPU |
| **REST API (CRUD, легкая логика)** | 1–2 vCPU |
| **Сложная бизнес-логика (много вычислений)** | 3–5 vCPU |
| **Шифрование / компрессия** | 2–4 vCPU |
| **ML инференс (CPU)** | 5–10 vCPU |
| **ML инференс (GPU)** | Считаем в GPU часах |

### 2.2. Расчет CPU для сервиса

**Формула:**
```
Total CPU = Peak RPS × CPU_per_1k_RPS / 1000
```

**Пример для REST API со сложной логикой:**
```
Peak RPS = 5800
CPU_per_1k_RPS = 3 vCPU
Total CPU = 5800 × 3 / 1000 = 17.4 vCPU
```

### 2.3. Учет запаса и отказоустойчивости

| Фактор | Коэффициент | Пояснение |
|--------|-------------|-----------|
| **Buffer (запас)** | ×1.2–1.3 | Чтобы не работать на 100% |
| **N+1 redundancy** | ×2 (если 2 ноды) | Для отказоустойчивости |
| **Rolling update** | ×1.2–1.3 | Чтобы выдерживать обновление без потери мощности |

**Пример:**
```
Сырая оценка = 17.4 vCPU
+ 30% buffer = 22.6 vCPU
N+1: берем 2 ноды → каждая 12 vCPU (итого 24 vCPU)
Или 3 ноды по 8 vCPU (24 vCPU)
```

**На интервью:** *«Для пиковой нагрузки 5800 RPS нам нужно около 18 vCPU. С учетом запаса и отказоустойчивости я возьму 3 инстанса по 8 vCPU — это даст 24 vCPU общей мощности и позволит безопасно обновлять сервис rolling update'ом»*.

---

## Шаг 3. Расчет RAM

RAM рассчитывается из трех компонентов:

```
Total RAM = Working Set + Application Memory + Buffer
```

### 3.1. Working Set (рабочий набор данных)

**Что это:** Данные, которые активно используются и должны находиться в памяти для быстрого доступа.

**Для кэша (Redis):**
```
Working Set = активные_объекты × средний_размер_объекта
```

**Пример:**
```
Активных пользователей в кэше = 5 млн (из 10 млн DAU)
Размер профиля = 2 КБ
Working Set = 5M × 2 КБ = 10 ГБ
```

**Для базы данных:**
- **In-memory БД (Redis):** весь набор данных должен помещаться в RAM
- **Дисковая БД (PostgreSQL):** достаточно буферного кэша для hot data (обычно 10–25% от общего объема данных)

**Правило для PostgreSQL:**
```
RAM для буферного кэша = 25% от общего объема данных (если данные > RAM)
или полный объем данных (если данные помещаются в RAM)
```

**Пример:**
```
Объем данных = 500 ГБ
Hot data (активно используемые) = 100 ГБ
Рекомендуемая RAM для буфера = 64–128 ГБ
```

### 3.2. Application Memory (память приложения)

| Тип приложения | RAM на 1 vCPU |
|----------------|---------------|
| **Go / Rust / C++** | 0.5–2 ГБ |
| **Java (JVM)** | 2–4 ГБ (плюс heap) |
| **Python / Ruby / Node.js** | 1–3 ГБ (зависит от нагрузки) |

**Пример для Java-сервиса:**
```
8 vCPU × 3 ГБ = 24 ГБ (JVM heap + overhead)
```

### 3.3. Итоговый расчет RAM

**Пример для сервиса на Java с Redis кэшем:**

| Компонент | Расчет |
|-----------|--------|
| Redis кэш (working set) | 10 ГБ |
| Redis overhead (RSS) | +30% = 13 ГБ |
| Приложение (8 vCPU × 3 ГБ) | 24 ГБ |
| Buffer (20%) | (13+24) × 0.2 = 7.4 ГБ |
| **Итого на ноду** | ≈ 45 ГБ |

**Выбор инстанса:** AWS r5.4xlarge (16 vCPU, 128 ГБ RAM) — запас есть.

**На интервью:** *«Redis требует 13 ГБ RAM для рабочего набора. Java-приложению на 8 vCPU нужно около 24 ГБ. Итого 37 ГБ плюс запас — выбираем инстансы с 64 ГБ RAM»*.

---

## Шаг 4. Расчет дискового хранилища

### 4.1. Объем данных

**Формула:**
```
Total Raw Data = объектов_в_день × размер_объекта × дней_хранения
```

**Пример для хранения постов:**
```
Новых постов в день = 1 000 000
Средний размер поста = 5 КБ
Срок хранения = 3 года = 1095 дней

Raw Data = 1M × 5 КБ × 1095 = 5.475 ТБ
```

### 4.2. Учет индексов и служебных данных

| Фактор | Коэффициент | Пояснение |
|--------|-------------|-----------|
| **Индексы (БД)** | ×1.5–2 | Индексы могут занимать больше данных |
| **WAL / журналы** | ×1.1–1.2 | Write-Ahead Log |
| **Репликация** | ×3 (для 3 реплик) | Если данные реплицируются |
| **Бэкапы** | ×1.5–2 | Полные + инкрементальные |
| **Свободное место** | ×1.2–1.3 | Для нормальной работы БД |

**Пример для PostgreSQL с репликацией:**
```
Raw Data = 5.5 ТБ
+ индексы (×1.5) = 8.25 ТБ
+ WAL (×1.1) = 9.1 ТБ
+ свободное место (×1.3) = 11.8 ТБ
+ 3 реплики = 11.8 × 3 = 35.4 ТБ
```

**На интервью:** *«Объем сырых данных за 3 года — 5.5 ТБ. С учетом индексов, WAL и запаса — около 12 ТБ на мастере. С тремя репликами общий объем хранилища — 36 ТБ»*.

---

### 4.3. IOPS (Input/Output Operations Per Second)

Критично для transactional БД (PostgreSQL, MySQL).

**Оценка IOPS:**
```
IOPS = Write RPS × write_amplification + Read RPS × read_amplification
```

| Тип нагрузки | IOPS на 1k RPS |
|--------------|----------------|
| **Точечные чтения по индексу** | 1–2 |
| **Сканирование таблицы** | 10–100 |
| **Вставка (без индексов)** | 1–2 |
| **Вставка (с индексами)** | 5–10 |

**Пример:**
```
Write RPS = 530, write_amplification = 5 → 2650 IOPS
Read RPS = 5270, read_amplification = 2 → 10 540 IOPS
Total IOPS ≈ 13 190

Вывод: нужен NVMe SSD (от 10k IOPS) или provisioned IOPS в облаке.
```

---

## Шаг 5. Расчет сети (Network)

### 5.1. Внутренний трафик (дата-центр)

| Компонент | Трафик |
|-----------|--------|
| **Сервис ↔ БД** | Запросы и ответы |
| **Сервис ↔ Кэш** | Запросы и ответы |
| **Репликация БД** | Весь объем записи (синхронно/асинхронно) |
| **Сервис ↔ Сервис** | gRPC / HTTP вызовы |

**Оценка:**
```
Сервис ↔ БД: 5800 RPS × 0.05 МБ ≈ 290 МБ/с
Репликация: 530 write RPS × 0.01 МБ = 5.3 МБ/с (но синхронно может дублироваться)
```

### 5.2. Внешний трафик (to clients)

Мы уже посчитали ранее: `≈ 270 МБ/с исходящего, 5 МБ/с входящего`.

### 5.3. Пропускная способность сети

| Тип инстанса | Сеть (Gbps) |
|--------------|-------------|
| Малые (t3, c5) | до 5 Gbps |
| Средние (c5.4xlarge) | до 10 Gbps |
| Крупные (c5.12xlarge) | до 25 Gbps |

Наш трафик ~2.2 Gbps → любой средний инстанс справится.

## Шаг 6. Учет отказоустойчивости и масштабирования

### 6.1. Факторы запаса

| Фактор | Коэффициент | Почему |
|--------|-------------|--------|
| **Buffer (не работать на пределе)** | ×1.2–1.3 | Оставляем место для пиков и роста |
| **N+1 redundancy** | +1 нода | Если одна нода падает, остальные выдерживают нагрузку |
| **Rolling update capacity** | +20% | Во время обновления одна нода недоступна |
| **Future growth** | ×1.5–2 на 1–2 года | Система будет расти |

### 6.2. Итоговый расчет ресурсов (пример)

**Исходные данные:**
- Peak RPS = 5800
- Чтение/запись = 10:1
- Хранение данных: 5.5 ТБ за 3 года
- Нужна отказоустойчивость

| Компонент | Расчет | Итог |
|-----------|--------|------|

| **Кэш (Redis)** | 10 ГБ working set → 3 ноды (кластер) | 3 × 8 ГБ RAM (16 ГБ нода) |
| **База данных (PostgreSQL)** | 12 ТБ данных + реплики | 1 мастер + 2 реплики, каждая: 16 vCPU, 64 ГБ RAM, 12 ТБ SSD |
| **Объектное хранилище (S3)** | Для файлов/медиа | По мере роста |

| Компонент | Инстанс | Кол-во | Мес. стоимость (AWS us-east-1) |

## Шаг 7. Готовый шаблон для интервью

Когда на интервью доходите до расчета ресурсов, используйте эту структуру:

> *«Давайте рассчитаем необходимые ресурсы. Я приму следующие допущения: [перечисляете].*

> *1. **RPS:** Средний RPS = X, пиковый = X × 5 = Y. Из них чтение = Z, запись = W.*

> *2. **Сеть:** Исходящий трафик = Y_read × размер_ответа = Z МБ/с, входящий = ...*

> *3. **CPU:** Для нашего типа нагрузки (CRUD со сложной логикой) нужно примерно 3 vCPU на 1000 RPS. Итого = Y × 3 / 1000 = N vCPU. С запасом 30% и N+1: возьмем M нод по K vCPU.*

> *4. **RAM:** Приложению на каждую ноду нужно L ГБ. Кэш (Redis) требует M ГБ рабочего набора. Итого на ноду = ...*

> *5. **Хранилище:** За X лет накопится Y ТБ данных. С учетом индексов, WAL и репликации нужно Z ТБ. Выбираем SSD с достаточным IOPS (около W IOPS).*

> *6. **Отказоустойчивость:** Ставим минимум 2 балансировщика, 3 ноды приложения (для rolling update), кластер Redis из 3 нод, мастер + 2 реплики PostgreSQL.*

> *Таким образом, нам потребуется: [суммарно ресурсов]».*

## Частые ошибки на интервью

| Ошибка | Как исправить |
|--------|---------------|
| **Нет цифр** | Всегда называйте числа (RPS, ГБ, vCPU). Допущения — нормально, но цифры должны быть. |
| **Забывают про пики** | Учитывайте peak factor (3–10x). Интервьюер проверяет, думаете ли вы о пиковых нагрузках. |
| **Одна нода на всё** | Всегда закладывайте отказоустойчивость (N+1, реплики). |
| **Нет запаса (buffer)** | Добавляйте 20–30% к CPU, RAM, диску. |
| **Смешивают in-memory и disk БД** | Для Redis — весь working set в RAM. Для PostgreSQL — буферный кэш 10–25% от данных. |
| **Забывают про IOPS** | Для transactional БД упомяните про IOPS и SSD. |
