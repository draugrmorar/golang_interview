# РЕПЛИКАЦИЯ

## Что это и зачем

**Репликация** — это процесс синхронизации данных между несколькими узлами (серверами). Цели:

| Цель | Описание |
|------|----------|
| **Высокая доступность (HA)** | При падении одного узла, другие продолжают работу |
| **Отказоустойчивость** | Данные не теряются, даже если несколько узлов вышли из строя |
| **Масштабирование чтения** | Распределение read-нагрузки между репликами |
| **Геораспределенность** | Реплики в разных регионах для низкой латентности |
| **Бэкапы без остановки** | Реплика может использоваться для бэкапов |

---

## 1. Синхронная / Асинхронная / Полу-синхронная репликация

Это классификация по **способу подтверждения записи**.

### Синхронная репликация (Synchronous)

**Как работает:**
Мастер ждет подтверждения от **всех** (или указанного количества) реплик, прежде чем подтвердить запись клиенту.

```
Клиент → Мастер
         Мастер → Реплика 1 (запись)
         Мастер → Реплика 2 (запись)
         Реплика 1 → Мастер (ACK)
         Реплика 2 → Мастер (ACK)
Мастер → Клиент (COMMIT)
```

**Характеристики:**
- **Долговечность:** Максимальная (данные на N узлах)
- **Консистентность:** Сильная (после COMMIT данные есть на всех)
- **Задержка:** Высокая (ждем всех)
- **Доступность:** Снижена (если реплика недоступна, запись блокируется)

**Где применяется:**
- Финансовые системы (нельзя потерять транзакцию)
- PostgreSQL с `synchronous_commit = on`
- MySQL Group Replication (синхронная)

**Пример (PostgreSQL):**
```yaml
synchronous_commit = on
synchronous_standby_names = 'replica1, replica2'
```

---

### Асинхронная репликация (Asynchronous)

**Как работает:**
Мастер подтверждает запись клиенту **до** того, как реплики получили данные.

```
Клиент → Мастер
Мастер → Клиент (COMMIT)  # сразу!
Мастер → Реплика 1 (запись)  # асинхронно
Мастер → Реплика 2 (запись)
```

**Характеристики:**
- **Долговечность:** Риск потери данных (при падении мастера до репликации)
- **Консистентность:** Eventual (реплики могут отставать)
- **Задержка:** Низкая
- **Доступность:** Высокая (реплики не блокируют запись)

**Где применяется:**
- Большинство веб-приложений
- Redis (асинхронная репликация по умолчанию)
- Cassandra (настраиваемая)

---

### Полу-синхронная репликация (Semi-synchronous)

**Как работает:**
Мастер ждет подтверждения **хотя бы от одной** реплики (но не обязательно от всех). Компромисс между синхронной и асинхронной.

```
Клиент → Мастер
         Мастер → Реплика 1 (запись)
         Реплика 1 → Мастер (ACK)  # ждем хотя бы одну
Мастер → Клиент (COMMIT)
         Мастер → Реплика 2 (запись)  # остальные асинхронно
```

**Характеристики:**
- **Долговечность:** Хорошая (данные минимум на 2 узлах)
- **Задержка:** Средняя (ждем одну реплику)
- **Доступность:** Высокая (если одна реплика упала, можно писать)

**Где применяется:**
- **MySQL Semi-Synchronous Replication**
- **PostgreSQL** с `synchronous_commit = remote_write` или `on` с указанием количества

**Пример (MySQL):**
```sql
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = 1;
SET GLOBAL rpl_semi_sync_master_wait_for_slave_count = 1;
```

---

### Сравнение

| Характеристика | Синхронная | Полу-синхронная | Асинхронная |
|----------------|------------|-----------------|-------------|
| **Потеря данных** | Нет (при N реплик) | Минимальна (при падении мастера и реплики) | Возможна |
| **Задержка** | Высокая | Средняя | Низкая |
| **Доступность записи** | Снижена (ждем реплики) | Хорошая | Отличная |
| **RPO (Recovery Point Objective)** | 0 | Секунды | Секунды-минуты |
| **Использование** | Финансы, платежи | Баланс надежности и скорости | Веб-приложения, кэши |

**RPO** — сколько данных может быть потеряно при сбое.

## 2. Топологии репликации

### С одним ведущим узлом (Single-Leader / Master-Slave)

**Как работает:**
Один узел — **мастер** (primary), принимает все записи. Один или несколько **реплик** (slave, standby) получают данные от мастера и обслуживают чтение.

```
        ┌─────────┐
        │ Мастер  │ ← запись
        └────┬────┘
      ┌──────┼──────┐
      ▼      ▼      ▼
┌─────────┐┌─────────┐┌─────────┐
│Реплика 1││Реплика 2││Реплика 3│ ← чтение
└─────────┘└─────────┘└─────────┘
```

**Преимущества:**
- Простота (нет конфликтов записи)
- Strong consistency возможна (синхронная репликация)
- Знакомая модель

**Недостатки:**
- Мастер — единая точка отказа (SPOF) без failover
- Масштабирование записи ограничено мощностью мастера
- Задержка для удаленных реплик

**Где применяется:**
- PostgreSQL (streaming replication)
- MySQL (classic replication)
- Redis (master-replica)

---

### С несколькими ведущими узлами (Multi-Leader / Master-Master)

**Как работает:**
Несколько узлов принимают **записи**. Изменения реплицируются между всеми лидерами.

```
┌─────────┐     ┌─────────┐
│ Лидер A │ ←→  │ Лидер B │
│(EU, запись)   │(US, запись)
└────┬────┘     └────┬────┘
     ▼               ▼
┌─────────┐     ┌─────────┐
│Реплика A│     │Реплика B│
└─────────┘     └─────────┘
```

**Преимущества:**
- Высокая доступность записи (пишем в любой узел)
- Геораспределенность (низкая латентность для локальных записей)
- Масштабирование записи

**Недостатки:**
- **Конфликты записи** — нужно разрешать (last-write-wins, application-level)
- Сложность (асинхронная репликация между лидерами)
- Eventual consistency между лидерами

**Где применяется:**
- **MySQL Group Replication** (multi-primary mode)
- **Cassandra** (masterless, но концептуально multi-leader)
- **PostgreSQL BDR** (Bi-Directional Replication)
- **Active-Active кластеры**

**Разрешение конфликтов:**
| Метод | Описание |
|-------|----------|
| **Last-Write-Wins (LWW)** | Побеждает запись с большим timestamp (просто, но может терять данные) |
| **CRDT** | Conflict-Free Replicated Data Types (например, счетчики, множества) |
| **Application-level** | Приложение само разрешает конфликт (пользователю) |
| **Слияние** | Объединение изменений (например, в документах) |

---

### Без ведущих узлов (Leaderless / Masterless)

**Как работает:**
Все узлы равны. Клиент пишет на **несколько узлов** (quorum) и читает с нескольких для обеспечения консистентности.

```
Клиент → [Узел A] → запись (success)
Клиент → [Узел B] → запись (success)
Клиент → [Узел C] → запись (timeout)
```

**Quorum:**
```
W + R > N
где:
N = общее количество узлов
W = количество узлов, подтверждающих запись
R = количество узлов, опрашиваемых при чтении
```

**Пример (N=3, W=2, R=2):**
- Запись успешна, если 2 узла подтвердили
- Чтение опрашивает 2 узла, берет самую свежую версию

**Преимущества:**
- **Нет единой точки отказа**
- Высокая доступность
- Линейное масштабирование

**Недостатки:**
- Eventual consistency (требует quorum для strong)
- Сложность (векторные часы, устранение конфликтов)

**Где применяется:**
- **Cassandra** (настраиваемый consistency level)
- **Amazon DynamoDB**
- **Riak**
- **ScyllaDB**

**Consistency Levels в Cassandra:**
| Level | Описание |
|-------|----------|
| ONE | Один узел подтвердил (быстро, возможна потеря) |
| QUORUM | Большинство (W = (N/2)+1) |
| ALL | Все узлы (медленно, но максимальная надежность) |

---

### Сравнение топологий

| Характеристика | Single-Leader | Multi-Leader | Leaderless |
|----------------|---------------|--------------|------------|
| **Запись** | Единая точка | Много точек | Много точек |
| **Конфликты** | Нет | Есть | Есть |
| **Сложность** | Низкая | Средняя | Высокая |
| **Consistency** | Strong (возможна) | Eventual | Tunable (quorum) |
| **Примеры** | PostgreSQL, MySQL | MySQL Group Replication, BDR | Cassandra, DynamoDB |

## 3. Логическая / Физическая репликация

### Физическая репликация (Physical Replication)

**Как работает:**
Реплицируются **изменения на уровне дисковых блоков** (WAL). Реплика — точная копия мастера на уровне байтов.

**Характеристики:**
- **Точность:** Байт-в-байт копия
- **Версия БД:** Должна быть одинаковой на мастере и реплике
- **DDL:** Реплицируются автоматически
- **Гибкость:** Низкая (нельзя выбрать, какие таблицы реплицировать)

**Где применяется:**
- **PostgreSQL Streaming Replication** (физическая)
- **MySQL** с binary log (row-based — это логическая, но физические снапшоты)

---

### Логическая репликация (Logical Replication)

**Как работает:**
Реплицируются **изменения данных на логическом уровне** (INSERT, UPDATE, DELETE). Реплика получает поток операций, которые может применять выборочно.

**Характеристики:**
- **Гибкость:** Можно выбирать таблицы, фильтровать строки
- **Разные версии:** Мастер и реплика могут быть разных версий
- **Разные типы БД:** Можно реплицировать между разными СУБД (CDC)
- **DDL:** Не реплицируется автоматически (нужно управлять)

**Способы логической репликации:**

| Способ | Описание | Пример |
|--------|----------|--------|
| **SBR (Statement-Based Replication)** | Реплицируются SQL-запросы | MySQL с binlog_format=STATEMENT |
| **RBR (Row-Based Replication)** | Реплицируются измененные строки | MySQL с binlog_format=ROW, PostgreSQL logical replication |

**SBR vs RBR:**

| Характеристика | SBR | RBR |
|----------------|-----|-----|
| **Объем** | Маленький (сам запрос) | Большой (все измененные строки) |
| **Недетерминированные функции** | Проблема (NOW(), RAND()) | Безопасно |
| **Триггеры** | Могут выполняться на реплике | Только данные |
| **Надежность** | Ниже | Выше |

**Где применяется логическая репликация:**
- **PostgreSQL Logical Replication** (начиная с 10)
- **MySQL** (binlog format ROW)
- **CDC (Change Data Capture):** Debezium, Kafka Connect

---

### Физическая vs Логическая

| Характеристика | Физическая | Логическая |
|----------------|------------|------------|
| **Уровень** | Дисковые блоки (WAL) | Логические операции (SQL/строки) |
| **Гибкость** | Низкая (все база) | Высокая (таблицы, фильтры) |
| **Версии БД** | Должны совпадать | Могут отличаться |
| **DDL** | Реплицируется | Не реплицируется |
| **Разные СУБД** | Нет | Да (через CDC) |
| **Назначение** | HA, бэкапы | Агрегация данных, миграции, CDC |

## 4. Hot Standby / Warm Standby

### Hot Standby (горячий резерв)

**Что это:**
Реплика, которая **принимает запросы на чтение** в обычном режиме. При падении мастера, hot standby может быть повышен до мастера с минимальным downtime.

**Характеристики:**
- **Чтение:** Разрешено
- **Запись:** Только от мастера
- **Время переключения:** Секунды
- **Использование:** Масштабирование чтения + HA

**Примеры:**
- PostgreSQL с `hot_standby = on`
- MySQL replica (может читать)
- Redis replica (read-only)

---

### Warm Standby (теплый резерв)

**Что это:**
Реплика, которая **не принимает запросы** в обычном режиме. Она только применяет изменения от мастера. При падении мастера требуется активация.

**Характеристики:**
- **Чтение:** Запрещено (или отключено)
- **Запись:** Только от мастера
- **Время переключения:** Минуты (нужно активировать)
- **Использование:** Только HA, не для масштабирования

**Примеры:**
- PostgreSQL с `standby_mode = on` без hot_standby
- MySQL replica в режиме только recovery

---

### Cold Standby (холодный резерв)

**Что это:**
Резервный сервер, который **не запущен**. При падении мастера его нужно запустить и восстановить данные.

**Время переключения:** Часы (восстановление из бэкапов)

## 5. Фильтрация репликации (Replication Filtering)

**Что это:**
Возможность реплицировать **только определенные таблицы или строки**.

### Типы фильтрации

| Тип | Пример | Использование |
|-----|--------|---------------|
| **По таблицам** | Реплицировать только таблицы `users`, `orders` | Аналитика только на нужных данных |
| **По строкам** | `WHERE region = 'EU'` | Геораспределение (европейские данные на европейскую реплику) |
| **По колонкам** | Реплицировать без колонки `ssn` | Безопасность (не хранить PII на аналитической реплике) |

### Где поддерживается

| БД | Фильтрация |
|----|------------|
| **MySQL** | `replicate-do-table`, `replicate-ignore-table`, `replicate-wild-do-table` |
| **PostgreSQL** | Логическая репликация: `CREATE PUBLICATION mypub FOR TABLE users, orders WHERE (region = 'EU')` |
| **MongoDB** | Фильтрация на уровне коллекций |

### Пример (PostgreSQL Logical Replication)

```sql
-- Публикация только для европейских пользователей
CREATE PUBLICATION eu_users FOR TABLE users WHERE (region = 'EU');

-- Подписка на реплике
CREATE SUBSCRIPTION eu_users_sub 
CONNECTION 'host=master dbname=mydb' 
PUBLICATION eu_users;
```

## 6. Split Brain (разделение мозга)

### Что это

**Split Brain** — критическая ситуация, когда в системе с отказоустойчивостью **два узла одновременно считают себя мастером** и оба принимают запись.

```
Нормальная работа:
    ┌─────────┐
    │ Мастер  │
    └─────────┘

Сбой сети (network partition):
    ┌─────────┐          ┌─────────┐
    │ Мастер A│  ──❌──  │ Мастер B│
    │(считает себя    │(считает себя
    │ мастером)       │ мастером)
    └─────────┘          └─────────┘
    
Проблема: в обе базы пишутся данные → конфликты, потеря данных
```

### Почему возникает

1. **Сетевое разделение:** Мастер и реплика не видят друг друга
2. **Неудачный failover:** Реплика повышена, но старый мастер не узнал об этом и продолжает работать

### Как предотвратить

| Механизм | Как работает | Примеры |
|----------|--------------|---------|
| **Quorum / Majority** | Для выбора мастера нужно большинство голосов | etcd, ZooKeeper, Consul |
| **Fencing (понижение)** | Отключение старого мастера физически или через STONITH | Pacemaker, cloud API |
| **Lease / TTL** | Мастер должен продлевать аренду. Если не продлил — теряет статус | Kubernetes leader election |
| **Manual intervention** | Ручное вмешательство (не автоматический failover) | Низкая автоматизация |

### Решения

**1. Использовать внешний координатор (etcd, ZooKeeper):**
```
Узлы регистрируются в etcd
etcd гарантирует, что только один узел — мастер
При разделении etcd требует quorum
```

**2. STONITH (Shoot The Other Node In The Head):**
```
При сбое, новый мастер физически отключает старый (через IPMI, cloud API)
```

**3. Третий узел-арбитр:**
```
Мастер A и B + арбитр C (не хранит данные)
При конфликте арбитр определяет, кто мастер
```

### На интервью

> *«Для предотвращения split brain мы используем внешний координатор (etcd) для выбора лидера. При сетевом разделении etcd требует кворума (большинства узлов) для работы. Это гарантирует, что только один мастер может быть активен. Дополнительно мы реализуем fencing — при повышении новой реплики старый мастер отключается через API облака или IPMI»*.

## 7. Failover (переключение)

### Что это

**Failover** — процесс автоматического (или ручного) переключения с отказавшего мастера на реплику.

### Типы failover

| Тип | Описание | Время |
|-----|----------|-------|
| **Автоматический** | Система сама детектирует сбой и повышает реплику | Секунды-минуты |
| **Ручной** | Оператор принимает решение и выполняет команды | Минуты-часы |
| **Planned (switchover)** | Плановое переключение (обновление, обслуживание) | Секунды (минимальный downtime) |

### Процесс failover

```
1. DETECT: Мониторинг детектирует, что мастер не отвечает
   - Health checks, timeout, heartbeat
   - Обычно 2-3 неудачных попытки

2. ELECT: Выбор новой реплики для повышения
   - Самая актуальная (последний WAL)
   - Настроенный приоритет

3. PROMOTE: Повышение реплики до мастера
   - Реплика становится read-write
   - Может переключить DNS / балансировщик

4. REPOINT: Перенаправление клиентов на новый мастер
   - Обновление connection strings
   - Перезапуск приложений

5. REINTEGRATE: Восстановление старого мастера как реплики (после починки)
```

### Методы

| Метод | Как работает |
|-------|--------------|
| **Virtual IP** | VIP плавает между мастером и репликой. Клиенты всегда ходят на VIP |
| **DNS** | Обновление DNS записи (может кэшироваться) |
| **Балансировщик** | Health check на бекенды, переключение трафика |
| **Connection string** | Клиенты обновляют строку подключения (требует перезапуска) |

### Инструменты

| Инструмент | Для каких БД | Особенности |
|------------|--------------|-------------|
| **Patroni** | PostgreSQL | Использует etcd/Consul, автоматический failover |
| **Repmgr** | PostgreSQL | Управление репликацией и failover |
| **Orchestrator** | MySQL | Топология, failover, обнаружение |
| **MHA (Master High Availability)** | MySQL | Автоматический failover |
| **Redis Sentinel** | Redis | Мониторинг, failover, уведомления |

### RPO и RTO

| Параметр | Описание | Для failover |
|----------|----------|--------------|
| **RPO (Recovery Point Objective)** | Сколько данных можно потерять | Зависит от типа репликации (синхронная → 0, асинхронная → секунды) |
| **RTO (Recovery Time Objective)** | Время восстановления | Время детекции + продвижения + переключения (обычно 30-120 секунд) |

---

## Сводная таблица: репликация

| Параметр | Варианты | Ключевое отличие |
|----------|----------|------------------|
| **Синхронность** | Синхронная / Полу-синхронная / Асинхронная | Надежность vs производительность |
| **Топология** | Single-Leader / Multi-Leader / Leaderless | Запись, конфликты, сложность |
| **Уровень** | Физическая / Логическая | Гибкость, версии БД |
| **Готовность** | Hot Standby / Warm Standby | Чтение на реплике |
| **Безопасность** | Split Brain prevention | Quorum, fencing, внешний координатор |

*«Для нашей системы я использую PostgreSQL с одной репликой в hot standby режиме для масштабирования чтения и отказоустойчивости. Репликация асинхронная для низкой задержки записи, но критичные операции (платежи) идут через синхронную репликацию с подтверждением хотя бы одной реплики. Для failover используем Patroni с etcd — это дает автоматическое переключение с RTO около 30 секунд. # ПАРТИЦИОНИРОВАНИЕ (PARTITIONING)

## Что это и зачем

**Партиционирование** — это разделение **логически единой таблицы** на **физические фрагменты** (партиции), которые хранятся отдельно, но при этом остаются частью одной таблицы.

### Зачем нужно партиционирование

| Цель | Описание |
|------|----------|
| **Управляемость** | Можно удалять/архивировать старые партиции одной командой (DROP PARTITION) |
| **Производительность** | Запросы с фильтром по ключу партиции сканируют только нужные партиции (partition pruning) |
| **Изоляция** | Нагрузка распределяется между партициями |
| **Параллелизм** | Можно сканировать партиции параллельно |

### Важное отличие: партиционирование vs шардирование

| Характеристика | Партиционирование | Шардирование |
|----------------|-------------------|--------------|
| **Уровень** | Внутри одной БД | Между разными БД/серверами |
| **Прозрачность** | Полностью прозрачно для приложения | Требует логики маршрутизации |
| **Масштабирование** | Ограничено мощностью одного сервера | Горизонтальное (много серверов) |
| **Сложность** | Низкая | Высокая |

**Ключевое правило:**
- **Партиционирование** — данные на одном сервере, но разделены физически
- **Шардирование** — данные распределены между разными серверами

---

## 1. Вертикальное / Горизонтальное партиционирование

### Вертикальное партиционирование (Vertical Partitioning)

**Что это:**
Разделение таблицы **по колонкам**. Разные группы колонок хранятся в разных физических таблицах (или партициях).

**Пример:**
```
Исходная таблица users (50 колонок):

users_full:
| id | login | email | password_hash | last_login | avatar | ... (еще 45 колонок) |

Вертикальное партиционирование:

users_auth:           users_profile:          users_metadata:
| id | login |        | id | email |        | id | last_login |
| password_hash |     | avatar | ...        | ...          |
```

**Когда применять:**

| Сценарий | Почему |
|----------|--------|
| **Редко используемые колонки** | Вынести большие BLOB/JSON поля в отдельную таблицу |
| **Разный доступ** | Поля аутентификации и профиля имеют разную частоту доступа |
| **Row size ограничения** | Некоторые БД имеют ограничение на размер строки (например, InnoDB 64KB) |

**Преимущества:**
- Меньше строк считывается при частых запросах
- Лучшая локализация кэша
- Можно использовать разные storage engine для разных колонок (в некоторых БД)

**Недостатки:**
- Требует JOIN-ов для сборки полной строки
- Усложняет транзакции (обновление двух таблиц)

---

### Горизонтальное партиционирование (Horizontal Partitioning)

**Что это:**
Разделение таблицы **по строкам**. Каждая партиция хранит подмножество строк с одинаковой структурой.

**Пример:**
```
Таблица orders с партиционированием по created_at (месяцам):

orders_2024_01:  заказы января 2024
orders_2024_02:  заказы февраля 2024
orders_2024_03:  заказы марта 2024
...
```

**Когда применять:**

| Сценарий | Почему |
|----------|--------|
| **Временные данные** | Логи, события, заказы — старые партиции можно удалять/архивировать |
| **Огромные таблицы** | Таблицы с миллиардами строк — партиции по ключу |
| **Локальность запросов** | Запросы часто фильтруют по ключу партиции (например, по user_id, дате) |

**Преимущества:**
- **Partition pruning:** запросы сканируют только нужные партиции
- **Управление жизненным циклом:** DROP PARTITION быстрее DELETE
- **Параллельная загрузка:** можно загружать данные в разные партиции

**Недостатки:**
- Все партиции на одном сервере (ограничение по ресурсам)
- Не все типы индексов поддерживают партиционирование

---

### Сравнение

| Характеристика | Вертикальное | Горизонтальное |
|----------------|--------------|----------------|
| **Разделение** | По колонкам | По строкам |
| **Структура** | Разные наборы колонок | Одинаковая структура |
| **Ключ** | Не нужен | Нужен ключ партиции |
| **JOIN** | Требуется | Не требуется |
| **Основная цель** | Оптимизация доступа | Масштабирование, управление |

---

## 2. Стратегии партиционирования

### 2.1. Range Based (диапазонное партиционирование)

**Что это:**
Партиции создаются по **диапазонам значений** ключа. Самая распространенная стратегия.

**Структура:**
```
orders:
├── orders_2024_Q1 (created_at < '2024-04-01')
├── orders_2024_Q2 (created_at BETWEEN '2024-04-01' AND '2024-06-30')
├── orders_2024_Q3 (created_at BETWEEN '2024-07-01' AND '2024-09-30')
└── orders_2024_Q4 (created_at >= '2024-10-01')
```

**Пример (PostgreSQL):**
```sql
CREATE TABLE orders (
    id SERIAL,
    user_id INT,
    total DECIMAL,
    created_at TIMESTAMP
) PARTITION BY RANGE (created_at);

CREATE TABLE orders_2024_q1 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');

CREATE TABLE orders_2024_q2 PARTITION OF orders
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');
```

**Когда применять:**

| Сценарий | Пример |
|----------|--------|
| **Временные данные** | Логи, события, заказы |
| **Актуальные vs архивные** | Горячие данные в свежих партициях, холодные — в старых |
| **Time-series данные** | Метрики, IoT показания |

**Преимущества:**
- Простое управление жизненным циклом (DROP старых партиций)
- Partition pruning по дате (очень эффективно)

**Недостатки:**
- **Skew (перекос):** может быть неравномерное распределение данных
- **Hot partition:** если все запросы идут в последнюю партицию (сегодняшнюю)

---

### 2.2. Key Based (хэш-партиционирование)

**Что это:**
Партиция определяется **хешем от ключа**. Равномерно распределяет данные, но не поддерживает диапазонные запросы.

**Структура:**
```
hash(key) → partition_id

users:
├── users_p0 (hash(user_id) % 4 = 0)
├── users_p1 (hash(user_id) % 4 = 1)
├── users_p2 (hash(user_id) % 4 = 2)
└── users_p3 (hash(user_id) % 4 = 3)
```

**Пример (PostgreSQL):**
```sql
CREATE TABLE users (
    id SERIAL,
    email TEXT,
    name TEXT
) PARTITION BY HASH (id);

CREATE TABLE users_p0 PARTITION OF users
    FOR VALUES WITH (MODULUS 4, REMAINDER 0);

CREATE TABLE users_p1 PARTITION OF users
    FOR VALUES WITH (MODULUS 4, REMAINDER 1);
```

**Когда применять:**

| Сценарий | Пример |
|----------|--------|
| **Равномерное распределение** | Пользователи, ключи без естественного диапазона |
| **Нет запросов по диапазону** | Только точечные запросы (SELECT * FROM users WHERE id = X) |
| **Предотвращение hot partition** | Запросы равномерно распределены |

**Преимущества:**
- **Равномерное распределение** (нет перекосов)
- Хорошо для равномерной нагрузки

**Недостатки:**
- **Нет range queries:** `WHERE id > 100` не использует partition pruning
- **Добавление партиций:** требует ребалансировки (хеш-функция меняется)

---

### 2.3. Directory Based (списочное партиционирование)

**Что это:**
Партиции создаются по **явному списку значений** ключа.

**Структура:**
```
orders:
├── orders_us (country = 'US')
├── orders_eu (country = 'EU')
├── orders_asia (country IN ('JP', 'CN', 'KR'))
└── orders_other (country NOT IN ('US', 'EU', 'JP', 'CN', 'KR'))
```

**Пример (PostgreSQL):**
```sql
CREATE TABLE orders (
    id SERIAL,
    country TEXT,
    total DECIMAL
) PARTITION BY LIST (country);

CREATE TABLE orders_us PARTITION OF orders
    FOR VALUES IN ('US');

CREATE TABLE orders_eu PARTITION OF orders
    FOR VALUES IN ('FR', 'DE', 'IT', 'ES');

CREATE TABLE orders_other PARTITION OF orders
    DEFAULT;
```

**Когда применять:**

| Сценарий | Пример |
|----------|--------|
| **Геораспределение** | Данные по регионам |
| **Категории** | Товары по категориям |
| **Тенанты (SaaS)** | Данные по клиентам (tenant_id) |
| **Статусы** | Заказы по статусам |

**Преимущества:**
- **Логическая изоляция** (каждый регион/тенант в своей партиции)
- **Управление по группам** (можно выключить/включить партицию)

**Недостатки:**
- **Skew:** неравномерное распределение (если один регион доминирует)
- **Ограниченное количество значений:** не подходит для уникальных ключей

---

### Сравнение стратегий

| Характеристика | Range | Key (Hash) | List (Directory) |
|----------------|-------|------------|------------------|
| **Ключ** | Диапазон (дата, ID) | Хеш от ключа | Список значений |
| **Равномерность** | Может быть skew | Равномерно | Зависит от данных |
| **Range queries** | ✅ Отлично | ❌ Плохо | ⚠️ Только по списку |
| **Partition pruning** | По диапазону | По точному ключу | По списку |
| **Добавление партиций** | Просто (новая дата) | Сложно (reshard) | Просто (новый регион) |
| **Управление TTL** | ✅ DROP старых | ❌ Сложно | ⚠️ Можно по группам |

---

## Практические аспекты

### Partition Pruning (отсечение партиций)

Оптимизатор БД анализирует запрос и исключает партиции, которые не могут содержать нужные данные.

```sql
-- Без партиционирования: сканирование всей таблицы (100 млн строк)
-- С range партиционированием: сканирование только 1 партиции (10 млн строк)
SELECT * FROM orders 
WHERE created_at BETWEEN '2024-03-01' AND '2024-03-31';

EXPLAIN показывает:
-> Seq Scan on orders_2024_03  (условие на created_at)
```

### Управление партициями

**Создание новой партиции (например, для следующего месяца):**
```sql
-- Автоматизация через CRON или событие
CREATE TABLE orders_2024_04 PARTITION OF orders
    FOR VALUES FROM ('2024-04-01') TO ('2024-05-01');
```

**Удаление старой партиции:**
```sql
-- Намного быстрее, чем DELETE FROM orders WHERE created_at < '2024-01-01'
DROP TABLE orders_2024_q1;
```

**Архивирование партиции:**
```sql
-- Выгрузить данные в S3, затем удалить партицию
COPY orders_2024_q1 TO 's3://archive/orders_2024_q1.parquet';
DROP TABLE orders_2024_q1;
```

### Выбор ключа партиционирования

**Ключевое правило:** выбирайте ключ, который:
1. **Часто используется в WHERE** (для partition pruning)
2. **Равномерно распределяет данные** (избегайте перекосов)
3. **Соответствует жизненному циклу** (для временных данных — дата)

**Примеры выбора:**

| Данные | Ключ | Почему |
|--------|------|--------|
| Логи | `created_at` (range) | TTL, запросы по дате |
| Пользователи | `user_id` (hash) | Равномерно, точечные запросы |
| Мультитенантный SaaS | `tenant_id` (list) | Изоляция тенантов |
| Геораспределенные | `region` (list) | Локальность, изоляция |

## Ограничения и особенности в разных БД

| БД | Поддержка | Особенности |
|----|-----------|-------------|
| **PostgreSQL** | Range, List, Hash (с 11) | Мощная, но требует ручного управления партициями |
| **MySQL (InnoDB)** | Range, List, Hash, Key | Ограничения на внешние ключи, уникальные индексы |
| **Oracle** | Все типы, автоматическое | Очень зрелая, автоматическое управление |
| **SQL Server** | Range (через filegroups) | Менее гибкая, чем PostgreSQL |
| **ClickHouse** | Range, Hash | Для временных рядов — идеально |

> *«Для таблицы заказов я выбираю **range партиционирование по created_at** по месяцам. Это дает несколько преимуществ:*
> 1. *Запросы за последний месяц (самые частые) сканируют только одну партицию (partition pruning)*

**Шардирование** — это следующий уровень масштабирования, когда данные уже не помещаются на одном сервере даже с партиционированием. На интервью шардирование — это одна из самых важных тем для high-load систем.

# ШАРДИРОВАНИЕ (SHARDING)

## Что это и зачем

**Шардирование** — это **горизонтальное масштабирование** базы данных, при котором данные распределяются между **независимыми серверами (шардами)**, каждый из которых хранит только часть данных.

### Шардирование vs Партиционирование

| Характеристика | Партиционирование | Шардирование |
|----------------|-------------------|--------------|
| **Физическое размещение** | Один сервер (разные таблицы/файлы) | Множество серверов |
| **Масштабирование** | Ограничено мощностью одного сервера | Горизонтальное (добавляем серверы) |
| **Прозрачность** | Полностью прозрачно для приложения | Требует логики маршрутизации |
| **Сложность** | Низкая | Высокая |
| **Транзакции** | Поддерживаются (в пределах БД) | Сложные (distributed transactions) |

### Когда нужно шардирование

1. **Данные > 1-10 ТБ** — один сервер не вмещает
2. **Write нагрузка > 10-50k RPS** — один мастер не справляется
3. **Чтение не масштабируется репликами** — реплики добавляют задержку, требуется шардирование
4. **Геораспределенность** — данные должны находиться ближе к пользователям

---

## 1. Перебалансировка (Rebalancing)

### Проблема

При шардировании мы распределяем данные между фиксированным количеством шардов. Но со временем:
- **Данные растут неравномерно** (один шард переполняется)
- **Добавляются новые шарды** (нужно перераспределить данные)
- **Шарды выходят из строя** (нужно перераспределить нагрузку)

### Virtual Buckets (виртуальные корзины)

**Что это:**
Промежуточный уровень между ключом и физическим шардом. Ключ → хеш → виртуальная корзина → физический шард.

**Структура:**
```
[Ключ] → Hash(key) → Bucket ID → Map: [Bucket → Physical Shard]

Пример (N=4 виртуальных корзины, 2 шарда):

Ключ "user_123" → hash=42 → bucket=42 % 4 = 2 → bucket 2 → shard A
Ключ "user_456" → hash=73 → bucket=73 % 4 = 1 → bucket 1 → shard A
Ключ "user_789" → hash=15 → bucket=15 % 4 = 3 → bucket 3 → shard B

Mapping:
Bucket 0 → Shard A
Bucket 1 → Shard A
Bucket 2 → Shard A
Bucket 3 → Shard B
```

**Преимущества:**
- **Гибкость:** меняем только mapping, данные не перемещаются
- **Перебалансировка:** просто переназначаем корзины на новые шарды
- **Добавление шарда:** перераспределяем *только некоторые* корзины

**Перебалансировка при добавлении шарда:**

```
Было: 4 корзины, 2 шарда
Bucket 0 → Shard A
Bucket 1 → Shard A
Bucket 2 → Shard B
Bucket 3 → Shard B

Добавляем Shard C (3 шарда). Перераспределяем корзины:

Bucket 0 → Shard A (остался)
Bucket 1 → Shard C (переехал)
Bucket 2 → Shard B (остался)
Bucket 3 → Shard C (переехал)

Переехало только 50% корзин (2 из 4)
```

**Где используется:**
- **Cassandra** — virtual nodes (vnodes), каждый физический узел отвечает за множество виртуальных токенов
- **Elasticsearch** — shards, которые могут перемещаться между узлами
- **Redis Cluster** — 16384 хеш-слота (virtual buckets)
- **Amazon DynamoDB** — partition keys, внутреннее распределение

---

### Пример: Redis Cluster (16384 хеш-слотов)

```
[Key] → CRC16(key) % 16384 → Slot ID

Слоты распределены между узлами:

Master A: слоты 0-5460
Master B: слоты 5461-10922
Master C: слоты 10923-16383

При добавлении нового мастера:
- Некоторые слоты перемещаются на новый узел
- Redis обрабатывает перемещение слотов (асинхронно)
- Клиенты обновляют mapping (MOVED redirect)
```

---

## 2. Resharding (перешардирование)

### Проблема

При изменении количества шардов нужно перемещать данные. Как минимизировать объем перемещений?

### Consistent Hashing (консистентное хеширование)

**Что это:**
Алгоритм, который минимизирует количество перемещаемых ключей при изменении количества узлов.

**Как работает:**

1. **Хеш-кольцо:** Все узлы и ключи хешируются в одно кольцо (0...2^32-1)

```
Кольцо:

        0/2^32
          |
     Node B├───┐
          │    │
          │    │
    ┌─────┘    └─────┐
    │                │
Node A├──┐        ┌──├ Node C
       │  │        │  │
       │  │        │  │
       └──┼────────┼──┘
          │        │
          └───Node D
```

2. **Определение шарда:** Двигаемся по кольцу по часовой стрелке до первого узла

```
Key X → hash(X) = 1000 → идем по часовой стрелке → Node A (в точке 2000)
Key Y → hash(Y) = 5000 → идем по часовой стрелке → Node C (в точке 6000)
```

3. **Добавление узла:** Только ключи в сегменте между новым узлом и следующим узлом перемещаются

```
Добавляем Node E в точку 3000:
Ключи между 2000 (Node A) и 3000 (Node E) → перемещаются с Node A на Node E
Остальные ключи не перемещаются

Перемещается только ~1/N данных
```

**Преимущества:**
- **Минимальное перемещение данных** при добавлении/удалении узлов (O(1/N))
- **Равномерное распределение** (с виртуальными узлами)

**Недостатки:**
- **Сложность реализации**
- **Неравномерность** без виртуальных узлов (если узлов мало)

**Где используется:**
- **Cassandra** (vnodes — virtual nodes, по сути consistent hashing с множеством токенов)
- **DynamoDB**
- **Akka Cluster**
- **Memcached** (client-side consistent hashing)

---

### Rendezvous Hashing (HRW — Highest Random Weight)

**Что это:**
Альтернатива consistent hashing. Для каждого ключа вычисляется вес для каждого узла, выбирается узел с максимальным весом.

**Как работает:**
```
weight(key, node) = hash(key + node)  # или hash(key) ^ hash(node)

node = argmax(weight(key, node_i))
```

**Пример:**
```
Key = "user_123"

weight(key, Node A) = hash("user_123:nodeA") = 0.9
weight(key, Node B) = hash("user_123:nodeB") = 0.3
weight(key, Node C) = hash("user_123:nodeC") = 0.7

Выбираем Node A (максимальный вес)
```

**Преимущества:**
- **Равномерное распределение** без виртуальных узлов
- **Минимальное перемещение** при изменении узлов (только ключи, у которых изменился максимальный вес)
- **Простота** — нет кольца

**Недостатки:**
- **Вычислительная сложность:** O(N) на запрос (но можно кэшировать mapping)
- **Меньше известен,** чем consistent hashing

**Где используется:**
- **DynamoDB** (внутренне)
- **Akka Distributed Data**
- **Couchbase** (vBuckets с rendezvous)

---

### Consistent Hashing vs Rendezvous Hashing

| Характеристика | Consistent Hashing | Rendezvous Hashing |
|----------------|-------------------|-------------------|
| **Структура** | Кольцо + виртуальные узлы | Прямое вычисление |
| **Сложность запроса** | O(log N) (бинарный поиск по кольцу) | O(N) (вычисление весов) |
| **Равномерность** | Требует виртуальных узлов | Равномерно из коробки |
| **Перемещение при добавлении узла** | ~1/N данных | ~1/N данных |
| **Кэширование** | Можно кэшировать mapping | Можно кэшировать mapping |

---

## 3. Роутинг (Routing)

Клиенту нужно знать, на какой шард отправлять запрос. Существует три основных подхода.

### 3.1. Client Routing (клиентский роутинг)

**Как работает:**
Клиент сам знает mapping (шард → узлы) и вычисляет, на какой шард идти.

```
┌─────────┐
│ Клиент  │ → вычисляет hash(key) → шард → прямой запрос
└─────────┘
     │
     ├──────────┐
     ▼          ▼
┌─────────┐ ┌─────────┐
│ Шард A  │ │ Шард B  │
└─────────┘ └─────────┘
```

**Преимущества:**
- **Нет единой точки отказа** (нет промежуточного слоя)
- **Минимальная задержка** (один сетевой hop)
- **Максимальная производительность**

**Недостатки:**
- **Сложность клиента:** каждый клиент должен знать топологию
- **Обновление mapping:** клиенты должны узнавать о добавлении/удалении шардов (нужен service discovery)
- **Сложность для разных языков:** логику нужно реализовывать на каждом языке

**Где используется:**
- **Cassandra** (driver знает топологию, использует gossip для обнаружения)
- **Redis Cluster** (клиенты знают mapping слотов, при MOVED обновляют кэш)
- **Memcached** (consistent hashing на клиенте)

**Пример (Redis Cluster client):**
```python
# Клиент вычисляет слот
slot = crc16(key) % 16384

# Проверяет локальный кэш: слот → узел
node = slot_map[slot]

# Если получил MOVED ответ, обновляет кэш
```

---

### 3.2. Proxy Routing (прокси-роутинг)

**Как работает:**
Клиент обращается к **прокси-сервису**, который знает mapping и перенаправляет запрос на нужный шард.

```
┌─────────┐
│ Клиент  │ → запрос на прокси
└─────────┘
     │
     ▼
┌─────────────┐
│   Прокси    │ → вычисляет hash(key) → шард
│ (Smart LB)  │
└──────┬──────┘
       │
       ├──────────┐
       ▼          ▼
┌─────────┐ ┌─────────┐
│ Шард A  │ │ Шард B  │
└─────────┘ └─────────┘
```

**Преимущества:**
- **Прозрачность для клиента:** клиент не знает о шардировании
- **Единая точка управления:** легко менять топологию
- **Поддержка разных языков:** клиенты простые

**Недостатки:**
- **Дополнительная задержка** (еще один hop)
- **Единая точка отказа** (прокси может упасть — нужна репликация)
- **Прокси может стать узким местом** (нужно масштабировать)

**Где используется:**
- **Vitess** (прокси для MySQL шардирования)
- **Citus** (прокси для PostgreSQL шардирования)
- **MongoDB** (mongos — proxy router)
- **TiDB** (TiDB Server — proxy/coordinator)

**Пример (Vitess):**
```
App → vtgate (прокси) → vttablet (шард A)
                       → vttablet (шард B)
```

---

### 3.3. Coordinator Routing (координатор-роутинг)

**Как работает:**
Специализированный сервис (координатор) управляет топологией и может выполнять distributed transactions, перераспределять данные.

```
┌─────────┐
│ Клиент  │
└─────────┘
     │
     ▼
┌─────────────┐
│ Координатор │ → знает топологию, может координировать distributed транзакции
│ (Smart LB)  │ → может перемещать данные (resharding)
└──────┬──────┘
       │
       ├──────────┐
       ▼          ▼
┌─────────┐ ┌─────────┐
│ Шард A  │ │ Шард B  │
└─────────┘ └─────────┘
```

**Преимущества:**
- **Поддержка distributed transactions:** 2PC, SAGA
- **Сложная логика перераспределения** (resharding, rebalancing)
- **Централизованное управление**

**Недостатки:**
- **Сложность**
- **Координатор — единая точка отказа** (нужен кластер координаторов)
- **Задержка** (может быть выше, чем proxy)

**Где используется:**
- **CockroachDB** (все узлы — координаторы)
- **Spanner** (координаторы + TrueTime)
- **YugabyteDB**

---

### Сравнение подходов роутинга

| Характеристика | Client | Proxy | Coordinator |
|----------------|--------|-------|-------------|
| **Задержка** | Минимальная | Средняя | Максимальная |
| **Прозрачность** | Нет | Да | Да |
| **Языковая поддержка** | На каждом языке | Один прокси | Один координатор |
| **Single point of failure** | Нет (клиенты распределены) | Есть (нужна репликация) | Есть (нужен кластер) |
| **Distributed transactions** | Сложно | Сложно | Поддерживает |
| **Сложность** | Средняя | Средняя | Высокая |
| **Примеры** | Cassandra, Redis | Vitess, MongoDB | CockroachDB, Spanner |

## Практические аспекты шардирования

### Выбор ключа шардирования (Shard Key)

**Ключевое правило:** ключ должен быть **равномерным** и **использоваться в большинстве запросов**.

**Хорошие шард-ключи:**
- `user_id` (для запросов конкретного пользователя)
- `tenant_id` (для мультитенантных SaaS)
- `hash(device_id)` (для IoT)

**Плохие шард-ключи:**
- `timestamp` (все запросы в один шард "сегодня")
- `status` (низкая селективность)
- `country` (перекос, если одна страна доминирует)

### Проблемы шардирования

| Проблема | Решение |
|----------|---------|
| **Join между шардами** | Денормализация, хранить связанные данные в одном шарде (co-location) |
| **Cross-shard транзакции** | Избегать, использовать SAGA, 2PC, или выбрать координатор |
| **Global уникальность (auto-increment)** | Использовать UUID, Snowflake ID, или sequence на координаторе |
| **Secondary indexes** | Локальные индексы (в шарде) или глобальные (отдельная таблица) |
| **Hot shard** | Разбить по составному ключу (например, user_id + date) |

### Co-location (размещение связанных данных в одном шарде)

Чтобы избежать cross-shard запросов, размещаем связанные данные в одном шарде.

**Пример (e-commerce):**
```
Шард-ключ: user_id
В одном шарде хранятся:
- Профиль пользователя
- Заказы пользователя
- Корзина пользователя

Все запросы по user_id идут в один шард
```

**В Cassandra:**
```sql
-- Данные пользователя в одной партиции
CREATE TABLE users (
    user_id UUID,
    order_id UUID,
    order_date TIMESTAMP,
    -- данные
    PRIMARY KEY (user_id, order_date, order_id)
) WITH CLUSTERING ORDER BY (order_date DESC);
```

## Архитектурные паттерны шардирования

### 1. Range Sharding

Данные разбиваются по диапазонам ключа (например, user_id 1-1M → шард 1, 1M-2M → шард 2).

**Плюсы:** хорош для range запросов.
**Минусы:** может быть перекос (hot shard), сложная перебалансировка.

### 2. Hash Sharding

Хеш от ключа определяет шард.

**Плюсы:** равномерное распределение.
**Минусы:** нет range запросов.

### 3. Directory Sharding

Отдельная таблица (lookup) хранит mapping ключ → шард.

**Плюсы:** максимальная гибкость, можно перемещать данные.
**Минусы:** lookup — единая точка отказа и узкое место.

### 4. Consistent Hash Sharding

Virtual buckets + consistent hashing (Cassandra, Redis Cluster).

**Плюсы:** минимальное перемещение при изменении шардов.
**Минусы:** сложность реализации.

---

## На интервью: пример ответа

> *«Для шардирования я выбираю **хеш-шардирование по user_id** с **virtual buckets**. Это даст равномерное распределение нагрузки и возможность перебалансировки при добавлении новых шардов. В качестве роутинга использую **client-side routing** для минимизации задержки, клиенты узнают топологию через service discovery (Consul).*
>
> *Для ключевых запросов (заказы пользователя) я использую **co-location** — все данные пользователя хранятся в одном шарде, что позволяет избежать cross-shard JOIN-ов. Для глобальных уникальных идентификаторов использую **Snowflake ID**, который встраивает в себя шард-ключ и timestamp, позволяя идемпотентность и сортировку.*
>
> *При добавлении нового шарда перебалансировка происходит через переназначение **virtual buckets**: только часть корзин (около 1/N данных) перемещается на новый шард, что минимизирует даунтайм и нагрузку на систему»*.

## 1. Бэкапы (Backups)

### Что это и зачем

**Бэкап** — это копия данных, которая может быть использована для восстановления после:
- Сбоя оборудования
- Ошибки оператора (DROP TABLE)
- Атаки (ransomware, удаление данных)
- Бага в приложении

### Виды бэкапов

| Вид | Описание | Размер | Время восстановления |
|-----|----------|--------|---------------------|
| **Полный (Full)** | Копия всех данных | Максимальный | Среднее |
| **Инкрементальный (Incremental)** | Только изменения с момента последнего бэкапа | Минимальный | Дольше (нужно восстановить full + все инкременты) |
| **Дифференциальный (Differential)** | Только изменения с момента последнего полного бэкапа | Средний | Быстрее инкрементального |

**Схема бэкапов:**
```
Full (воскресенье) → Diff (пн) → Diff (вт) → Diff (ср) → Full (след. воскресенье)
Full → Inc (пн) → Inc (вт) → Inc (ср) → Full
```

### Стратегии бэкапов

**1. RPO / RTO**

| Параметр | Описание | Пример |
|----------|----------|--------|
| **RPO (Recovery Point Objective)** | Максимально допустимая потеря данных | 5 минут → бэкапы каждые 5 минут |
| **RTO (Recovery Time Objective)** | Максимально допустимое время восстановления | 1 час → восстановление должно занять < 1 часа |

**2. 3-2-1 правило**
- **3** копии данных
- **2** разных носителя (например, SSD и S3)
- **1** копия вне основной площадки (offsite)

### Типы бэкапов в разных БД

| БД | Механизм бэкапа |
|----|-----------------|
| **PostgreSQL** | `pg_dump` (логический), `pg_basebackup` (физический), WAL архивация (PITR) |
| **MySQL** | `mysqldump`, `mysqlpump`, физические бэкапы (Percona XtraBackup) |
| **MongoDB** | `mongodump`, `mongorestore`, файловые снапшоты |
| **Cassandra** | `nodetool snapshot`, SSTable бэкапы |
| **S3** | Versioning, Cross-Region Replication, Object Lock |

### Point-in-Time Recovery (PITR)

**Что это:**
Восстановление базы данных на **конкретный момент времени** (например, за 5 секунд до случайного DROP TABLE).

**Как работает:**
1. Периодические полные бэкапы (например, каждые 24 часа)
2. Непрерывный архив WAL (write-ahead log) — все изменения
3. При восстановлении: full backup + replay WAL до нужного времени

**Пример (PostgreSQL):**
```bash
# Восстановить на 2024-03-22 14:30:00
restore_command = 'cp /archive/%f %p'
recovery_target_time = '2024-03-22 14:30:00'
```

### Бэкапы и шардирование

В шардированной системе бэкапы сложнее:
- **Каждый шард бэкапится отдельно**
- **Согласованность между шардами** — если восстанавливаем все шарды, нужно гарантировать, что они восстановлены на один момент времени
- **Инструменты:** для Cassandra есть `nodetool snapshot`, для Vitess — координация бэкапов

### На интервью

> *«Для бэкапов я использую стратегию 3-2-1: полный бэкап раз в день, WAL архивация для PITR (каждые 5 минут). Храним бэкапы на отдельном S3 (cross-region replication) и на ленточной библиотеке (Glacier) для долгосрочного хранения. RPO = 5 минут, RTO = 1 час. Регулярно проверяем восстановление — самые важные бэкапы восстанавливаем на staging»*.

## 2. CAP теорема

### Что это

**CAP теорема** (теорема Брюера) — фундаментальное ограничение распределенных систем. Утверждает, что в любой распределенной системе можно гарантировать **только два из трех** свойств одновременно:

| Свойство | Описание |
|----------|----------|
| **C — Consistency (Согласованность)** | Все узлы видят одни и те же данные в один момент времени. После записи чтение вернет последнее значение |
| **A — Availability (Доступность)** | Каждый запрос получает ответ (успешный или ошибка), даже если узлы недоступны |
| **P — Partition Tolerance (Устойчивость к разделению)** | Система продолжает работать даже при разрыве сети между узлами |

### Классическая диаграмма

```
        C (Consistency)
       /\
      /  \
     /    \
    /      \
   /  CP    \
  /          \  AP
 /            \
/              \
A (Availability) ────── P (Partition Tolerance)

CA — не существует в распределенных системах (только в single-node)
```

### Выбор в распределенной системе

| Выбор | Системы | Описание |
|-------|---------|----------|
| **CP (Consistency + Partition Tolerance)** | PostgreSQL (синхронная репликация), HBase, ZooKeeper, etcd | При разделении сети жертвуют доступностью, чтобы гарантировать консистентность |
| **AP (Availability + Partition Tolerance)** | Cassandra, DynamoDB, CouchDB | При разделении сети жертвуют консистентностью (eventual consistency), но остаются доступными |

### Примеры

**CP система (PostgreSQL с синхронной репликацией):**
```
Мастер и реплика соединены. При разрыве сети между ними:
- Мастер не может получить подтверждение от реплики
- Мастер блокирует запись (жертвует доступностью)
- Запись недоступна, но консистентность гарантирована
```

**AP система (Cassandra с QUORUM):**
```
При разрыве сети:
- Обе части кластера продолжают принимать записи
- Данные расходятся (консистентность нарушена)
- Но система доступна для записи
```

### Важное уточнение

CAP теорема применима **только при network partition**. В нормальном состоянии система может обеспечивать и C, и A.

### На интервью

> *«При проектировании распределенной системы я учитываю CAP теорему. Для платежной системы мне нужна **CP** — я готов пожертвовать доступностью в момент сетевого разделения, но не могу допустить расхождения данных. Для ленты новостей выбираю **AP** — пользователи могут видеть немного устаревшие посты, но система должна оставаться доступной всегда»*.

## 3. PACELC теорема

### Что это

**PACELC** — расширение CAP теоремы, учитывающее, что происходит **в нормальном режиме (без разделения)**.

**Акроним:**
- **P** — Partition (разделение)
- **A** — Availability (доступность)
- **C** — Consistency (согласованность)
- **E** — Else (иначе)
- **L** — Latency (задержка)
- **C** — Consistency (согласованность)

### Формулировка

> **If there is a partition (P), the system must choose between Availability (A) and Consistency (C).**
> **Else (E), when there is no partition, the system must choose between Latency (L) and Consistency (C).**

### Примеры систем

| Система | При разделении (P) | В нормальном режиме (E) |
|---------|-------------------|------------------------|
| **Cassandra** | AP | L (низкая задержка, eventual consistency) |
| **DynamoDB** | AP | L (низкая задержка, настраиваемая консистентность) |
| **PostgreSQL (синхронная)** | CP | C (сильная консистентность, выше задержка) |
| **MongoDB (по умолчанию)** | CP (primary) | C (strong consistency на primary) |
| **MySQL (асинхронная)** | CP (если failover) | L (низкая задержка, возможна потеря данных) |

### CAP vs PACELC

| Теорема | Что описывает |
|---------|---------------|
| **CAP** | Только поведение при network partition |
| **PACELC** | Поведение и при partition, и в нормальном режиме |

**Важность PACELC:** многие системы делают выбор **всегда**, а не только при сбоях. Например, Cassandra выбрала AP + L — даже в нормальном режиме она предпочитает низкую задержку строгой консистентности.

### На интервью

> *«PACELC теорема дает более полную картину. Например, Cassandra — это **AP/L** система: при разделении выбирает доступность, а в нормальном режиме предпочитает низкую задержку (eventual consistency). PostgreSQL с синхронной репликацией — **CP/C**: при разделении жертвует доступностью, а в нормальном режиме гарантирует сильную консистентность ценой большей задержки»*.

## 4. CDN (Content Delivery Network)

### Что это

**CDN** — географически распределенная сеть серверов (edge-узлов), которая доставляет контент пользователям с ближайшего узла.

### Как работает

```
1. Пользователь запрашивает image.example.com/photo.jpg
2. DNS направляет его на ближайший CDN edge-узел
3. Если контент есть в кэше CDN → отдается (HIT)
4. Если нет (MISS) → CDN загружает с origin (S3 / origin server), кэширует, отдает
```

### Зачем нужен CDN

| Цель | Описание |
|------|----------|
| **Снижение задержки** | Контент доставляется с ближайшего узла (географически) |
| **Разгрузка origin** | Основной сервер/S3 получает меньше запросов |
| **Масштабируемость** | CDN выдерживает огромные пики нагрузки (млн RPS) |
| **Безопасность** | DDoS protection, WAF, скрытие origin IP |
| **Стоимость** | Исходящий трафик CDN часто дешевле, чем direct из облака |

### Типы контента для CDN

| Тип | Примеры | Стратегия кэширования |
|-----|---------|----------------------|
| **Статика** | изображения, CSS, JS, видео | Долгий TTL (дни, годы) |
| **Динамика** | HTML страницы, API ответы | Короткий TTL (секунды, минуты) |
| **Потоковое видео** | HLS, DASH фрагменты | Долгий TTL, pre-fetching |
| **Пользовательский контент** | аватары, загруженные фото | TTL по согласованию |

### Популярные CDN

| CDN | Особенности |
|-----|-------------|
| **Cloudflare** | Бесплатный слой, WAF, DDoS, интеграция DNS |
| **AWS CloudFront** | Интеграция с S3, Lambda@Edge, AWS ecosystem |
| **Fastly** | Высокая производительность, real-time purging, VCL |
| **Akamai** | Enterprise, крупнейшая сеть |

### Инвалидация кэша CDN

| Метод | Описание |
|-------|----------|
| **TTL** | Автоматическое истечение (заголовок Cache-Control: max-age=3600) |
| **Purge / Invalidate** | Ручная инвалидация (CloudFront: `/*`, Fastly: purge by key) |
| **Versioning** | Добавление версии в URL (`image.jpg?v=2`) |
| **Cache Keys** | Учет заголовков (Accept-Encoding, Cookie) |

### CDN и динамический контент

**Edge Computing:** CDN может выполнять код на edge-узлах (Lambda@Edge, Cloudflare Workers) для:
- Персонализации контента
- A/B тестирования
- Авторизации (JWT проверка на edge)
- API aggregation

### На интервью

> *«Для статического контента (изображения, CSS, JS) я использую CDN. Конфигурирую CloudFront перед S3 — это снижает задержку для глобальной аудитории и разгружает origin. Для статики TTL = 1 год, версионирование через хеш в имени файла для инвалидации. Для динамических API с низкой латентностью рассматриваю CloudFront с коротким TTL или Cloudflare Workers для edge computing»*.

## 5. CDC (Change Data Capture)

### Что это

**CDC** — технология для отслеживания и обработки изменений в базе данных в реальном времени. Изменения (INSERT, UPDATE, DELETE) фиксируются и передаются в другие системы.

### Как работает CDC

```
БД (PostgreSQL) → WAL → Debezium → Kafka → Consumer (Analytics, Search, Cache)
```

### Основные подходы к CDC

| Подход | Описание | Плюсы | Минусы |
|--------|----------|-------|--------|
| **Trigger-based** | Триггеры в БД записывают изменения в отдельную таблицу | Простота, работает везде | Нагрузка на БД, сложность с DDL |
| **Log-based (WAL)** | Чтение WAL (транзакционного журнала) | Низкая нагрузка, capture всех изменений, PITR | Требует поддержки в БД (logical replication) |
| **Query-based** | Периодические запросы (SELECT с timestamp) | Простота | Высокая нагрузка, задержка |

### Популярные инструменты

| Инструмент | Поддержка БД | Особенности |
|------------|--------------|-------------|
| **Debezium** | PostgreSQL, MySQL, MongoDB, SQL Server, etc. | Kafka Connect, log-based, schema evolution, exactly-once |
| **AWS DMS** | Множество БД, в облаке | Managed, репликация, миграции |
| **Flink CDC** | MySQL, PostgreSQL, MongoDB | Интеграция с Apache Flink, stream processing |
| **Airbyte** | 100+ источников | ELT, open-source, cloud |

### Сценарии использования

| Сценарий | Пример |
|----------|--------|
| **Event-driven архитектура** | Изменения в БД → Kafka → микросервисы |
| **Аналитика в реальном времени** | БД заказов → ClickHouse для дашбордов |
| **Поисковая индексация** | Изменения в БД → Elasticsearch |
| **Инвалидация кэша** | Изменения в БД → Redis (delete/update) |
| **Аудит** | Все изменения → отдельное хранилище |
| **Data Lake** | CDC → Kafka → S3 (Parquet) → аналитика |

### Архитектура с CDC (Debezium + Kafka)

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ PostgreSQL  │────▶│  Debezium   │────▶│    Kafka    │
│   Master    │ WAL │ (Kafka Connect) │   (Topics)   │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                                │
                     ┌──────────────────────────┼──────────────────────────┐
                     ▼                          ▼                          ▼
              ┌─────────────┐           ┌─────────────┐           ┌─────────────┐
              │   Elastic   │           │   Redis     │           │   ClickHouse│
              │   Search    │           │   (Cache)   │           │  (Analytics)│
              └─────────────┘           └─────────────┘           └─────────────┘
```

### Exactly-once гарантии

Современные CDC инструменты (Debezium + Kafka) обеспечивают **exactly-once** семантику:
- Каждое изменение доставляется ровно один раз
- Сохранение позиции в WAL в Kafka (offset)
- При сбое восстановление с последнего offset

### Schema Evolution

CDC инструменты работают с эволюцией схемы:
- Debezium использует Avro + Schema Registry
- Новые колонки добавляются без остановки
- Совместимость схемы (forward/backward)

### На интервью

> *«Для интеграции между микросервисами и синхронизации данных я использую CDC. Debezium читает WAL PostgreSQL, публикует изменения в Kafka. Это дает нам:*
> - *Event-driven архитектуру без дублирования логики (микросервисы реагируют на события)*
> - *Real-time инвалидацию кэша (Redis подписывается на изменения)*
> - *Аналитику в ClickHouse с минимальной задержкой*
> - *Exactly-once гарантии и автоматическое восстановление при сбоях*
>
> *Альтернатива — dual-writes (приложение пишет и в БД, и в Kafka), но это сложнее с точки зрения атомарности и может привести к расхождениям»*.
