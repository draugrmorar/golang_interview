## 2. Оценка нагрузки (3-5 минут)

### Исходные допущения

При отсутствии точных цифр от интервьюера, берем реалистичные:
- 1 млрд активных пользователей (DAU)
- Каждый отправляет в среднем 10 сообщений в день
- Пик — 3x от среднего (праздники, утро)
- Средний размер сообщения: текст — 1 КБ, медиа — 100 КБ (но медиа — только 10% сообщений)

### Расчет RPS (запросов в секунду)

| Метрика | Формула | Результат |
|---------|---------|----------|
| Сообщений в день | 1B × 10 | 10 млрд сообщений/день |
| Сообщений в секунду (среднее) | 10B / 86,400 | ~115,000 сообщений/сек |
| Пик (×3) | 115,000 × 3 | ~350,000 сообщений/сек |

Это **только отправка**. Чтение (доставка получателю) — примерно столько же, плюс дополнительные запросы на синхронизацию, статусы, медиа.

**Итог:** система должна выдерживать ~500,000 RPS в пик.

### Расчет трафика

| Тип | Объем в сутки | Расчет |
|-----|---------------|--------|
| Текст | 10B × 1 КБ | 10 ТБ/день |
| Медиа (10% сообщений) | 1B × 100 КБ | 100 ТБ/день |
| **Итого** | | **110 ТБ/день** |

**Исходящий трафик** (пользователям) примерно такой же. В пик — 3-5 ТБ/сек.

### Хранение сообщений

| Период | Расчет | Объем |
|---------|--------|-------|
| За день | 110 ТБ | 110 ТБ |
| За год (с репликацией ×3) | 110 × 365 × 3 | ~120 ПБ |

Это очень много. На практике старые сообщения архивируются или удаляются.

## 3. Проектирование API

API должен быть простым и эффективным. Для мессенджера ключевое — поддержка реального времени.

### Основные эндпоинты (REST для "тяжелых" операций)

| Эндпоинт | Метод | Описание |
|----------|-------|----------|
| `/api/v1/users` | POST | Регистрация |
| `/api/v1/users/{id}/contacts` | GET | Список контактов |
| `/api/v1/messages/sync` | POST | Синхронизация истории (с курсором) |

### Real-time API (WebSocket)

Основной канал — WebSocket (или gRPC streaming). Одно соединение на устройство.

**Сообщения от клиента (client → server):**
```json
{
  "type": "message",
  "msg_id": "uuid",           // идемпотентный ключ
  "chat_id": "uuid",
  "text": "hello",
  "timestamp": 1700000000
}
```

**Сообщения от сервера (server → client):**
```json
{
  "type": "message",
  "msg_id": "uuid",
  "chat_id": "uuid",
  "from_user_id": "uuid",
  "text": "hello",
  "timestamp": 1700000000,
  "delivery_status": "delivered"
}
```

### Ключевые особенности API

1. **Идемпотентность:** каждое сообщение имеет msg_id, сгенерированный клиентом. Сервер использует его для дедупликации.

2. **Синхронизация:** при подключении клиент отправляет `last_sync_timestamp` или `last_msg_id`, сервер возвращает все пропущенные сообщения.

3. **Медиафайлы:** загружаются отдельно через HTTP (S3 signed URL), в сообщении передается только ссылка.

## Пояснение ролей сервисов

| Сервис | Тип | Что делает | Куда пишет / откуда читает |
|--------|-----|------------|---------------------------|
| **Presence Service** | Stateful | Держит WebSocket соединения, хранит mapping `user_id → connection_id` | Redis Cluster |
| **Message Service** | Stateless | Принимает сообщения от клиентов, кладет в Kafka, сразу отвечает "получено" | Kafka (`messages.inbound`) |
| **Delivery Service** | Stateless | Читает из Kafka, проверяет статус получателя (через Presence Service), отправляет онлайн-получателям, сохраняет офлайн-сообщения | Kafka (`messages.outbound`), Cassandra |
| **Media Service** | Stateless | Загружает медиафайлы в S3, выдает подписанные URL для скачивания | S3, CDN |

## Поток данных (пошагово)

1. **Отправка сообщения:**
    - Клиент A → API Gateway → Message Service
    - Message Service → кладет в Kafka (`messages.inbound`)
    - Message Service → отвечает клиенту: "сообщение получено" (одна галочка)

2. **Асинхронная обработка:**
    - Delivery Service → читает из Kafka (`messages.inbound`)
    - Delivery Service → сохраняет сообщение в Cassandra (таблицы `messages_by_chat` и `messages_by_user`)
    - Delivery Service → определяет получателя, через Presence Service узнает, онлайн ли он
    - Если онлайн → отправляет через WebSocket (Presence Service использует сохраненное соединение)
    - Если офлайн → сохраняет в Redis (очередь офлайн-сообщений) или помечает в Cassandra, что сообщение не доставлено

3. **Получение сообщения:**
    - Клиент B (онлайн) → получает сообщение через WebSocket → отправляет подтверждение (две галочки)
    - Клиент B (офлайн) → при подключении отправляет `last_sync_timestamp` → Delivery Service возвращает все сообщения с timestamp больше указанного

## Почему такое разделение важно

| Сервис | Обоснование |
|--------|-------------|
| **Message Service отдельно от Delivery Service** | Прием сообщений должен быть максимально быстрым и не зависеть от того, онлайн получатель или нет. Очередь (Kafka) позволяет разорвать эту связь. |
| **Presence Service отдельно** | Он stateful (держит WebSocket), его сложнее масштабировать. Выделение в отдельный сервис позволяет управлять им независимо. |
| **Media Service отдельно** | Загрузка больших файлов требует другой инфраструктуры (S3, CDN) и не должна блокировать доставку текстовых сообщений. |

## Как это звучит в ответе на интервью

> *«Архитектура строится вокруг четырех основных сервисов:*

> *1. **Message Service** (stateless) — принимает сообщения от клиентов, сразу кладет их в Kafka и отвечает клиенту "получено". Это дает низкую задержку и развязывает прием от доставки.*

> *2. **Kafka** — центральная очередь. Топик `messages.inbound` для входящих сообщений, `messages.outbound` — для исходящих. Это сглаживает пики и гарантирует, что сообщения не потеряются.*

> *3. **Delivery Service** (stateless) — читает из Kafka, сохраняет сообщения в Cassandra, и через **Presence Service** определяет, онлайн ли получатель. Если онлайн — отправляет через WebSocket, если нет — сохраняет для последующей синхронизации.*

> *4. **Presence Service** (stateful) — держит все WebSocket соединения и хранит в Redis mapping `user_id → connection_id`. Он шардирован по user_id, чтобы запросы одного пользователя всегда попадали на один инстанс.*

> *5. **Media Service** (stateless) — отдельный сервис для загрузки и раздачи медиафайлов через S3 и CDN, чтобы не перегружать основную инфраструктуру.*

> *Долговременное хранение — **Cassandra**: высокая скорость записи, линейное масштабирование, репликация. Денормализация: сообщения хранятся дважды — по чату (для отображения истории) и по пользователю (для быстрой синхронизации устройств)»*.

### Модель данных (Cassandra)

Cassandra — идеальный выбор для мессенджера: высокая скорость записи, линейное масштабирование, отказоустойчивость.

**Таблица сообщений в чате:**
```sql
CREATE TABLE messages_by_chat (
    chat_id uuid,
    created_at timestamp,
    message_id uuid,
    from_user_id uuid,
    text text,
    media_url text,
    PRIMARY KEY (chat_id, created_at, message_id)
) WITH CLUSTERING ORDER BY (created_at DESC);
```
- `chat_id` — partition key (все сообщения чата в одной партиции)
- `created_at` + `message_id` — clustering key (сортировка по времени)

**Таблица сообщений пользователя (для синхронизации):**
```sql
CREATE TABLE messages_by_user (
    user_id uuid,
    created_at timestamp,
    message_id uuid,
    chat_id uuid,
    text text,
    from_user_id uuid,
    PRIMARY KEY (user_id, created_at, message_id)
);
```
- Денормализация: те же сообщения дублируются для каждого участника чата
- Позволяет быстро синхронизировать историю пользователя

**Таблица метаданных чата:**
```sql
CREATE TABLE chats (
    chat_id uuid PRIMARY KEY,
    chat_type text,          -- 'one_to_one', 'group'
    participants set<uuid>,  -- список участников
    created_at timestamp,
    last_message text,
    last_message_time timestamp
);
```

### Очереди сообщений (Kafka)

**Зачем:** отделить прием сообщений от их обработки, сгладить пики, гарантировать доставку.

**Топики:**
| Топик | Назначение |
|-------|------------|
| `messages.inbound` | Входящие сообщения от клиентов |
| `messages.outbound` | Исходящие сообщения для доставки |
| `presence.updates` | События изменения статуса (онлайн/офлайн) |

**Поток обработки сообщения:**
1. Клиент отправляет сообщение через WebSocket
2. API Gateway → Message Service → сохраняет в Kafka (`messages.inbound`)
3. Message Service сразу отвечает клиенту "получено"
4. Consumer:
    - Сохраняет сообщение в Cassandra
    - Определяет получателей (из метаданных чата)
    - Отправляет событие в `messages.outbound` для каждого получателя
5. Delivery Service:
    - Читает `messages.outbound`
    - Если получатель онлайн → отправляет через WebSocket
    - Если офлайн → сохраняет в очередь офлайн-сообщений (в Redis или Cassandra)

## 5. Отказоустойчивость и масштабируемость

### Stateless vs Stateful компоненты

| Компонент | Тип | Почему | Масштабирование |
|-----------|-----|--------|-----------------|
| **API Gateway** | Stateless | Нет локального состояния, любая нода обрабатывает любой запрос | Горизонтальное (добавляем ноды) |
| **Message Service** | Stateless | Не хранит состояние, вся логика в Kafka и Cassandra | Горизонтальное |
| **Delivery Service** | Stateless | Доставляет сообщения, не хранит состояние | Горизонтальное |
| **Presence Service** | Stateful | Хранит mapping user_id → WebSocket connection (в памяти) | Шардирование по user_id (consistent hashing) |
| **Cassandra** | Stateful | Хранит данные | Репликация, шардирование |
| **Redis (presence)** | Stateful | In-memory хранилище сессий | Redis Cluster с шардированием |

### Отказоустойчивость по компонентам

| Компонент | Механизм защиты |
|-----------|-----------------|
| **API Gateway** | Несколько экземпляров, L4/L7 балансировщик перед ними |
| **Message Service / Delivery Service** | Запущены в Kubernetes с HPA (автомасштабирование), liveness/readiness probes |
| **Kafka** | Репликация (replication factor = 3), минимум 3 брокера |
| **Cassandra** | Репликация (RF=3), consistency level = QUORUM для записи, ONE для чтения |
| **Redis Cluster** | Репликация (master-replica), автоматический failover |
| **S3 / CDN** | Встроенная репликация, 11 девяток |

### Что происходит при отказе?

| Сценарий | Поведение |
|----------|-----------|
| **Падение одного экземпляра Message Service** | Балансировщик перестает отправлять трафик, Kubernetes создает новый pod |
| **Падение мастера Kafka** | Один из реплик становится лидером (контроллер выбирает новый) |
| **Падение узла Cassandra** | Клиенты перенаправляются на другие реплики, данные восстанавливаются из hinted handoff |
| **Потеря соединения WebSocket** | Клиент переподключается, отправляет курсор (последний полученный msg_id), сервер отправляет пропущенные сообщения |

### Масштабирование

- **Message Service:** горизонтальное масштабирование (Kubernetes HPA по CPU/RPS)
- **Kafka:** добавление брокеров, перераспределение партиций
- **Cassandra:** добавление узлов, виртуальные токены (vnodes) автоматически перераспределяют данные
- **Presence Service:** шардирование по user_id (consistent hashing), чтобы один пользователь всегда попадал на один и тот же экземпляр (иначе нельзя найти его WebSocket)

## 6. Расчет ресурсов

### Исходные данные
- 1 млрд DAU
- 350,000 сообщений/сек в пик
- Каждое сообщение обрабатывается несколькими сервисами (Message Service → Delivery Service)

### Расчет CPU для stateless сервисов

**Message Service:** принимает сообщения, сохраняет в Kafka
- Оценка: 1 vCPU на 2000 RPS (средняя нагрузка, легкая логика)
- Требуется: 350,000 / 2000 = 175 vCPU

**Delivery Service:** читает из Kafka, отправляет через WebSocket
- Оценка: 1 vCPU на 1000 RPS (чуть тяжелее, нужно держать WebSocket соединения)
- Требуется: 350,000 / 1000 = 350 vCPU

**С учетом отказоустойчивости и запаса (×1.5):**
- Message: 175 × 1.5 ≈ 260 vCPU → 65 инстансов по 4 vCPU
- Delivery: 350 × 1.5 ≈ 525 vCPU → 130 инстансов по 4 vCPU

### Расчет памяти для Presence Service (stateful)

Presence Service хранит mapping: `user_id → websocket_connection_id` в памяти.

| Метрика | Значение |
|---------|----------|
| Одновременно онлайн | ~500 млн (50% от DAU) |
| Память на запись | 200 байт (user_id + connection_id + метаданные) |
| **Итого RAM** | 500M × 200 байт ≈ 100 ГБ |

С репликацией и запасом → 150-200 ГБ RAM для кластера. Шардируем на 20 нод по 8-16 ГБ каждая.

### Расчет хранилища Cassandra

| Метрика | Значение |
|---------|----------|
| Сообщений в день | 10 млрд |
| Средний размер сообщения | 1 КБ (текст) + 0.5 КБ метаданные = 1.5 КБ |
| Данных в день | 10B × 1.5 КБ ≈ 15 ТБ |
| За 90 дней (retention) | 15 × 90 = 1.35 ПБ |
| С репликацией (RF=3) | 1.35 × 3 = 4 ПБ |
| С учетом индексов и запаса | ~5 ПБ |

**Диски:** 5 ПБ / 20 узлов = 250 ТБ на узел (HDD или NVMe в зависимости от требований к латентности).

### Расчет Kafka

| Метрика | Значение |
|---------|----------|
| Сообщений в день | 10 млрд (входящих) + 10 млрд (исходящих) = 20 млрд |
| Retention | 7 дней |
| Объем | 20B × 7 × 1.5 КБ ≈ 210 ТБ |
| С репликацией (RF=3) | 210 × 3 = 630 ТБ |

Kafka брокеры: 10-15 брокеров по 10-15 ТБ диска, 16-32 ГБ RAM.

### Расчет сети

| Направление | Трафик в пик |
|-------------|--------------|
| Входящие сообщения (текст) | 350k RPS × 1 КБ = 350 МБ/с |
| Входящие медиа | 35k RPS × 100 КБ = 3.5 ГБ/с |
| Исходящие (доставка) | аналогично входящим |

**Итог:** ~7-10 ГБ/с в пик, что требует сети 10-25 Гбит/с на дата-центр.
