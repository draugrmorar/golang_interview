> ## Архитектура информационных систем
> - Файл-сервер / клиент-сервер / P2P

---

**Архитектура информационных систем** — это эволюция подходов к организации взаимодействия между компонентами. Понимание этих архитектур помогает выбрать правильную модель для конкретной задачи и объяснить, почему в современных системах доминирует клиент-серверная архитектура, а P2P используется для специфических сценариев.

---

# АРХИТЕКТУРА ИНФОРМАЦИОННЫХ СИСТЕМ

## Введение

Эволюция архитектур отражает развитие вычислительных мощностей, сетей и требований к масштабируемости:

```
Файл-сервер → Клиент-сервер (2-tier) → Многоуровневая (N-tier) → Микросервисы
                      ↓
                  Peer-to-Peer (P2P)
```

---

## 1. Файл-серверная архитектура (File-Server Architecture)

### Что это

**Файл-серверная архитектура** — одна из самых ранних моделей, где данные хранятся на центральном сервере, но вся бизнес-логика и обработка выполняются **на клиенте**.

### Структура

```
┌─────────────────────────────────────────────────────────────┐
│                      Файловый сервер                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                   Файловое хранилище                 │   │
│  │  *.DBF, *.MDB, *.XLS, *.TXT, общие файлы            │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
         │              │              │              │
         ▼              ▼              ▼              ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│   Клиент 1  │ │   Клиент 2  │ │   Клиент 3  │ │   Клиент 4  │
│ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │
│ │Бизнес-  │ │ │ │Бизнес-  │ │ │ │Бизнес-  │ │ │ │Бизнес-  │ │
│ │логика   │ │ │ │логика   │ │ │ │логика   │ │ │ │логика   │ │
│ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │
│ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │
│ │UI       │ │ │ │UI       │ │ │ │UI       │ │ │ │UI       │ │
│ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │
└─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘
```

### Как работает

1. **Хранение данных:** Централизованный сервер предоставляет доступ к общим файлам (SMB/CIFS, NFS)
2. **Обработка:** Клиентские приложения загружают файлы целиком, выполняют операции локально
3. **Конкурентный доступ:** Используются файловые блокировки (locking) для предотвращения конфликтов

### Примеры

| Пример | Описание |
|--------|----------|
| **Microsoft Access** | .MDB файл на общем сетевом диске, несколько пользователей открывают и редактируют |
| **Paradox / dBase** | Старые СУБД с файловым доступом (.DBF файлы) |
| **1С:Предприятие (файловый режим)** | Общая папка с базой данных на сетевом диске |
| **Общие документы** | Word/Excel файлы в общей папке |

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Простота** | Минимум настройки, не требуется серверное ПО |
| **Низкая стоимость** | Нет лицензий на серверные СУБД |
| **Малая инфраструктура** | Достаточно сетевой папки |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Высокая сетевая нагрузка** | Передаются целые файлы, а не только нужные данные |
| **Конкурентность** | Файловые блокировки примитивны, часты конфликты |
| **Безопасность** | Доступ на уровне файлов, сложно разграничить права |
| **Целостность** | Нет транзакций, риск повреждения данных |
| **Масштабируемость** | Ограничена возможностями файловой системы и сети |
| **Локальность бизнес-логики** | Обновление логики требует обновления всех клиентов |

### Когда применяется (исторически)

- Малые организации (1-10 пользователей)
- Простые приложения с низкой интенсивностью изменений
- Сценарии, где важна простота, а не надежность

## 2. Клиент-серверная архитектура (Client-Server Architecture)

### Что это

**Клиент-серверная архитектура** — модель, где **сервер** управляет данными и бизнес-логикой, а **клиент** отвечает за представление (UI). Это доминирующая архитектура современных приложений.

### Структура (2-tier)

```
┌─────────────────────────────────────────────────────────────┐
│                        Сервер                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                  СУБД (PostgreSQL, MySQL)           │   │
│  │  ┌─────────────────────────────────────────────┐   │   │
│  │  │  Хранимые процедуры (опционально)           │   │   │
│  │  └─────────────────────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
         │              │              │              │
         ▼              ▼              ▼              ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│   Клиент 1  │ │   Клиент 2  │ │   Клиент 3  │ │   Клиент 4  │
│ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │
│ │UI       │ │ │ │UI       │ │ │ │UI       │ │ │ │UI       │ │
│ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │
│ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │ │ ┌─────────┐ │
│ │Бизнес-  │ │ │ │Бизнес-  │ │ │ │Бизнес-  │ │ │ │Бизнес-  │ │
│ │логика   │ │ │ │логика   │ │ │ │логика   │ │ │ │логика   │ │
│ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │ │ └─────────┘ │
└─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘
```

### Как работает

1. **Сервер:** Хранит данные, обеспечивает ACID, конкурентный доступ, транзакции
2. **Клиент:** Отправляет SQL-запросы или вызовы API, получает результаты, отображает UI
3. **Сеть:** Передаются только результаты запросов (не целые файлы)

### Эволюция: 2-tier → N-tier

**2-tier (Fat Client):**
- Клиент содержит UI + бизнес-логику
- Сервер — только СУБД
- Проблема: обновление логики требует обновления всех клиентов

```
[Клиент: UI + бизнес-логика] ←→ [Сервер: СУБД]
```

**3-tier (Thin Client):**
- Добавлен промежуточный слой — сервер приложений (Application Server)
- Бизнес-логика вынесена на сервер
- Клиент только UI (браузер или тонкий клиент)

```
[Клиент: UI] ←→ [Сервер приложений: бизнес-логика] ←→ [Сервер: СУБД]
```

**N-tier / Многоуровневая:**
- Web-слой, application-слой, data-слой, кэш, очереди, etc.
- Основа современных систем

```
[Браузер] ←→ [CDN] ←→ [Web Server] ←→ [App Server] ←→ [Cache] ←→ [DB]
                              ↓
                         [Queue] ←→ [Worker] ←→ [Analytics DB]
```

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Централизованное управление** | Данные и бизнес-логика на сервере |
| **Безопасность** | Контроль доступа на уровне БД/приложения |
| **Целостность** | ACID транзакции, констрейнты |
| **Масштабируемость** | Возможность горизонтального масштабирования |
| **Тонкий клиент** | Обновления на сервере, клиент не требует обновлений |
| **Современные протоколы** | HTTP/HTTPS, REST, gRPC, GraphQL |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Единая точка отказа** | Сервер — SPOF (решается кластеризацией) |
| **Сложность инфраструктуры** | Требует администрирования серверов |
| **Стоимость** | Лицензии, хостинг, администрирование |
| **Сетевая зависимость** | Не работает без сети |

### Где применяется

- **Все современные веб-приложения** (браузер → сервер)
- **Мобильные приложения** (app → API → БД)
- **Корпоративные системы** (ERP, CRM)
- **Сервисы с централизованными данными** (банки, соцсети, e-commerce)

## 3. Peer-to-Peer (P2P) архитектура

### Что это

**Peer-to-Peer** — децентрализованная архитектура, где **все узлы равноправны**. Каждый узел может выступать и как клиент, и как сервер. Нет центрального сервера.

### Структура

```
                    ┌─────────────┐
                    │   Peer 5    │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
       ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
       │   Peer 1    │ │   Peer 2    │ │   Peer 3    │
       └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
              │               │               │
              └───────────────┼───────────────┘
                              │
                       ┌─────────────┐
                       │   Peer 4    │
                       └─────────────┘

Каждый узел может быть:
- Клиентом (запрашивать данные)
- Сервером (отдавать данные)
- Маршрутизатором (передавать запросы)
```

### Типы P2P сетей

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Некструктурированные** | Случайные связи, поиск широковещательный | Gnutella, ранний Napster |
| **Структурированные** | DHT (Distributed Hash Table) для детерминированного поиска | BitTorrent DHT, Kademlia, IPFS |
| **Гибридные** | Есть центральные узлы (суперпиры) для координации | Napster (центральный индекс), Skype (ранний) |
| **Полностью децентрализованные** | Нет никаких центральных компонентов | BitTorrent (без трекера), IPFS |

### Как работает (на примере BitTorrent)

```
1. Файл разбивается на маленькие блоки (pieces)
2. Клиенты обмениваются блоками между собой
3. Трекер (или DHT) координирует, у кого какие блоки есть
4. Чем больше отдаешь, тем быстрее скачиваешь (tit-for-tat)
```

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Масштабируемость** | Чем больше пользователей, тем выше пропускная способность |
| **Отказоустойчивость** | Нет единой точки отказа |
| **Дешевизна** | Нет затрат на серверную инфраструктуру |
| **Анонимность** | Труднее цензурировать |
| **Устойчивость к нагрузкам** | Нагрузка распределяется между всеми узлами |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Сложность координации** | Нет центрального контроля, сложно управлять |
| **Безопасность** | Трудно аутентифицировать узлы, риск вредоносных пиров |
| **Консистентность** | Нет строгих гарантий согласованности |
| **Сложность разработки** | Сложные протоколы (DHT, Kademlia) |
| **Юридические вопросы** | Ассоциируется с пиратством |

### Где применяется

| Применение | Примеры | Почему P2P |
|------------|---------|------------|
| **Файлообмен** | BitTorrent, eMule | Масштабируемость, дешевизна |
| **Блокчейн** | Bitcoin, Ethereum | Децентрализация, доверие без центра |
| **Мессенджеры** | Tox, Matrix (децентрализованные) | Приватность, независимость |
| **Облачное хранилище** | IPFS, Filecoin | Децентрализованное хранение |
| **Видеоконференции** | WebRTC (P2P каналы) | Низкая задержка, экономия серверов |
| **Стриминг** | PeerTube | Альтернатива централизованным платформам |

### P2P в современных системах

Даже в централизованных системах часто используются P2P-элементы:

| Компонент | Использование |
|-----------|---------------|
| **WebRTC** | P2P каналы для видео/аудио в браузере (Zoom, Google Meet) |
| **CDN P2P** | Peer-to-peer CDN для стриминга (приставки обмениваются данными) |
| **Service Discovery** | Gossip протоколы (Consul, Cassandra) |
| **Кластеры БД** | Gossip для обнаружения узлов (Cassandra, Riak) |

## Сравнение архитектур

| Характеристика | Файл-сервер | Клиент-сервер | P2P |
|----------------|-------------|---------------|-----|
| **Централизация** | Высокая | Высокая | Нет |
| **SPOF** | Да | Да (без кластера) | Нет |
| **Масштабируемость** | Плохая | Хорошая (вертикально) | Отличная (горизонтально) |
| **Безопасность** | Низкая | Высокая | Средняя |
| **Целостность данных** | Низкая | Высокая (ACID) | Низкая/средняя |
| **Сложность реализации** | Низкая | Средняя | Высокая |
| **Сетевой трафик** | Высокий (весь файл) | Низкий (только данные) | Средний |
| **Стоимость инфраструктуры** | Низкая | Средняя/Высокая | Низкая |
| **Примеры** | Access, 1С (файловый) | Веб-приложения, API | BitTorrent, Blockchain |

## Эволюция и современное применение

### 1. Файл-сервер → Клиент-сервер

Переход произошел в 1990-х годах с распространением SQL-серверов и сетей. Причины:
- Снижение сетевого трафика
- Транзакции и целостность
- Безопасность

### 2. Клиент-сервер → Многоуровневая архитектура

С развитием интернета и веб-приложений:
- Web-серверы (Apache, Nginx)
- Серверы приложений (Tomcat, Node.js)
- Разделение слоев

### 3. P2P как специализированное решение

P2P не заменил клиент-сервер, но занял свои ниши:
- **Файлообмен** — где масштабируемость важнее управления
- **Блокчейн** — где нужна децентрализация
- **Медиастриминг** — где нужно экономить серверные ресурсы

### 4. Гибридные архитектуры

Современные системы часто комбинируют подходы:

```
┌─────────────────────────────────────────────────────────────┐
│                     Гибридная архитектура                   │
├─────────────────────────────────────────────────────────────┤
│  Централизованная часть (клиент-сервер):                    │
│  - Аутентификация и авторизация                             │
│  - Хранение метаданных                                      │
│  - Платежи и транзакции                                     │
├─────────────────────────────────────────────────────────────┤
│  P2P часть:                                                 │
│  - Передача больших файлов (WebRTC, BitTorrent)            │
│  - Распределенное хранение (IPFS)                          │
│  - Децентрализованные коммуникации                          │
└─────────────────────────────────────────────────────────────┘
```

## На интервью: пример ответа

> *«При проектировании архитектуры я исхожу из требований:*

> *Для **централизованного веб-приложения** я выбираю **клиент-серверную архитектуру** с N-tier разбиением: веб-сервер (Nginx), сервер приложений (Node.js), база данных (PostgreSQL), Redis для кэша. Это дает централизованное управление, безопасность и ACID-гарантии для транзакций.*

> *Если бы система требовала **максимальной масштабируемости при передаче больших файлов** (например, дистрибутивы ОС), я бы добавил **P2P-компонент** (BitTorrent) для разгрузки серверов. Пользователи скачивали бы друг у друга, а не с центрального сервера.*

> *Для **децентрализованного приложения** (например, криптовалютного кошелька) я бы выбрал **P2P архитектуру** полностью, используя DHT (Kademlia) для обнаружения узлов и распределенного хранения.*

> *В современных системах часто используется **гибридный подход**: централизованные компоненты для критических операций (аутентификация, платежи), и P2P-слои для масштабируемых операций (передача контента)»*.

> ## Паттерны развертывания сервисов
> - Rolling / blue-green / канареечный

---

**Паттерны развертывания** — это то, что обеспечивает **безопасность изменений**, **минимальный downtime** и **быстрый откат** при проблемах. На интервью понимание этих паттернов показывает, что вы умеете не только проектировать систему, но и поддерживать ее в production.

---

# ПАТТЕРНЫ РАЗВЕРТЫВАНИЯ СЕРВИСОВ

## Введение

### Зачем нужны паттерны развертывания

| Цель | Описание |
|------|----------|
| **Zero downtime** | Пользователи не замечают обновления |
| **Безопасность** | Возможность быстро откатить при проблемах |
| **Валидация** | Проверка новой версии на реальном трафике |
| **Минимизация риска** | Постепенное введение изменений |
| **Сохранение данных** | Совместимость схемы данных между версиями |

### Ключевые метрики

| Метрика | Описание |
|---------|----------|
| **MTTR (Mean Time To Recover)** | Среднее время восстановления при сбое |
| **Change Failure Rate** | Процент изменений, вызвавших инцидент |
| **Deployment Frequency** | Частота развертываний |

## 1. Rolling Deployment (катящееся развертывание)

### Что это

**Rolling deployment** — обновление сервиса **постепенно**, один инстанс за другим. В любой момент времени часть инстансов работает на старой версии, часть — на новой.

### Как работает

```
Начальное состояние (3 инстанса на версии v1):
┌─────────┐ ┌─────────┐ ┌─────────┐
│  v1     │ │  v1     │ │  v1     │
│Instance1│ │Instance2│ │Instance3│
└─────────┘ └─────────┘ └─────────┘
Балансировщик: трафик на все 3

Шаг 1: Обновляем Instance1
┌─────────┐ ┌─────────┐ ┌─────────┐
│  v2     │ │  v1     │ │  v1     │
│Instance1│ │Instance2│ │Instance3│
└─────────┘ └─────────┘ └─────────┘
Балансировщик: трафик на v1 и v2 (но Instance1 не принимает трафик до готовности)

Шаг 2: Instance1 прошел health check → трафик возвращается
┌─────────┐ ┌─────────┐ ┌─────────┐
│  v2     │ │  v1     │ │  v1     │
│Instance1│ │Instance2│ │Instance3│
└─────────┘ └─────────┘ └─────────┘
Балансировщик: трафик на все 3

Шаг 3: Обновляем Instance2
┌─────────┐ ┌─────────┐ ┌─────────┐
│  v2     │ │  v2     │ │  v1     │
│Instance1│ │Instance2│ │Instance3│
└─────────┘ └─────────┘ └─────────┘

Шаг 4: Обновляем Instance3
┌─────────┐ ┌─────────┐ ┌─────────┐
│  v2     │ │  v2     │ │  v2     │
│Instance1│ │Instance2│ │Instance3│
└─────────┘ └─────────┘ └─────────┘
Развертывание завершено
```

### Параметры настройки

| Параметр | Описание | Типичное значение |
|----------|----------|-------------------|
| **Max Surge** | Сколько лишних инстансов можно создать сверх желаемого количества | 1 или 25% |
| **Max Unavailable** | Сколько инстансов может быть недоступно во время обновления | 1 или 25% |
| **Health Check** | Как долго ждать готовности нового инстанса | 30-60 секунд |

### Пример (Kubernetes)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # можно создать 1 лишний инстанс
      maxUnavailable: 1  # 1 инстанс может быть недоступен
  template:
    spec:
      containers:
      - name: myapp
        image: myapp:v2
```

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Эффективность ресурсов** | Не требует дополнительных ресурсов (или минимально) |
| **Zero downtime** | Пользователи не замечают обновления |
| **Простота** | Легко автоматизировать |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Сложный откат** | Откат требует такого же поэтапного процесса (медленно) |
| **Совместимость версий** | v1 и v2 работают одновременно — нужна обратная совместимость API и схемы данных |
| **Длительность** | Может занимать много времени при большом количестве инстансов |
| **Сложность детекции проблем** | Проблемы могут проявиться не сразу, а только после обновления части инстансов |

### Когда применять

- **Большинство production-сервисов** (стандартный выбор)
- **Сервисы с горизонтальным масштабированием**
- **Когда обратная совместимость обеспечена**

## 2. Blue-Green Deployment (сине-зеленое развертывание)

### Что это

**Blue-Green deployment** — поддерживаются **две полностью изолированные среды**:
- **Blue** — текущая production-версия
- **Green** — новая версия, готовая к релизу

Переключение происходит **мгновенно** путем смены маршрутизации.

### Как работает

```
┌─────────────────────────────────────────────────────────────────┐
│                       Балансировщик                             │
│                    (Router / Load Balancer)                     │
│                         │                                       │
│              ┌──────────┴──────────┐                            │
│              │                     │                            │
│              ▼                     ▼                            │
│     ┌─────────────────┐   ┌─────────────────┐                   │
│     │   Blue (v1)     │   │  Green (v2)     │                   │
│     │   Production    │   │   Staging       │                   │
│     │  ┌───────────┐  │   │  ┌───────────┐  │                   │
│     │  │App v1     │  │   │  │App v2     │  │                   │
│     │  │DB (v1)    │  │   │  │DB (v2)    │  │                   │
│     │  └───────────┘  │   │  └───────────┘  │                   │
│     └─────────────────┘   └─────────────────┘                   │
│              │                     │                            │
│              └──────────┬──────────┘                            │
│                         │                                       │
│                         ▼                                       │
│              ┌─────────────────────┐                            │
│              │      Traffic        │                            │
│              │  100% → Blue        │                            │
│              └─────────────────────┘                            │
└─────────────────────────────────────────────────────────────────┘

После переключения (изменение routing):
┌─────────────────────────────────────────────────────────────────┐
│     ┌─────────────────┐   ┌─────────────────┐                   │
│     │   Blue (v1)     │   │  Green (v2)     │                   │
│     │   (ожидает)     │   │   Production    │                   │
│     └─────────────────┘   └─────────────────┘                   │
│              │                     │                            │
│              └──────────┬──────────┘                            │
│                         │                                       │
│                         ▼                                       │
│              ┌─────────────────────┐                            │
│              │      Traffic        │                            │
│              │  100% → Green       │                            │
│              └─────────────────────┘                            │
└─────────────────────────────────────────────────────────────────┘
```

### Этапы развертывания

| Этап | Действие |
|------|----------|
| **1. Подготовка** | Среда Green развернута с новой версией, но не принимает трафик |
| **2. Тестирование** | Green проверяется (smoke tests, интеграционные тесты) |
| **3. Переключение** | Балансировщик перенаправляет трафик на Green |
| **4. Мониторинг** | Наблюдение за метриками Green |
| **5. Откат (при необходимости)** | Переключение обратно на Blue |
| **6. Очистка** | Blue может оставаться для быстрого отката или удаляется |

### Инструменты для переключения

| Инструмент | Механизм |
|------------|----------|
| **DNS** | Изменение A/AAAA записи (может кэшироваться, не мгновенно) |
| **Load Balancer** | Изменение target groups (ALB, NLB) |
| **Service Mesh** | Изменение routing правил (Istio, Linkerd) |
| **Kubernetes Service** | Изменение селектора (мгновенно) |

### Пример (Kubernetes)

```yaml
# Blue deployment (v1)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: myapp
        image: myapp:v1

---
# Green deployment (v2)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: myapp
        image: myapp:v2

---
# Service (меняем селектор при переключении)
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    version: blue  # или green
  ports:
  - port: 80
```

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Мгновенный откат** | Простое переключение обратно на Blue |
| **Полная изоляция** | v1 и v2 не влияют друг на друга |
| **Тестирование до продакшн** | Можно тестировать Green на реальных данных (например, дублирование трафика) |
| **Предсказуемость** | Переключение — одна операция, а не последовательность |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Удвоение ресурсов** | Нужны дополнительные ресурсы для Green среды |
| **Совместимость БД** | Если схема БД меняется, нужно обеспечить совместимость с обеими версиями |
| **Сложность с состоянием** | Если Green пишет в ту же БД, что и Blue, могут быть конфликты |
| **Время развертывания** | Подготовка всей среды может занимать время |

### Когда применять

- **Критичные сервисы** (минимальный риск, быстрый откат)
- **Когда важна скорость отката**
- **Когда можно позволить дополнительные ресурсы**
- **Сервисы без состояния (stateless)** или с совместимой схемой БД

---

## 3. Canary Deployment (канареечное развертывание)

### Что это

**Canary deployment** — новая версия запускается для **небольшой части пользователей** (1-10%). Если метрики в норме, процент трафика постепенно увеличивается.

Название происходит от практики шахтеров, которые брали с собой канареек в шахты — птицы чувствовали угарный газ раньше людей.

### Как работает

```
Начальное состояние (100% на v1):
┌─────────────────────────────────────────────────────────────────┐
│  Балансировщик                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                     Traffic                             │   │
│  │                    100% → v1                            │   │
│  │                        │                                │   │
│  │         ┌──────────────┼──────────────┐                │   │
│  │         ▼              ▼              ▼                │   │
│  │    ┌─────────┐   ┌─────────┐   ┌─────────┐            │   │
│  │    │  v1     │   │  v1     │   │  v1     │            │   │
│  │    └─────────┘   └─────────┘   └─────────┘            │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘

Шаг 1: Запускаем 1 инстанс v2 (0% трафика)
┌─────────────────────────────────────────────────────────────────┐
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐        │
│  │  v1     │   │  v1     │   │  v1     │   │  v2     │        │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘        │
│  Traffic: 100% → v1, 0% → v2                                   │
└─────────────────────────────────────────────────────────────────┘

Шаг 2: Направляем 1% трафика на v2, мониторим
┌─────────────────────────────────────────────────────────────────┐
│  Traffic: 99% → v1, 1% → v2                                    │
│  Мониторинг: ошибки, латентность, CPU, память                  │
└─────────────────────────────────────────────────────────────────┘

Шаг 3: Если метрики OK → увеличиваем до 10%, затем 50%, затем 100%
┌─────────────────────────────────────────────────────────────────┐
│  Traffic: 50% → v1, 50% → v2                                   │
│  Добавляем инстансы v2 по мере роста трафика                   │
└─────────────────────────────────────────────────────────────────┘

Шаг 4: 100% на v2
┌─────────────────────────────────────────────────────────────────┐
│  Traffic: 100% → v2                                            │
│  v1 инстансы убираются                                         │
└─────────────────────────────────────────────────────────────────┘
```

### Стратегии разделения трафика

| Стратегия | Описание | Пример |
|-----------|----------|--------|
| **По проценту** | Случайное распределение | 5% пользователей на v2 |
| **По пользователям** | Определенная группа пользователей | VIP-пользователи на v2 |
| **По географическому признаку** | Определенный регион | EU регион на v2 |
| **По заголовкам** | Тестовые пользователи | `X-Canary: true` |
| **По IP** | Белый список | Внутренние IP на v2 |

### Автоматическое увеличение (Auto-promotion)

```yaml
canary:
  steps:
    - weight: 1%      # начальный вес
      duration: 5m    # наблюдение 5 минут
    - weight: 10%     # увеличиваем до 10%
      duration: 10m
    - weight: 50%
      duration: 30m
    - weight: 100%    # полный rollout
    
  analysis:
    metrics:
      - name: error_rate
        threshold: 1%  # если ошибок > 1% → откат
      - name: latency_p99
        threshold: 500ms
    duration: 2m       # анализ метрик каждые 2 минуты
```

### Инструменты

| Инструмент | Описание |
|------------|----------|
| **Kubernetes + Istio** | Traffic splitting через VirtualService |
| **Argo Rollouts** | Canary стратегии для Kubernetes |
| **Flagger** | Автоматический canary анализ |
| **Nginx / HAProxy** | Weighted routing |
| **AWS CodeDeploy** | Canary deployments |
| **LaunchDarkly** | Feature flags для canary |

### Пример (Istio VirtualService)

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
  - myapp
  http:
  - match:
    - headers:
        x-canary:
          exact: "true"
    route:
    - destination:
        host: myapp-v2
        weight: 100
  - route:
    - destination:
        host: myapp-v1
        weight: 90
    - destination:
        host: myapp-v2
        weight: 10
```

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Минимизация риска** | Проблемы затрагивают только малую часть пользователей |
| **Реальное тестирование** | v2 проверяется на реальном трафике, а не в staging |
| **Объективные метрики** | Сравнение метрик v1 и v2 под нагрузкой |
| **Постепенный rollout** | Можно остановить в любой момент |
| **А/B тестирование** | Можно сравнивать поведение версий |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Сложность** | Требует инфраструктуру для трафик-сплиттинга |
| **Совместимость версий** | v1 и v2 работают одновременно с одними данными |
| **Время** | Полный rollout занимает больше времени |
| **Сложность анализа** | Нужны четкие метрики для автоматического решения |

### Когда применять

- **Высокорисковые изменения** (критическая бизнес-логика)
- **Когда нужна уверенность** перед полным rollout
- **Сервисы с большим трафиком** (статистически значимая выборка)
- **A/B тестирование** новых фич

## Сравнение паттернов развертывания

| Характеристика | Rolling | Blue-Green | Canary |
|----------------|---------|------------|--------|
| **Дополнительные ресурсы** | Нет | +100% (временные) | +N% (процент канареек) |
| **Время развертывания** | Среднее (зависит от кол-ва инстансов) | Мгновенное (переключение) | Долгое (постепенное) |
| **Время отката** | Медленное (обратный rolling) | Мгновенное (переключение) | Мгновенное (убрать canary) |
| **Сложность** | Низкая | Средняя | Высокая |
| **Риск** | Средний | Низкий | Минимальный |
| **Тестирование на реальном трафике** | Нет | Ограниченное (только после переключения) | Да (на canary группе) |
| **Совместимость версий** | Нужна (v1 и v2 работают вместе) | Не нужна (изолированные среды) | Нужна (v1 и v2 работают вместе) |
| **Где применяется** | Стандартные сервисы | Критичные сервисы, быстрый откат | Высокорисковые изменения, A/B тесты |

## Совместимость схемы данных

**Ключевая проблема** всех паттернов, где v1 и v2 работают одновременно:

### Проблема
```
v1 пишет в БД → v2 читает → непонимание новой схемы
v2 пишет в БД → v1 читает → поломка
```

### Решения

| Решение | Описание |
|---------|----------|
| **Backward compatibility** | v2 умеет читать формат v1 |
| **Forward compatibility** | v1 игнорирует новые поля |
| **Database migrations** | Миграции с сохранением совместимости (3-фазные миграции) |
| **Feature flags** | Новая функциональность отключена, пока все на v2 |

### 3-фазные миграции БД

```
Фаза 1: Добавляем новые колонки (NULLABLE)
       v1 и v2 работают, v2 пишет в старые и новые

Фаза 2: Backfill (заполняем новые колонки)
       Все еще оба работают

Фаза 3: Переключаем v2 на чтение из новых колонок
       Убираем v1

Фаза 4: Удаляем старые колонки
       Только v2
```

## На интервью: пример ответа

> *«Для развертывания сервиса я выбираю стратегию в зависимости от риска:*

> *Для **обычных сервисов с низким риском** я использую **rolling deployment** — он эффективен по ресурсам и дает zero downtime. В Kubernetes это настройка по умолчанию с maxSurge=1 и maxUnavailable=1.*

> *Для **критичных сервисов** (платежи, аутентификация) я выбираю **blue-green deployment**. Это требует вдвое больше ресурсов на время обновления, но дает мгновенный откат при проблемах — просто переключаем балансировщик обратно. Откат занимает секунды, а не минуты.*

> *Для **высокорисковых изменений** (рефакторинг критической логики, изменения схемы данных) я использую **canary deployment**. Сначала 1% трафика на новую версию, мониторим ошибки и латентность. Если метрики в норме, увеличиваем до 10%, 50%, 100%. При проблемах откат мгновенный — просто убираем canary-инстансы.*

> *Для **всех стратегий** я обеспечиваю **обратную совместимость API и схемы данных** — новые версии умеют читать старые форматы, старые версии игнорируют новые поля. Миграции БД провожу в 3 фазы, чтобы v1 и v2 могли работать одновременно»*.

**Архитектура бэкенда** — это один из самых важных архитектурных выборов, который определяет, как будет развиваться система, как команды будут работать, и как система будет масштабироваться. На интервью этот вопрос возникает практически всегда, когда речь идет о проектировании крупной системы.

# АРХИТЕКТУРА БЭКЕНДА

## Введение

Эволюция архитектур бэкенда отражает рост сложности систем и команд:

```
Монолит → Модульный монолит → SOA → Микросервисы
                                    ↓
                              Микросервисы + Serverless
```

**Ключевой принцип:** нет "лучшей" архитектуры — есть **компромисс** между сложностью, скоростью разработки и масштабируемостью.

---

## 1. Монолитная архитектура (Monolithic Architecture)

### Что это

**Монолит** — это единое приложение, где все компоненты (UI, бизнес-логика, доступ к данным, интеграции) собраны в один исполняемый файл (или один кодовая база), который развертывается как единое целое.

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                      Монолитное приложение                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    Web / API Layer                       │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐  │   │
│  │  │  Users   │ │  Orders  │ │Products  │ │Payments  │  │   │
│  │  │ Module   │ │  Module  │ │ Module   │ │ Module   │  │   │
│  │  └──────────┘ └──────────┘ └──────────┘ └──────────┘  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                    Data Access Layer                     │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │                     Shared Database                      │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │  Единый деплой  │
                    │  (JAR, WAR,     │
                    │   Docker image) │
                    └─────────────────┘
```

### Разновидности монолитов

| Тип | Описание | Пример |
|-----|----------|--------|
| **Single-process monolith** | Один процесс, все модули внутри | Rails app, Spring Boot app |
| **Modular monolith** | Модули внутри, но с четкими границами | Хорошо структурированный код |
| **Multiple-process monolith** | Несколько процессов, но единый деплой и база данных | Несколько сервисов, но общий release cycle |

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Простота разработки** | Одна кодовая база, один язык, один инструментарий |
| **Простота деплоя** | Один артефакт, один pipeline |
| **Простота отладки** | Все логи в одном месте, легко трассировать |
| **Производительность** | Нет сетевых вызовов между модулями (in-memory вызовы) |
| **ACID транзакции** | Единая база данных, простота транзакций |
| **Меньше операционной сложности** | Меньше движущихся частей |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Сложность масштабирования** | Масштабировать можно только целиком (даже если нагружен только один модуль) |
| **Сложность изменений** | Любое изменение требует деплоя всего приложения |
| **Связанность команд** | Несколько команд работают в одной кодовой базе → конфликты, координация |
| **Технологическая стагнация** | Трудно внедрить новый язык/фреймворк |
| **Риск "big bang"** | Ошибка в одном модуле может уронить все приложение |
| **Долгий CI/CD** | Сборка и тестирование занимают много времени |
| **Сложность выноса** | Со временем монолит становится "big ball of mud" |

### Когда выбирать монолит

✅ **Подходит для:**
- **Startup / MVP** — быстрая разработка, неизвестные требования
- **Небольшие команды** (1-5 разработчиков)
- **Приложения с простой бизнес-логикой**
- **Когда скорость разработки важнее масштабируемости**

❌ **Не подходит для:**
- **Крупных команд** (> 10 разработчиков)
- **Сложных систем с множеством доменов**
- **Систем, требующих независимого масштабирования компонентов**

---

## 2. SOA (Service-Oriented Architecture)

### Что это

**SOA** — архитектура, ориентированная на сервисы. Это **промежуточный этап** между монолитом и микросервисами. Сервисы — это крупные бизнес-компоненты, которые обмениваются данными через **ESB** (Enterprise Service Bus).

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                        Enterprise Service Bus (ESB)             │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Маршрутизация / Трансформация / Оркестрация / Протоколы│   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
         │              │              │              │
         ▼              ▼              ▼              ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│   Сервис    │ │   Сервис    │ │   Сервис    │ │   Сервис    │
│ Пользователи│ │   Заказы    │ │   Товары    │ │   Платежи   │
│   (крупный) │ │  (крупный)  │ │  (крупный)  │ │  (крупный)  │
└─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘
      │               │               │               │
      └───────────────┼───────────────┼───────────────┘
                      ▼
            ┌─────────────────┐
            │  Shared Data    │
            │    Layer        │
            │ (общие данные)  │
            └─────────────────┘
```

### Ключевые особенности SOA

| Характеристика | Описание |
|----------------|----------|
| **ESB (Enterprise Service Bus)** | Центральный компонент для маршрутизации, трансформации, оркестрации |
| **Крупные сервисы** | Сервисы соответствуют бизнес-доменам (CRM, ERP, Billing) |
| **Общие данные** | Сервисы могут иметь общую базу данных или общий слой данных |
| **SOAP / XML** | Часто используется SOAP, WSDL, XML |
| **Централизованное управление** | Единая точка управления сервисами |

### Преимущества SOA

| Преимущество | Описание |
|--------------|----------|
| **Повторное использование** | Сервисы могут использоваться разными приложениями |
| **Стандартизация** | Единые протоколы, форматы (SOAP, XML, WS-*) |
| **Централизованное управление** | ESB дает единую точку мониторинга, маршрутизации |
| **Интеграция legacy** | ESB может соединять старые системы |

### Недостатки SOA

| Недостаток | Описание |
|------------|----------|
| **ESB становится монолитом** | ESB становится центральной точкой отказа и сложности |
| **Сложность** | SOAP, WSDL, XML — тяжеловесные протоколы |
| **Медленные изменения** | Изменения требуют координации через ESB |
| **Общие данные** | Сложность изменений схемы данных |
| **Технологическая связанность** | Все сервисы часто привязаны к ESB |

### SOA vs Микросервисы

| Характеристика | SOA | Микросервисы |
|----------------|-----|--------------|
| **Размер сервисов** | Крупные (доменные) | Мелкие (один бизнес-капус) |
| **Коммуникация** | ESB (централизованная) | Прямая (HTTP/gRPC) + очередь |
| **Протоколы** | SOAP, XML, WS-* | REST, gRPC, GraphQL |
| **Управление данными** | Общая БД или общий слой | Каждый сервис владеет своей БД |
| **Деплой** | Совместный или независимый | Независимый |
| **Сложность** | Высокая (ESB) | Высокая (распределенная система) |

---

## 3. Микросервисная архитектура (Microservices Architecture — MSA)

### Что это

**Микросервисная архитектура** — это подход, при котором приложение строится как набор **независимо развертываемых сервисов**, каждый из которых реализует **один бизнес-капус** и владеет **своими данными**.

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                         API Gateway                             │
│  (Routing, Auth, Rate Limiting, Caching, Aggregation)          │
└─────────────────────────────────────────────────────────────────┘
         │              │              │              │
         ▼              ▼              ▼              ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│  User       │ │  Order      │ │  Product    │ │  Payment    │
│  Service    │ │  Service    │ │  Service    │ │  Service    │
│  ┌───────┐  │ │  ┌───────┐  │ │  ┌───────┐  │ │  ┌───────┐  │
│  │  DB   │  │ │  │  DB   │  │ │  │  DB   │  │ │  │  DB   │  │
│  └───────┘  │ │  └───────┘  │ │  └───────┘  │ │  └───────┘  │
└─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘
      │               │               │               │
      └───────────────┼───────────────┼───────────────┘
                      ▼
            ┌─────────────────┐
            │   Message Bus   │
            │   (Kafka,       │
            │    RabbitMQ)    │
            └─────────────────┘
```

### Ключевые принципы

| Принцип | Описание |
|---------|----------|
| **Single Responsibility** | Каждый сервис отвечает за один бизнес-капус |
| **Database per Service** | Каждый сервис владеет своей БД (нет shared database) |
| **Independent Deployment** | Сервисы деплоятся независимо |
| **Decentralized Governance** | Команды выбирают технологии для своих сервисов |
| **Failure Isolation** | Отказ одного сервиса не должен падать всю систему |
| **API-first** | Сервисы общаются через четкие API |

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Независимое масштабирование** | Масштабируем только те сервисы, которые нуждаются |
| **Независимые деплои** | Изменение в одном сервисе не требует деплоя других |
| **Технологическая свобода** | Каждый сервис может использовать свой стек |
| **Автономность команд** | Команды могут работать независимо |
| **Изоляция отказов** | Падение одного сервиса не убивает всю систему |
| **Упрощение кодовой базы** | Каждый сервис меньше и проще монолита |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Операционная сложность** | Много сервисов = много деплоев, конфигураций, мониторинга |
| **Сетевая задержка** | Вместо in-memory вызовов — сетевые вызовы |
| **Сложность транзакций** | ACID транзакции между сервисами невозможны (SAGA, eventual consistency) |
| **Распределенная трассировка** | Трудно отлаживать запросы, проходящие через много сервисов |
| **Сложность тестирования** | Интеграционное тестирование сложнее |
| **Управление данными** | Денормализация, синхронизация данных |
| **Versioning API** | Необходимость поддерживать совместимость версий |

### Когда выбирать микросервисы

✅ **Подходит для:**
- **Крупные команды** (> 20 разработчиков, несколько команд)
- **Сложные системы с четкими доменами**
- **Разные требования к масштабированию** компонентов
- **Необходимость независимого деплоя**
- **Разные технологии** для разных компонентов

❌ **Не подходит для:**
- **MVP / Startups** (слишком большая операционная сложность)
- **Небольшие команды** (< 10 разработчиков)
- **Простые приложения**
- **Системы, где важны ACID транзакции между компонентами**

---

## Сравнение архитектур

| Характеристика | Монолит | SOA | Микросервисы |
|----------------|---------|-----|--------------|
| **Размер кодовой базы** | Один большой репозиторий | Крупные сервисы | Много маленьких |
| **Деплой** | Единый | Координированный | Независимый |
| **Масштабирование** | Вертикальное (или копии) | Покомпонентное | Покомпонентное |
| **Команды** | Одна команда | Несколько команд | Много команд |
| **Технологический стек** | Единый | Ограниченно гибкий | Полностью гибкий |
| **Транзакции** | ACID (просто) | ACID (сложнее) | Eventual (SAGA) |
| **Коммуникация** | In-memory (быстро) | ESB | HTTP/gRPC + Queue |
| **Операционная сложность** | Низкая | Средняя | Высокая |
| **Скорость разработки (начало)** | Высокая | Средняя | Низкая |
| **Скорость разработки (масштаб)** | Низкая | Средняя | Высокая |
| **Изоляция отказов** | Нет (все падает) | Частичная | Высокая |

## Стратегии перехода: Монолит → Микросервисы

### 1. Strangler Pattern (паттерн "душитель")

Постепенное вытеснение монолита микросервисами.

```
┌─────────────────────────────────────────────────────────────────┐
│                         API Gateway                             │
│  (маршрутизирует запросы на монолит или микросервисы)          │
└─────────────────────────────────────────────────────────────────┘
         │              │              │
         ▼              ▼              ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────────────────────┐
│   User      │ │   Order     │ │        Монолит              │
│   Service   │ │   Service   │ │  (постепенно уменьшается)  │
│   (новый)   │ │   (новый)   │ │                            │
└─────────────┘ └─────────────┘ └─────────────────────────────┘
```

### 2. Выделение сервисов по доменам

1. **Выделить модуль** (например, авторизация)
2. **Создать API** для монолита
3. **Перенести данные** (сложный этап)
4. **Переключить трафик** через API Gateway
5. **Повторить** для следующего модуля

### 3. Пример этапов миграции

| Этап | Действие |
|------|----------|
| **1** | Монолит + общая БД |
| **2** | Выделяем User Service, но данные остаются в общей БД |
| **3** | User Service получает свою БД, синхронизация через CDC |
| **4** | Монолит начинает читать данные через API User Service |
| **5** | Монолит перестает писать в User таблицы |
| **6** | Повторяем для следующего модуля |

---

## Гибридные подходы

### Модульный монолит + Микросервисы

Некоторые компоненты вынесены в микросервисы, остальное — хорошо структурированный монолит.

```
┌─────────────────────────────────────────────────────────────────┐
│                    Модульный монолит                            │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                       │
│  │  Users   │ │  Orders  │ │Products  │                       │
│  │ Module   │ │  Module  │ │ Module   │                       │
│  └──────────┘ └──────────┘ └──────────┘                       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │   Payment       │
                    │   Service       │  ← вынесен как микросервис
                    │   (Node.js)     │
                    └─────────────────┘
```

### Когда применять

- **Масштабируемый компонент** вынесен (например, платежи)
- **Компонент на другом языке** (например, ML-сервис на Python)
- **Команда работает над отдельным модулем**

## На интервью: пример ответа

> *«При выборе архитектуры я исхожу из размера команды, сложности системы и требований к масштабированию.*

> *Если это **MVP или стартап с командой 3-5 человек**, я выбираю **модульный монолит** — он дает быструю разработку, простой деплой и возможность позже выделить сервисы, когда требования станут ясны.*

> *Для **средней системы с командой 10-20 человек**, где есть четкие бизнес-домены, я выбираю **SOA или гибрид** — несколько крупных сервисов с общей инфраструктурой.*

> *Для **крупной системы с несколькими командами (20+ разработчиков)**, где компоненты имеют разные требования к масштабированию, я выбираю **микросервисную архитектуру**. Это позволяет командам работать независимо, масштабировать компоненты по отдельности и выбирать технологии под задачи.*

> *При переходе от монолита к микросервисам я использую **strangler pattern** — постепенное вытеснение монолита, начиная с наименее связанных компонентов. Критически важно начать с **изоляции данных**: каждый сервис должен владеть своей БД, а не разделять общую. Без этого микросервисы превращаются в "распределенный монолит" со всеми недостатками обоих подходов»*.

> ## Событийно-ориентированная архитектура
> - Event notification / state transfer / event collaboration

---

**Событийно-ориентированная архитектура (Event-Driven Architecture — EDA)** — это подход, при котором компоненты системы взаимодействуют через **асинхронные события**, а не через прямые синхронные вызовы. Это фундамент для построения слабосвязанных, масштабируемых и отказоустойчивых систем.

# СОБЫТИЙНО-ОРИЕНТИРОВАННАЯ АРХИТЕКТУРА

## Введение

### Что такое событие

**Событие** — это факт, который произошел в системе и представляет интерес для других компонентов. Событие — это **имя существительное в прошедшем времени**: `OrderCreated`, `PaymentProcessed`, `UserRegistered`.

### Синхронная vs Асинхронная коммуникация

| Характеристика | Синхронная (Request-Response) | Асинхронная (Event-Driven) |
|----------------|-------------------------------|----------------------------|
| **Связность** | Высокая (знаем о потребителе) | Низкая (не знаем о потребителях) |
| **Задержка** | Суммарная (ждем всех) | Минимальная (только отправка) |
| **Отказоустойчивость** | Каскадные отказы | Изоляция отказов |
| **Масштабируемость** | Ограничена | Высокая |
| **Трассировка** | Простая | Сложная |

---

## Ключевые компоненты EDA

```
┌─────────────────────────────────────────────────────────────────┐
│                      Event Bus (Kafka, RabbitMQ)                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Topic: orders                                          │   │
│  │  ├── OrderCreated (event)                               │   │
│  │  ├── OrderPaid (event)                                  │   │
│  │  └── OrderShipped (event)                               │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
         ▲                             │
         │                             ▼
┌─────────────────┐           ┌─────────────────┐
│   Producer      │           │   Consumer      │
│ (публикует      │           │ (подписывается  │
│  события)       │           │  на события)    │
│                 │           │                 │
│ Order Service   │           │ Payment Service │
│                 │           │ Inventory Svc   │
│                 │           │ Notification   │
└─────────────────┘           └─────────────────┘
```

## 1. Event Notification (уведомление о событии)

### Что это

**Event Notification** — самый простой и распространенный паттерн. Сервис публикует событие, чтобы **уведомить** другие сервисы о том, что **что-то произошло**. Другие сервисы могут **реагировать** на это событие.

### Ключевая идея

- **Производитель** не знает, кто получит событие
- **Потребитель** получает событие и выполняет свои действия
- Событие содержит **минимальный объем данных** (обычно ID сущности)

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                      Order Service                              │
│                                                                 │
│  1. Создает заказ в своей БД                                    │
│  2. Публикует событие OrderCreated { orderId: 123 }            │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                         Event Bus                               │
│  Topic: orders                                                  │
│  └── OrderCreated { orderId: 123, userId: 456, total: 100 }    │
└─────────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ Payment Service │ │Inventory Service│ │Notification Svc │
│                 │ │                 │ │                 │
│ Получает        │ │ Получает        │ │ Получает        │
│ orderId         │ │ orderId         │ │ orderId         │
│ Запрашивает     │ │ Резервирует     │ │ Отправляет      │
│ детали заказа   │ │ товары          │ │ email           │
│ через API       │ │                 │ │ пользователю    │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

### Пример события

```json
{
  "eventId": "evt_123456",
  "eventType": "OrderCreated",
  "timestamp": "2024-03-22T10:30:00Z",
  "data": {
    "orderId": 12345,
    "userId": 67890,
    "total": 99.99
  }
}
```

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Простота** | Легко понять и реализовать |
| **Слабая связность** | Производитель не знает о потребителях |
| **Асинхронность** | Основной сервис не ждет потребителей |
| **Масштабируемость** | Легко добавлять новых потребителей |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Дополнительные запросы** | Потребителям может потребоваться запросить дополнительные данные через API |
| **Сложность консистентности** | Потребители могут видеть несогласованные данные |
| **Задержка** | Между событием и реакцией есть задержка |

### Когда применять

- **Уведомления** (email, push)
- **Обновление кэша** (инвалидация)
- **Асинхронная обработка** (генерация отчетов)
- **Интеграции**, где задержка допустима

---

## 2. State Transfer (передача состояния)

### Что это

**State Transfer** — паттерн, при котором событие содержит **все необходимые данные** (или их достаточно), чтобы потребитель мог обработать событие **без дополнительных запросов** к производителю.

### Ключевая идея

- Событие содержит **полное состояние** объекта
- Потребитель может обновить свой локальный кэш или хранилище
- Снижается количество синхронных вызовов между сервисами

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                      Order Service                              │
│                                                                 │
```

## 3. Event Collaboration (совместная работа через события)

### Что это

**Event Collaboration** (также известен как **Event Sourcing**) — наиболее сложный паттерн, где **события являются источником истины**. Состояние системы не хранится напрямую, а **вычисляется** из последовательности событий.

### Ключевая идея

- **События** — это единственный источник истины
- **Текущее состояние** — это результат воспроизведения всех событий
- Сервисы **коллаборируют**, публикуя события, на которые реагируют другие

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                      Event Store (Kafka, EventStoreDB)         │
│                                                                 │
│  Order Stream / Topic:                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 1. OrderCreated { ... }                                 │   │
│  │ 2. OrderItemAdded { ... }                               │   │
│  │ 3. OrderPaymentInitiated { ... }                        │   │
│  │ 4. OrderPaid { ... }                                    │   │
│  │ 5. OrderShipped { ... }                                 │   │
│  │ 6. OrderDelivered { ... }                               │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│ Order Service   │ │ Payment Service │ │ Shipping Service│
│ (read model)    │ │ (read model)    │ │ (read model)    │
│                 │ │                 │ │                 │
│ Слушает события │ │ Слушает события │ │ Слушает события │
│ Строит свое     │ │ Строит свое     │ │ Строит свое     │
│ представление   │ │ представление   │ │ представление   │
│ (projection)    │ │ (projection)    │ │ (projection)    │
└─────────────────┘ └─────────────────┘ └─────────────────┘
```

### Event Store (Хранилище событий)

```
Event Store:
┌─────────────────────────────────────────────────────────────────┐
│ Aggregate: Order-123                                           │
├─────┬─────────────────────────────┬───────────────────────────┤
│ Seq │ Event Type                  │ Data                      │
├─────┼─────────────────────────────┼───────────────────────────┤
│ 1   │ OrderCreated                │ { userId: 456, ... }      │
│ 2   │ OrderItemAdded              │ { productId: 1, qty: 2 }  │
│ 3   │ OrderPaymentInitiated       │ { paymentId: 789 }        │
│ 4   │ OrderPaid                   │ { amount: 99.99 }         │

Текущее состояние Order-123 вычисляется воспроизведением:
- Created → статус "created"
- ItemAdded → добавить товар
- PaymentInitiated → статус "payment_pending"
- Paid → статус "paid"
- Shipped → статус "shipped"
```

### Projections (представления)

Разные сервисы строят разные представления из одних и тех же событий:

| Сервис | Projection (что хранит) |
|--------|------------------------|
| **Order Service** | Текущий статус заказа, товары, сумму |
| **Payment Service** | Платежи по заказам |
| **Analytics Service** | Суммы, категории, временные ряды |
| **Notification Service** | Последнее событие для уведомления |

### Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Полная аудитория** | Каждое изменение зафиксировано навсегда |
| **Time travel** | Можно восстановить состояние на любой момент времени |
| **Отладка** | Можно воспроизвести проблему, переиграв события |
| **Свободная эволюция** | Можно добавить новое представление из старых событий |
| **Слабая связность** | Сервисы коллаборируют через события |
| **Консистентность** | События — единственная истина |

### Недостатки

| Недостаток | Описание |
|------------|----------|
| **Сложность** | Значительно сложнее CRUD-подхода |
| **Событийная модель** | Требует переосмысления бизнес-логики |
| **Согласованность** | Eventual consistency между представлениями |
| **Хранение** | Много событий, нужна компактификация (snapshots) |
| **Миграции** | Изменение структуры события требует работы со старыми событиями |

### Snapshots (снимки)

Чтобы не воспроизводить миллионы событий каждый раз, делают **снапшоты**:

```
Order-123:
├── Snapshot (после 1000 событий) → состояние
├── Событие 1001
├── Событие 1002
└── ...
```

### Когда применять

- **Финансовые системы** (полный аудит, невозможно потерять изменения)
- **Системы бронирования** (нужен history)
- **Системы с требованиями к аудиту**
- **Сложные бизнес-процессы** с множеством состояний
- **Когда нужно "time travel" и отладка**

| Характеристика | Event Notification | State Transfer | Event Collaboration (Event Sourcing) |
|----------------|-------------------|----------------|-------------------------------------|

| **Сложность** | Низкая | Средняя | Высокая |
| **Когда применять** | Уведомления, инвалидация кэша | Поиск, аналитика, кэши | Финансы, аудит, сложные процессы |

```python
    def create_order(self, user_id, items):

        order = Order(user_id=user_id, items=items)

class PaymentService:
    def handle_order_created(self, event):

        total = event.total
        
        # Инициируем платеж
        payment = self.initiate_payment(order_id, total)
        
        # Публикуем результат
        self.event_bus.publish("payments", PaymentInitiated(
            payment_id=payment.id,
            order_id=order_id,
            amount=total
        ))
```

### 1. Гарантии доставки

```python
def handle_event(event):
```

> *«Для построения слабосвязанной, масштабируемой системы я выбираю **событийно-ориентированную архитектуру**.*

> *Я использую комбинацию паттернов:*
> - ***Event Notification** для уведомлений и обновления кэша — событие содержит только ID, потребители при необходимости запрашивают детали через API.*
> - ***State Transfer** для поискового индекса (Elasticsearch) — событие содержит полные данные заказа, чтобы индекс мог обновляться без дополнительных вызовов.*
> - ***Event Collaboration (Event Sourcing)** для финансовой части — каждая транзакция фиксируется как событие, что дает полный аудит и возможность восстановления состояния на любой момент времени.*

> *В качестве брокера сообщений я использую **Kafka**. Это дает:*
> - *Персистентное хранение событий (data retention)*
> - *Возможность перечитывать историю (replay)*
> - *Exactly-once гарантии для критических операций*

> *Для обеспечения консистентности между сервисами я использую **outbox pattern**: изменения в БД и публикация события происходят в одной транзакции (через таблицу outbox), а отдельный процесс публикует события из этой таблицы. Это гарантирует, что событие не будет потеряно и не будет опубликовано без соответствующего изменения данных»*.

# КОНСЕНСУС

## Введение

### Что такое консенсус

**Консенсус** — это процесс достижения соглашения между несколькими узлами в распределенной системе. Задачи консенсуса:

| Задача | Описание |
|--------|----------|
| **Выбор лидера** | Определить, какой узел будет координатором |
| **Распределенная блокировка** | Гарантировать, что только один узел владеет ресурсом |
| **Распределенные транзакции** | Согласовать изменения на нескольких узлах |
| **Репликация** | Договориться о порядке операций |

### Теоретическая основа

- **FLP impossibility** — в асинхронной системе невозможно гарантировать консенсус, если хотя бы один узел может упасть
- **Paxos / Raft** — практические алгоритмы консенсуса, работающие при большинстве работающих узлов

---

## 1. Выбор лидера (Leader Election)

### Что это

**Выбор лидера** — процесс, в котором группа узлов выбирает один узел, который будет выполнять координационные функции (например, принимать записи, назначать задачи).

### Зачем нужен лидер

| Сценарий | Почему нужен лидер |
|----------|-------------------|
| **Кластер БД** | Только мастер принимает записи (PostgreSQL, MySQL) |
| **Распределенная блокировка** | Координатор назначает блокировки |
| **Координатор задач** | Распределяет задачи между воркерами |
| **Репликация** | Определяет порядок операций |

### Алгоритмы выбора лидера

#### 1. Bully Algorithm

**Как работает:**
- Каждый узел имеет ID (чем больше, тем "сильнее")
- При обнаружении отсутствия лидера, узел инициирует выборы
- Узел рассылает "вызов" (election) узлам с большим ID
- Если никто не ответил, узел становится лидером
- Лидер рассылает "я лидер" (coordinator) всем

```
Узлы: 1, 2, 3, 4 (4 — самый сильный)

Узел 3 обнаруживает, что лидер (4) упал:
1. 3 рассылает ELECTION узлам 4 (больше его)
2. 4 не отвечает (упал)
3. 3 становится лидером
4. 3 рассылает COORDINATOR всем
```

**Преимущества:** Простота
**Недостатки:** Много сообщений, не устойчив к сетевым задержкам

#### 2. Paxos

**Что это:**
Семейство алгоритмов консенсуса, теоретический стандарт. Сложен для понимания и реализации.

**Компоненты Paxos:**
| Роль | Описание |
|------|----------|
| **Proposer** | Предлагает значение |
| **Acceptor** | Принимает/отклоняет предложения |
| **Learner** | Узнает результат консенсуса |

**Фазы Paxos:**
```
Phase 1: Prepare
  Proposer → Acceptors: Prepare(N)
  Acceptors → Proposer: Promise(N, lastAccepted)

Phase 2: Accept
  Proposer → Acceptors: Accept(N, value)
  Acceptors → Proposer: Accepted(N, value)

Result: Значение выбрано, когда большинство Acceptors приняли
```

#### 3. Raft

**Что это:**
Алгоритм консенсуса, разработанный как более понятная альтернатива Paxos. Используется во многих современных системах.

**Компоненты Raft:**

| Роль | Описание |
|------|----------|
| **Leader** | Принимает все запросы, реплицирует логи |
| **Follower** | Пассивный, реплицирует от лидера |
| **Candidate** | Состояние при выборах |

**Состояния узлов:**
```
        ┌──────────┐
        │ Follower │ ←─── timeout без heartbeat
        └────┬─────┘
             │
             ▼
        ┌──────────┐
        │ Candidate│ ←─── получает большинство голосов
        └────┬─────┘
             │
             ▼
        ┌──────────┐
        │  Leader  │
        └──────────┘
```

**Выборы в Raft:**
1. Follower не получает heartbeat от лидера (timeout)
2. Переходит в Candidate, увеличивает term
3. Голосует за себя, рассылает RequestVote
4. Получает голоса от большинства → становится Leader
5. Рассылает AppendEntries (heartbeat) всем

**Term (срок):**
```
Term 1:  Leader A
Term 2:  (сбой A) → выборы → Leader B
Term 3:  (сбой B) → выборы → Leader C
```

**Логи репликация:**
```
Leader:    [1] [2] [3] [4] [5]
Follower1: [1] [2] [3] [4] [5]
Follower2: [1] [2] [3] [4]

Leader отправляет AppendEntries с записями 4,5
Follower2 догоняет
```

### Где применяется

| Система | Алгоритм |
|---------|----------|
| **etcd** | Raft |
| **Consul** | Raft |
| **ZooKeeper** | ZAB (ZooKeeper Atomic Broadcast) — похож на Paxos |
| **MongoDB** | Raft (в replica sets) |
| **CockroachDB** | Raft |
| **TiDB / TiKV** | Raft |
| **Kafka** | KRaft (Kafka Raft) — собственный Raft |

## 2. Распределенная блокировка (Distributed Lock)

### Что это

**Распределенная блокировка** — механизм, позволяющий гарантировать, что только один узел (или процесс) выполняет критическую операцию в распределенной системе.

### Зачем нужна

| Сценарий | Пример |
|----------|--------|
| **Обработка заказа** | Только один воркер обрабатывает конкретный заказ |
| **Scheduled jobs** | Только один экземпляр cron-задачи запускается |
| **Операции с ресурсами** | Запись в shared storage |
| **Leader election** | Выбор лидера для координации |

### Проблемы распределенных блокировок

| Проблема | Описание | Решение |
|----------|----------|---------|
| **Deadlock** | Узел с блокировкой упал, не освободил | TTL / аренда (lease) |
| **Lock expiry** | Операция дольше TTL → другой узел забрал | Fencing tokens |
| **Clock drift** | Часы на узлах расходятся | Использовать монотонное время, внешние часы |
| **Split brain** | Два узла считают, что владеют блокировкой | Quorum, Fencing |

### Реализации

#### 1. Redis + Redlock

**Как работает:**
1. Клиент пытается установить блокировку на N независимых Redis узлах
2. Успех, если удалось на большинстве (N/2 + 1)
3. Блокировка имеет TTL

```
SET lock_key random_value NX PX 30000
```

**Проблемы Redlock:** спорная надежность при сетевых разделениях и GC паузах.

#### 2. ZooKeeper / etcd / Consul

**Как работает (ZooKeeper):**
1. Создание эфемерного узла (ephemeral node) с последовательным номером
2. Узел с наименьшим номером — владелец блокировки
3. При потере соединения узел удаляется автоматически

```
/locks/resource_1/
├── lock-000000001  ← владелец (наименьший номер)
├── lock-000000002
└── lock-000000003
```

**Преимущества:** Сильная консистентность, наблюдатели (watchers)

#### 3. Fencing Token (защита от проблем с TTL)

**Проблема:**
```
1. Клиент A берет блокировку с TTL 30 сек
2. GC пауза 40 сек (блокировка истекла)
3. Клиент B берет блокировку
4. Клиент A просыпается и продолжает писать → конфликт
```

**Решение: Fencing Token**
```
Блокировка выдает монотонно возрастающий токен:
Client A → lock(1) → token=10
Client B → lock(2) → token=11

Storage проверяет token при записи:
- Принимает только если token > последнего записанного
- Запись Client A (token=10) отклоняется
```

### Выбор инструмента

| Инструмент | Консистентность | TTL | Fencing | Сложность |
|------------|-----------------|-----|---------|-----------|
| **Redis (Redlock)** | AP (eventual) | Да | Нет | Низкая |
| **ZooKeeper** | CP (strong) | Да (эфемерные узлы) | Да (zxid) | Средняя |
| **etcd** | CP (strong) | Да (lease) | Да (mod_revision) | Средняя |
| **PostgreSQL (advisory locks)** | CP | Нет (ручное освобождение) | Нет | Низкая |

## 3. Распределенные транзакции

### Проблема

В распределенной системе (микросервисы, шардированная БД) одна бизнес-операция может затрагивать несколько сервисов или шардов. Как обеспечить **атомарность**?

### SAGA

#### Что это

**SAGA** — паттерн для управления распределенными транзакциями без использования 2PC. SAGA разбивает длинную транзакцию на последовательность локальных транзакций, каждая из которых имеет **компенсирующую транзакцию** (undo).

#### Типы SAGA

**1. Хореография (Choreography)**

**Как работает:**
Каждый сервис публикует события, на которые реагируют другие сервисы. Нет центрального координатора.

```
Order Service                Payment Service              Inventory Service
     │                              │                            │
     │ 1. OrderCreated             │                            │
     ├─────────────────────────────►                            │
     │                              │                            │
     │                              │ 2. PaymentProcessed       │
     │                              ├───────────────────────────►
     │                              │                            │
     │                              │                            │ 3. StockReserved
     │                              │◄───────────────────────────┤
     │                              │                            │
     │ 4. OrderConfirmed            │                            │
     │◄─────────────────────────────┤                            │
     │                              │                            │

При ошибке (например, PaymentFailed):
     │                              │                            │
     │                              │ 2. PaymentFailed           │
     │                              ├───────────────────────────►
     │                              │                            │
     │ 3. OrderCancelled            │                            │
     │◄─────────────────────────────┤                            │
     │                              │                            │
     │                              │ 4. StockReleased           │
     │                              ├───────────────────────────►
```

**Преимущества:**
- Нет единой точки отказа
- Слабая связность (сервисы не знают друг о друге)
- Легко добавлять новые участники

**Недостатки:**
- Сложность отслеживания (нет единого места с состоянием)
- Риск циклических зависимостей
- Сложность отладки

**2. Оркестрация (Orchestration)**

**Как работает:**
Центральный **оркестратор** (SAGA Coordinator) управляет последовательностью шагов и вызывает компенсации при ошибках.

```
                    ┌─────────────────────────────────────┐
                    │        SAGA Coordinator             │
                    │                                     │
                    │  1. Create Order                    │
                    │  2. Process Payment                 │
                    │  3. Reserve Stock                   │
                    │  4. Confirm Order                   │
                    │                                     │
                    │  Если ошибка:                       │
                    │  - Compensation: Release Stock      │
                    │  - Compensation: Refund Payment     │
                    │  - Compensation: Cancel Order       │
                    └─────────────────────────────────────┘
                         │        │        │
                         ▼        ▼        ▼
                    ┌─────────┐ ┌─────────┐ ┌─────────┐
                    │ Order   │ │Payment  │ │Inventory│
                    │ Service │ │ Service │ │ Service │
                    └─────────┘ └─────────┘ └─────────┘
```

**Пример кода (оркестрация):**

```python
class CreateOrderSaga:
    def execute(self, order_data):
        try:
            # Шаг 1: Создать заказ
            order = self.order_service.create(order_data)
            
            # Шаг 2: Обработать платеж
            payment = self.payment_service.process(order.id, order.total)
            
            # Шаг 3: Зарезервировать товары
            inventory = self.inventory_service.reserve(order.id, order.items)
            
            # Шаг 4: Подтвердить заказ
            self.order_service.confirm(order.id)
            
            return Success(order)
            
        except PaymentFailedException:
            # Компенсация: отменить заказ
            self.order_service.cancel(order.id)
            raise
            
        except InventoryFailedException:
            # Компенсация: вернуть платеж
            self.payment_service.refund(payment.id)
            self.order_service.cancel(order.id)
            raise
```

**Преимущества:**
- Централизованное управление состоянием
- Проще отладка и мониторинг
- Явная последовательность шагов

**Недостатки:**
- Оркестратор — единая точка отказа
- Риск "толстого" координатора
- Сложность при большом количестве шагов

#### SAGA: сравнение подходов

| Характеристика | Хореография | Оркестрация |
|----------------|-------------|-------------|
| **Связность** | Низкая | Средняя |
| **SPOF** | Нет | Оркестратор |
| **Отслеживание** | Сложное | Простое |
| **Изменение потока** | Сложно (менять много сервисов) | Просто (только оркестратор) |
| **Циклические зависимости** | Риск | Нет |
| **Масштабируемость** | Высокая | Ограничена оркестратором |

---

### 2PC (Two-Phase Commit)

#### Что это

**2PC** — классический протокол распределенных транзакций, обеспечивающий атомарность. Используется в распределенных БД (XA транзакции).

#### Фазы 2PC

```
Координатор (TM)                    Участники (RM)
     │                                    │
     │  PHASE 1: PREPARE                   │
     ├───────────────────────────────────►│
     │                                    │
     │  Can you commit?                   │
     │◄───────────────────────────────────┤
     │           YES / NO                  │
     │                                    │
     │  PHASE 2: COMMIT / ROLLBACK        │
     ├───────────────────────────────────►│
     │                                    │
     │  Commit (if all YES)               │
     │  Rollback (if any NO)              │
     │◄───────────────────────────────────┤
     │           ACK                       │
```

**Фаза 1: Prepare**
- Координатор отправляет `prepare` всем участникам
- Каждый участник:
    - Подготавливает изменения (пишет в WAL)
    - Блокирует ресурсы
    - Отвечает `YES` или `NO`

**Фаза 2: Commit / Rollback**
- Если все ответили `YES`:
    - Координатор отправляет `commit`
    - Участники фиксируют изменения
    - Освобождают блокировки
- Если кто-то ответил `NO` или не ответил:
    - Координатор отправляет `rollback`
    - Участники откатывают изменения

#### Проблемы 2PC

| Проблема | Описание |
|----------|----------|
| **Блокирующий протокол** | Если координатор упал после PREPARE, участники ждут indefinitely |
| **Single point of failure** | Падение координатора блокирует все транзакции |
| **Блокировки ресурсов** | Ресурсы заблокированы до завершения фазы 2 |
| **Производительность** | Высокая задержка из-за двух фаз |

#### Когда применяется

- **XA транзакции** (Java JTA, MS DTC)
- **Распределенные БД** (PostgreSQL с 2PC, MySQL XA)
- **Системы, требующие строгой согласованности**

---

### 3PC (Three-Phase Commit)

#### Что это

**3PC** — улучшение 2PC, добавляющее фазу `pre-commit` для уменьшения времени блокировки при сбое координатора.

#### Фазы 3PC

```
Фаза 1: CanCommit
  Координатор проверяет, могут ли участники выполнить транзакцию

Фаза 2: PreCommit
  Все участники подтвердили → отправка pre-commit
  Участники готовятся, но не блокируют ресурсы

Фаза 3: DoCommit
  Фиксация транзакции
```

#### 2PC vs 3PC

| Характеристика | 2PC | 3PC |
|----------------|-----|-----|
| **Блокирующий** | Да | Нет (с ограничениями) |
| **Фазы** | 2 | 3 |
| **Задержка** | Меньше | Больше |
| **Сложность** | Ниже | Выше |
| **Использование** | Широкое | Ограниченное |

---

## Сравнение протоколов распределенных транзакций

| Характеристика | SAGA | 2PC | 3PC |
|----------------|------|-----|-----|
| **Атомарность** | Eventual | Strong | Strong |
| **Блокировки** | Нет (компенсации) | Да (ресурсы заблокированы) | Минимальные |
| **Задержка** | Низкая | Высокая | Очень высокая |
| **Отказоустойчивость** | Высокая | Низкая | Средняя |
| **Сложность реализации** | Средняя | Средняя | Высокая |
| **Обработка сбоев** | Компенсации | Ожидание | Pre-commit |
| **Когда применять** | Микросервисы, long-running | Распределенные БД, строгая ACID | Редко (избыточно) |

> *«При проектировании распределенных транзакций я выбираю подход в зависимости от требований:*

> *Для **распределенных транзакций в микросервисной архитектуре** я использую **SAGA**. Если процесс простой и сервисов немного — выбираю **хореографию**: сервисы публикуют события в Kafka, другие реагируют. Это дает слабую связность и хорошую масштабируемость. Если процесс сложный (много шагов, ветвления) — выбираю **оркестрацию**: центральный координатор управляет последовательностью шагов, что упрощает мониторинг и отладку.*

> *Для **координации внутри одного сервиса с шардированной БД** (например, транзакция между шардами PostgreSQL) я использую **2PC**. Это дает строгую атомарность, но ценой производительности.*

> *Для **выбора лидера в кластере** я использую **Raft** — он понятнее Paxos и широко применяется (etcd, Consul, MongoDB). Для распределенных блокировок выбираю **etcd** или **ZooKeeper** — они дают сильную консистентность, fencing tokens и автоматическое освобождение при потере соединения.*

> *Для защиты от проблем с TTL и GC паузами при блокировках я использую **fencing tokens** — хранилище проверяет, что токен записи больше предыдущего, что предотвращает "зомби-записи" после истечения блокировки»*.

## 1. Point-to-Point (очередь)

### Что это

**Point-to-Point** — модель взаимодействия, где **одно сообщение доставляется одному потребителю**. Если несколько потребителей слушают одну очередь, сообщение получит **только один** из них.

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                         Queue                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ [Msg1] [Msg2] [Msg3] [Msg4] [Msg5]                    │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
         │              │              │
         ▼              ▼              ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ Consumer 1  │ │ Consumer 2  │ │ Consumer 3  │
│             │ │             │ │             │
│ Msg1, Msg4  │ │ Msg2, Msg5  │ │ Msg3        │
└─────────────┘ └─────────────┘ └─────────────┘
```

### Ключевые особенности

| Особенность | Описание |
|-------------|----------|
| **Один получатель** | Каждое сообщение обрабатывается только одним консьюмером |
| **Балансировка нагрузки** | Сообщения распределяются между консьюмерами |
| **Гарантии доставки** | Настраиваемые (at-least-once, exactly-once) |
| **Ack механизм** | Подтверждение обработки (удаление из очереди) |

### Примеры

| Система | Реализация P2P |
|---------|----------------|
| **RabbitMQ** | Очередь (queue) с несколькими консьюмерами |
| **AWS SQS** | Стандартная очередь (Standard Queue) |
| **Kafka** | Consumer group — каждая партиция отдается одному консьюмеру в группе |
| **ActiveMQ** | Очереди (queues) |

### RabbitMQ пример

```python
# Producer
channel.basic_publish(
    exchange='',
    routing_key='task_queue',
    body=json.dumps({'task': 'send_email', 'user_id': 123})
)

# Consumer
def callback(ch, method, properties, body):
    process_task(body)
    ch.basic_ack(delivery_tag=method.delivery_tag)  # подтверждение

channel.basic_consume(queue='task_queue', on_message_callback=callback)
```

### Когда применять

| Сценарий | Почему |
|----------|--------|
| **Распределение задач** | Задачи (обработка изображений, отправка email) распределяются между воркерами |
| **Параллельная обработка** | Много воркеров обрабатывают сообщения из одной очереди |
| **Балансировка нагрузки** | Сообщения распределяются равномерно |
| **Буферизация** | Сглаживание пиков нагрузки |

## 2. Publish-Subscribe (pub-sub)

### Что это

**Publish-Subscribe** — модель взаимодействия, где **одно сообщение доставляется всем подписчикам**. Издатель (publisher) не знает о подписчиках (subscribers).

### Структура

```
┌─────────────────────────────────────────────────────────────────┐
│                          Topic                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ [Msg1]                                                │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│ Subscriber 1│ │ Subscriber 2│ │ Subscriber 3│
│             │ │             │ │             │
│   Msg1      │ │   Msg1      │ │   Msg1      │
└─────────────┘ └─────────────┘ └─────────────┘
```

### Ключевые особенности

| Особенность | Описание |
|-------------|----------|
| **Много получателей** | Каждое сообщение получают все подписчики |
| **Слабая связность** | Издатель не знает о подписчиках |
| **Типы подписки** | По точному имени, по шаблону (wildcards) |
| **Фильтрация** | Подписчики могут получать только нужные сообщения |

### Примеры

| Система | Реализация Pub-Sub |
|---------|-------------------|
| **Kafka** | Consumer groups — разные группы получают все сообщения |
| **RabbitMQ** | Fanout exchange, topic exchange |
| **AWS SNS** | Топики, подписки (SQS, Lambda, Email) |
| **Redis** | Pub/Sub каналы |
| **Google Pub/Sub** | Топики и подписки |

### RabbitMQ пример (Topic Exchange)

```python
# Publisher
channel.basic_publish(
    exchange='order_events',
    routing_key='order.created',
    body=json.dumps({'order_id': 123})
)

# Subscriber 1 (обрабатывает все события заказов)
channel.queue_bind(exchange='order_events', queue='all_orders', routing_key='order.*')

# Subscriber 2 (только созданные заказы)
channel.queue_bind(exchange='order_events', queue='new_orders', routing_key='order.created')
```

### Kafka пример (Consumer Groups)

```python
# Producer
producer.send('order_events', value={'order_id': 123, 'event': 'created'})

# Consumer Group 1: Analytics (обрабатывает все)
consumer_analytics.subscribe(['order_events'])

# Consumer Group 2: Notifications (обрабатывает все — отдельно)
consumer_notifications.subscribe(['order_events'])

# Consumer Group 3: Parallel workers (делят нагрузку — P2P внутри группы)
consumer_workers.subscribe(['order_events'])  # 3 консьюмера делят партиции
```

### Когда применять

| Сценарий | Почему |
|----------|--------|
| **События в микросервисах** | Много сервисов реагируют на одни события |
| **Логи и метрики** | Сбор данных в разные системы |
| **Уведомления** | Отправка в разные каналы (email, push, sms) |
| **CDC** | Одно изменение БД → несколько потребителей |

---

## 3. Request-Response

### Что это

**Request-Response** — классический паттерн синхронного взаимодействия, но может быть реализован **асинхронно** через очереди.

### Синхронная версия

```
Client ── Request ──► Server
Client ◄── Response ── Server
```

### Асинхронная версия (через очереди)

```
┌─────────────────────────────────────────────────────────────────┐
│  Client                                                         │
│    │                                                            │
│    │ 1. Отправляет запрос в Request Queue с correlation_id     │
│    ▼                                                            │
│  ┌─────────────────┐                                            │
│  │  Request Queue  │                                            │
│  └─────────────────┘                                            │
│         │                                                       │
│         ▼                                                       │
│  ┌─────────────────┐                                            │
│  │    Server       │                                            │
│  └─────────────────┘                                            │
│         │                                                       │
│         ▼                                                       │
│  ┌─────────────────┐                                            │
│  │  Response Queue │                                            │
│  └─────────────────┘                                            │
│         │                                                       │
│         ▼                                                       │
│  Client слушает Response Queue, сопоставляет correlation_id    │
└─────────────────────────────────────────────────────────────────┘
```

### Пример (асинхронный Request-Response)

```python
# Client
import uuid

def send_request(server_queue, data):
    correlation_id = str(uuid.uuid4())
    response_queue = f"response_{correlation_id}"
    
    # Создаем временную очередь для ответа
    channel.queue_declare(queue=response_queue, auto_delete=True)
    
    # Отправляем запрос с указанием очереди для ответа
    channel.basic_publish(
        exchange='',
        routing_key=server_queue,
        properties={'correlation_id': correlation_id, 'reply_to': response_queue},
        body=json.dumps(data)
    )
    
    # Ждем ответа
    def callback(ch, method, properties, body):
        if properties.correlation_id == correlation_id:
            result = json.loads(body)
            # обрабатываем результат
    
    channel.basic_consume(queue=response_queue, on_message_callback=callback)

# Server
def handle_request(ch, method, properties, body):
    result = process(body)
    
    # Отправляем ответ в очередь, указанную клиентом
    ch.basic_publish(
        exchange='',
        routing_key=properties.reply_to,
        properties={'correlation_id': properties.correlation_id},
        body=json.dumps(result)
    )
    
    ch.basic_ack(delivery_tag=method.delivery_tag)
```

### Где применяется

| Система | Реализация |
|---------|------------|
| **RabbitMQ** | RPC (Remote Procedure Call) через очереди |
| **gRPC** | Синхронный, но может быть с streaming |
| **HTTP** | Классический request-response |
| **Kafka** | request-response сложнее (требует отдельных топиков) |

### Когда применять

| Сценарий | Почему |
|----------|--------|
| **API вызовы** | Классические REST/gRPC API |
| **Взаимодействие сервисов** | Когда нужен результат синхронно |
| **Long-running операции** | Асинхронная версия с callback |

---

## Сравнение P2P vs Pub-Sub vs Request-Response

| Характеристика | Point-to-Point | Publish-Subscribe | Request-Response |
|----------------|----------------|-------------------|------------------|
| **Получатели** | Один | Много (все подписчики) | Один (адресат) |
| **Связность** | Средняя | Низкая | Высокая |
| **Ожидание ответа** | Нет | Нет | Да |
| **Балансировка** | Да (между консьюмерами) | Нет | Нет (маршрутизация) |
| **Основное использование** | Задачи, воркеры | События, уведомления | API, запросы |
| **Пример** | SQS, RabbitMQ queue | Kafka topic, SNS | HTTP, gRPC, RPC |

## 4. Отложенное выполнение задач (Delayed / Scheduled Tasks)

### Что это

**Отложенное выполнение** — механизм, позволяющий запланировать выполнение задачи на определенное время или через определенный интервал.

### Сценарии использования

| Сценарий | Пример |
|----------|--------|
| **Напоминания** | Отправить email через 24 часа после регистрации |
| **Отмена заказов** | Отменить заказ, если не оплачен через 30 минут |
| **Повторные попытки** | Exponential backoff для failed операций |
| **Cron-задачи** | Ежедневные отчеты, очистка данных |
| **Стадии бизнес-процессов** | Перевести заказ в статус "доставлен" через N дней |

### Реализации

#### 1. Dead Letter Queue с задержкой (RabbitMQ)

```python
# RabbitMQ с TTL и DLX (Dead Letter Exchange)
channel.queue_declare(
    queue='delayed_tasks',
    arguments={
        'x-dead-letter-exchange': 'dlx',           # куда отправлять по истечении
        'x-message-ttl': 30000,                    # TTL 30 секунд
        'x-dead-letter-routing-key': 'process'     # после TTL → в очередь process
    }
)
```

#### 2. Redis + Sorted Sets

```python
import redis
import time

def schedule_task(task_id, execute_at):
    redis.zadd('scheduled_tasks', {task_id: execute_at})

def process_scheduled():
    now = time.time()
    # Получаем задачи, которые пора выполнять
    tasks = redis.zrangebyscore('scheduled_tasks', 0, now)
    for task_id in tasks:
        # Выполняем задачу
        process_task(task_id)
        # Удаляем из расписания
        redis.zrem('scheduled_tasks', task_id)
```

#### 3. Kafka + Timestamp

```python
# Producer с указанием timestamp
producer.send('tasks', value=task, timestamp_ms=execute_time_ms)

# Consumer читает с определенного смещения по времени
consumer = KafkaConsumer('tasks', consumer_timeout_ms=1000)
# В consumer можно искать сообщения по timestamp
partitions = consumer.partitions_for_topic('tasks')
for partition in partitions:
    offset = consumer.offsets_for_times({partition: execute_time_ms})
```

#### 4. Внешние инструменты

| Инструмент | Описание |
|------------|----------|
| **Celery** | Python, отложенные задачи, cron, retry, результат |
| **Bull** | Node.js, Redis-based, отложенные задачи |
| **Airflow** | DAG, сложные зависимости, расписание |
| **AWS Step Functions** | State machine, таймеры, retry |
| **AWS EventBridge** | Расписание + события |

### Exponential Backoff

```python
def schedule_retry(task, attempt):
    backoff = min(300, 2 ** attempt)  # 2, 4, 8, 16, 32, 64, 128, 256, 300 сек
    execute_at = time.time() + backoff
    schedule_task(task.id, execute_at)
```

## 5. Dead Letter Queue (DLQ)

### Что это

**Dead Letter Queue** — специальная очередь для сообщений, которые **не удалось обработать** после нескольких попыток. DLQ позволяет:
- Не терять проблемные сообщения
- Анализировать причины ошибок
- Перезапускать обработку после исправления

### Жизненный цикл сообщения

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  ┌─────────────┐                                               │
│  │   Queue     │                                               │
│  └──────┬──────┘                                               │
│         │                                                      │
│         ▼                                                      │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐      │
│  │ Consumer    │────►│    Retry    │────►│   DLQ       │      │
│  │  (attempt)  │     │   Queue     │     │             │      │
│  └─────────────┘     └─────────────┘     └─────────────┘      │
│         │                                        │             │
│         │ Success                                │             │
│         ▼                                        ▼             │
│  ┌─────────────┐                         ┌─────────────┐      │
│  │   Ack       │                         │  Analyze    │      │
│  │   Delete    │                         │  Fix        │      │
│  └─────────────┘                         │  Replay     │      │
│                                          └─────────────┘      │
└─────────────────────────────────────────────────────────────────┘
```

### Настройка DLQ

#### RabbitMQ

```python
# Объявляем основную очередь с DLQ
channel.queue_declare(
    queue='main_queue',
    arguments={
        'x-dead-letter-exchange': 'dlx',           # DLX для недоставленных
        'x-dead-letter-routing-key': 'dead_letters',
        'x-max-retries': 3                         # максимум попыток
    }
)

# Объявляем очередь DLQ
channel.queue_declare(queue='dead_letter_queue')
```

#### AWS SQS

#### Kafka (manual)

```python
class KafkaConsumerWithDLQ:
    def __init__(self, main_topic, dlq_topic, max_retries=3):
        self.main_topic = main_topic
        self.dlq_topic = dlq_topic
        self.max_retries = max_retries
        self.retry_counts = {}
    
    def handle_message(self, msg):
        msg_id = msg['id']
        retry_count = self.retry_counts.get(msg_id, 0)
        
        try:
            process(msg)
            del self.retry_counts[msg_id]  # успех, очищаем
            return True
        except Exception as e:
            if retry_count + 1 >= self.max_retries:
                # отправляем в DLQ
                self.dlq_producer.send(self.dlq_topic, msg)
                del self.retry_counts[msg_id]
            else:
                # увеличиваем счетчик и повторяем
                self.retry_counts[msg_id] = retry_count + 1
                self.schedule_retry(msg, retry_count + 1)
            return False
```

### Типичные причины попадания в DLQ

| Причина | Решение |
|---------|---------|
| **Ошибка в формате данных** | Исправить код consumer, заново обработать из DLQ |
| **Зависимость недоступна** | Увеличить retry, проверить availability |
| **Бизнес-ошибка** | Проверить логику, возможно требуется ручное вмешательство |
| **Исчерпаны ресурсы** | Масштабировать consumer |
| **Таймауты** | Увеличить timeout, оптимизировать обработку |

### Мониторинг DLQ

```
Важные метрики:
- DLQ message count (должно быть близко к 0)
- Age of oldest message (время в DLQ)
- Rate of messages entering DLQ (рост → проблема)
```

### На интервью: пример ответа

> *«Для асинхронной обработки задач я использую комбинацию паттернов:*

> *Для **распределения задач между воркерами** (например, обработка загруженных изображений) я выбираю **point-to-point** с очередью (RabbitMQ или SQS). Это дает балансировку нагрузки и гарантирует, что каждая задача будет обработана одним воркером.*

> *Для **событийной архитектуры** (сервисы реагируют на изменения) я использую **publish-subscribe** с Kafka. Каждый сервис имеет свою consumer group, что позволяет им независимо читать одни и те же события.*

> *Для **задач, не требующих немедленного результата**, я использую **отложенное выполнение**. Например, при создании заказа я планирую отмену через 30 минут, если он не оплачен. Для этого использую RabbitMQ с TTL и DLX или Redis с sorted sets.*

> *Для **сообщений, которые не удалось обработать**, я настраиваю **Dead Letter Queue**. После 3 неудачных попыток сообщение попадает в DLQ. Это позволяет не терять проблемные сообщения, анализировать причины ошибок и перезапускать обработку после исправления. Инженеры мониторят размер DLQ и при его росте получают алерты»*.
