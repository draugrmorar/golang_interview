# ООП (Объектно-Ориентированное Программирование)

**ООП** — это методология программирования, основанная на представлении программы в виде совокупности взаимодействующих **ОБЪЕКТОВ**.

---

## Основные принципы

| Принцип | Описание |
|---------|----------|
| **Инкапсуляция (Encapsulation)** | Объединение свойств и функций в единое целое. Скрытие внутренней реализации. |
| **Наследование (Inheritance)** | Создание новых классов на основе существующих ("is-a" отношение). Дочерний класс получает все поля и методы родителя и может их расширять или переопределять. |
| **Полиморфизм (Polymorphism)** | Один интерфейс — множество реализаций. Объекты разных классов могут обрабатывать один и тот же вызов метода по-своему. |
| **Абстракция (Abstraction)** | Выделение существенных характеристик, отбрасывание несущественных. Работа с объектами на уровне интерфейсов/абстрактных классов, а не конкретных деталей. |

---

## Другие парадигмы

| Парадигма | Фокус на... | Плюсы | Минусы |
|-----------|-------------|-------|--------|
| **ООП** | Объекты и их взаимодействие | Моделирует реальный мир, переиспользование | Сложность, оверхед |
| **Функциональное** | Чистые функции, неизменяемость | Предсказуемость, легко тестировать | Сложно моделировать состояние |
| **Процедурное** | Последовательность действий | Простота, скорость | Сложно поддерживать большие программы |

# SOLID (5 принципов хорошего дизайна классов)

| Принцип | Суть | Нарушение ведет к... |
|---------|------|----------------------|
| **Single Responsibility** (Принцип единственной ответственности) | У класса одна причина для изменения | Огромные классы, трудно тестировать |
| **Open/Closed** (Принцип открытости/закрытости) | Открыт для расширения, закрыт для изменения | Любое новое требование ломает старый код |
| **Liskov Substitution** (Принцип подстановки Барбары Лисков) | Наследники не ломают логику родителя | Неожиданное поведение, проверки на тип |
| **Interface Segregation** (Принцип разделения интерфейсов) | Много специализированных интерфейсов лучше одного жирного | Классы зависят от методов, которые не используют |
| **Dependency Inversion** (Принцип инверсии зависимостей) | Зависеть от абстракций, а не от конкретики | Жесткая связность, невозможно подменить компонент |

# ООП в Golang

| Вопрос | Ответ |
|--------|-------|
| **Go — ООП-язык?** | Нет, если под ООП понимать классическую триаду (классы, наследование, полиморфизм). |
| **Можно ли писать ООП-код на Go?** | Да, используя интерфейсы, структуры и методы. |
| **Go поддерживает инкапсуляцию?** | Да, на уровне пакета. Есть публичные и приватные методы/поля. |
| **Go поддерживает полиморфизм?** | Да, через интерфейсы + встраивание. |
| **Go поддерживает наследование?** | Нет, только композицию. Композиция как встраивание. |
| **Go ближе к процедурному или ООП?** | Гибрид: процедурная основа + ООП-возможности. |
